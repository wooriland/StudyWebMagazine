28. 예외 처리

예외 처리는 프로그램 실행 중 오류가 발생해도 비정상 종료되지 않도록 대응하는 구조입니다.
try, catch, finally, throw, throws,
Checked Exception, Unchecked Exception, 사용자 정의 예외까지 공부합니다.

이번 페이지의 학습 목표

예외 처리는 오류가 발생할 수 있는 상황을 미리 대비하는 문법입니다.
프로그램이 중간에 멈추지 않도록 예외를 잡고, 사용자에게 안내하거나, 자원을 정리하고, 호출한 쪽으로 예외 처리를 넘길 수 있습니다.

예외 개념

실행 중 발생하는 오류 상황과 예외 처리의 목적을 이해합니다.

try-catch

예외 발생 가능 코드와 예외 처리 코드를 분리합니다.

throw / throws

예외를 직접 발생시키는 것과 호출자에게 넘기는 것을 구분합니다.

실무 예외 처리

입력값 검증, 파일, DB, 외부 API 처리에서 예외를 활용합니다.

1. 예외 처리의 기본 개념

1-1. 예외란?

예외는 프로그램 실행 중에 발생할 수 있는 오류 상황입니다. 예외가 발생했는데 처리하지 않으면 프로그램이 중간에 비정상 종료될 수 있습니다. 

int result = 10 / 0;
10 / 0 실행
    ↓
ArithmeticException 발생
    ↓
예외 처리가 없으면 프로그램 비정상 종료

1-2. 예외 처리를 하는 이유

예외 처리는 오류 상황이 발생해도 프로그램이 바로 죽지 않고, 적절한 안내나 대체 처리를 하도록 만드는 방법입니다. 

try {
    int result = 10 / 0;
} catch (ArithmeticException e) {
    System.out.println("0으로 나눌 수 없습니다.");
}
0으로 나눌 수 없습니다.

핵심: 예외 처리는 오류를 없애는 문법이 아니라, 오류가 발생했을 때 프로그램 흐름을 안전하게 관리하는 문법입니다.

1-3. 예외 처리 흐름

try 블록 진입
    ↓
예외 발생 가능 코드 실행
    ↓
예외 없음 → catch 건너뜀
예외 있음 → 해당 catch 실행
    ↓
finally가 있으면 실행
    ↓
다음 코드 계속 실행

2. 예외의 종류

2-1. Checked Exception

Checked Exception은 컴파일러가 미리 확인하는 예외입니다. 반드시 try ~ catch로 직접 처리하거나 throws로 호출한 쪽에 넘겨야 합니다. 

대표 예:
IOException
SQLException
ClassNotFoundException

특징:
컴파일 시점에 예외 처리 강제

import java.io.IOException;

public class ExceptionTest {

    public static void main(String[] args) throws IOException {
        int data = System.in.read();
        System.out.println(data);
    }

}

2-2. Unchecked Exception

Unchecked Exception은 컴파일러가 처리를 강제하지 않는 예외입니다. 컴파일은 되지만 실행 중에 발생할 수 있으며, 주로 RuntimeException 계열입니다. 

대표 예:
NullPointerException
ArithmeticException
ArrayIndexOutOfBoundsException
NumberFormatException

특징:
실행 시점에 발생
컴파일러가 예외 처리를 강제하지 않음

String str = null;
System.out.println(str.length()); // NullPointerException

2-3. Error

Error는 예외보다 더 심각한 문제입니다. JVM 수준의 치명적인 문제에 가까우며, 일반적으로 개발자가 복구 처리하기 어렵습니다.

구분 의미 처리 가능성
Exception 프로그램에서 대응 가능한 오류 상황 처리 대상
Error JVM 수준의 치명적 문제 일반적으로 처리 대상 아님

2-4. Checked와 Unchecked 비교

구분 Checked Exception Unchecked Exception
확인 시점 컴파일 시점 실행 시점
처리 강제 여부 예외 처리 강제 예외 처리 강제 아님
대표 부모 Exception 계열 중 Runtime 제외 RuntimeException 계열
대표 예 IOException, SQLException NullPointerException, ArithmeticException
주요 의미 외부 환경 문제 가능성 코드 로직 문제 가능성

3. try ~ catch

3-1. try ~ catch 기본 구조

try {
    // 예외가 발생할 수 있는 코드
} catch (예외클래스 e) {
    // 예외 처리 코드
}
블록 역할
try 예외가 발생할 가능성이 있는 코드를 작성
catch 예외가 발생했을 때 처리할 코드를 작성

3-2. 기본 예제

public class ExceptionExample {

    public static void main(String[] args) {

        try {
            int result = 10 / 0;
            System.out.println(result);
        } catch (ArithmeticException e) {
            System.out.println("0으로 나눌 수 없습니다.");
        }

        System.out.println("프로그램 종료");
    }

}
0으로 나눌 수 없습니다.
프로그램 종료

예외가 발생했지만 catch에서 처리했기 때문에 프로그램은 중간에 종료되지 않고 다음 코드까지 실행됩니다.

3-3. 예외가 없으면 catch는 실행되지 않는다

try {
    int result = 10 / 2;
    System.out.println(result);
} catch (ArithmeticException e) {
    System.out.println("예외 발생");
}

3-4. 여러 catch 사용

public class MultiCatchExample {

    public static void main(String[] args) {

        String input = "abc";

        try {
            int number = Integer.parseInt(input);
            int result = 10 / number;
            System.out.println(result);
        } catch (NumberFormatException e) {
            System.out.println("숫자로 변환할 수 없습니다.");
        } catch (ArithmeticException e) {
            System.out.println("0으로 나눌 수 없습니다.");
        } catch (Exception e) {
            System.out.println("알 수 없는 예외가 발생했습니다.");
        }
    }
}

3-5. catch 순서 주의

여러 catch를 사용할 때는 구체적인 예외를 먼저 작성하고, 부모 예외인 Exception은 마지막에 작성해야 합니다. 

try {
    int number = Integer.parseInt("abc");
} catch (Exception e) {
    System.out.println("전체 예외");
} catch (NumberFormatException e) {
    System.out.println("숫자 변환 예외");
}
try {
    int number = Integer.parseInt("abc");
} catch (NumberFormatException e) {
    System.out.println("숫자 변환 예외");
} catch (Exception e) {
    System.out.println("전체 예외");
}

Exception이 먼저 모든 예외를 잡아버리면, 뒤의 구체적인 예외 catch 블록은 실행될 수 없습니다.

4. throw와 throws

4-1. throws

throws는 예외를 직접 처리하지 않고, 메서드를 호출한 쪽으로 예외 처리를 넘기겠다고 선언하는 키워드입니다. 

import java.io.IOException;

public class ThrowsExample {

    public static void readInput() throws IOException {
        int data = System.in.read();
        System.out.println(data);
    }

    public static void main(String[] args) throws IOException {
        readInput();
    }
}
readInput()에서 IOException 발생 가능
    ↓
readInput()이 직접 처리하지 않음
    ↓
throws IOException으로 호출한 쪽에 넘김
    ↓
main도 throws IOException으로 넘김

4-2. throw

throw는 예외 객체를 직접 발생시키는 키워드입니다. 

throw new Exception("예외 메시지");
public static void checkAge(int age) throws Exception {

    if (age < 20) {
        throw new Exception("미성년자는 입장할 수 없습니다.");
    }

    System.out.println("입장 가능");
}
try {
    checkAge(17);
} catch (Exception e) {
    System.out.println(e.getMessage());
}
미성년자는 입장할 수 없습니다.

4-3. throw와 throws 차이

구분 throw throws
위치 메서드 내부 메서드 선언부
역할 예외 객체를 직접 발생 예외 처리를 호출한 쪽으로 넘김
예시 throw new Exception() public void test() throws Exception
개수 한 번에 하나의 예외 객체 여러 예외 선언 가능 
public void test() throws Exception {
    throw new Exception("예외 발생");
}

5. finally

5-1. finally란?

finally는 예외 발생 여부와 상관없이 반드시 한 번 실행되는 블록입니다. 주로 파일, DB, 네트워크 연결 같은 자원을 정리할 때 사용합니다. 

try {
    System.out.println("try 실행");
} catch (Exception e) {
    System.out.println("catch 실행");
} finally {
    System.out.println("finally 실행");
}
try 실행
finally 실행

5-2. 예외가 발생해도 finally 실행

public class FinallyExample {

    public static void main(String[] args) {

        try {
            System.out.println("파일 읽기");
            int result = 10 / 0;
        } catch (ArithmeticException e) {
            System.out.println("계산 중 예외 발생");
        } finally {
            System.out.println("자원 정리");
        }

        System.out.println("프로그램 종료");
    }
}
파일 읽기
계산 중 예외 발생
자원 정리
프로그램 종료

5-3. finally를 사용하는 이유

finally 사용 목적:
- 파일 닫기
- DB 연결 닫기
- 네트워크 연결 닫기
- 자원 해제
- 로그 출력

주의:
return이 있어도 finally는 실행된다.
단, System.exit(0)으로 JVM이 종료되면 실행되지 않을 수 있다.

6. 사용자 정의 예외

6-1. 사용자 정의 예외란?

사용자 정의 예외는 개발자가 직접 만드는 예외 클래스입니다. 업무 규칙을 코드로 명확하게 표현할 때 유용합니다. 

사용자 정의 예외가 어울리는 상황:
- 잔액 부족
- 권한 없음
- 이미 가입한 회원
- 삭제된 계정
- 예약 불가능한 시간
- 잘못된 인증 코드

6-2. 사용자 정의 예외 클래스 만들기

class InsufficientBalanceException extends Exception {

    public InsufficientBalanceException(String message) {
        super(message);
    }
}

생성자에서 super(message)를 호출하면 예외 메시지를 부모 Exception 클래스에 전달할 수 있습니다. 이 메시지는 getMessage()로 꺼낼 수 있습니다.

6-3. 사용자 정의 예외 사용 예제

class Account {

    private int balance = 10000;

    public void withdraw(int amount) throws InsufficientBalanceException {

        if (balance < amount) {
            throw new InsufficientBalanceException("잔액이 부족합니다.");
        }

        balance -= amount;
        System.out.println("출금 완료. 잔액: " + balance);
    }
}
public class AccountMain {

    public static void main(String[] args) {

        Account account = new Account();

        try {
            account.withdraw(20000);
        } catch (InsufficientBalanceException e) {
            System.out.println(e.getMessage());
        }
    }
}
잔액이 부족합니다.

6-4. getMessage()

try {
    throw new Exception("사용자 정의 예외 메시지");
} catch (Exception e) {
    System.out.println(e.getMessage());
}
사용자 정의 예외 메시지

7. 실무 연결

7-1. 실무에서 예외 처리가 필요한 곳

  1. 사용자 입력값 검증
  2. 파일 업로드와 다운로드
  3. DB 조회, 저장, 수정, 삭제
  4. 외부 API 호출
  5. 로그인과 회원가입 인증 처리
  6. 권한 검증
  7. 결제 처리
  8. 네트워크 통신

7-2. 입력값 검증 예외

if (loginId == null || loginId.isBlank()) {
    throw new IllegalArgumentException("로그인 ID는 필수입니다.");
}

값이 잘못 들어왔을 때 단순히 false를 반환하는 대신, 예외를 발생시켜 문제 상황을 명확하게 표현할 수 있습니다.

7-3. Spring 전역 예외 처리 흐름

Controller
    ↓
Service에서 예외 발생
    ↓
ControllerAdvice에서 예외 처리
    ↓
JSON 응답으로 변환
    ↓
사용자에게 오류 메시지 전달

실무에서는 각 메서드마다 예외 응답을 따로 만들기보다, 전역 예외 처리기를 통해 일관된 응답 구조로 변환하는 경우가 많습니다.

실수 2. throw와 throws를 혼동

throw
→ 예외를 실제로 발생시킨다.

throws
→ 예외를 호출한 쪽으로 넘긴다.

실수 3. finally가 예외가 있을 때만 실행된다고 생각

try {
    System.out.println("정상 실행");
} finally {
    System.out.println("무조건 실행");
}

finally는 예외가 없어도 실행됩니다.

실수 4. 예외를 무조건 Exception 하나로만 처리

catch (Exception e) {
    System.out.println("오류 발생");
}
catch (NumberFormatException e) {
    System.out.println("숫자 형식 오류");
} catch (ArithmeticException e) {
    System.out.println("계산 오류");
} catch (Exception e) {
    System.out.println("기타 오류");
}

실수 5. Checked Exception을 throw만 하고 처리하지 않음

public void test() {
    throw new Exception("예외 발생"); // 오류
}
public void test() throws Exception {
    throw new Exception("예외 발생");
}
public void test() {
    try {
        throw new Exception("예외 발생");
    } catch (Exception e) {
        System.out.println(e.getMessage());
    }
}

10. 면접 포인트

Q1. 예외란 무엇인가요?

답변: 예외는 프로그램 실행 중에 발생할 수 있는 오류 상황입니다. 예외 처리를 통해 프로그램이 비정상 종료되지 않고 적절히 대응하도록 만들 수 있습니다.

Q2. Checked Exception과 Unchecked Exception의 차이는 무엇인가요?

답변: Checked Exception은 컴파일 시점에 처리를 강제하는 예외입니다. IOException, SQLException 등이 대표적입니다.
Unchecked Exception은 컴파일러가 처리를 강제하지 않고 실행 중 발생하는 예외입니다.
RuntimeException 계열이며 NullPointerException, ArithmeticException 등이 대표적입니다.

Q3. try ~ catch는 무엇인가요?

답변: try 블록에는 예외가 발생할 수 있는 코드를 작성하고, catch 블록에는 예외가 발생했을 때 처리할 코드를 작성합니다.

Q4. catch 순서가 중요한 이유는 무엇인가요?

답변: 예외 클래스에도 상속 관계가 있기 때문입니다. 부모 예외인 Exception을 먼저 catch하면 자식 예외 catch 블록은 실행될 수 없습니다.
따라서 구체적인 예외를 먼저 작성하고, Exception은 마지막에 작성해야 합니다.

Q5. throw와 throws의 차이는 무엇인가요?

답변: throw는 메서드 내부에서 예외 객체를 직접 발생시키는 키워드입니다. throws는 메서드 선언부에서 예외 처리를 호출한 쪽으로 넘기겠다고 선언하는 키워드입니다.

Q6. finally는 언제 실행되나요?

답변: finally는 예외 발생 여부와 상관없이 실행됩니다. 예외가 발생해도 실행되고, 발생하지 않아도 실행됩니다. 심지어 return이 있어도 실행됩니다.
단, System.exit(0)처럼 JVM이 종료되는 경우에는 실행되지 않을 수 있습니다.

Q7. 사용자 정의 예외는 어떻게 만드나요?

답변: Exception 클래스를 상속받아 사용자 정의 예외 클래스를 만들 수 있습니다. 생성자에서 super(message)를 호출하면 getMessage()로 예외 메시지를 꺼낼 수 있습니다.

Q8. RuntimeException은 반드시 try ~ catch 해야 하나요?

답변: 반드시 해야 하는 것은 아닙니다. RuntimeException 계열은 Unchecked Exception이므로 컴파일러가 처리를 강제하지 않습니다.
다만 필요한 경우 try ~ catch로 처리할 수 있습니다.

Q9. 예외 처리를 실무에서 어디에 사용하나요?

답변: 입력값 검증, 로그인 인증, 권한 검증, 파일 처리, DB 처리, 외부 API 호출, 결제 처리, 네트워크 통신 등에서 사용합니다.

Q10. 예외를 무조건 catch(Exception e)로 잡는 것은 좋은가요?

답변: 항상 좋은 방식은 아닙니다. 모든 예외를 한 번에 잡을 수는 있지만, 어떤 문제가 발생했는지 구체적으로 구분하기 어렵습니다.
가능하면 구체적인 예외를 먼저 처리하고 마지막에 Exception으로 공통 예외를 처리하는 것이 좋습니다.

11. 오늘의 핵심 정리

  1. 예외는 프로그램 실행 중 발생할 수 있는 오류 상황이다.
  2. 예외 처리는 프로그램이 비정상 종료되지 않도록 적절히 대응하는 방법이다.
  3. Checked Exception은 컴파일 시점에 처리를 강제한다.
  4. Unchecked Exception은 실행 시점에 발생하며 RuntimeException 계열이다.
  5. Error는 치명적 문제로 일반적으로 처리 대상이 아니다.
  6. try 블록에는 예외 발생 가능 코드를 작성한다.
  7. catch 블록에는 예외 처리 코드를 작성한다.
  8. 여러 catch를 사용할 때는 구체적인 예외를 먼저, Exception을 마지막에 둔다.
  9. throws는 예외 처리를 호출한 쪽으로 넘기는 선언이다.
  10. throw는 예외 객체를 직접 발생시키는 키워드다.
  11. finally는 예외 발생 여부와 상관없이 실행된다.
  12. return이 있어도 finally는 실행된다.
  13. 사용자 정의 예외는 Exception을 상속받아 만들 수 있다.
  14. 사용자 정의 예외 메시지는 super(message)로 전달하고 getMessage()로 꺼낸다.
  15. 실무에서는 입력값 검증, 파일 처리, DB 처리, 외부 API 호출, 인증/권한 처리에서 예외 처리를 자주 사용한다.

학습 메모

이 페이지는 Java 예외 처리 단원입니다.
예외 처리는 프로그램 실행 중 오류가 발생했을 때 프로그램이 비정상 종료되지 않도록 처리하는 구조입니다.
try에는 예외 발생 가능 코드를 작성하고, catch에는 예외 처리 코드를 작성합니다.
throw는 예외를 직접 발생시키고, throws는 예외 처리를 호출한 쪽으로 넘기는 선언입니다. finally는 예외 발생 여부와 상관없이 실행되며, 주로 자원 정리에 사용됩니다.
