17. 캡슐화

캡슐화는 객체의 필드를 외부에서 직접 접근하지 못하게 숨기고,
공개된 메서드를 통해 안전하게 접근하도록 만드는 객체지향의 핵심 개념입니다.
이 페이지에서는 private 필드, getter, setter, this, 검증 로직, 의미 있는 메서드를 공부합니다.

이번 페이지의 학습 목표

캡슐화는 접근 지정자 단원과 직접 연결됩니다. 필드는 private으로 숨기고, 외부에서는 public 메서드를 통해서만 객체의 값을 읽거나 변경하게 만드는 방식입니다.
단순히 getter/setter를 만드는 것이 아니라, 객체 상태를 안전하게 보호하는 것이 핵심입니다.

캡슐화 개념

필드를 숨기고 메서드로 접근하게 하는 이유를 이해합니다.

getter / setter

값을 넣는 setter와 값을 꺼내는 getter의 구조를 익힙니다.

this 이해

필드명과 매개변수명이 같을 때 this로 구분하는 방법을 익힙니다.

객체 보호

검증 로직과 의미 있는 메서드로 객체 상태를 안전하게 관리합니다.

1. 캡슐화의 기본 개념

1-1. 캡슐화란?

캡슐화는 객체의 내부 데이터를 외부에서 직접 건드리지 못하게 숨기고, 정해진 메서드를 통해서만 접근하게 만드는 방식입니다. 

class Account {

    private int balance;

    public void setBalance(int money) {
        balance = money;
    }

    public int getBalance() {
        return balance;
    }

}
private int balance
→ 외부에서 직접 접근 불가

public setBalance()
→ 값을 넣는 통로

public getBalance()
→ 값을 꺼내는 통로

캡슐화는 필드를 완전히 못 쓰게 막는 것이 아니라, 안전한 통로를 만들어 그 통로로만 접근하게 하는 것입니다.

1-2. 캡슐화가 필요한 이유

캡슐화가 없으면 외부에서 객체의 값을 마음대로 바꿀 수 있습니다. 예를 들어 계좌의 잔고를 외부에서 직접 수정할 수 있다면 위험한 상태가 만들어질 수 있습니다. 

class Account {
    int balance;
}

Account account = new Account();

account.balance = -100000;

위 코드는 계좌 잔고를 음수로 만들어버립니다. 객체가 가져서는 안 되는 상태가 된 것입니다. 

class Account {

    private int balance;

    public void setBalance(int balance) {
        if (balance >= 0) {
            this.balance = balance;
        }
    }

    public int getBalance() {
        return balance;
    }

}

이렇게 만들면 외부에서 직접 값을 바꾸지 못하고, setBalance() 안에서 값 검증을 할 수 있습니다.

1-3. 캡슐화의 기본 방법

캡슐화의 기본 흐름은 두 단계입니다. 

1. 멤버 변수 앞에 private 접근 지정자를 붙인다.
2. 값을 넣거나 꺼낼 수 있는 public 메서드를 만든다.

private int balance;

public void setBalance(int money) {
    balance = money;
}

public int getBalance() {
    return balance;
}

2. getter와 setter

2-1. setter 메서드

setter는 필드에 값을 넣는 메서드입니다. 보통 접근 지정자는 public, 반환 타입은 void, 매개변수는 필드에 넣을 값을 받는 형태입니다. 

public void setBalance(int money) {
    balance = money;
}
구성 요소 의미
public 외부에서 호출할 수 있게 공개
void 값을 넣기만 하고 반환값은 없음
setBalance balance 값을 설정한다는 의미
int money 외부에서 전달받는 값
balance = money 전달받은 값을 필드에 저장

2-2. getter 메서드

getter는 필드에 저장된 값을 꺼내오는 메서드입니다. 보통 접근 지정자는 public, 반환 타입은 필드의 자료형과 같고, 매개변수는 받지 않습니다. 

public int getBalance() {
    return balance;
}
구성 요소 의미
public 외부에서 호출할 수 있게 공개
int balance의 자료형이 int이므로 int 반환
getBalance balance 값을 꺼낸다는 의미
return balance 필드 값을 반환

2-3. getter/setter 이름 규칙

getter와 setter는 일반적으로 필드 이름을 기준으로 만듭니다. 

private String name;

public void setName(String name) {
    this.name = name;
}

public String getName() {
    return name;
}
필드명: name
setter: setName()
getter: getName()

필드명: balance
setter: setBalance()
getter: getBalance()

필드명: email
setter: setEmail()
getter: getEmail()

2-4. boolean 필드의 getter

boolean 타입은 getter 이름을 get 대신 is로 쓰는 경우가 많습니다. 

private boolean active;

public boolean isActive() {
    return active;
}

정리: getActive()도 가능하지만, Java 관례에서는 상태를 묻는 boolean 값에 isActive() 형태를 자주 사용합니다.

2-5. this 키워드와 캡슐화

setter를 만들 때 필드명과 매개변수명을 같게 쓰는 경우가 많습니다. 이때 멤버 변수와 매개변수를 구분하기 위해 this를 사용합니다. 

private int balance;

public void setBalance(int balance) {
    balance = balance;
}

위 코드는 멤버 변수에 값을 넣는 것이 아니라, 매개변수에 매개변수를 대입하는 의미가 될 수 있습니다. 

private int balance;

public void setBalance(int balance) {
    this.balance = balance;
}
this.balance
→ 이 객체의 멤버 변수 balance

balance
→ 메서드로 전달받은 매개변수 balance

3. 캡슐화는 단순 getter/setter가 아니다

3-1. 검증 없는 setter의 문제

필드를 private으로 숨겼더라도 setter가 아무 값이나 받아들이면 캡슐화 효과가 약해집니다. 

public void setBalance(int balance) {
    this.balance = balance;
}
account.setBalance(-100000);

위 코드는 형식상 필드는 숨겨졌지만, 외부에서 음수 잔고를 넣을 수 있으므로 안전하지 않습니다. 

public void setBalance(int balance) {
    if (balance >= 0) {
        this.balance = balance;
    }
}

3-2. 의미 있는 메서드로 상태 변경하기

더 좋은 캡슐화는 단순히 setBalance()로 값을 바꾸는 것이 아니라, 입금과 출금이라는 의미 있는 행위로 상태를 변경하게 만드는 것입니다. 

public void deposit(int amount) {
    if (amount > 0) {
        balance += amount;
    }
}

public int withdraw(int amount) {
    if (amount <= 0) {
        return 0;
    }

    if (balance < amount) {
        return 0;
    }

    balance -= amount;
    return amount;
}

핵심: 외부에서 잔고를 직접 설정하는 것이 아니라, deposit()과 withdraw()라는 의미 있는 행위로만 잔고가 바뀌게 합니다.

3-3. getter도 무조건 만들 필요는 없다

캡슐화라고 해서 모든 필드에 getter와 setter를 무조건 만드는 것은 좋은 설계가 아닐 수 있습니다. 특히 비밀번호 같은 민감한 값은 외부로 그대로 꺼내 보여주면 위험합니다. 

private String password;

public String getPassword() {
    return password;
}

더 좋은 방향은 비밀번호 원문을 반환하지 않고, 변경 메서드나 검증 메서드만 제공하는 것입니다. 

public void changePassword(String newPassword) {
    if (newPassword != null && newPassword.length() >= 8) {
        this.password = newPassword;
    }
}

4. 그림처럼 이해하기

4-1. 캡슐화 구조

외부 클래스
↓
직접 필드 접근 차단
↓
private field
↑
public method
↑
검증 로직

4-2. 계좌 객체 캡슐화

Account 객체

private balance
┌──────────────────────┐
│ balance 직접 접근 X   │
│                      │
│ deposit(amount)      │
│ withdraw(amount)     │
│ getBalance()         │
└──────────────────────┘

가능:
account.deposit(10000)
account.withdraw(3000)
account.getBalance()

불가능:
account.balance = -100000

4-3. 캡슐화 전후 비교

캡슐화 전

외부 클래스 → account.balance = -100000 가능
문제: 검증 없이 객체 상태 변경


캡슐화 후

외부 클래스 → account.deposit(10000)
외부 클래스 → account.withdraw(3000)

Account 내부에서 금액 검증 후 balance 변경

4-4. 정보 은닉

외부는 사용 방법만 안다.
내부 구현은 클래스 안에 숨긴다.

외부:
account.deposit(10000)

내부:
balance를 어떻게 계산하고 저장할지는 Account가 책임진다.

5. 코드 예제

5-1. 기본 getter/setter 예제

class Account {

    private int balance;

    public void setBalance(int money) {
        balance = money;
    }

    public int getBalance() {
        return balance;
    }

}
public class AccountMain {

    public static void main(String[] args) {

        Account account = new Account();

        account.setBalance(10000);

        System.out.println(account.getBalance());

    }

}
10000

5-2. 검증이 있는 setter

class Account {

    private int balance;

    public void setBalance(int balance) {
        if (balance >= 0) {
            this.balance = balance;
        }
    }

    public int getBalance() {
        return balance;
    }

}
Account account = new Account();

account.setBalance(-100000);

System.out.println(account.getBalance());
0

음수 값이 들어오면 조건문에서 막기 때문에 필드 값이 변경되지 않습니다.

5-3. 의미 있는 메서드로 상태 변경

class Account {

    private String name;
    private String accountNo;
    private int balance;

    public void setName(String name) {
        this.name = name;
    }

    public void setAccountNo(String accountNo) {
        this.accountNo = accountNo;
    }

    public void deposit(int amount) {
        if (amount > 0) {
            balance += amount;
        }
    }

    public int withdraw(int amount) {
        if (amount <= 0) {
            return 0;
        }

        if (balance < amount) {
            return 0;
        }

        balance -= amount;
        return amount;
    }

    public int getBalance() {
        return balance;
    }

    public void printAccountInfo() {
        System.out.println("예금주: " + name);
        System.out.println("계좌번호: " + accountNo);
        System.out.println("잔고: " + balance);
    }

}
public class AccountMain {

    public static void main(String[] args) {

        Account account = new Account();

        account.setName("홍길동");
        account.setAccountNo("123-456");

        account.deposit(10000);
        account.withdraw(3000);

        account.printAccountInfo();

    }

}
예금주: 홍길동
계좌번호: 123-456
잔고: 7000

5-4. User 캡슐화 예제

public class User {

    private Long userId;
    private String loginId;
    private String password;
    private String email;
    private String userType;

    public Long getUserId() {
        return userId;
    }

    public String getLoginId() {
        return loginId;
    }

    public String getEmail() {
        return email;
    }

    public String getUserType() {
        return userType;
    }

    public void changePassword(String newPassword) {
        if (newPassword != null && newPassword.length() >= 8) {
            this.password = newPassword;
        }
    }

    public void changeEmail(String email) {
        if (isValidEmail(email)) {
            this.email = email;
        }
    }

    private boolean isValidEmail(String email) {
        return email != null && email.contains("@");
    }

}

password getter는 제공하지 않고, changePassword()로만 변경하게 합니다. isValidEmail()은 내부 검증용이므로 private으로 숨깁니다.

6. 실무 연결

6-1. 실무에서의 캡슐화 흐름

필드
→ private

외부에 보여줘도 되는 값
→ public getter

외부에서 변경 가능한 값
→ public setter 또는 의미 있는 변경 메서드

내부 검증 로직
→ private method

민감한 값
→ getter 제공 주의

6-2. Entity에서 캡슐화

public class UserEntity {

    private Long userId;
    private String loginId;
    private String password;
    private String email;
    private String userType;

    public String getLoginId() {
        return loginId;
    }

    public void changeEmail(String email) {
        if (email != null && email.contains("@")) {
            this.email = email;
        }
    }

}

Entity는 DB 데이터와 가까운 객체이므로 외부에서 아무 값이나 직접 넣을 수 없도록 캡슐화하는 것이 중요합니다.

6-3. 유지보수에 유리한 이유

캡슐화를 하면 규칙이 바뀌어도 클래스 내부만 수정하면 됩니다. 

public void deposit(int amount) {
    if (amount > 0) {
        balance += amount;
    }
}

나중에 1회 입금 한도를 추가해야 한다면 메서드 내부만 바꾸면 됩니다.

public void deposit(int amount) {
    if (amount > 0 && amount <= 1_000_000) {
        balance += amount;
    }
}

외부 코드는 그대로 유지됩니다. 

account.deposit(50000);

7. 자주 하는 실수

실수 1. 필드를 public으로 열어둠

public int balance;
private int balance;

실수 2. setter에 검증이 없음

public void setBalance(int balance) {
    this.balance = balance;
}
public void setBalance(int balance) {
    if (balance >= 0) {
        this.balance = balance;
    }
}

실수 3. 모든 필드에 무조건 setter를 만듦

public void setUserId(Long userId) {
    this.userId = userId;
}

userId처럼 시스템이나 DB가 관리해야 하는 값은 외부에서 마음대로 바꾸지 못하게 setter를 만들지 않는 것이 나을 수 있습니다.

실수 4. 비밀번호 getter를 만듦

public String getPassword() {
    return password;
}

비밀번호 같은 민감한 값은 외부에 그대로 반환하지 않는 것이 좋습니다.

실수 5. private으로 숨겼지만 public setter로 다 열어둠

private int balance;

public void setBalance(int balance) {
    this.balance = balance;
}

형식상 필드는 private이지만, setter로 아무 값이나 넣을 수 있으면 캡슐화 효과가 약해집니다.

실수 6. this를 쓰지 않아 필드 대입이 제대로 안 됨

private int balance;

public void setBalance(int balance) {
    balance = balance;
}
public void setBalance(int balance) {
    this.balance = balance;
}

8. 면접 포인트

Q1. 캡슐화란 무엇인가요?

답변: 캡슐화는 객체의 필드를 외부에서 직접 접근하지 못하게 숨기고, 공개된 메서드를 통해서만 접근하도록 하는 객체지향의 특징입니다.
이를 통해 객체 내부 데이터를 보호하고 잘못된 값이 들어오는 것을 막을 수 있습니다.

Q2. 캡슐화는 어떻게 구현하나요?

답변: 일반적으로 필드를 private으로 선언하고, 필드 값을 읽거나 변경할 수 있는 public getter/setter 또는 의미 있는 메서드를 제공합니다.

Q3. setter 메서드란 무엇인가요?

답변: setter는 필드에 값을 설정하는 메서드입니다. 보통 public 접근 지정자, void 반환 타입, 매개변수를 가지며, 전달받은 값을 멤버 변수에 저장하는 역할을 합니다.

Q4. getter 메서드란 무엇인가요?

답변: getter는 필드에 저장된 값을 반환하는 메서드입니다. 보통 public 접근 지정자를 사용하고, 반환 타입은 필드의 자료형과 같으며, 매개변수는 받지 않습니다.

Q5. 캡슐화에서 this를 사용하는 이유는 무엇인가요?

답변: 필드명과 매개변수명이 같을 때 둘을 구분하기 위해 this를 사용합니다.
this.balance는 현재 객체의 멤버 변수 balance를 의미하고, balance는 메서드의 매개변수를 의미합니다.

Q6. 모든 필드에 getter/setter를 만드는 것이 좋은 캡슐화인가요?

답변: 항상 그렇지는 않습니다. 필드를 private으로 숨겼더라도 모든 setter를 열어두면 외부에서 객체 상태를 마음대로 바꿀 수 있습니다.
비밀번호처럼 민감한 값은 getter를 제공하지 않는 것이 좋고,
상태 변경은 setBalance()보다 deposit(), withdraw(), changePassword()처럼 의미 있는 메서드로 제한하는 것이 좋습니다.

Q7. 캡슐화의 장점은 무엇인가요?

답변: 캡슐화는 객체 내부 데이터를 보호하고, 잘못된 값이 들어오는 것을 막으며, 내부 구현 변경이 외부 코드에 미치는 영향을 줄입니다.
또한 객체가 자신의 상태 변경 규칙을 스스로 관리하게 하므로 유지보수성과 안정성이 좋아집니다.

9. 오늘의 핵심 정리

  1. 캡슐화는 필드를 숨기고 메서드로 접근하게 하는 것이다.
  2. 필드는 보통 private으로 선언한다.
  3. setter는 값을 넣는 메서드다.
  4. getter는 값을 꺼내는 메서드다.
  5. setter는 public, 반환 타입 void, 매개변수를 받는 형태가 일반적이다.
  6. getter는 public, 반환 타입이 있고, 매개변수를 받지 않는 형태가 일반적이다.
  7. this는 멤버 변수와 매개변수 이름이 같을 때 멤버 변수를 구분하는 데 사용된다.
  8. 캡슐화의 목적은 객체 내부 데이터를 보호하는 것이다.
  9. getter/setter를 무조건 만드는 것이 좋은 캡슐화는 아니다.
  10. 필드에 직접 접근하지 못하게 하고, 의미 있는 메서드로 상태를 변경하는 것이 좋다.
  11. 캡슐화는 접근 지정자와 직접 연결된다.
  12. 캡슐화는 유지보수성과 안정성을 높인다.

학습 메모

이 페이지는 Java 캡슐화 단원입니다.
캡슐화는 멤버 변수, 즉 필드를 private으로 숨기고, 필드에 접근할 때는 getter, setter 또는 의미 있는 public 메서드를 사용하게 하는 방식입니다.
중요한 점은 getter/setter를 무조건 만드는 것이 아니라, 객체의 상태가 안전하게 유지되도록 필요한 통로만 열어두는 것입니다.
