Spring Boot - Chapter 19

JWT 개념

JWT(Json Web Token)는 로그인 성공 후 서버가 발급하는 서명된 인증 토큰입니다.
사용자는 이 토큰을 저장했다가 API 요청마다 Authorization: Bearer <token> 헤더로 보내고, 서버는 토큰의 서명과 만료 시간을 검증해 사용자를 식별합니다.
이번 단원에서는 Header, Payload, Signature, Access Token, Refresh Token, Bearer, HttpOnly Cookie를 중심으로
로그인 후 토큰 저장과 인증 요청 구조를 정리합니다.

JWT Json Web Token Header Payload Signature Claim sub iat exp HS256 Secret Key Access Token Refresh Token Bearer Authorization Header HttpOnly Cookie Token Rotation React Axios

01

JWT를 공부한다는 것

JWT를 공부한다는 것은 단순히 긴 토큰 문자열을 외우는 것이 아닙니다.
로그인 성공 후 서버가 어떤 증명서를 만들어 주고, React가 그것을 어디에 저장하며, 이후 API 요청마다 어떻게 인증 정보로 보내는지를 이해하는 것입니다.

Token

서명된 인증 문자열

JWT는 로그인 성공 후
서버가 발급하는 인증용 문자열입니다.
서버는 나중에 이 문자열의 서명을 검증해
사용자를 식별합니다.

Structure

Header.Payload.Signature

JWT는 점(.)으로 구분된
세 영역으로 구성됩니다.
Header는 타입과 알고리즘,
Payload는 사용자 정보,
Signature는 서명입니다.

Request

Authorization Header

React는 저장한 Access Token을
API 요청마다 Authorization 헤더에
Bearer 형식으로 담아 보냅니다.

Validation

서명과 만료 시간 검증

서버는 토큰의 Signature와 exp를 확인합니다.
변조되었거나 만료된 토큰이면
요청을 거부합니다.

핵심 문장

JWT는 사용자의 로그인 상태를 증명하기 위해 서버가 발급하는 서명된 토큰이며, 클라이언트는 이 토큰을 저장했다가 API 요청마다 함께 보냅니다.

02

JWT 구조 카드

JWT는 하나의 문자열처럼 보이지만 실제로는 Header, Payload, Signature 세 부분으로 나뉩니다.

Header 토큰 타입 + 알고리즘

typ: JWT
alg: HS256

.
Payload 사용자 정보 + Claim

sub, roles iat, exp

.
Signature 변조 방지 서명

Header + Payload + Secret Key

Header

표지

이 토큰이 JWT이고 어떤 알고리즘으로 서명되었는지 알려줍니다. 

{
  "typ": "JWT",
  "alg": "HS256"
}
Payload

내용

사용자 식별자, 권한, 발급 시간, 만료 시간 같은
Claim이 들어갑니다.
단, 암호화가 아니므로 민감 정보는 넣으면 안 됩니다. 

{
  "sub": "user1",
  "roles": ["ROLE_USER"],
  "iat": 1720000000,
  "exp": 1720003600
}
Signature

위조 방지 도장

Header와 Payload가 중간에 바뀌지 않았는지
확인하는 서명입니다. 

HMACSHA256(
  base64UrlEncode(header) + "." +
  base64UrlEncode(payload),
  secretKey
)
주의

JWT의 Payload는 숨겨진 공간이 아닙니다.
Base64Url 인코딩일 뿐이므로 디코딩하면 내용을 볼 수 있습니다.
따라서 비밀번호, 주민등록번호, 계좌번호, 카드번호 같은 민감 정보는 절대 넣으면 안 됩니다.

03

로그인 후 JWT 인증 흐름도

JWT 인증 흐름은 로그인 요청 → 토큰 발급 → 토큰 저장 → 인증 요청 → 서버 검증으로 이어집니다. React는 토큰을 저장하고, Spring Boot는 요청마다 들어오는 토큰을 검증합니다.

1 로그인 요청

React가 username과 password를 담아
POST /api/auth/login 요청을 보냅니다.

→
2 사용자 검증

Spring Boot가 DB에서 사용자를 조회하고
비밀번호가 맞는지 확인합니다.

→
3 JWT 발급

검증에 성공하면 서버가 Access Token과
필요 시 Refresh Token을 발급합니다.

4 토큰 저장

React가 Access Token을 저장합니다.
Refresh Token은 HttpOnly Cookie로 저장하는 구조가
권장됩니다.

→
5 인증 요청

Axios가 API 요청마다
Authorization: Bearer <token> 헤더를 추가합니다.

→
6 서버 검증

Spring Security Filter가
토큰의 서명, 만료 시간, 사용자 정보를 검증합니다.

입장권 비유

JWT는 입장권

로그인은 입장권 발급이고, API 요청은 놀이기구 입장입니다.
서버는 요청마다 입장권이 진짜인지 확인합니다.

신분증 비유

Payload는 신분 정보

Payload에는 사용자 ID와 권한이 들어갑니다.
하지만 누구나 볼 수 있으므로 민감 정보는 넣지 않습니다.

도장 비유

Signature는 위조 방지 도장

Payload를 누군가 바꾸면 도장이 맞지 않습니다.
서버는 서명 검증 실패로 토큰을 거부합니다.

04

Access Token, Refresh Token, Bearer, HttpOnly Cookie

JWT 인증 구조에서 가장 많이 헷갈리는 부분은 Access Token과 Refresh Token의 역할 분리입니다. Access Token은 API 요청에 사용하고, Refresh Token은 Access Token 재발급에 사용합니다.

Access Token

API 요청 인증용

서버 리소스에 접근할 권한을 증명합니다.
보통 짧은 만료 시간을 가지며
Authorization 헤더에 담아 보냅니다. 

Authorization: Bearer <AccessToken>
Refresh Token

Access Token 재발급용

Access Token이 만료되었을 때
새 Access Token을 발급받기 위해 사용합니다.
Access Token보다 긴 수명을 가집니다. 

POST /api/auth/refresh
Cookie: refreshToken=...
Bearer

토큰 인증 접두사

Authorization 헤더에 토큰을 보낼 때
사용하는 표준적인 접두사입니다.
Bearer 뒤에는 반드시 공백이 필요합니다. 

Authorization: Bearer eyJ...
HttpOnly Cookie

JavaScript 접근 차단

JavaScript에서 읽을 수 없는 쿠키입니다.
XSS로 Refresh Token이 탈취되는 위험을
줄이는 데 사용됩니다. 

Set-Cookie: refreshToken=...
HttpOnly; SameSite=Lax

4) JwtTokenProvider

package com.example.study.auth.jwt;

import io.jsonwebtoken.*;
import io.jsonwebtoken.security.Keys;
import org.springframework.beans.factory.annotation.Value;
import org.springframework.stereotype.Component;

import javax.crypto.SecretKey;
import java.util.Date;
import java.util.List;

@Component
public class JwtTokenProvider {

    private final SecretKey secretKey;
    private final long accessTokenExpirationMs;

    public JwtTokenProvider(
            @Value("${jwt.secret}") String secret,
            @Value("${jwt.access-token-expiration-ms}") long accessTokenExpirationMs
    ) {
        this.secretKey = Keys.hmacShaKeyFor(secret.getBytes());
        this.accessTokenExpirationMs = accessTokenExpirationMs;
    }

    public String createAccessToken(String username, List<String> roles) {
        Date now = new Date();
        Date expiration = new Date(now.getTime() + accessTokenExpirationMs);

        return Jwts.builder()
                .setSubject(username)
                .claim("roles", roles)
                .setIssuedAt(now)
                .setExpiration(expiration)
                .signWith(secretKey, SignatureAlgorithm.HS256)
                .compact();
    }

    public boolean validateToken(String token) {
        try {
            Jwts.parserBuilder()
                    .setSigningKey(secretKey)
                    .build()
                    .parseClaimsJws(token);

            return true;

        } catch (ExpiredJwtException e) {
            System.out.println("JWT 만료: " + e.getMessage());
        } catch (JwtException | IllegalArgumentException e) {
            System.out.println("JWT 유효하지 않음: " + e.getMessage());
        }

        return false;
    }

    public String getUsername(String token) {
        return Jwts.parserBuilder()
                .setSigningKey(secretKey)
                .build()
                .parseClaimsJws(token)
                .getBody()
                .getSubject();
    }
}

5) AuthController - 로그인 성공 시 JWT 발급

package com.example.study.auth.controller;

import com.example.study.auth.dto.LoginRequest;
import com.example.study.auth.dto.LoginResponse;
import com.example.study.auth.jwt.JwtTokenProvider;
import lombok.RequiredArgsConstructor;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.*;

import java.util.List;

@RestController
@RequestMapping("/api/auth")
@RequiredArgsConstructor
public class AuthController {

    private final JwtTokenProvider jwtTokenProvider;

    @PostMapping("/login")
    public ResponseEntity<LoginResponse> login(@RequestBody LoginRequest request) {
        // 학습용 단순 예제
        // 실무에서는 DB 조회 + PasswordEncoder.matches() 검증이 필요합니다.
        if (!request.username().equals("user1") || !request.password().equals("pass1234")) {
            return ResponseEntity.status(401).build();
        }

        String accessToken = jwtTokenProvider.createAccessToken(
                request.username(),
                List.of("ROLE_USER")
        );

        return ResponseEntity.ok(
                new LoginResponse("Bearer", accessToken)
        );
    }
}

6) TokenTestController - 학습용 토큰 검증

package com.example.study.auth.controller;

import com.example.study.auth.jwt.JwtTokenProvider;
import lombok.RequiredArgsConstructor;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.*;

import java.util.Map;

@RestController
@RequestMapping("/api/token-test")
@RequiredArgsConstructor
public class TokenTestController {

    private final JwtTokenProvider jwtTokenProvider;

    @GetMapping("/me")
    public ResponseEntity<?> me(@RequestHeader("Authorization") String authorizationHeader) {
        if (authorizationHeader == null || !authorizationHeader.startsWith("Bearer ")) {
            return ResponseEntity.status(401).body("Authorization 헤더가 없습니다.");
        }

        String token = authorizationHeader.substring(7);

        if (!jwtTokenProvider.validateToken(token)) {
            return ResponseEntity.status(401).body("유효하지 않은 토큰입니다.");
        }

        String username = jwtTokenProvider.getUsername(token);

        return ResponseEntity.ok(
                Map.of(
                        "username", username,
                        "message", "토큰 검증 성공"
                )
        );
    }
}

import org.springframework.http.HttpHeaders;
import org.springframework.http.ResponseCookie;
import org.springframework.http.ResponseEntity;

ResponseCookie refreshCookie = ResponseCookie.from("refreshToken", refreshToken)
        .httpOnly(true)
        .secure(false) // 로컬 개발 환경 false, HTTPS 운영 환경 true 권장

2) 토큰 저장 함수

// src/api/tokenStorage.js

const ACCESS_TOKEN_KEY = "accessToken";

export function saveAccessToken(token, keepLogin) {
  if (keepLogin) {
    localStorage.setItem(ACCESS_TOKEN_KEY, token);
    sessionStorage.removeItem(ACCESS_TOKEN_KEY);
    return;
  }

  sessionStorage.setItem(ACCESS_TOKEN_KEY, token);
  localStorage.removeItem(ACCESS_TOKEN_KEY);
}

export function getAccessToken() {
  return (
    localStorage.getItem(ACCESS_TOKEN_KEY) ||
    sessionStorage.getItem(ACCESS_TOKEN_KEY)
  );
}

export function clearAccessToken() {
  localStorage.removeItem(ACCESS_TOKEN_KEY);
  sessionStorage.removeItem(ACCESS_TOKEN_KEY);
}

3) Axios 공통 인스턴스

// src/api/apiClient.js

import axios from "axios";
import { getAccessToken } from "./tokenStorage";

const apiClient = axios.create({
  baseURL: "http://localhost:8080",
  timeout: 5000,
  headers: {
    "Content-Type": "application/json",
  },
});

apiClient.interceptors.request.use(
  (config) => {
    const token = getAccessToken();

    if (token) {
      config.headers.Authorization = `Bearer ${token}`;
    }

    return config;
  },
  (error) => Promise.reject(error)
);

export default apiClient;

// src/pages/LoginPage.jsx

import { useState } from "react";
import { login } from "../api/authApi";
import { saveAccessToken } from "../api/tokenStorage";

function LoginPage() {
  const [form, setForm] = useState({
    username: "",
    password: "",
    keepLogin: false,
  });

  const handleChange = (e) => {
    const { name, value, type, checked } = e.target;

    setForm((prev) => ({
      ...prev,
      [name]: type === "checkbox" ? checked : value,
    }));
  };

  const handleSubmit = async (e) => {
    e.preventDefault();

    try {
      const result = await login({
        username: form.username,
        password: form.password,
      });

      saveAccessToken(result.accessToken, form.keepLogin);

      alert("로그인 성공");
    } catch (error) {
      console.error(error);
      alert("아이디 또는 비밀번호를 확인하세요.");
    }
  };

  return (
    <form onSubmit={handleSubmit}>
      <input
        name="username"
        value={form.username}
        onChange={handleChange}
        placeholder="아이디"
      />

      <input
        name="password"
        type="password"
        value={form.password}
        onChange={handleChange}
        placeholder="비밀번호"
      />

      <label>
        <input
          name="keepLogin"
          type="checkbox"
          checked={form.keepLogin}
          onChange={handleChange}
        />
        로그인 유지
      </label>

      <button type="submit">로그인</button>
    </form>
  );
}

export default LoginPage;

자주 하는 실수 카드

1. Payload에 민감 정보 저장

JWT Payload는 디코딩하면 볼 수 있습니다.
비밀번호, 주민등록번호, 계좌번호,
카드번호를 넣으면 안 됩니다.

2. JWT를 암호화로 오해

일반적인 JWT는 암호화가 아니라
인코딩과 서명입니다.
핵심은 숨김이 아니라 변조 검증입니다.

3. Access Token 만료 시간을
너무 길게 설정

Access Token이 탈취되면
만료 전까지 API 접근이 가능합니다.
짧게 유지하고
Refresh Token 구조를 고려합니다.

4. Refresh Token을
localStorage에 저장

Refresh Token은
재발급 권한이 있어 더 위험합니다.
실무에서는 HttpOnly Cookie 저장을
많이 고려합니다.

5. Bearer 접두사 누락

서버가 startsWith("Bearer ")로 검사하면
Bearer가 없을 때 토큰을 추출하지 못합니다.

6. Bearer 뒤 공백 누락

올바른 형식은
Authorization: Bearer <token>입니다.
Bearer와 토큰 사이에는 공백이 필요합니다.

7. Secret Key를 코드에 직접 작성

Secret Key는 환경변수로 관리해야 합니다.
Git에 올라가면
모든 토큰을 위조할 수 있는 위험이 생깁니다.

8. Secret Key 길이 부족

HS256에서는 충분히 긴 키가 필요합니다.
너무 짧으면 WeakKeyException이
발생할 수 있습니다.

9. 만료 처리 로직 누락

Access Token은 반드시 만료됩니다.
401 응답 감지, 재발급,
로그아웃 흐름을 준비해야 합니다.

10. 로그아웃을
클라이언트 삭제만으로 끝냄

Refresh Token을 운영한다면
서버 저장소 삭제, 블랙리스트, 쿠키 만료도
함께 고려해야 합니다.

11. 토큰에 너무 많은 정보 저장

JWT는 요청마다 Header에 실려 이동합니다.
토큰이 커지면 모든 API 요청이 무거워집니다.

12. CORS와 JWT 문제 혼동

Authorization 헤더와 application/json은
Preflight를 유발할 수 있습니다.
이때는 JWT가 아니라
CORS 설정 문제일 수 있습니다.

08

오류 메시지 카드

JWT 오류는 대부분 헤더 형식 오류, 토큰 만료, 서명 불일치, Secret Key 문제, CORS 설정 문제에서 발생합니다.

10

면접 포인트 카드

Q1. JWT란 무엇인가요?

JWT는 JSON Web Token의 약자로, 로그인 성공 후 서버가 발급하는 서명된 인증 토큰입니다. 클라이언트는 이 토큰을 저장했다가 API 요청마다 함께 보냅니다.

Q2. JWT는 어떤 구조인가요?

JWT는 Header, Payload, Signature 세 부분으로 구성됩니다. Header에는 타입과 알고리즘, Payload에는 Claim, Signature에는 변조 검증용 서명이 들어갑니다.

Q3. Payload에는 무엇을 넣나요?

사용자 식별자, 권한, 발급 시간, 만료 시간 같은 Claim을 넣습니다. 단, 암호화된 공간이 아니므로 비밀번호 같은 민감 정보는 넣으면 안 됩니다.

Q4. Signature의 역할은 무엇인가요?

Header와 Payload가 변조되지 않았는지 검증하는 역할입니다. 서버의 Secret Key로 다시 계산한 서명과 비교합니다.

Q5. JWT는 암호화인가요?

일반적인 JWT는 암호화가 아니라 Base64Url 인코딩과 서명입니다. Payload는 디코딩 가능하므로 민감 정보를 넣으면 안 됩니다.

Q6. Access Token과 Refresh Token의 차이는 무엇인가요?

Access Token은 API 요청 인증용이고 짧은 수명을 가집니다. Refresh Token은 Access Token 재발급용이고 더 긴 수명을 가집니다.

Q7. Bearer는 무엇인가요?

Authorization 헤더에서 토큰 인증 방식을 나타내는 접두사입니다. 형식은 Authorization: Bearer <JWT>입니다.

Q8. HttpOnly Cookie는 무엇인가요?

JavaScript로 접근할 수 없는 쿠키입니다. Refresh Token 저장 시 XSS로 인한 탈취 위험을 줄이는 데 사용됩니다.

Q9. JWT 방식과 세션 방식의 차이는 무엇인가요?

세션 방식은 서버가 로그인 상태를 저장하고, JWT 방식은 클라이언트가 토큰을 저장해 요청마다 보내면 서버가 검증합니다.

Q10. JWT 로그아웃은 어떻게 처리하나요?

단순 구조에서는 클라이언트 저장소의 토큰을 삭제합니다. Refresh Token을 운영한다면 서버 저장소 삭제, 블랙리스트, 쿠키 만료도 함께 처리합니다.

Q11. JWT 사용 시 DB 조회가 필요 없나요?

토큰만으로 사용자 식별은 가능하지만, 탈퇴, 정지, 권한 변경을 반영하려면 UserDetailsService로 DB를 다시 조회하는 구조를 많이 사용합니다.

Q12. JWT 인증 흐름을 설명해 주세요.

로그인 성공 후 서버가 JWT를 발급하고, React가 토큰을 저장한 뒤, API 요청마다 Authorization 헤더에 Bearer Token을 담아 보내면 Spring Security Filter가 검증합니다.

학습 메모

오늘의 암기 포인트

  • JWT는 JSON Web Token이며, 로그인 성공 후 서버가 발급하는 서명된 인증 토큰입니다.
  • JWT 구조는 Header.Payload.Signature입니다.
  • Header에는 토큰 타입과 서명 알고리즘이 들어갑니다.
  • Payload에는 사용자 식별자, 권한, 발급 시간, 만료 시간 같은 Claim이 들어갑니다.
  • Payload는 암호화가 아니므로 민감 정보를 넣으면 안 됩니다.
  • Signature는 Header와 Payload가 변조되지 않았는지 검증하는 서명입니다.
  • Access Token은 API 요청 인증용이고 짧은 수명을 가집니다.
  • Refresh Token은 Access Token 재발급용이고 긴 수명을 가집니다.
  • Bearer는 Authorization 헤더에 토큰을 실어 보낼 때 사용하는 접두사입니다.
  • HttpOnly Cookie는 JavaScript로 읽을 수 없어 Refresh Token 저장에 자주 사용됩니다.

헷갈리기 쉬운 개념

인코딩 vs 암호화

JWT Payload는 인코딩되어 있을 뿐입니다.
디코딩하면 볼 수 있으므로 민감 정보 금지입니다.

Signature vs Encryption

Signature는 내용을 숨기는 것이 아니라
변조 여부를 확인하는 서명입니다.

Access vs Refresh

Access Token은 API 요청용,
Refresh Token은 Access Token 재발급용입니다.

Bearer 공백

Bearer토큰이 아니라
Bearer 토큰처럼 공백이 필요합니다.

localStorage vs HttpOnly

localStorage는 사용이 쉽지만 XSS에 취약합니다.
HttpOnly Cookie는 JavaScript 접근을 막습니다.

401 vs 403

401은 인증 실패,
403은 인증은 되었지만 권한이 없는 상태입니다.
