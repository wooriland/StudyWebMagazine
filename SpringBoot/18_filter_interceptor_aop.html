Spring Boot - Chapter 18

Filter, Interceptor, AOP

Filter, Interceptor, AOP는 Spring Boot 애플리케이션에서 반복되는 공통 로직을 어느 위치에 배치할지 결정하는 핵심 기술입니다.
Filter는 DispatcherServlet에 도달하기 전 가장 바깥쪽에서 요청/응답을 가로채고,
Interceptor는 Controller 실행 전후를 가로채며,
AOP는 Service 같은 Spring Bean 메서드 실행 전후에 공통 로직을 적용합니다.
React에서 Axios로 API를 호출하면
Spring Boot 내부에서는 Filter → DispatcherServlet → Interceptor → Controller → AOP → Service 흐름으로 요청이 처리됩니다.

01

Filter, Interceptor, AOP를 공부한다는 것

Spring Boot 프로젝트에는 여러 기능에서 반복되는 공통 로직이 많습니다.
예를 들어 로그인 확인, JWT 토큰 검사, 권한 확인, 요청 로그 출력, 실행 시간 측정, CORS 처리, 인코딩 처리, 예외 로그 저장, 트랜잭션 처리 같은 로직입니다.
이런 코드를 모든 Controller와 Service에 직접 작성하면 비즈니스 로직이 지저분해지고, 정책이 바뀔 때 여러 파일을 수정해야 합니다.

Filter

가장 바깥쪽 요청 관문

Servlet Container,
즉 Tomcat 레벨에서 동작합니다. DispatcherServlet에 도달하기 전
요청/응답을 가로챕니다.

Interceptor

Controller 앞 요청 관문

Spring MVC 영역에서 동작합니다.
DispatcherServlet과 Controller 사이에서
요청을 가로챕니다.

AOP

메서드 실행 관문

Controller, Service, Repository 같은
Spring Bean의 메서드 실행 전후에
공통 로직을 끼워 넣습니다.

Common Logic

공통 로직 분리

인증, 권한, 로깅, 성능 측정, 트랜잭션 같은
반복 코드를 분리해
핵심 로직에 집중하게 합니다.

핵심 문장

Filter는 Spring MVC 바깥쪽, Interceptor는 Controller 앞, AOP는 Spring Bean 메서드 앞뒤에서 공통 로직을 처리합니다.

02

요청 처리 흐름도

React에서 Axios로 Spring Boot API를 호출하면 요청은 바로 Controller로 들어가지 않습니다.
먼저 Servlet Container가 요청을 받고, Filter Chain을 통과한 뒤 DispatcherServlet, Interceptor, Controller, AOP, Service 순서로 흘러갑니다.

1 React / Browser

Axios로 API 요청을 보냅니다.
JWT가 있다면 Authorization 헤더에 담습니다.

→
2 Servlet Container

Tomcat이 가장 먼저 HTTP 요청을 받습니다.
Spring MVC에 들어가기 전 단계입니다.

→
3 Filter Chain

JWT, CORS, 인코딩, 전체 요청 로그 등
가장 바깥쪽 공통 처리를 수행합니다.

4 DispatcherServlet

Spring MVC의 중앙 관문입니다.
어떤 Controller가 요청을 처리할지 찾습니다.

→
5 Interceptor

Controller 실행 전후에 동작합니다.
로그인 체크, 권한 확인, 실행 시간 측정에 사용합니다.

→
6 Controller

요청 DTO를 받고 Service를 호출합니다.
REST API에서는 JSON 응답을 반환합니다.

7 AOP Proxy

Service 메서드 실행 전후를 감쌉니다.
성능 측정, 감사 로그, 예외 로그를 처리합니다.

→
8 Service

핵심 비즈니스 로직을 처리합니다.
예배 기록 저장, 추천 생성, 게시글 등록 등이 여기에 해당합니다.

→
9 Repository / DB

DB에 접근합니다.
JPA Repository 또는 MyBatis, JDBC 등이 연결됩니다.

요청 중단 포인트

Filter

chain.doFilter()를 호출하지 않으면
다음 단계로 가지 않습니다.

Interceptor

preHandle()에서 false를 반환하면
Controller로 가지 않습니다.

AOP

joinPoint.proceed()를 호출하지 않으면
실제 메서드가 실행되지 않습니다.

03

그림처럼 이해하기

Filter, Interceptor, AOP는 모두 “가로채기”처럼 보이지만, 어느 문 앞에서 가로채는지가 다릅니다.

건물 입구 경비원

Filter

건물에 들어오기 전 출입증과 위험 요소를 검사합니다.
웹에서는 JWT 토큰, CORS, IP 차단, 인코딩처럼
가장 앞단 처리를 담당합니다.

엘리베이터 앞 안내 직원

Interceptor

건물 안으로 들어온 뒤 특정 층이나
방에 들어갈 수 있는지 다시 확인합니다.
웹에서는 로그인 여부, 관리자 URL 접근,
Controller 요청 로그를 담당합니다.

사무실 내부 업무 감시 장치

AOP

실제 업무가 실행될 때
시작 시간, 종료 시간, 오류 여부를 기록합니다.
웹에서는 Service 실행 시간, 트랜잭션, 감사 로그를 담당합니다.

어디에 둘지 판단하는 기준

질문 1 Spring MVC에 들어오기 전 처리해야 하나?

그렇다면 Filter에 둡니다. 예: JWT, CORS, 인코딩

질문 2 Controller 실행 전후
URL별로 처리해야 하나?

그렇다면 Interceptor에 둡니다. 예: 세션 로그인, 관리자 경로

질문 3 Service 메서드 실행 전후로 처리해야 하나?

그렇다면 AOP에 둡니다. 예: 실행 시간, 트랜잭션, 감사 로그

5) Filter - 요청/응답 로그

package com.example.study.global.filter;

import jakarta.servlet.Filter;
import jakarta.servlet.FilterChain;
import jakarta.servlet.FilterConfig;
import jakarta.servlet.ServletException;
import jakarta.servlet.ServletRequest;
import jakarta.servlet.ServletResponse;
import jakarta.servlet.http.HttpServletRequest;

import java.io.IOException;

public class RequestLoggingFilter implements Filter {

    @Override
    public void init(FilterConfig filterConfig) throws ServletException {
        System.out.println("[RequestLoggingFilter] init");
    }

    @Override
    public void doFilter(ServletRequest request,
                         ServletResponse response,
                         FilterChain chain)
            throws IOException, ServletException {

        HttpServletRequest httpRequest = (HttpServletRequest) request;

        System.out.println("[Filter 전처리]");
        System.out.println("HTTP Method: " + httpRequest.getMethod());
        System.out.println("Request URI: " + httpRequest.getRequestURI());

        // 이 코드를 호출해야 다음 Filter 또는 DispatcherServlet으로 요청이 넘어갑니다.
        chain.doFilter(request, response);

        System.out.println("[Filter 후처리]");
    }

    @Override
    public void destroy() {
        System.out.println("[RequestLoggingFilter] destroy");
    }
}

7) Interceptor - Controller 실행 시간 측정

package com.example.study.global.interceptor;

import jakarta.servlet.http.HttpServletRequest;
import jakarta.servlet.http.HttpServletResponse;
import org.springframework.web.servlet.HandlerInterceptor;

public class RuntimeInterceptor implements HandlerInterceptor {

    private static final String START_TIME = "startTime";

    @Override
    public boolean preHandle(HttpServletRequest request,
                             HttpServletResponse response,
                             Object handler) throws Exception {

        long startTime = System.currentTimeMillis();
        request.setAttribute(START_TIME, startTime);

        System.out.println("[Interceptor preHandle]");
        System.out.println("URI: " + request.getRequestURI());
        System.out.println("Handler: " + handler);

        // true이면 Controller로 진행, false이면 Controller로 가지 않습니다.
        return true;
    }

    @Override
    public void afterCompletion(HttpServletRequest request,
                                HttpServletResponse response,
                                Object handler,
                                Exception ex) throws Exception {

        Long startTime = (Long) request.getAttribute(START_TIME);

        if (startTime != null) {
            long endTime = System.currentTimeMillis();
            long duration = endTime - startTime;

            System.out.println("[Interceptor afterCompletion]");
            System.out.println("URI: " + request.getRequestURI());
            System.out.println("Controller 처리 시간: " + duration + "ms");
        }

        if (ex != null) {
            System.out.println("요청 처리 중 예외 발생: " + ex.getMessage());
        }
    }
}

9) AOP - Service 실행 시간 측정

package com.example.study.global.aop;

import org.aspectj.lang.ProceedingJoinPoint;
import org.aspectj.lang.annotation.*;
import org.springframework.stereotype.Component;

@Aspect
@Component
public class ServiceRuntimeAspect {

    @Around("execution(* com.example.study..service..*(..))")
    public Object measureServiceTime(ProceedingJoinPoint joinPoint) throws Throwable {
        String methodName = joinPoint.getSignature().toShortString();

        long startTime = System.currentTimeMillis();

        try {
            // 이 코드가 실제 대상 Service 메서드를 실행합니다.
            Object result = joinPoint.proceed();

            long endTime = System.currentTimeMillis();
            System.out.println("[AOP] " + methodName + " 실행 시간: " + (endTime - startTime) + "ms");

            return result;

        } catch (Exception e) {
            System.out.println("[AOP] " + methodName + " 실행 중 예외: " + e.getMessage());
            throw e;
        }
    }
}

React는 Filter, Interceptor, AOP를 직접 호출하지 않습니다.

1) 토큰 저장 함수

// src/api/tokenStorage.js

const ACCESS_TOKEN_KEY = "accessToken";

export function saveToken(token, keepLogin) {
  if (keepLogin) {
    localStorage.setItem(ACCESS_TOKEN_KEY, token);
    sessionStorage.removeItem(ACCESS_TOKEN_KEY);
    return;
  }

  sessionStorage.setItem(ACCESS_TOKEN_KEY, token);
  localStorage.removeItem(ACCESS_TOKEN_KEY);
}

export function getAccessToken() {
  return (
    localStorage.getItem(ACCESS_TOKEN_KEY) ||
    sessionStorage.getItem(ACCESS_TOKEN_KEY)
  );
}

export function clearToken() {
  localStorage.removeItem(ACCESS_TOKEN_KEY);
  sessionStorage.removeItem(ACCESS_TOKEN_KEY);
}

2) Axios 공통 인스턴스

// src/api/apiClient.js

import axios from "axios";

const apiClient = axios.create({
  baseURL: "http://localhost:8080",
  timeout: 5000,

    "Content-Type": "application/json",

apiClient.interceptors.request.use(
  (config) => {

    if (token) {
      config.headers.Authorization = `Bearer ${token}`;

    return config;
  },
  (error) => Promise.reject(error)
);

import { useState } from "react";
import { createPost } from "../api/postApi";

function PostCreatePage() {
  const [form, setForm] = useState({
    title: "",
    content: "",
    writerId: 1,
  });

    const { name, value } = e.target;

    setForm((prev) => ({
      ...prev,
      [name]: value,
    }));
  };

  const handleSubmit = async (e) => {
    e.preventDefault();

    const requestDto = {
      title: form.title,
      content: form.content,
      writerId: Number(form.writerId),
    };

    try {
      const result = await createPost(requestDto);

      alert("게시글 등록 성공");
    } catch (error) {
      console.error(error);
      alert("게시글 등록 실패");
    }
  };

  return (
    <form onSubmit={handleSubmit}>
      <input
        name="title"
        value={form.title}
        onChange={handleChange}
        placeholder="제목"
      />

      <textarea
        name="content"
        value={form.content}
        onChange={handleChange}
        placeholder="내용"
      />

      <input
        name="writerId"
        type="number"
        value={form.writerId}
        onChange={handleChange}
      />

      <button type="submit">등록</button>
    </form>
  );
}

JWT Filter와 다음 단원 연결 코드

JWT 인증은 보통 Interceptor보다 Spring Security Filter에서 처리합니다. 인증은 Controller에 도달하기 전에 처리되어야 하고, 인증 결과를 SecurityContextHolder에 저장해야 하기 때문입니다.

1) OncePerRequestFilter 기반 JWT Filter

package com.example.study.auth.jwt;

import jakarta.servlet.FilterChain;
import jakarta.servlet.ServletException;
import jakarta.servlet.http.*;
import lombok.RequiredArgsConstructor;
import org.springframework.security.core.context.SecurityContextHolder;
import org.springframework.web.filter.OncePerRequestFilter;

import java.io.IOException;

@RequiredArgsConstructor
public class JwtAuthenticationFilter extends OncePerRequestFilter {

    private final JwtTokenProvider jwtTokenProvider;

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain filterChain)
            throws ServletException, IOException {

        String token = jwtTokenProvider.resolveToken(request);

        if (token != null && jwtTokenProvider.validateToken(token)) {
            var authentication = jwtTokenProvider.getAuthentication(token);
            SecurityContextHolder.getContext().setAuthentication(authentication);
        }

        filterChain.doFilter(request, response);
    }
}

2) SecurityFilterChain 등록

package com.example.study.auth.config;

import com.example.study.auth.jwt.JwtAuthenticationFilter;
import com.example.study.auth.jwt.JwtTokenProvider;
import lombok.RequiredArgsConstructor;
import org.springframework.context.annotation.*;
import org.springframework.security.config.annotation.web.builders.HttpSecurity;
import org.springframework.security.config.http.SessionCreationPolicy;
import org.springframework.security.web.*;
import org.springframework.security.web.authentication.UsernamePasswordAuthenticationFilter;

@Configuration
@RequiredArgsConstructor
public class SecurityConfig {

    private final JwtTokenProvider jwtTokenProvider;

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        http
                .csrf(csrf -> csrf.disable())
                .sessionManagement(session ->
                        session.sessionCreationPolicy(SessionCreationPolicy.STATELESS)
                )
                .authorizeHttpRequests(auth -> auth
                        .requestMatchers("/api/auth/**").permitAll()
                        .anyRequest().authenticated()
                )
                .addFilterBefore(
                        new JwtAuthenticationFilter(jwtTokenProvider),
                        UsernamePasswordAuthenticationFilter.class
                );

        return http.build();
    }
}
주의

커스텀 Filter를 @Component로 Bean 등록하고, 동시에 addFilterBefore()로 SecurityFilterChain에도 등록하면 Filter가 두 번 실행될 수 있습니다.
JWT Filter는 보통 SecurityFilterChain 안에서 명확하게 등록합니다.

자주 하는 실수 카드

1. 세 기술의 위치를 구분하지 못함

Filter는 DispatcherServlet 전,
Interceptor는 Controller 전후,
AOP는 Spring Bean 메서드 전후입니다.

2. chain.doFilter() 누락

Filter에서 이 코드를 호출하지 않으면
요청이 다음 Filter나 Controller로
전달되지 않습니다.

3. preHandle()에서 false 반환

Interceptor에서 false를 반환하면
Controller가 실행되지 않습니다.
정상 진행은 true입니다.

4. proceed() 누락

AOP의 @Around에서
joinPoint.proceed()를 호출하지 않으면
실제 Service 메서드가 실행되지 않습니다.

5. javax.servlet 사용

Spring Boot 3에서는
javax.servlet이 아니라
jakarta.servlet 패키지를 사용해야 합니다.

6. @WebFilter만 붙이고 스캔 누락

@WebFilter 방식은 메인 클래스에
@ServletComponentScan이
필요할 수 있습니다.

7. Filter 중복 등록

@Component 등록과
addFilterBefore 등록을 동시에 하면
Filter가 두 번 실행될 수 있습니다.

8. JWT Filter 위치 오류

JWT Filter는 보통
UsernamePasswordAuthenticationFilter
앞에 addFilterBefore로 등록합니다.

9. URL 패턴 오해

/api/*와 /api/**는 다릅니다.
하위 경로 전체를 잡으려면
보통 /api/**를 사용합니다.

10. AOP Pointcut을 너무 넓게 작성

execution(* com.example..*(..))처럼
너무 넓으면 원하지 않는 Bean까지
AOP가 적용될 수 있습니다.

11. REST API에서
postHandle만 믿음

@RestController 응답은
ModelAndView가 없을 수 있습니다.
응답 body 처리는
ResponseBodyAdvice나
ControllerAdvice도 고려합니다.

12. 내부 메서드 호출에 AOP 기대

Spring AOP는 Proxy 기반이라
같은 클래스 내부 호출에는
적용되지 않을 수 있습니다.

10

면접 포인트 카드

Q1. Filter, Interceptor, AOP의 차이는 무엇인가요?

Filter는 Servlet Container에서 DispatcherServlet 전후에 동작하고, Interceptor는 Spring MVC에서 Controller 전후에 동작하며, AOP는 Spring Bean 메서드 실행 전후에 동작합니다.

Q2. Filter는 언제 사용하나요?

Spring MVC에 들어오기 전 전체 요청에 적용할 공통 처리에 사용합니다. JWT 인증, CORS, 인코딩, 요청 로그, IP 차단이 대표 예시입니다.

Q3. Interceptor는 언제 사용하나요?

Controller 실행 전후에 URL 패턴별 공통 처리가 필요할 때 사용합니다. 세션 로그인 체크, 관리자 경로 제한, Controller 실행 시간 측정에 적합합니다.

Q4. AOP는 언제 사용하나요?

Service나 Repository 같은 Spring Bean 메서드 실행 전후에 공통 로직을 적용할 때 사용합니다. 트랜잭션, 실행 시간 측정, 감사 로그, 예외 로깅이 대표 예시입니다.

Q5. chain.doFilter()는 무엇인가요?

다음 Filter 또는 DispatcherServlet으로 요청을 넘기는 코드입니다. 호출하지 않으면 요청은 Controller까지 가지 않습니다.

Q6. preHandle()의 true와 false 차이는 무엇인가요?

true를 반환하면 Controller로 진행하고, false를 반환하면 요청이 중단되어 Controller가 실행되지 않습니다.

Q7. joinPoint.proceed()는 무엇인가요?

@Around Advice에서 실제 대상 메서드를 실행하는 코드입니다. 호출하지 않으면 Service 메서드가 실행되지 않습니다.

Q8. JWT 인증은 어디에 구현하는 것이 좋나요?

보통 Spring Security의 SecurityFilterChain 내부에 커스텀 Filter로 구현합니다. Controller에 도달하기 전에 인증을 처리해야 하기 때문입니다.

Q9. OncePerRequestFilter를 사용하는 이유는 무엇인가요?

한 요청당 Filter가 한 번만 실행되도록 보장하기 위해 사용합니다. JWT 인증 Filter에 자주 사용합니다.

Q10. @Around를 많이 사용하는 이유는 무엇인가요?

메서드 실행 전과 실행 후를 모두 제어할 수 있기 때문입니다. 실행 시간 측정, 예외 처리, 결과 가공, 감사 로그에 유연합니다.

Q11. Controller 실행 시간과 Service 실행 시간은 각각 어디에서 측정하나요?

Controller 실행 시간은 Interceptor에서 측정하기 좋고, Service 비즈니스 로직 실행 시간은 AOP에서 측정하기 좋습니다.

Q12. CORS는 어디에서 처리하는 것이 좋나요?

Spring Security를 사용하는 프로젝트라면 Security Filter Chain 레벨에서 처리하는 것이 좋습니다. OPTIONS Preflight 요청이 Controller에 도달하기 전에 막힐 수 있기 때문입니다.

학습 메모

오늘의 암기 포인트

  • Filter는 Servlet Container 영역에서 DispatcherServlet에 도달하기 전에 실행됩니다.
  • Interceptor는 DispatcherServlet과 Controller 사이에서 실행됩니다.
  • AOP는 Controller, Service, Repository 같은 Spring Bean 메서드 실행 전후에 동작합니다.
  • Filter에서 다음 단계로 보내려면 chain.doFilter(request, response)를 호출해야 합니다.
  • Interceptor의 preHandle()은 true면 진행, false면 중단입니다.
  • AOP의 @Around에서는 joinPoint.proceed()가 실제 메서드 실행 코드입니다.
  • JWT 인증은 보통 Interceptor보다 Spring Security Filter에서 처리합니다.
  • Controller 처리 시간은 Interceptor, Service 처리 시간은 AOP에서 측정하기 좋습니다.
  • Spring Boot 3에서는 javax.servlet가 아니라 jakarta.servlet를 사용합니다.
  • Filter, Interceptor, AOP는 공통 로직을 분리해 핵심 비즈니스 로직을 깨끗하게 유지하기 위한 기술입니다.

헷갈리기 쉬운 개념

Filter vs Interceptor

Filter는 Spring MVC 바깥쪽,
Interceptor는 Spring MVC 안쪽 Controller 앞입니다.

Interceptor vs AOP

Interceptor는 HTTP 요청과 Controller 중심,
AOP는 Bean 메서드 실행 중심입니다.

chain.doFilter vs return true

Filter는 chain.doFilter가 진행 코드이고,
Interceptor는 return true가 진행 코드입니다.

return false vs 401/403

return false는 Interceptor에서 요청을 멈추는 방식이고,
401/403은 보안 필터나 권한 체크에서 응답할 수 있습니다.

@Around vs @Before

@Before는 실행 전만 처리하고,
@Around는 실행 전후를 모두 제어합니다.

javax vs jakarta

Spring Boot 3에서는 Servlet 관련 import가
jakarta.servlet 패키지로 바뀌었습니다.
