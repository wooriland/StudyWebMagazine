Spring Boot - Chapter 10

Validation과 Exception

Validation은 React가 보낸 폼 데이터가 Service와 DB까지 들어가기 전에 검사하는 서버의 1차 방어선입니다.
Exception Handling은 검증 실패, 비즈니스 오류, DB 오류를 React가 이해할 수 있는 JSON 응답으로 표준화하는 구조입니다.
이 단원에서는 @Service, @Repository, @Configuration, @Valid, @RestControllerAdvice를 연결해서
폼 검증 에러를 React에 JSON으로 내려주는 실무 흐름을 정리합니다.

DI @Service @Repository @Configuration @Bean @Valid @NotBlank @Email DTO Exception @ExceptionHandler @RestControllerAdvice MethodArgumentNotValidException ResponseEntity React Error JSON
학습 목표 1

DI와 주요 Annotation을 통해 Controller, Service,
Repository가 어떻게 연결되는지 이해합니다.

학습 목표 2

@Valid와 DTO 검증 어노테이션으로
잘못된 입력값을 Service 이전에 차단합니다.

학습 목표 3

@RestControllerAdvice로 Validation과
Business Exception을 JSON 오류 응답으로 통일합니다.

01

Validation과 Exception을 공부한다는 것

Controller는 React 요청을 받는 입구입니다. 하지만 요청을 받는 것만으로는 충분하지 않습니다.
사용자가 제목을 비워 보내거나, 이메일 형식이 틀리거나, 존재하지 않는 게시글을 수정하려고 할 수 있습니다.
이런 요청을 그대로 Service와 DB에 넘기면 데이터가 오염되고 React는 어떤 오류가 났는지 알기 어렵습니다.
그래서 Spring Boot API 서버는 입력값 검증과 예외 응답 표준화가 필요합니다.
입력값 형식은 DTO에서 Validation으로 막고, 업무 규칙 위반이나 DB 오류는 Exception으로 표현한 뒤, @RestControllerAdvice에서 JSON 응답으로 통일합니다.

DI

객체 자동 연결

Controller, Service, Repository를
Spring Container가
Bean으로 만들고 생성자로 주입합니다.

Validation

입력값 검증

@Valid와 DTO 검증 어노테이션으로
잘못된 요청이 Service까지
들어가지 못하게 합니다.

Exception

오류 상황 표현

게시글 없음, 중복 아이디,
DB 제약 조건 위반 같은 문제를
예외로 표현합니다.

Advice

JSON 오류 응답

@RestControllerAdvice에서
예외를 공통 JSON 구조로 변환해
React에 전달합니다.

핵심 문장

Validation은 입력값을 막는 문이고, Exception Handling은 문제가 생겼을 때 React가 이해할 수 있는 JSON 안내문을 만드는 구조입니다.

04

Validation Annotation 카드

문자열 필수 입력에는 보통 @NotBlank를 사용합니다. @NotNull은 null만 막고 공백 문자열은 막지 못하므로, 폼 입력값 검증에서는 차이를 정확히 알아야 합니다.

@NotNull

null 불가. 빈 문자열은 통과할 수 있습니다.

@NotEmpty

null과 빈 문자열 불가.
공백 문자열은 통과할 수 있습니다.

@NotBlank

null, 빈 문자열, 공백 문자열 모두 불가.
제목, 이름, 내용 필수값에 자주 사용합니다.

@Size

문자열, 배열,
컬렉션의 최소/최대 크기를 제한합니다.

@Pattern

정규 표현식으로
전화번호, 아이디 형식 등을 검증합니다.

@Email

이메일 형식을 검증합니다.

@Min / @Max

숫자의 최소값과 최대값을 검증합니다.

@Positive

수량, 금액처럼 양수여야 하는 값을 검증합니다.

import java.util.Map;

public record ApiResponse<T>(
        boolean ok,
        String message,
        T data,
        Map<String, String> errors,
        LocalDateTime timestamp

    public static <T> ApiResponse<T> success(T data) {

                null,

                data,
                null,

        );
    }

    public static ApiResponse<Void> fail(String message) {
        return new ApiResponse<>(
                false,
                message,
                null,
                null,
                LocalDateTime.now()
        );
    }

    public static ApiResponse<Void> fail(String message, Map<String, String> errors) {
        return new ApiResponse<>(
                false,
                message,
                null,
                errors,
                LocalDateTime.now()
        );
    }
}

3) Controller

import jakarta.validation.Valid;
import lombok.RequiredArgsConstructor;
import org.springframework.http.HttpStatus;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.*;

@RestController
@RequestMapping("/api/posts")
@RequiredArgsConstructor
public class PostController {

    private final PostService postService;

    @PostMapping
    public ResponseEntity<ApiResponse<PostResponse>> create(
            @Valid @RequestBody PostCreateRequest request
    ) {
        PostResponse response = postService.create(request);

        return ResponseEntity
                .status(HttpStatus.CREATED)
                .body(ApiResponse.success("게시글이 등록되었습니다.", response));
    }
}

4) Service

import lombok.RequiredArgsConstructor;
import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;

@Service
@RequiredArgsConstructor
public class PostService {

    private final PostRepository postRepository;

    @Transactional
    public PostResponse create(PostCreateRequest request) {
        Post post = new Post(
                request.title(),
                request.content()
        );

        Post savedPost = postRepository.save(post);

        return PostResponse.from(savedPost);
    }
}

GlobalExceptionHandler 전체 구조

전역 예외 처리는 REST API에서 매우 중요합니다. 검증 실패, JSON 파싱 실패, 타입 변환 실패, 자원 없음, 중복 데이터, DB 제약 조건 위반을 각각 적절한 HTTP 상태 코드와 JSON 구조로 내려줍니다.

@RestControllerAdvice

import jakarta.validation.ConstraintViolationException;
import org.springframework.dao.DataIntegrityViolationException;
import org.springframework.http.HttpStatus;
import org.springframework.http.ResponseEntity;
import org.springframework.http.converter.HttpMessageNotReadableException;
import org.springframework.validation.BindException;
import org.springframework.web.bind.MissingServletRequestParameterException;
import org.springframework.web.bind.MethodArgumentNotValidException;
import org.springframework.web.bind.annotation.ExceptionHandler;
import org.springframework.web.bind.annotation.RestControllerAdvice;
import org.springframework.web.method.annotation.MethodArgumentTypeMismatchException;

import java.util.LinkedHashMap;
import java.util.Map;

@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<ApiResponse<Void>> handleValidation(
            MethodArgumentNotValidException e
    ) {
        Map<String, String> errors = new LinkedHashMap<>();

        e.getBindingResult().getFieldErrors()
                .forEach(error ->
                        errors.put(error.getField(), error.getDefaultMessage())
                );

        return ResponseEntity
                .status(HttpStatus.BAD_REQUEST)
                .body(ApiResponse.fail("입력값이 올바르지 않습니다.", errors));
    }

    @ExceptionHandler(HttpMessageNotReadableException.class)
    public ResponseEntity<ApiResponse<Void>> handleJsonError(
            HttpMessageNotReadableException e
    ) {
        return ResponseEntity
                .status(HttpStatus.BAD_REQUEST)
                .body(ApiResponse.fail("JSON 요청 형식이 올바르지 않습니다."));
    }

    @ExceptionHandler(MethodArgumentTypeMismatchException.class)
    public ResponseEntity<ApiResponse<Void>> handleTypeMismatch(
            MethodArgumentTypeMismatchException e
    ) {
        return ResponseEntity
                .status(HttpStatus.BAD_REQUEST)
                .body(ApiResponse.fail("요청 값의 타입이 올바르지 않습니다: " + e.getName()));
    }

    @ExceptionHandler(PostNotFoundException.class)
    public ResponseEntity<ApiResponse<Void>> handlePostNotFound(
            PostNotFoundException e
    ) {
        return ResponseEntity
                .status(HttpStatus.NOT_FOUND)
                .body(ApiResponse.fail(e.getMessage()));
    }

    @ExceptionHandler(DataIntegrityViolationException.class)
    public ResponseEntity<ApiResponse<Void>> handleDataIntegrity(
            DataIntegrityViolationException e
    ) {
        return ResponseEntity
                .status(HttpStatus.CONFLICT)
                .body(ApiResponse.fail("데이터 제약 조건을 위반했습니다."));
    }

    @ExceptionHandler(Exception.class)
    public ResponseEntity<ApiResponse<Void>> handleException(
            Exception e
    ) {
        return ResponseEntity
                .status(HttpStatus.INTERNAL_SERVER_ERROR)
                .body(ApiResponse.fail("서버 내부 오류가 발생했습니다."));
    }
}

1) 토큰 저장 함수

export function saveToken(token, keepLogin) {
  if (keepLogin) {
    localStorage.setItem("accessToken", token);
    sessionStorage.removeItem("accessToken");
  } else {
    sessionStorage.setItem("accessToken", token);
    localStorage.removeItem("accessToken");
  }
}

export function getAccessToken() {
  return (
    localStorage.getItem("accessToken") ||
    sessionStorage.getItem("accessToken")
  );
}

export function clearToken() {
  localStorage.removeItem("accessToken");
  sessionStorage.removeItem("accessToken");
}

2) Axios Interceptor

import axios from "axios";
import { getAccessToken, clearToken } from "../utils/tokenStorage";

const apiClient = axios.create({
  baseURL: "http://localhost:8080",
  headers: {
    "Content-Type": "application/json",
  },
  timeout: 5000,
});

apiClient.interceptors.request.use((config) => {
  const token = getAccessToken();

  if (token) {
    config.headers.Authorization = `Bearer ${token}`;
  }

  return config;
});

apiClient.interceptors.response.use(
  (response) => response,
  (error) => {
    if (error.response?.status === 401) {
      clearToken();
      window.location.href = "/login";
    }

    return Promise.reject(error);
  }
);

export default apiClient;

3) 폼 입력값을 DTO로 보내고 오류 받기

import { useState } from "react";
import apiClient from "../api/apiClient";

function PostCreateForm() {
  const [form, setForm] = useState({
    title: "",
    content: "",
  });

  const [fieldErrors, setFieldErrors] = useState({});
  const [message, setMessage] = useState("");

  function handleChange(event) {
    const { name, value } = event.target;

    setForm((prev) => ({
      ...prev,
      [name]: value,
    }));

    setFieldErrors((prev) => ({
      ...prev,
      [name]: "",
    }));
  }

  async function handleSubmit(event) {
    event.preventDefault();

    setMessage("");
    setFieldErrors({});

    try {
      const response = await apiClient.post("/api/posts", form);
      const result = response.data;

      if (result.ok) {
        setMessage(result.message);
        setForm({
          title: "",
          content: "",
        });
      }
    } catch (error) {
      const response = error.response?.data;

      if (response?.errors) {
        setFieldErrors(response.errors);
      }

      setMessage(response?.message || "게시글 등록에 실패했습니다.");
    }
  }

  return (
    <form onSubmit={handleSubmit}>
      <input
        name="title"
        value={form.title}
        onChange={handleChange}
        placeholder="제목"
      />
      {fieldErrors.title && <p>{fieldErrors.title}</p>}

      <textarea
        name="content"
        value={form.content}
        onChange={handleChange}
        placeholder="내용"
      />
      {fieldErrors.content && <p>{fieldErrors.content}</p>}

      <button type="submit">등록</button>

      {message && <p>{message}</p>}
    </form>
  );
}

09

자주 하는 실수

1. @Valid 누락

DTO에 @NotBlank를 붙여도
Controller 파라미터에 @Valid가 없으면
검증이 실행되지 않습니다.

2. @RequestBody 누락

React가 JSON을 보내는 경우
@RequestBody가 없으면
DTO에 값이 제대로 들어오지 않습니다.

3. Entity에 Validation 집중

API 검증 규칙은 Request DTO에 두고
Entity는 DB 매핑에 집중시키는 것이 좋습니다.

4. 모든 예외를 Exception 하나로 처리

Validation, NotFound, Duplicate,
DB 오류는 각각 다른 상태 코드로 처리해야
React가 원인을 구분합니다.

5. @NotNull과 @NotBlank 혼동

문자열 필수 입력은 보통
@NotBlank를 사용해야
공백 문자열까지 막을 수 있습니다.

6. JSON key와 DTO 필드명 불일치

React의 form name과 DTO 필드명이 다르면
값이 null로 들어올 수 있습니다.

7. 예외 메시지 그대로 노출

내부 DB 오류나 stack trace를
그대로 React에 내려주면 보안상 위험합니다.

8. Security 예외를
ControllerAdvice만으로 처리

JWT 인증 실패는
Controller 전에 Filter에서 발생할 수 있어
Security 예외 처리를 별도로 고려해야 합니다.

11

면접 포인트

Q1. DI란 무엇인가요?

객체가 필요한 의존 객체를 직접 생성하지 않고 Spring Container가 생성한 Bean을 외부에서 주입받는 방식입니다.

Q2. @Valid는 어떤 역할을 하나요?

DTO에 선언된 Validation 어노테이션을 Controller 요청 처리 시점에 실행하는 트리거입니다.

Q3. @NotNull, @NotEmpty, @NotBlank의 차이는 무엇인가요?

@NotNull은 null만 막고, @NotEmpty는 null과 빈 문자열을 막고, @NotBlank는 공백 문자열까지 막습니다.

Q4. Validation 실패 시 어떤 예외가 발생하나요?

@Valid @RequestBody DTO 검증 실패 시 보통 MethodArgumentNotValidException이 발생합니다.

Q5. @ExceptionHandler와 @RestControllerAdvice의 차이는 무엇인가요?

@ExceptionHandler는 특정 예외 처리 메서드이고, @RestControllerAdvice는 여러 Controller 예외를 전역 JSON 응답으로 처리합니다.

Q6. DTO와 Entity를 분리하는 이유는 무엇인가요?

DB 구조 노출을 막고, 필요한 필드만 주고받고, API 검증 규칙과 DB 매핑 규칙을 분리하기 위해서입니다.

Q7. ResponseEntity를 쓰는 이유는 무엇인가요?

응답 body뿐 아니라 HTTP status와 header까지 명확하게 제어하기 위해 사용합니다.

Q8. React에서 Validation 오류를 어떻게 처리하나요?

error.response.data.errors를 fieldErrors 상태에 저장하고 각 input 아래에 메시지를 표시합니다.

학습 메모

오늘의 암기 포인트

  • DI는 Spring Container가 Bean을 만들고 필요한 곳에 주입하는 구조입니다.
  • @Service는 비즈니스 로직, @Repository는 DB 접근, @Configuration은 설정 Bean 등록 역할입니다.
  • Validation은 잘못된 요청이 Service와 DB까지 들어가지 못하게 막는 1차 방어선입니다.
  • @Valid는 DTO 검증을 실행하는 트리거입니다.
  • @NotBlank는 null, 빈 문자열, 공백 문자열을 모두 막습니다.
  • Validation 실패 시 MethodArgumentNotValidException이 발생할 수 있습니다.
  • @RestControllerAdvice는 예외를 공통 JSON 응답으로 바꾸는 통제센터입니다.
  • React는 error.response.data.errors를 사용해 필드별 오류를 표시할 수 있습니다.

헷갈리기 쉬운 개념

@Valid vs 검증 어노테이션

@NotBlank는 조건이고,
@Valid는 그 조건을 실행시키는
트리거입니다.

@NotNull vs @NotBlank

문자열 필수 입력은
공백까지 막는
@NotBlank가 더 적합합니다.

DTO vs Entity

DTO는 API 요청/응답 구조이고,
Entity는
DB 테이블 매핑 구조입니다.

@ExceptionHandler vs
@RestControllerAdvice

ExceptionHandler는
처리 메서드이고,
RestControllerAdvice는
전역 JSON 예외 처리 클래스입니다.

Validation Exception vs
Business Exception

Validation은 형식 오류,
Business Exception은
업무 규칙 위반입니다.
