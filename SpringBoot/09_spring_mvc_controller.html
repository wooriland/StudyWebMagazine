Spring MVC와 Controller

Controller는 React의 HTTP 요청을 Java 코드로 연결하는 Spring MVC의 입구입니다.
요청 URL과 HTTP Method를 분석하고, PathVariable, RequestParam, RequestBody로 요청 데이터를 받아
DTO로 변환한 뒤, Service를 호출하고 JSON 응답을 반환합니다.
React 입장에서는 이 Controller 메서드 하나하나가 실제로 호출하는 REST API 엔드포인트입니다.

01

Controller는 HTTP 세계와 Java 세계를 연결하는 입구

Spring MVC에서 Controller는 사용자의 요청을 받아 분석하고, 필요한 비즈니스 로직을 호출한 뒤, 결과를 View 또는 JSON 응답으로 돌려주는 역할을 합니다.
React와 함께 사용하는 REST API 서버에서는 대부분 @RestController를 사용하여 HTML이 아니라 JSON 데이터를 반환합니다.
Controller는 비즈니스 로직을 모두 처리하는 장소가 아닙니다.
Controller는 URL, HTTP Method, 요청 데이터, 응답 상태 코드를 다루는 HTTP 경계이고, 실제 업무 규칙은 Service가 담당하며, DB 접근은 Repository가 담당합니다.

데이터 접근

@Controller와 @RestController

@Controller는 주로 Thymeleaf 같은 서버 템플릿 화면을 반환할 때 사용합니다.
반면 @RestController는 React나 모바일 앱에 JSON 데이터를 반환할 때 사용합니다. @RestController는 @Controller + @ResponseBody처럼 이해할 수 있습니다.

View Controller

@Controller

문자열 반환값을 View 이름으로 해석합니다.

@Controller
public class PageController {

    @GetMapping("/posts")
    public String posts(Model model) {
        model.addAttribute("posts", postService.findAll());
        return "posts/list";
    }
}

반환 객체를 JSON 응답 body로 변환합니다.

@RestController
@RequestMapping("/api/posts")
public class PostController {

    @GetMapping("/{id}")
    public PostResponse findById(@PathVariable Long id) {
        return postService.findById(id);
    }

결과: PostResponse 객체를 JSON으로 응답

public record ApiResponse<T>(
        boolean ok,
        String message,
        T data
) {
    public static <T> ApiResponse<T> success(T data) {
        return new ApiResponse<>(true, "요청이 성공했습니다.", data);
    }

    public static <T> ApiResponse<T> success(String message, T data) {
        return new ApiResponse<>(true, message, data);
    }

    public static <T> ApiResponse<T> fail(String message) {
        return new ApiResponse<>(false, message, null);
    }
}

3) Controller

@RestController
@RequestMapping("/api/posts")
@RequiredArgsConstructor
public class PostController {

    private final PostService postService;

    @GetMapping
    public ResponseEntity<ApiResponse<List<PostResponse>>> findAll(
            @RequestParam(required = false) String keyword
    ) {
        List<PostResponse> response = postService.findAll(keyword);

        return ResponseEntity.ok(
                ApiResponse.success(response)
        );
    }

    @GetMapping("/{id}")
    public ResponseEntity<ApiResponse<PostResponse>> findById(
            @PathVariable Long id
    ) {
        PostResponse response = postService.findById(id);

        return ResponseEntity.ok(
                ApiResponse.success(response)
        );
    }

    @PostMapping
    public ResponseEntity<ApiResponse<PostResponse>> create(
            @RequestBody PostCreateRequest request
    ) {
        PostResponse response = postService.create(request);

        return ResponseEntity
                .status(HttpStatus.CREATED)
                .body(ApiResponse.success("게시글이 등록되었습니다.", response));
    }

    @PutMapping("/{id}")
    public ResponseEntity<ApiResponse<PostResponse>> update(
            @PathVariable Long id,
            @RequestBody PostUpdateRequest request
    ) {
        PostResponse response = postService.update(id, request);

        return ResponseEntity.ok(
                ApiResponse.success("게시글이 수정되었습니다.", response)
        );
    }

    @DeleteMapping("/{id}")
    public ResponseEntity<Void> delete(
            @PathVariable Long id
    ) {
        postService.delete(id);

        return ResponseEntity.noContent().build();
    }
}

Service, Repository, Entity 구조

Controller는 요청을 받은 뒤 직접 DB에 접근하지 않고 Service를 호출합니다. Service는 비즈니스 로직과 트랜잭션을 담당하고, Repository는 DB 접근을 담당합니다.

1) Entity

@Entity
public class Post {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String title;

    @Column(length = 3000)
    private String content;

    protected Post() {
    }

    public Post(String title, String content) {
        this.title = title;
        this.content = content;
    }

    public void update(String title, String content) {
        this.title = title;
        this.content = content;
    }

    public Long getId() {
        return id;
    }

    public String getTitle() {
        return title;
    }

    public String getContent() {
        return content;
    }
}

2) Repository

public interface PostRepository extends JpaRepository<Post, Long> {

    List<Post> findByTitleContaining(String keyword);
}

3) Service

@Service
@RequiredArgsConstructor
public class PostService {

    private final PostRepository postRepository;

    @Transactional(readOnly = true)
    public List<PostResponse> findAll(String keyword) {
        List<Post> posts;

        if (keyword == null || keyword.isBlank()) {
            posts = postRepository.findAll();
        } else {
            posts = postRepository.findByTitleContaining(keyword);
        }

        return posts.stream()
                .map(PostResponse::from)
                .toList();
    }

    @Transactional(readOnly = true)
    public PostResponse findById(Long id) {
        Post post = postRepository.findById(id)
                .orElseThrow(() -> new IllegalArgumentException("게시글이 없습니다. id=" + id));

        return PostResponse.from(post);
    }

    @Transactional
    public PostResponse create(PostCreateRequest request) {
        Post post = new Post(
                request.title(),
                request.content()
        );

        Post savedPost = postRepository.save(post);

        return PostResponse.from(savedPost);
    }

    @Transactional
    public PostResponse update(Long id, PostUpdateRequest request) {
        Post post = postRepository.findById(id)
                .orElseThrow(() -> new IllegalArgumentException("게시글이 없습니다. id=" + id));

        post.update(request.title(), request.content());

        return PostResponse.from(post);
    }

    @Transactional
    public void delete(Long id) {
        if (!postRepository.existsById(id)) {
            throw new IllegalArgumentException("게시글이 없습니다. id=" + id);
        }

        postRepository.deleteById(id);
    }
}

React에서 Axios로 API 호출하기

React는 Java Service나 Repository를 직접 호출하지 않습니다.
React는 Axios로 HTTP 요청을 보내고, Spring Boot Controller가 그 요청을 받아 처리합니다.
인증이 필요한 API라면 Axios Interceptor에서 JWT 토큰을 Authorization 헤더에 자동으로 붙일 수 있습니다.

1) 토큰 저장 함수

export function saveToken(token, keepLogin) {
  if (keepLogin) {
    localStorage.setItem("accessToken", token);
    sessionStorage.removeItem("accessToken");
  } else {
    sessionStorage.setItem("accessToken", token);
    localStorage.removeItem("accessToken");
  }
}

export function getAccessToken() {
  return (
    localStorage.getItem("accessToken") ||
    sessionStorage.getItem("accessToken")
  );
}

export function clearToken() {
  localStorage.removeItem("accessToken");
  sessionStorage.removeItem("accessToken");
}

2) Axios 공통 인스턴스

import axios from "axios";
import { getAccessToken, clearToken } from "../utils/tokenStorage";

const apiClient = axios.create({
  baseURL: "http://localhost:8080",
  headers: {
    "Content-Type": "application/json",
  },
  timeout: 5000,
});

apiClient.interceptors.request.use(
  (config) => {
    const token = getAccessToken();

    if (token) {
      config.headers.Authorization = `Bearer ${token}`;
    }

    return config;
  },
  (error) => Promise.reject(error)
);

apiClient.interceptors.response.use(
  (response) => response,
  (error) => {
    if (error.response?.status === 401) {
      clearToken();
      window.location.href = "/login";
    }

    return Promise.reject(error);
  }
);

export default apiClient;

폼 입력값을 DTO로 보내기

React 폼 상태 객체는 Axios 요청 body로 전송되고, Spring Boot에서는 @RequestBody가 JSON을 DTO로 변환합니다.
이때 React의 JSON key와 Java DTO 필드명이 일치해야 자연스럽게 매핑됩니다.

React 폼 컴포넌트

import { useState } from "react";
import { createPost } from "../api/postApi";

function PostCreateForm() {
  const [form, setForm] = useState({
    title: "",
    content: "",
  });

  const [message, setMessage] = useState("");

  function handleChange(event) {
    const { name, value } = event.target;

    setForm((prev) => ({
      ...prev,
      [name]: value,
    }));
  }

  async function handleSubmit(event) {
    event.preventDefault();

    try {
      const result = await createPost(form);

      if (result.ok) {
        setMessage("게시글이 등록되었습니다.");
        setForm({
          title: "",
          content: "",
        });
      } else {
        setMessage(result.message);
      }
    } catch (error) {
      setMessage("게시글 등록 중 오류가 발생했습니다.");
    }
  }

  return (
    <form onSubmit={handleSubmit}>
      <input
        name="title"
        value={form.title}
        onChange={handleChange}
        placeholder="제목"
      />

      <textarea
        name="content"
        value={form.content}
        onChange={handleChange}
        placeholder="내용"
      />

      <button type="submit">등록</button>

      {message && <p>{message}</p>}
    </form>
  );
}

export default PostCreateForm;

예외 처리 기본 구조

Controller에서 발생하는 예외는 각 Controller 안에서 @ExceptionHandler로 지역 처리할 수도 있고, @RestControllerAdvice로 전역 처리할 수도 있습니다.
REST API에서는 전역 예외 처리를 통해 공통 JSON 오류 응답을 만드는 것이 좋습니다.

지역 예외 처리

@RestController
@RequestMapping("/api/sample")
public class SampleController {

    @GetMapping("/error")
    public String error() {
        throw new IllegalArgumentException("잘못된 요청입니다.");
    }

    @ExceptionHandler(IllegalArgumentException.class)
    public ResponseEntity<ApiResponse<Void>> handleIllegalArgument(
            IllegalArgumentException e
    ) {
        return ResponseEntity.badRequest()
                .body(ApiResponse.fail(e.getMessage()));
    }
}

전역 예외 처리

@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(IllegalArgumentException.class)
    public ResponseEntity<ApiResponse<Void>> handleIllegalArgument(
            IllegalArgumentException e
    ) {
        return ResponseEntity.badRequest()
                .body(ApiResponse.fail(e.getMessage()));
    }
}

오류 메시지 읽는 법

Spring MVC Controller 오류는 대부분 URL 매핑, HTTP Method, 요청 파라미터, JSON 변환, Content-Type, View 반환, 인증/인가, CORS에서 발생합니다.

오류 메시지
의미
확인할 것
404 Not Found
요청 URL에 매핑된 Controller 메서드가 없습니다.
@RequestMapping, @GetMapping 경로와 React 요청 URL을 비교합니다.
405 Method Not Allowed
URL은 맞지만 HTTP Method가 다릅니다.
GET, POST, PUT, PATCH, DELETE 요청 방식과 Mapping을 확인합니다.

MissingServletRequestParameterException

자주 하는 실수

1. API 서버인데 @Controller 사용

React API는 JSON 응답이 필요하므로
대부분 @RestController를 사용합니다.

2. @RequestBody 누락

React가 JSON body를 보내면
Controller 파라미터에
@RequestBody가 필요합니다.

3. JSON key와 DTO 필드명 불일치

React의 title, content와
Java DTO의 title, content
이름이 맞아야 합니다.

4. @PathVariable 이름 불일치

/{postId}를 쓰면
@PathVariable("postId")처럼
이름을 명시하는 것이 안전합니다.

5. 필수 @RequestParam 누락

검색어가 선택값이면
required=false 또는 defaultValue를
사용합니다.

6. GET 요청에 JSON body 사용

간단한 검색 조건은
GET + RequestParam으로
보내는 것이 자연스럽습니다.

7. Controller에서
Repository 직접 호출

Controller는 Service를 호출하고,
DB 접근은 Repository가 담당하게 분리합니다.

8. Entity를 그대로 요청/응답에 사용

DB 구조 노출과
API 변경 위험이 있으므로 DTO를 사용합니다.

9. DELETE 후
response.data를 기대

204 No Content 응답은 body가 없으므로
React에서 data.ok를 기대하면 안 됩니다.

10. Authorization 헤더 수동 반복

Axios Interceptor를 사용하면
인증 헤더 누락을 줄일 수 있습니다.

11. filterChain.doFilter 누락

JWT Filter에서 다음 필터로 넘기지 않으면
Controller까지 요청이 도달하지 않습니다.

12. HTTP status를
모두 200으로 처리

등록은 201, 삭제는 204,
요청 오류는 400처럼
의미 있는 상태 코드를 사용합니다.

면접 포인트

Q1. Spring MVC에서 Controller의 역할은 무엇인가요?

Controller는 HTTP 요청을 받아 요청 데이터를 Java 타입이나 DTO로 변환하고, Service를 호출한 뒤 View 또는 JSON 응답을 반환하는 Spring MVC의 입구입니다.

Q2. @Controller와 @RestController의 차이는 무엇인가요?

@Controller는 View 이름을 반환해 Thymeleaf 같은 템플릿을 렌더링할 때 사용하고, @RestController는 반환 객체를 response body에 직접 넣어 JSON으로 응답할 때 사용합니다.

Q3. @RestController는 내부적으로 어떻게 이해할 수 있나요?

@RestController는 @Controller와 @ResponseBody를 합친 것처럼 이해할 수 있습니다. 모든 메서드 반환값이 View 이름이 아니라 응답 body 데이터로 처리됩니다.

Q4. @PathVariable, @RequestParam, @RequestBody의 차이는 무엇인가요?

@PathVariable은 URL 경로 값, @RequestParam은 query string 값, @RequestBody는 JSON body 값을 Java DTO로 받을 때 사용합니다.

Q5. DispatcherServlet은 무엇인가요?

DispatcherServlet은 모든 HTTP 요청을 앞단에서 받아 적절한 Controller로 배분하는 Spring MVC의 핵심 서블릿입니다. Servlet Container와 Spring Container 사이의 Bridge 역할도 합니다.

Q6. Filter, Interceptor, Controller의 실행 순서는 어떻게 되나요?

요청은 Servlet Container, Filter Chain, DispatcherServlet, Interceptor, Controller 순서로 이동합니다. 응답은 역순으로 돌아갑니다.

Q7. ResponseEntity는 왜 사용하나요?

응답 body뿐 아니라 HTTP status와 header까지 제어할 수 있기 때문입니다. 예를 들어 등록 성공은 201 Created, 삭제 성공은 204 No Content로 표현할 수 있습니다.

Q8. Controller에서 Entity를 직접 반환하지 않는 이유는 무엇인가요?

Entity를 직접 반환하면 DB 구조가 API에 노출되고, 불필요한 필드나 순환 참조 문제가 생길 수 있습니다. 그래서 요청 DTO와 응답 DTO를 분리해 사용하는 것이 좋습니다.

Q9. JWT 인증 요청은 Controller에 오기 전에 어떤 과정을 거치나요?

React가 Authorization Bearer 토큰을 보내면 JWT Filter가 먼저 토큰을 추출하고 검증합니다. 유효하면 Authentication을 SecurityContext에 저장한 뒤 Controller로 요청을 전달합니다.

Q10. REST API URL 설계 시 주의할 점은 무엇인가요?

URL에는 행동보다 자원을 표현하고, 행동은 HTTP Method로 표현하는 것이 좋습니다. 예를 들어 게시글 등록은 /api/createPost가 아니라 POST /api/posts로 설계합니다.

학습 메모

오늘의 암기 포인트

  • Controller는 HTTP 요청을 Java 코드로 연결하는 Spring MVC의 입구입니다.
  • @Controller는 View를 반환하고, @RestController는 JSON을 반환합니다.
  • @RestController는 @Controller + @ResponseBody처럼 이해할 수 있습니다.
  • @RequestMapping은 공통 URL을 지정합니다.
  • @PathVariable은 URL 경로 값, @RequestParam은 query string, @RequestBody는 JSON body를 받습니다.
  • ResponseEntity는 HTTP status, header, body를 함께 제어합니다.
  • React는 Java 메서드를 직접 호출하지 않고 Axios로 Controller의 REST API를 호출합니다.
  • JWT 인증 요청은 Controller 전에 Security Filter에서 먼저 처리됩니다.

헷갈리기 쉬운 개념

@Controller vs
@RestController

Controller는 View 이름,
RestController는
JSON 응답입니다.

@PathVariable vs
@RequestParam

PathVariable은 URL 경로 값,
RequestParam은
?key=value 값입니다.

@RequestBody vs
@ModelAttribute

JSON body는 RequestBody,
form/query 객체 바인딩은
ModelAttribute를 사용합니다.

Filter vs Controller

Filter는 Controller 전에 실행되고,
JWT 인증은 보통
Filter에서 처리됩니다.

Entity vs DTO

Entity는 DB 구조,
DTO는 API 요청/응답 구조입니다.
