01

JDBC 쿼리 실행은 Connection에서 시작된다

JDBC에서 SQL을 실행하려면 먼저 DB와 연결된 Connection이 필요합니다.
Connection은 단순한 연결 표시가 아니라, SQL 실행 객체인 Statement, PreparedStatement, CallableStatement를 만드는 출발점입니다.
이 객체들은 모두 “SQL을 DB로 보내는 도구”이지만, 어떤 SQL을 어떻게 보내느냐에 따라 역할이 달라집니다.
단순 SQL은 Statement, 파라미터가 있는 SQL은 PreparedStatement, DB 프로시저나 함수 호출은 CallableStatement를 사용합니다.

Statement

완성된 SQL 문자열을 그대로 실행합니다.
학습용 기본 구조를 이해하기 좋지만,
사용자 입력값이 들어가는 SQL에는 권장되지 않습니다.

PreparedStatement

SQL 틀을 미리 준비하고,
값이 들어갈 위치를 ?로 표시한 뒤
안전하게 값을 바인딩합니다.

CallableStatement

Oracle 프로시저나 함수처럼
DB 내부에 저장된 업무 로직을
Java에서 호출할 때 사용합니다.

핵심 문장

JDBC 쿼리 실행은 Connection → Statement 계열 객체 → execute 계열 메서드 → ResultSet 또는 영향받은 행 수 순서로 이해합니다.

04

executeQuery(), executeUpdate(), execute()

SQL을 실행할 때는 SQL의 목적에 맞는 execute 메서드를 사용해야 합니다. 조회는 executeQuery(), 입력·수정·삭제는 executeUpdate(), 결과가 복합적인 경우는 execute()를 사용합니다.

SELECT

executeQuery()

조회 결과를 ResultSet으로 반환합니다.

ResultSet rs = ps.executeQuery();
DML

executeUpdate()

INSERT, UPDATE, DELETE 실행 후
영향받은 행 수를 반환합니다.

int count = ps.executeUpdate();
ANY

execute()

SELECT, DML, 프로시저처럼
결과가 복합적인 경우 사용합니다.

boolean result = stmt.execute(sql);
판단 기준

데이터를 가져오는 SQL이면 executeQuery(), 데이터를 바꾸는 SQL이면 executeUpdate(), 결과 형태가 복합적이면 execute()로 구분합니다.

05

ResultSet을 그림처럼 이해하기

ResultSet은 SELECT 결과를 담는 객체입니다. 하지만 배열이나 리스트처럼 바로 값을 꺼내는 구조가 아니라, 커서를 한 행씩 이동시키면서 읽는 구조입니다.

초기 상태 커서는 첫 번째 행 바로 앞에 있음
id username email
커서 ↓ rs.next() 전
1 kim kim@example.com
2 lee lee@example.com
rs.next() 커서가 실제 행으로 이동한 뒤 값을 읽음
id username email
1 kim kim@example.com
2 lee lee@example.com

ResultSet 기본 읽기

while (rs.next()) {
    Long id = rs.getLong("id");
    String username = rs.getString("username");
    String email = rs.getString("email");
}
06

순수 Java JDBC 예제

아래 예제는 순수 JDBC로 사용자 목록을 조회하고, ResultSet에서 값을 꺼내 DTO 목록으로 변환하는 기본 흐름입니다.

1) SELECT → ResultSet → DTO List

import java.sql.*;
import java.util.ArrayList;
import java.util.List;

public class UserSelectExample {

    public static void main(String[] args) throws Exception {
        String url = "jdbc:oracle:thin:@localhost:1521/XEPDB1";
        String username = "JAVA";
        String password = "JAVA";

        String sql = """
            SELECT id, username, email
            FROM users
            ORDER BY id DESC
            """;

        Class.forName("oracle.jdbc.OracleDriver");

        try (
            Connection con = DriverManager.getConnection(url, username, password);
            PreparedStatement ps = con.prepareStatement(sql);
            ResultSet rs = ps.executeQuery()
        ) {
            List<UserResponse> users = new ArrayList<>();

            while (rs.next()) {
                users.add(new UserResponse(
                    rs.getLong("id"),
                    rs.getString("username"),
                    rs.getString("email")
                ));
            }

            for (UserResponse user : users) {
                System.out.println(user);
            }
        }
    }

    public record UserResponse(
        Long id,
        String username,
        String email
    ) {
    }
}

Spring Boot DataSource + JDBC Repository

Spring Boot에서는 보통 DriverManager를 직접 사용하지 않고, DataSource를 주입받아 Connection을 얻습니다.
이렇게 하면 HikariCP 커넥션 풀을 통해 DB 연결을 효율적으로 관리할 수 있습니다.

1) 응답 DTO

public record UserResponse(
    Long id,
    String username,
    String email
) {
}

2) JDBC Repository

import lombok.RequiredArgsConstructor;
import org.springframework.stereotype.Repository;

import javax.sql.DataSource;
import java.sql.*;
import java.util.ArrayList;
import java.util.List;

@Repository
@RequiredArgsConstructor
public class UserJdbcRepository {

    private final DataSource dataSource;

    public List<UserResponse> findAll() {
        String sql = """
            SELECT id, username, email
            FROM users
            ORDER BY id DESC
            """;

        List<UserResponse> users = new ArrayList<>();

        try (
            Connection con = dataSource.getConnection();
            PreparedStatement ps = con.prepareStatement(sql);
            ResultSet rs = ps.executeQuery()
        ) {
            while (rs.next()) {
                users.add(new UserResponse(
                    rs.getLong("id"),
                    rs.getString("username"),
                    rs.getString("email")
                ));
            }

            return users;

        } catch (SQLException e) {
            throw new RuntimeException("사용자 목록 조회 중 오류가 발생했습니다.", e);
        }
    }

    public List<UserResponse> searchByUsername(String keyword) {
        String sql = """
            SELECT id, username, email
            FROM users
            WHERE username LIKE '%' || ? || '%'
            ORDER BY id DESC
            """;

        List<UserResponse> users = new ArrayList<>();

        try (
            Connection con = dataSource.getConnection();
            PreparedStatement ps = con.prepareStatement(sql)
        ) {
            ps.setString(1, keyword);

            try (ResultSet rs = ps.executeQuery()) {
                while (rs.next()) {
                    users.add(new UserResponse(
                        rs.getLong("id"),
                        rs.getString("username"),
                        rs.getString("email")
                    ));
                }
            }

            return users;

        } catch (SQLException e) {
            throw new RuntimeException("사용자 검색 중 오류가 발생했습니다.", e);
        }
    }
}

3) Service

import lombok.RequiredArgsConstructor;
import org.springframework.stereotype.Service;

import java.util.List;

@Service
@RequiredArgsConstructor
public class UserService {

    private final UserJdbcRepository userJdbcRepository;

    public List<UserResponse> findAllUsers() {
        return userJdbcRepository.findAll();
    }

    public List<UserResponse> searchUsers(String keyword) {
        return userJdbcRepository.searchByUsername(keyword);
    }
}

4) Controller

import lombok.RequiredArgsConstructor;
import org.springframework.web.bind.annotation.*;

import java.util.List;

@RestController
@RequestMapping("/api/users")
@RequiredArgsConstructor
public class UserController {

import axios from "axios";
import { getAccessToken, clearToken } from "../utils/tokenStorage";

const apiClient = axios.create({
  baseURL: "http://localhost:8080",
  headers: {
    "Content-Type": "application/json",
  },
});

apiClient.interceptors.request.use(
  (config) => {
    const token = getAccessToken();

    if (token) {
      config.headers.Authorization = `Bearer ${token}`;
    }

    return config;
  },
  (error) => Promise.reject(error)
);

apiClient.interceptors.response.use(
  (response) => response,
  (error) => {
    if (error.response?.status === 401) {
      clearToken();
      window.location.href = "/login";
    }

    return Promise.reject(error);
  }
);

export default apiClient;

2) 토큰 저장 함수

export function saveToken(token, keepLogin) {
  if (keepLogin) {
    localStorage.setItem("accessToken", token);
    sessionStorage.removeItem("accessToken");
  } else {
    sessionStorage.setItem("accessToken", token);
    localStorage.removeItem("accessToken");
  }
}

export function getAccessToken() {
  return (
    localStorage.getItem("accessToken") ||
    sessionStorage.getItem("accessToken")
  );
}

export function clearToken() {
  localStorage.removeItem("accessToken");
  sessionStorage.removeItem("accessToken");
}

5) React 검색 폼 컴포넌트

import { useState } from "react";
import { searchUsers } from "../api/userApi";

function UserSearchForm() {
  const [form, setForm] = useState({
    keyword: "",
  });

  const [users, setUsers] = useState([]);
  const [message, setMessage] = useState("");

  function handleChange(event) {
    const { name, value } = event.target;

    setForm((prevForm) => ({
      ...prevForm,
      [name]: value,
    }));
  }

  async function handleSubmit(event) {
    event.preventDefault();

    try {
      const data = await searchUsers(form);
      setUsers(data);
      setMessage("");
    } catch (error) {
      setUsers([]);
      setMessage("검색 중 오류가 발생했습니다.");
    }
  }

  return (
    <section>
      <h2>사용자 검색</h2>

      <form onSubmit={handleSubmit}>
        <input
          type="text"
          name="keyword"
          value={form.keyword}
          onChange={handleChange}
          placeholder="아이디 검색"
        />

        <button type="submit">검색</button>
      </form>

      {message && <p>{message}</p>}

      <ul>
        {users.map((user) => (
          <li key={user.id}>
            {user.username} / {user.email}
          </li>
        ))}
      </ul>
    </section>
  );
}

export default UserSearchForm;

오류 메시지 읽는 법

JDBC 쿼리 실행 오류는 대부분 SQL 문법, 파라미터 바인딩, ResultSet 커서, 컬럼명, 자원 반납에서 발생합니다. 오류 메시지를 보면 어느 단계에서 막혔는지 먼저 찾아야 합니다.

오류 메시지
의미
확인할 것
SQLSyntaxErrorException
SQL 문법, 테이블명, 컬럼명 중 하나가 잘못되었습니다.
SQL Developer에서 쿼리를 직접 실행하고 테이블명과 컬럼명을 확인합니다.
ORA-00942
테이블 또는 뷰가 존재하지 않거나 권한이 없습니다.
접속 계정, 스키마명, 테이블명, SELECT 권한을 확인합니다.
ORA-00904
컬럼명 또는 식별자가 잘못되었습니다.
SELECT에 포함된 컬럼명과 rs.getXXX("컬럼명")이 일치하는지 확인합니다.
ORA-01008
SQL의 모든 ?에 값이 바인딩되지 않았습니다.
setXXX() 누락 여부와 파라미터 개수를 확인합니다.
Invalid column index
파라미터 인덱스가 잘못되었습니다.
PreparedStatement의 인덱스는 1부터 시작합니다.
Invalid column name
ResultSet에서 읽으려는 컬럼명이 결과에 없습니다.
SELECT 컬럼과 별칭을 확인합니다.
ResultSet closed
ResultSet이 닫힌 뒤 값을 읽으려고 했습니다.
try-with-resources 블록 안에서 DTO 변환까지 끝내야 합니다.
Invalid cursor state
커서가 실제 행을 가리키지 않는데 값을 읽으려고 했습니다.
rs.next() 호출 후 값을 읽어야 합니다.
executeQuery() 오류
INSERT, UPDATE, DELETE를 executeQuery로 실행했을 수 있습니다.
DML은 executeUpdate()를 사용합니다.
400 Bad Request
React JSON과 Spring DTO 구조가 맞지 않을 수 있습니다.
@RequestBody, JSON key, DTO 필드명을 확인합니다.
10

자주 하는 실수

1. Statement에
사용자 입력을 직접 붙임

SQL Injection 위험이 있으므로
사용자 입력값은
PreparedStatement의 ?에
바인딩해야 합니다.

2. 파라미터 인덱스를 0부터 사용함

JDBC의 setXXX() 인덱스는
0이 아니라 1부터 시작합니다.

3. setXXX 순서를 SQL과 다르게 넣음

?의 순서와
setXXX()의 인덱스 순서는
반드시 일치해야 합니다.

4. SELECT에 executeUpdate 사용

SELECT는 executeQuery(),
INSERT/UPDATE/DELETE는
executeUpdate()입니다.

5. rs.next() 없이 값을 꺼냄

ResultSet 커서는
처음에 첫 행 앞에 있으므로
반드시 rs.next() 후 값을 읽어야 합니다.

6. ResultSet을 블록 밖에서 사용함

Connection이나
PreparedStatement가 닫히면
ResultSet도 사용할 수 없습니다.
DTO 변환까지 블록 안에서 끝냅니다.

7. LIKE 검색에 '%?%'를 씀

'%?%'는 파라미터가 아닙니다.
Oracle에서는 '%' || ? || '%'를
사용합니다.

8. React JSON과 DTO 필드명이 다름

React의 JSON key와
Spring Boot DTO 필드명이 다르면
값이 제대로 들어가지 않을 수 있습니다.

면접 포인트

Q1. Statement와 PreparedStatement의 차이는 무엇인가요?

Statement는 완성된 SQL 문자열을 그대로 실행하는 객체입니다. PreparedStatement는 SQL을 미리 준비하고 값이 들어갈 부분을 ?로 둔 뒤 setXXX()로 값을 바인딩합니다.
SQL Injection 방지와 유지보수성 때문에 실무에서는 PreparedStatement를 주로 사용합니다.

Q2. CallableStatement는 언제 사용하나요?

DB에 저장된 프로시저나 함수를 Java에서 호출할 때 사용합니다. IN 파라미터는 setXXX()로 넣고, OUT 파라미터는 registerOutParameter()로 등록한 뒤 실행 후 getXXX()로 읽습니다.

Q3. executeQuery()와 executeUpdate()의 차이는 무엇인가요?

executeQuery()는 SELECT처럼 조회 결과가 필요한 SQL에 사용하고 ResultSet을 반환합니다.
executeUpdate()는 INSERT, UPDATE, DELETE처럼 데이터를 변경하는 SQL에 사용하고 영향받은 행 수를 int로 반환합니다.

Q4. ResultSet이란 무엇인가요?

ResultSet은 SELECT 결과를 담는 JDBC 객체입니다. 커서 기반으로 동작하며, 처음에는 첫 번째 행 앞에 위치합니다.
rs.next()로 한 행씩 이동하면서 getString(), getLong() 같은 메서드로 값을 읽습니다.

Q5. rs.next()를 호출해야 하는 이유는 무엇인가요?

ResultSet 생성 직후 커서는 실제 데이터 행을 가리키지 않습니다. 따라서 데이터를 읽기 전에 rs.next()를 호출해 첫 번째 행으로 이동해야 합니다.

Q6. JDBC 조회 결과를 API 응답으로 만드는 과정은 무엇인가요?

executeQuery()로 ResultSet을 받고, while(rs.next())로 한 행씩 읽습니다. 필요한 컬럼 값을 꺼내 DTO 객체를 만들고 List에 담습니다.
Controller가 이 List를 반환하면 Spring Boot가 JSON으로 변환해 React에 응답합니다.

Q7. SQL Injection을 방지하려면 어떻게 해야 하나요?

사용자 입력값을 SQL 문자열에 직접 붙이지 않고 PreparedStatement를 사용합니다. SQL의 값 부분을 ?로 두고 setXXX()로 바인딩하면 입력값이 SQL 구조가 아니라 값으로 처리됩니다.

Q8. React의 response.data는 JDBC와 어떻게 연결되나요?

React는 Axios로 API를 호출하고 JSON을 받습니다. 서버 내부에서는 JDBC가 ResultSet을 읽고 DTO로 변환한 뒤, Spring Boot가 그 DTO를 JSON으로 변환합니다.
그 결과가 React의 response.data가 됩니다.

학습 메모

오늘의 암기 포인트

  • Statement는 완성된 SQL 문자열을 직접 실행합니다.
  • PreparedStatement는 SQL의 값 부분을 ?로 두고 안전하게 바인딩합니다.
  • CallableStatement는 프로시저나 함수를 호출할 때 사용합니다.
  • executeQuery()는 SELECT용이고 ResultSet을 반환합니다.
  • executeUpdate()는 INSERT, UPDATE, DELETE용이고 영향받은 행 수를 반환합니다.
  • ResultSet은 커서 기반 객체이므로 반드시 rs.next() 후 값을 읽습니다.
  • React의 response.data는 서버가 ResultSet을 DTO로 바꾼 결과일 수 있습니다.

헷갈리기 쉬운 개념

Statement vs
PreparedStatement

Statement는 완성 SQL 실행,
PreparedStatement는
SQL 틀과 값을 분리합니다.

executeQuery vs
executeUpdate

조회는 executeQuery,
입력·수정·삭제는
executeUpdate입니다.

ResultSet vs List

ResultSet은 바로 리스트가 아니라
커서로 한 행씩 읽는 객체입니다.

컬럼명 vs 별칭

SQL에서 별칭을 주면
ResultSet에서도
별칭으로 읽어야 합니다.

DTO vs DB 컬럼

DTO는 화면/API 응답용이고,
DB 컬럼 전체를
그대로 노출하지 않습니다.
