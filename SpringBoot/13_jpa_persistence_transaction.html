JPA 영속성 컨텍스트와 트랜잭션

영속성 컨텍스트는 JPA가 Entity를 보관하고 추적하는 작업 공간입니다.
@Transactional은 그 작업 공간이 유지되는 범위를 만들고, 그 안에서 조회한 Entity는 Managed 상태가 됩니다.
그래서 수정 API에서 save()를 다시 호출하지 않아도, JPA가 Dirty Checking으로 변경을 감지하고 Flush 시점에 UPDATE SQL을 만들어 DB에 반영할 수 있습니다.

01

영속성 컨텍스트를 공부한다는 것

앞 단원에서는 @Entity와 JpaRepository를 이용해서 CRUD API를 만드는 방법을 공부했습니다.
이번 단원에서는 그 CRUD가 내부에서 어떻게 동작하는지 봅니다. 특히 수정 API에서 postRepository.save(post)를 다시 호출하지 않아도 DB UPDATE가 발생하는 이유가 핵심입니다.
JPA는 Repository가 호출될 때 바로 DB에 SQL만 던지는 단순한 도구가 아닙니다.
중간에 영속성 컨텍스트라는 작업 공간을 두고, Entity를 1차 캐시에 보관하고, 처음 상태의 스냅샷을 저장하고, 트랜잭션 커밋 시점에 변경된 부분을 감지합니다.

04

1차 캐시와 쓰기 지연 SQL 저장소

영속성 컨텍스트는 Entity를 1차 캐시에 보관합니다. 같은 트랜잭션 안에서 같은 Entity를 다시 조회하면 DB를 다시 조회하지 않고 캐시된 Entity를 반환할 수 있습니다.

1차 캐시

같은 Entity 재사용

첫 번째 조회에서는 DB SELECT가 발생하고,
두 번째 같은 id 조회에서는 1차 캐시의 Entity를 반환할 수 있습니다. 

Post post1 = repository
    .findById(1L)
    .orElseThrow();

Post post2 = repository
    .findById(1L)
    .orElseThrow();

post1 == post2 // true 가능
쓰기 지연

SQL을 잠시 모아둠

save()를 호출하면 Entity를 영속성 컨텍스트에 올리고,
INSERT SQL은 쓰기 지연 SQL 저장소에 쌓였다가 Flush 시점에 DB로 전송됩니다. 

Post post = new Post(...);

postRepository.save(post);

// INSERT SQL은
// 커밋 직전 Flush 시점에
// 나갈 수 있음
영속성 컨텍스트

1차 캐시와 쓰기 지연 SQL 저장소를 포함하는 JPA 작업 공간

1차 캐시

쓰기 지연 SQL 저장소

INSERT, UPDATE, DELETE SQL을 Flush 전까지 보관

05

Flush와 Commit의 차이

Flush와 Commit은 같은 개념이 아닙니다. Flush는 영속성 컨텍스트의 변경 내용을 SQL로 DB에 보내는 동작이고, Commit은 트랜잭션의 변경을 최종 확정하는 동작입니다.

동기화

Flush

영속성 컨텍스트의 변경 내용을 SQL로 DB에 전송합니다.

entityManager.flush();

Flush가 되었다고 해서 트랜잭션이 확정된 것은 아닙니다.

확정

Commit

트랜잭션의 변경을 최종 확정합니다.

// @Transactional 메서드 정상 종료
// → commit

Commit 전에 예외가 발생하면 Rollback될 수 있습니다.

중요한 구분

Flush는 SQL을 DB에 보내는 것이고, Commit은 그 변경을 확정하는 것입니다. Flush 후에도 트랜잭션이 Rollback되면 변경은 취소될 수 있습니다.

Post Entity

package com.example.studyapi.post.entity;

import jakarta.persistence.*;
import org.hibernate.annotations.CreationTimestamp;

import java.time.LocalDateTime;

@Entity
@Table(name = "posts")
public class Post {

    @Id
    @GeneratedValue(strategy = GenerationType.SEQUENCE, generator = "seq_posts")
    @SequenceGenerator(
            name = "seq_posts",
            sequenceName = "seq_posts",
            allocationSize = 1
    )
    private Long id;

    @Column(nullable = false, length = 100)
    private String title;

    @Lob
    @Column(nullable = false)
    private String content;

    @Column(nullable = false, length = 50)
    private String writer;

    @CreationTimestamp
    @Column(nullable = false, updatable = false)
    private LocalDateTime createdAt;

    protected Post() {
    }

    public Post(String title, String content, String writer) {
        this.title = title;
        this.content = content;
        this.writer = writer;
    }

    public void update(String title, String content) {
        if (title == null || title.isBlank()) {
            throw new IllegalArgumentException("제목은 필수입니다.");
        }

        if (content == null || content.isBlank()) {
            throw new IllegalArgumentException("내용은 필수입니다.");
        }

        this.title = title;
        this.content = content;
    }

    public Long getId() {
        return id;
    }

    public String getTitle() {
        return title;
    }

    public String getContent() {
        return content;
    }

    public String getWriter() {
        return writer;
    }

    public LocalDateTime getCreatedAt() {
        return createdAt;
    }
}

Service와 Controller 예제

수정 메서드에서 핵심은 @Transactional 안에서 기존 Entity를 조회하고,
Entity의 상태를 변경하는 것입니다. 이때 save()를 다시 호출하지 않아도 Dirty Checking으로 UPDATE SQL이 생성될 수 있습니다.

PostService

package com.example.studyapi.post.service;

import com.example.studyapi.post.dto.PostCreateRequest;
import com.example.studyapi.post.dto.PostResponse;
import com.example.studyapi.post.dto.PostUpdateRequest;
import com.example.studyapi.post.entity.Post;
import com.example.studyapi.post.exception.PostNotFoundException;
import com.example.studyapi.post.repository.PostRepository;
import lombok.RequiredArgsConstructor;
import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;

import java.util.List;

@Service
@RequiredArgsConstructor
public class PostService {

    private final PostRepository postRepository;

    @Transactional
    public PostResponse create(PostCreateRequest request) {
        Post post = new Post(
                request.title(),
                request.content(),
                request.writer()
        );

        Post savedPost = postRepository.save(post);

        return PostResponse.from(savedPost);
    }

    @Transactional(readOnly = true)
    public List<PostResponse> findAll() {
        return postRepository.findAll()
                .stream()
                .map(PostResponse::from)
                .toList();
    }

    @Transactional(readOnly = true)
    public PostResponse findById(Long id) {
        Post post = postRepository.findById(id)
                .orElseThrow(() -> new PostNotFoundException(id));

        return PostResponse.from(post);
    }

    @Transactional
    public PostResponse update(Long id, PostUpdateRequest request) {
        Post post = postRepository.findById(id)
                .orElseThrow(() -> new PostNotFoundException(id));

        post.update(request.title(), request.content());

        return PostResponse.from(post);
    }

    @Transactional
    public void delete(Long id) {
        Post post = postRepository.findById(id)
                .orElseThrow(() -> new PostNotFoundException(id));

        postRepository.delete(post);
    }

    @Transactional(readOnly = true)
    public void firstLevelCacheTest(Long id) {
        Post post1 = postRepository.findById(id)
                .orElseThrow(() -> new PostNotFoundException(id));

        Post post2 = postRepository.findById(id)
                .orElseThrow(() -> new PostNotFoundException(id));

        System.out.println("같은 객체인가? " + (post1 == post2));
    }
}

PostController

package com.example.studyapi.post.controller;

import com.example.studyapi.common.ApiResponse;
import com.example.studyapi.post.dto.PostCreateRequest;
import com.example.studyapi.post.dto.PostResponse;
import com.example.studyapi.post.dto.PostUpdateRequest;
import com.example.studyapi.post.service.PostService;
import jakarta.validation.Valid;
import lombok.RequiredArgsConstructor;
import org.springframework.http.HttpStatus;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.*;

@RestController
@RequestMapping("/api/posts")
@RequiredArgsConstructor
public class PostController {

    private final PostService postService;

    @PostMapping
    public ResponseEntity<ApiResponse<PostResponse>> create(
            @Valid @RequestBody PostCreateRequest request
    ) {
        PostResponse response = postService.create(request);

        return ResponseEntity
                .status(HttpStatus.CREATED)
                .body(ApiResponse.success("게시글이 등록되었습니다.", response));
    }

    @PutMapping("/{id}")
    public ResponseEntity<ApiResponse<PostResponse>> update(
            @PathVariable Long id,
            @Valid @RequestBody PostUpdateRequest request
    ) {
        PostResponse response = postService.update(id, request);

        return ResponseEntity.ok(
                ApiResponse.success("게시글이 수정되었습니다.", response)
        );
    }

    @DeleteMapping("/{id}")
    public ResponseEntity<Void> delete(
            @PathVariable Long id
    ) {
        postService.delete(id);

        return ResponseEntity.noContent().build();
    }
}

토큰 저장 함수

export function saveToken(token, keepLogin) {
  if (keepLogin) {
    localStorage.setItem("accessToken", token);
    sessionStorage.removeItem("accessToken");
  } else {
    sessionStorage.setItem("accessToken", token);
    localStorage.removeItem("accessToken");
  }
}

export function getAccessToken() {
  return (
    localStorage.getItem("accessToken") ||
    sessionStorage.getItem("accessToken")
  );
}

export function clearToken() {
  localStorage.removeItem("accessToken");
  sessionStorage.removeItem("accessToken");
}

Axios 공통 인스턴스

import axios from "axios";
import { getAccessToken, clearToken } from "../utils/tokenStorage";

const apiClient = axios.create({
  baseURL: "http://localhost:8080",
  headers: {
    "Content-Type": "application/json",
  },
  timeout: 5000,
});

apiClient.interceptors.request.use(
  (config) => {
    const token = getAccessToken();

    if (token) {
      config.headers.Authorization = `Bearer ${token}`;
    }

    return config;
  },
  (error) => Promise.reject(error)
);

apiClient.interceptors.response.use(
  (response) => response,
  (error) => {
    if (error.response?.status === 401) {
      clearToken();
      window.location.href = "/login";
    }

    return Promise.reject(error);
  }
);

export default apiClient;

모든 컬럼이 UPDATE됨
JPA 기본 UPDATE 전략에 따라 여러 컬럼이 함께 업데이트될 수 있습니다.
정말 필요한 경우에만 @DynamicUpdate를 검토합니다.
CORS / 401
수정 요청이 Controller까지 도달하지 못했을 수 있습니다.
OPTIONS, Authorization 헤더, Security CORS 설정을 확인합니다.
11

자주 하는 실수

1. 수정 시 새 Entity 생성

수정은 새 Entity를 만드는 것이 아니라
기존 Entity를 조회한 뒤
값을 변경하는 방식이 기본입니다.

2. @Transactional 누락

Dirty Checking과 Flush가
의도대로 동작하려면
Service 메서드에 트랜잭션 경계가 필요합니다.

3. readOnly에서 수정

@Transactional(readOnly = true)는
조회 전용입니다.
수정 메서드에 붙이면 안 됩니다.

4. Detached Entity 수정

영속성 컨텍스트가
추적하지 않는 Entity를 변경해도
자동 UPDATE가 되지 않습니다.

5. Entity 직접 반환

LazyInitializationException, 순환 참조,
민감 정보 노출 위험이 있습니다.
DTO로 반환합니다.

6. findById().get() 사용

값이 없으면 오류가 나므로
orElseThrow로 명확히 처리합니다.

7. setter 무분별 사용

Entity 상태 변경은
update(), use(), cancel()처럼
의미 있는 메서드로 제한하는 것이 좋습니다.

8. Flush와 Commit 혼동

Flush는 SQL 전송, Commit은
트랜잭션 확정입니다.
둘은 같은 개념이 아닙니다.

실무 연결 카드

게시글 수정 API

findById로 기존 게시글을 조회하고
update 메서드로 값을 변경하면
Dirty Checking으로 UPDATE됩니다.

회원 프로필 수정

User Entity를 조회해
nickname, profileImageUrl 등을
변경하는 구조에 그대로 적용됩니다.

주문 생성

주문 저장, 재고 차감, 결제 로그 저장을
하나의 @Transactional로 묶어야 합니다.

포인트 사용

지갑 잔액 차감과 사용 이력 저장은
함께 성공하거나 함께 rollback되어야 합니다.

예배 기록 수정

WorshipService와 TimetableItem을
조회한 뒤 Managed 상태에서 변경해야
수정 흐름이 안정적입니다.

성능 튜닝

조회 전용 메서드는 readOnly를 사용하고,
연관관계 조회는
다음 단원의 Lazy/N+1 문제와 연결됩니다.

면접 포인트

Q1. 영속성 컨텍스트란 무엇인가요?

JPA가 Entity를 보관하고 추적하는 논리적 작업 공간입니다. 1차 캐시, 쓰기 지연, Dirty Checking, Flush의 기반입니다.

Q2. 1차 캐시란 무엇인가요?

영속성 컨텍스트 내부의 Entity 저장소입니다. 같은 트랜잭션 안에서 같은 Entity를 다시 조회할 때 재사용할 수 있습니다.

Q3. Entity의 상태에는 무엇이 있나요?

Transient, Managed, Detached, Removed가 있습니다. Dirty Checking은 Managed 상태에서 의미가 있습니다.

Q4. Dirty Checking이란 무엇인가요?

Managed Entity의 처음 스냅샷과 현재 상태를 비교해 변경이 있으면 UPDATE SQL을 생성하는 기능입니다.

Q5. 수정 API에서 save() 없이 UPDATE가 되는 이유는 무엇인가요?

@Transactional 범위 안에서 findById로 조회한 Entity가 Managed 상태이고, 값을 변경하면 Dirty Checking이 동작하기 때문입니다.

Q6. Flush와 Commit의 차이는 무엇인가요?

Flush는 SQL을 DB에 전송하는 동작이고, Commit은 트랜잭션 변경을 최종 확정하는 동작입니다.

Q7. @Transactional은 왜 필요한가요?

트랜잭션 경계를 만들고, 영속성 컨텍스트가 유지되는 범위와 Dirty Checking, Flush, commit/rollback을 관리하기 위해 필요합니다.

Q8. readOnly = true는 언제 사용하나요?

조회 전용 메서드에서 사용합니다. 불필요한 Flush와 Dirty Checking 부담을 줄이는 데 도움이 됩니다.

학습 메모

오늘의 암기 포인트

  • 영속성 컨텍스트는 JPA가 Entity를 보관하고 추적하는 작업 공간입니다.
  • 1차 캐시는 같은 트랜잭션 안에서 같은 Entity를 재사용할 수 있게 합니다.
  • Entity 상태는 Transient, Managed, Detached, Removed로 나눌 수 있습니다.
  • save() 후 INSERT SQL은 쓰기 지연 SQL 저장소에 쌓였다가 Flush 시점에 나갈 수 있습니다.
  • Flush는 SQL 전송이고, Commit은 트랜잭션 확정입니다.
  • Dirty Checking은 Managed Entity의 변경을 감지해 UPDATE SQL을 생성하는 기능입니다.
  • 수정 API의 핵심은 save()가 아니라 findById()로 Managed Entity를 조회하고 값을 변경하는 것입니다.
  • @Transactional은 영속성 컨텍스트와 Dirty Checking이 동작하는 범위를 만듭니다.

헷갈리기 쉬운 개념

Flush vs Commit

Flush는 SQL 전송,
Commit은 변경 확정입니다.

Managed vs Detached

Managed는 JPA가 추적하고,
Detached는
더 이상 추적하지 않습니다.

save() vs Dirty Checking

등록은 save(),
수정은 조회 후
변경 감지가 기본 흐름입니다.

1차 캐시 vs Redis

1차 캐시는
영속성 컨텍스트 내부 캐시이고
요청 간 공유 캐시가 아닙니다.

readOnly 트랜잭션

조회 전용 메서드에 사용하고
수정 메서드에는 사용하지 않습니다.
