1. 이번 단원은 무엇을 배우는가?

React 컴포넌트의 기본 역할은 현재 state와 props를 기준으로 JSX를 반환하는 것입니다. 하지만 실제 앱에서는 JSX를 반환하는 것만으로 끝나지 않습니다.
화면이 렌더링된 뒤 문서 제목을 바꾸거나, 서버에 데이터를 요청하거나, 타이머를 시작하거나, input에 focus를 주는 일이 필요합니다.
이런 작업은 렌더링 과정 안에서 직접 처리하기보다 useEffect로 분리하는 것이 좋습니다.
또한 DOM 요소를 직접 가리키거나, 타이머 ID처럼 화면에는 표시하지 않지만 컴포넌트 생애 동안 유지해야 하는 값은 useRef로 관리할 수 있습니다.

useEffect 렌더링 이후 실행할 부가 작업을 처리합니다.
cleanup 타이머, 이벤트 리스너 같은 등록 작업을 정리합니다.
useRef DOM 요소나 리렌더링 없는 값을 책갈피처럼 기억합니다.

이번 단원의 한 줄 정리

useEffect는 렌더링 후 울리는 알람이고, cleanup은 알람을 끄고 정리하는 작업이며, useRef는 DOM 요소나 리렌더링 없는 값을 가리키는 책갈피입니다.

4. 핵심 개념 설명

4-1. useEffect는 렌더링 이후 실행할 부가 작업을 담당한다

React 컴포넌트의 렌더링은 JSX를 계산하는 과정입니다. 하지만 문서 제목 변경, 서버 요청, 타이머 등록, 이벤트 리스너 등록, localStorage 접근 같은 작업은 JSX를 계산하는 일과 다릅니다.
이런 작업은 렌더링 이후에 실행되는 부가 작업이므로 useEffect에서 처리합니다.

권장하지 않는 방향

export default function App() {
    document.title = "App";

    return <h1>App</h1>;
}

렌더링 중 외부 작업을 직접 실행하는 습관은 좋지 않습니다.

useEffect 사용

useEffect(() => {
    document.title = "App";
}, []);

렌더링 이후 실행할 작업을 effect로 분리합니다.

4-2. useEffect의 세 가지 실행 패턴

의존성 배열 없음 매 렌더링 후 실행 
useEffect(() => {
    console.log("매번 실행");
});

Mount 후에도 실행되고, Update 후에도 실행됩니다.

빈 배열 [] 처음 한 번 실행 
useEffect(() => {
    console.log("처음 한 번");
}, []);

컴포넌트가 처음 화면에 나타난 뒤 한 번만 실행됩니다.

값이 있는 배열 값 변경 시 실행 
useEffect(() => {
    document.title = `count: ${count}`;
}, [count]);

처음 실행되고, 이후 count가 바뀔 때마다 실행됩니다.

cleanup 포함 정리 작업 
useEffect(() => {
    const id = setInterval(() => {}, 1000);

    return () => clearInterval(id);
}, []);

Unmount 시 또는 다음 effect 실행 전에 정리합니다.

4-3. 의존성 배열은 effect의 실행 조건이다

의존성 배열은 effect가 다시 실행되어야 하는 기준값 목록입니다. effect 안에서 사용하는 state나 props가 있고, 그 값이 바뀔 때 effect도 다시 실행되어야 한다면 의존성 배열에 넣어야 합니다.

의존성 누락

useEffect(() => {
    document.title = `count: ${count}`;
}, []);

처음 한 번만 실행되므로 count가 바뀌어도 최신 값이 반영되지 않습니다.

의존성 포함

useEffect(() => {
    document.title = `count: ${count}`;
}, [count]);

count가 바뀔 때마다 effect가 다시 실행됩니다.

의존성 배열 판단 공식

effect 안에서 state나 props를 사용한다.
↓
그 값이 바뀔 때 effect도 다시 실행되어야 한다.
↓
의존성 배열에 그 값을 넣는다.

4-4. cleanup 함수는 등록한 작업을 정리한다

타이머, 이벤트 리스너, 구독 작업처럼 “등록하는 작업”은 나중에 정리해야 합니다. useEffect에서 함수를 반환하면 그 함수가 cleanup 함수가 됩니다. 

useEffect(() => {
    const timerId = setInterval(() => {
        console.log("timer");
    }, 1000);

    return () => {
        clearInterval(timerId);
    };
}, []);

cleanup은 컴포넌트가 화면에서 사라질 때 또는 다음 effect가 실행되기 전에 호출됩니다. 타이머를 정리하지 않으면 컴포넌트가 사라져도 타이머가 계속 실행될 수 있습니다.

4-5. useEffect에서 무한 루프가 생기는 경우

effect 안에서 의존성 배열에 포함된 state를 조건 없이 계속 변경하면 무한 루프가 생길 수 있습니다. 

const [count, setCount] = useState(0);

useEffect(() => {
    setCount(count + 1);
}, [count]);

무한 루프 흐름

count 변경
↓
useEffect 실행
↓
setCount(count + 1)
↓
count 변경
↓
리렌더링
↓
useEffect 다시 실행
↓
반복

4-6. useRef는 DOM 요소나 렌더링과 관계없는 값을 기억한다

useRef는 DOM 요소에 직접 접근하거나, 값은 기억해야 하지만 그 값이 바뀐다고 화면을 다시 그릴 필요는 없을 때 사용합니다.

DOM 요소 접근

const inputRef = useRef(null);

<input ref={inputRef} />

inputRef.current.focus();

inputRef.current가 실제 input DOM 요소를 가리킵니다.

렌더링 없는 값 저장

const timerIdRef = useRef(null);

timerIdRef.current = setInterval(() => {
    console.log("timer");
}, 1000);

타이머 ID처럼 화면에 표시하지 않는 값을 저장하기 좋습니다.

4-7. useState와 useRef의 차이

구분 useState useRef
목적 화면에 반영되는 변하는 값 관리 DOM 접근 또는 리렌더링과 관계없는 값 저장
값 저장 위치 state 값 ref.current
값 변경 방식 setState() 호출 ref.current = 값
변경 시 리렌더링 발생함 발생하지 않음
대표 예 카운트, 입력값, 열림/닫힘 input focus, timer ID, 이전 값, 렌더링 횟수

5-2. 의존성 배열은 알람 조건표다

조건표 없음 
useEffect(() => {})

빈 조건표 
useEffect(() => {}, [])

처음 한 번만 알람이 울립니다.

count 조건 
useEffect(() => {}, [count])

count가 바뀔 때 알람이 울립니다.

5-3. cleanup은 방을 나가기 전 정리다

1 Mount

컴포넌트가 화면에 나타납니다.

2 작업 등록

타이머나 이벤트 리스너를 등록합니다.

3 Update / Unmount

다음 effect 실행 전 또는 화면에서 사라지는 시점이 옵니다.

4 cleanup

등록한 타이머나 이벤트 리스너를 정리합니다.

5-4. useRef는 책갈피다

책을 읽다가 중요한 페이지에 책갈피를 꽂으면 나중에 그 위치를 바로 찾을 수 있습니다. React에서도 특정 DOM 요소에 ref라는 책갈피를 꽂아두면, 나중에 ref.current로 접근할 수 있습니다.

ref 객체 생성 책갈피 만들기 
const inputRef = useRef(null);
DOM에 연결 책갈피 꽂기 
<input ref={inputRef} />
DOM 접근 책갈피 위치 찾기 
inputRef.current.focus();

6-2. src/App.jsx

import { useEffect, useRef, useState } from "react";
import "./App.css";

export default function App() {
    const [count, setCount] = useState(0);
    const [seconds, setSeconds] = useState(0);
    const [isRunning, setIsRunning] = useState(false);
    const [name, setName] = useState("");

    const inputRef = useRef(null);
    const renderCountRef = useRef(0);

    useEffect(() => {
        renderCountRef.current += 1;
    });

    useEffect(() => {
        document.title = `count: ${count}`;
    }, [count]);

    useEffect(() => {
        if (!isRunning) {
            return;
        }

        const timerId = setInterval(() => {
            setSeconds((prevSeconds) => prevSeconds + 1);
        }, 1000);

        return () => {
            clearInterval(timerId);
        };
    }, [isRunning]);

    function focusInput() {
        if (inputRef.current) {
            inputRef.current.focus();
        }
    }

    function resetTimer() {
        setSeconds(0);
        setIsRunning(false);
    }

    return (
        <main className="app-page">
            <header className="page-hero">
                <p>React.js With Vite</p>
                <h1>useEffect와 useRef</h1>
                <p>
                    렌더링 이후 작업, cleanup, DOM 접근을 공부합니다.
                </p>
            </header>

            <section className="card">
                <h2>1. count 변경과 document.title</h2>
                <p>현재 count: {count}</p>
                <button onClick={() => setCount(count + 1)}>
                    count 증가
                </button>
            </section>

            <section className="card">
                <h2>2. 타이머와 cleanup</h2>
                <p>진행 시간: {seconds}초</p>

                <div className="button-group">
                    <button onClick={() => setIsRunning(true)}>
                        시작
                    </button>
                    <button onClick={() => setIsRunning(false)}>
                        정지
                    </button>
                    <button onClick={resetTimer}>
                        초기화
                    </button>
                </div>
            </section>

            <section className="card">
                <h2>3. useRef로 input focus</h2>

                <label>
                    이름
                    <input
                        ref={inputRef}
                        value={name}
                        onChange={(event) => setName(event.target.value)}
                        placeholder="이름을 입력하세요"
                    />
                </label>

                <button onClick={focusInput}>
                    입력창에 focus 주기
                </button>

                <p>입력한 이름: {name || "아직 입력하지 않음"}</p>
            </section>

            <section className="card summary-card">
                <h2>4. 렌더링 횟수 ref</h2>
                <p>
                    이 값은 ref에 저장되어 렌더링 이후 증가하지만,
                    ref 값을 바꾸는 것 자체는 리렌더링을 발생시키지 않습니다.
                </p>
                <p>렌더링 이후 기록된 횟수: {renderCountRef.current}</p>
            </section>
        </main>
    );
}

Mistakes

7. 자주 하는 실수

  • 실수 1. 렌더링 중에 부가 작업을 직접 실행함 document.title 변경, 타이머 등록, 이벤트 리스너 등록 같은 작업은 렌더링 본문이 아니라 useEffect에서 처리하는 것이 좋습니다.
  • 실수 2. 의존성 배열을 빼먹음 effect 안에서 count를 사용하면서 []를 넣으면 처음 값만 보고 이후 변경을 반영하지 못할 수 있습니다.
  • 실수 3. 의존성 배열을 아예 안 써서 effect가 너무 자주 실행됨 의존성 배열이 없으면 렌더링 후 매번 실행됩니다. 서버 요청이나 타이머 등록을 넣으면 과도하게 실행될 수 있습니다.
  • 실수 4. effect 안에서 의존성 배열의 state를 무조건 변경함 useEffect(() => setCount(count + 1), [count])처럼 작성하면 무한 루프가 발생할 수 있습니다.
  • 실수 5. cleanup을 하지 않음 타이머나 이벤트 리스너를 등록했다면 cleanup에서 clearInterval, removeEventListener로 정리해야 합니다.
  • 실수 6. 이벤트 리스너를 제거하지 않음 window.addEventListener를 사용했다면 cleanup에서 같은 함수 참조로 window.removeEventListener를 호출해야 합니다.
  • 실수 7. ref 값을 바꾸면 화면도 바로 바뀐다고 생각함 ref.current 변경은 리렌더링을 발생시키지 않습니다. 화면에 보여야 하는 값은 useState로 관리해야 합니다.
  • 실수 8. 모든 DOM 조작을 ref로 해결하려고 함 입력값, 열림/닫힘, 선택 상태처럼 화면 상태는 state로 관리하고, focus나 scroll 같은 DOM 접근이 필요할 때 ref를 사용합니다.
  • 실수 9. ref.current가 null인 시점을 고려하지 않음 DOM이 아직 연결되지 않은 시점에는 ref.current가 null일 수 있으므로 필요한 경우 if (inputRef.current)로 확인합니다.
  • 실수 10. 지역변수를 의존성 배열에 넣고 React가 추적한다고 생각함 지역변수는 컴포넌트 함수가 호출될 때마다 새로 생성되는 일시적인 값입니다. 화면과 연결되면 state, 렌더링 없는 유지값이면 ref를 고려합니다.
Interview

8. 면접 포인트

Q1. useEffect란 무엇인가요?

useEffect는 함수형 컴포넌트에서 렌더링 이후 실행할 side effect를 처리하는 Hook입니다.
문서 제목 변경, 데이터 요청, 타이머 등록, 이벤트 리스너 등록 같은 작업을 처리할 때 사용합니다.

Q2. useEffect는 언제 실행되나요?

의존성 배열이 없으면 Mount와 모든 Update 후 실행되고, 빈 배열이면 Mount 후 한 번만 실행됩니다. 의존성 배열에 값이 있으면 처음 실행된 뒤 해당 값이 변경될 때마다 실행됩니다.

Q3. 의존성 배열이란 무엇인가요?

의존성 배열은 effect가 어떤 값의 변경에 반응해야 하는지를 알려주는 배열입니다. effect 안에서 사용하는 state나 props를 넣어 최신 값을 기준으로 effect가 실행되게 합니다.

Q4. 의존성 배열을 누락하면 어떤 문제가 생길 수 있나요?

effect가 최신 state나 props를 기준으로 실행되지 않을 수 있습니다. 예를 들어 count를 사용하면서 빈 배열을 넣으면 처음 count만 기준으로 실행되고 이후 변경은 반영되지 않습니다.

Q5. cleanup 함수란 무엇인가요?

cleanup 함수는 useEffect에서 반환하는 정리 함수입니다.
컴포넌트가 unmount될 때 또는 다음 effect가 실행되기 전에 호출되며, 타이머 해제, 이벤트 리스너 제거, 구독 해제 같은 작업에 사용합니다.

Q6. cleanup이 필요한 이유는 무엇인가요?

타이머나 이벤트 리스너를 등록한 뒤 정리하지 않으면 컴포넌트가 사라진 후에도 계속 실행될 수 있습니다.
이로 인해 중복 실행, 메모리 누수, 예측하기 어려운 동작이 발생할 수 있습니다.

Q7. useEffect에서 무한 루프는 언제 발생하나요?

effect 안에서 의존성 배열에 포함된 state를 조건 없이 변경하면 state 변경 → 리렌더링 → effect 실행 → state 변경이 반복되어 무한 루프가 발생할 수 있습니다.

Q8. useRef란 무엇인가요?

useRef는 DOM 요소를 참조하거나 리렌더링과 관계없는 값을 저장하기 위한 Hook입니다.
반환된 ref 객체의 current 속성에 값이 저장되며, current가 바뀌어도 리렌더링은 발생하지 않습니다.

Q9. useRef로 input focus를 어떻게 구현하나요?

const inputRef = useRef(null)로 ref 객체를 만들고, input의 ref 속성에 연결합니다. 이후 inputRef.current.focus()를 호출하면 해당 input에 focus를 줄 수 있습니다.

Q10. useState와 useRef의 차이는 무엇인가요?

useState는 값이 바뀌면 리렌더링이 발생하고 화면에 반영됩니다.
useRef는 값을 기억하지만 ref.current를 변경해도 리렌더링이 발생하지 않습니다.
화면에 반영되어야 하는 값은 state, DOM 접근이나 화면과 관계없는 값은 ref를 사용합니다.

Q11. 타이머 ID는 state와 ref 중 무엇으로 관리하는 것이 좋은가요?

타이머 ID는 화면에 표시할 값이 아니고 나중에 clearInterval로 정리하기 위해 기억하면 되는 값입니다.
그래서 useRef로 관리하는 것이 적절합니다. 반면 화면에 표시되는 초 단위 시간은 useState로 관리해야 합니다.

Q12. useRef를 사용하면 document.querySelector를 안 써도 되나요?

특정 컴포넌트 안의 DOM 요소에 접근할 때는 document.querySelector보다 useRef가 React 구조에 더 잘 맞습니다.
ref를 JSX 요소에 직접 연결하므로 컴포넌트 내부의 특정 요소를 명확히 참조할 수 있습니다.

Summary

9. 이번 단원 정리

반드시 기억할 구분

useEffect
→ 렌더링 이후 실행할 부가 작업
→ document.title 변경
→ 데이터 요청
→ 타이머 등록
→ 이벤트 리스너 등록

의존성 배열 없음
→ 모든 렌더링 후 실행

빈 배열 []
→ Mount 후 한 번 실행

[count]
→ Mount 후 실행 + count 변경 시 실행

의존성 배열
→ effect가 다시 실행될 기준값 목록
→ effect 안에서 사용하는 state/props를 넣어야 최신 값 보장

cleanup
→ effect에서 반환하는 정리 함수
→ Unmount 시 또는 다음 effect 실행 전 호출
→ 타이머 해제, 이벤트 리스너 제거

무한 루프 주의
→ effect 안에서 의존성 배열에 있는 state를 계속 변경하면 위험

useRef
→ { current: 초기값 } 객체 반환
→ DOM 요소 접근
→ 리렌더링과 관계없는 값 저장
→ current 변경만으로는 리렌더링 없음

useState vs useRef
→ 화면에 반영되어야 하는 값은 useState
→ 화면과 관계없이 기억해야 하는 값은 useRef

핵심 체크리스트

  • useEffect는 렌더링 이후 실행할 부가 작업을 처리합니다.
  • 의존성 배열이 없으면 렌더링 후 매번 실행됩니다.
  • 빈 배열 []은 Mount 후 한 번만 실행됩니다.
  • [count]는 처음 실행 후 count가 바뀔 때마다 실행됩니다.
  • effect 안에서 사용하는 state나 props는 의존성 배열에 포함해야 최신 값을 보장할 수 있습니다.
  • cleanup 함수는 타이머, 이벤트 리스너, 구독 작업을 정리합니다.
  • 의존성 배열에 포함된 state를 effect 안에서 계속 변경하면 무한 루프가 될 수 있습니다.
  • useRef는 { current: 초기값 } 형태의 객체를 반환합니다.
  • ref.current로 DOM 요소에 접근할 수 있습니다.
  • ref.current 값을 변경해도 리렌더링은 발생하지 않습니다.
  • 화면에 반영되어야 하는 값은 useState, 화면과 관계없이 기억해야 하는 값은 useRef를 사용합니다.

학습 메모

이번 단원은 React가 화면을 그린 뒤에 무엇을 해야 하는지 다루는 단원입니다.
useEffect는 렌더링 이후 실행할 작업을 정리하는 장소이고, cleanup은 등록한 작업을 안전하게 해제하는 장치입니다.
useRef는 DOM 요소나 리렌더링 없이 유지해야 하는 값을 가리키는 책갈피입니다.

오늘의 핵심 문법

useEffect(() => {
    const timerId = setInterval(() => {
        console.log("timer");
    }, 1000);

    return () => {
        clearInterval(timerId);
    };
}, []);

const inputRef = useRef(null);

<input ref={inputRef} />

inputRef.current.focus();

가장 많이 틀리는 부분

useEffect에서 의존성 배열을 빼먹거나, 타이머와 이벤트 리스너를 cleanup하지 않는 실수가 많습니다.
또한 ref.current를 바꾸면 화면도 자동으로 바뀐다고 오해하기 쉽습니다. 화면에 반영할 값은 state, DOM 접근이나 리렌더링 없는 기억값은 ref로 구분해야 합니다.

다음 단원 연결

다음 단원에서는 useReducer와 useContext를 공부합니다.
useState가 단순한 상태 관리에 적합하고, useEffect가 렌더링 이후 작업을 담당한다면,
useReducer는 복잡한 상태 변경 로직을 분리하고 useContext는 여러 컴포넌트가 공유할 값을 전달하는 데 도움을 줍니다.
