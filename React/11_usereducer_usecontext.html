useReducer와 useContext

useState가 단순한 상태를 관리하기 좋은 Hook이라면,
useReducer는 복잡한 상태 변경 로직을 action, dispatch, reducer 구조로 정리하는 Hook입니다.
그리고 useContext와 Context API는 여러 컴포넌트가 공유해야 하는 상태와 함수를 props를 여러 단계로 내려보내지 않고 사용할 수 있게 해 줍니다.
이번 단원에서는 로그인 상태를 예제로 복잡한 상태 관리, props drilling 해결, Provider 범위, 중앙 안내 데스크 비유까지 함께 정리합니다.

Overview

1. 이번 단원은 무엇을 배우는가?

지금까지는 useState를 사용해 카운터, 입력값, 체크박스처럼 비교적 단순한 상태를 관리했습니다.
하지만 실제 프로젝트에서는 로그인 상태, 사용자 정보, 권한, 에러 메시지, 전역 알림, 장바구니처럼 여러 값이 함께 움직이는 복잡한 상태를 자주 만나게 됩니다.
이런 상태를 모두 useState 여러 개로 나누어 관리하면 상태 변경 함수가 컴포넌트 곳곳에 흩어질 수 있습니다.
이때 useReducer를 사용하면 상태 변경 규칙을 reducer 함수에 모아 관리할 수 있습니다.
또한 여러 컴포넌트가 같은 로그인 상태나 dispatch 함수를 필요로 할 때, props를 계속 아래로 전달하면 코드가 복잡해집니다.
이 문제를 줄이기 위해 Context API와 useContext를 사용합니다.

useReducer 복잡한 상태 변경 로직을

dispatch / action action 요청서를 dispatch로 보내
reducer가 새 state를 만듭니다.

이번 단원의 한 줄 정리

useReducer는 action이라는 요청서를 dispatch로 접수해 reducer가 새 state를 만드는 구조이고,
useContext는 Provider가 제공한 공통 데이터를 하위 컴포넌트가 props drilling 없이 꺼내 쓰게 해 주는 구조입니다.

3. useReducer와 useContext 한눈에 보기

useReducer와 useContext는 서로 다른 문제를 해결합니다. useReducer는 상태 변경 로직을 정리하고, useContext는 여러 컴포넌트가 같은 값을 공유하도록 도와줍니다.

개념 역할 대표 예 주의점
useReducer 복잡한 상태 변경 로직 관리 로그인, 장바구니, 게시판 필터 단순 state까지 모두 reducer로 만들 필요는 없습니다.
state 현재 상태값 { isLogin, user, message } 기존 객체를 직접 수정하지 않습니다.
action 상태 변경 요청 객체 { type: "LOGIN", payload: user } type 오타와 payload 구조를 주의합니다.
dispatch action을 reducer에게 보내는 함수 dispatch({ type: "LOGOUT" }) 직접 state를 바꾸지 않고 dispatch로 요청합니다.
reducer 현재 state와 action으로 새 state 반환 authReducer(state, action) default에서는 기존 state를 반환해야 합니다.
Context API 여러 컴포넌트에 공통 값 제공 로그인 상태, 테마, 전역 알림 Provider 범위 안에서만 값 사용이 가능합니다.
useContext Provider가 제공한 값을 꺼내 사용 const { state } = useAuth() Provider 밖에서 사용하면 기본값이 반환될 수 있습니다.

4. 핵심 개념 설명

4-1. useReducer는 복잡한 상태 변경 로직을 정리한다

useState는 단순한 상태를 관리할 때 좋습니다. 하지만 상태가 객체이고 변경 방식이 여러 개라면, 상태 변경 함수가 컴포넌트 안에 계속 쌓이게 됩니다.

useState가 적합한 경우

const [count, setCount] = useState(0);
const [name, setName] = useState("");
const [isOpen, setIsOpen] = useState(false);

숫자 하나, 문자열 하나, boolean 하나처럼 단순한 상태는
useState가 더 간단하고 읽기 쉽습니다.

useReducer가 적합한 경우

const [state, dispatch] =
    useReducer(authReducer, initialState);

로그인, 로그아웃, 에러 처리, 메시지 변경처럼
상태 변경 종류가 많으면 useReducer가 유리합니다.

4-2. useReducer의 기본 형식

const [state, dispatch] = useReducer(reducer, initialState);
state 현재 상태

컴포넌트가 현재 가지고 있는
상태값입니다.

dispatch 요청 접수 함수

action을 reducer에게 전달하는
함수입니다.

action 요청 객체

어떤 상태 변경을 원하는지 표현합니다.

reducer 상태 변경 담당자

현재 state와 action을 받아
새 state를 반환합니다.

4-3. action은 상태 변경 요청서다

action은 reducer에게 보내는 요청 객체입니다. 보통 type으로 작업 종류를 구분하고, 필요한 추가 데이터는 payload에 담습니다.

로그인 요청

{
    type: "LOGIN",
    payload: {
        id: 1,
        name: "가길동",
        role: "USER"
    }
}

로그인에는 사용자 정보가 필요하므로 payload를 함께 전달합니다.

로그아웃 요청

{
    type: "LOGOUT"
}

로그아웃은 추가 데이터 없이 상태를 초기화할 수 있습니다.

4-4. dispatch는 action을 reducer에게 보내는 함수다

컴포넌트는 state를 직접 수정하지 않습니다. 대신 dispatch로 action을 보내고, reducer가 그 action에 따라 새 state를 만듭니다. 

dispatch({
    type: "LOGIN",
    payload: {
        id: 1,
        name: "가길동",
        role: "USER"
    }
});

dispatch 실행 흐름

사용자 클릭
↓
dispatch(action)
↓
reducer(currentState, action)
↓
newState 반환
↓
React가 state 변경 반영
↓
컴포넌트 리렌더링

4-5. reducer는 기존 state를 직접 수정하지 않고 새 state를 반환한다

잘못된 예

function authReducer(state, action) {
    state.isLogin = true;
    state.user = action.payload;

    return state;
}

기존 state 객체를 직접 수정하면 상태 변경 추적이 어려워질 수 있습니다.

올바른 예

function authReducer(state, action) {
    return {
        ...state,
        isLogin: true,
        user: action.payload
    };
}

기존 state를 복사하고 변경할 값만 덮어쓴 새 객체를 반환합니다.

4-6. Context API는 공통 데이터를 제공하는 구조다

React의 기본 데이터 전달 방식은 props입니다.
그러나 로그인 상태, 테마, 전역 알림처럼 여러 컴포넌트가 공통으로 사용하는 값은 props만으로 전달하면 중간 컴포넌트가 전달 통로가 될 수 있습니다.

Context 공유 컨테이너 
const AuthContext = createContext(null);

공유할 데이터를 담는 컨테이너 역할을 합니다.

Provider 값 제공자 
<AuthContext.Provider value={{ state, dispatch }}>
    {children}
</AuthContext.Provider>

하위 컴포넌트에게 Context 값을 제공합니다.

useContext 값 사용자 
const { state, dispatch } =
    useContext(AuthContext);

Provider가 제공한 값을 하위 컴포넌트에서 꺼내 씁니다.

props drilling 해결 중간 전달 감소 
const { dispatch } = useAuth();

dispatch({ type: "LOGOUT" });

중간 컴포넌트를 거치지 않고 필요한 컴포넌트가 직접 사용합니다.

4-7. useReducer와 useContext를 함께 쓰는 이유

useReducer는 상태 변경 로직을 정리합니다. useContext는 state와 dispatch를 여러 컴포넌트가 공유하도록 도와줍니다.
둘을 함께 쓰면 로그인 상태처럼 여러 컴포넌트에서 필요한 값을 깔끔하게 관리할 수 있습니다.

조합 구조

AuthProvider
├─ useReducer로 state와 dispatch 생성
├─ Context.Provider로 state와 dispatch 제공
└─ 하위 컴포넌트는 useContext로 필요한 값만 사용

6-2. src/context/AuthContext.jsx

import { createContext, useContext, useReducer } from "react";

const AuthContext = createContext(null);

const initialState = {
    isLogin: false,
    user: null,
    message: "로그인이 필요합니다.",
    error: null
};

function authReducer(state, action) {
    switch (action.type) {
        case "LOGIN":
            return {
                ...state,
                isLogin: true,
                user: action.payload,
                message: `${action.payload.name}님, 환영합니다.`,
                error: null
            };

        case "LOGOUT":
            return {
                ...state,
                isLogin: false,
                user: null,
                message: "로그아웃되었습니다.",
                error: null
            };

        case "SET_MESSAGE":
            return {
                ...state,
                message: action.payload
            };

        case "SET_ERROR":
            return {
                ...state,
                error: action.payload
            };

        case "CLEAR_ERROR":
            return {
                ...state,
                error: null
            };

        default:
            return state;
    }
}

export function AuthProvider({ children }) {
    const [state, dispatch] = useReducer(authReducer, initialState);

    return (
        <AuthContext.Provider value={{ state, dispatch }}>
            {children}
        </AuthContext.Provider>
    );
}

export function useAuth() {
    const context = useContext(AuthContext);

    if (!context) {
        throw new Error("useAuth는 AuthProvider 안에서 사용해야 합니다.");
    }

    return context;
}

import { useAuth } from "../context/AuthContext.jsx";

export default function ProfilePanel() {
    const { state } = useAuth();

    if (!state.isLogin) {
        return (
            <section className="card">
                <h2>프로필</h2>
                <p>로그인 후 사용자 정보를 확인할 수 있습니다.</p>

    return (
        <section className="card">

}

6-5. src/components/LoginButton.jsx

import { useAuth } from "../context/AuthContext.jsx";

export default function LoginButton() {
    const { dispatch } = useAuth();

    function handleLogin() {
        dispatch({
            type: "LOGIN",
            payload: {
                id: 1,
                name: "가길동",
                role: "USER"
            }
        });
    }

    return (
        <button onClick={handleLogin}>
            로그인
        </button>
    );
}

Mistakes

7. 자주 하는 실수

  • 실수 1. useReducer를 모든 상태에 사용하려고 함 숫자 하나, 문자열 하나, boolean 하나처럼 단순한 상태는 useState가 더 간단합니다. 상태 구조가 복잡하거나 action 종류가 많을 때 useReducer를 고려합니다.
  • 실수 2. reducer 안에서 state를 직접 수정함 state.isLogin = true처럼 직접 수정하지 말고, { ...state, isLogin: true }처럼 새 객체를 반환해야 합니다.
  • 실수 3. default에서 state를 반환하지 않음 알 수 없는 action이 들어왔을 때 undefined가 반환되지 않도록 default: return state를 작성해야 합니다.
  • 실수 4. action.type 오타 LOGIN 대신 LOGN처럼 오타가 나면 reducer의 case와 매칭되지 않습니다. 프로젝트가 커지면 action type을 상수로 분리할 수 있습니다.
  • 실수 5. payload 구조를 reducer와 다르게 보냄 reducer가 action.payload.name을 기대한다면 payload는 문자열이 아니라 { name: "가길동" } 같은 객체여야 합니다.
  • 실수 6. Provider 범위를 잘못 잡음 Context를 사용하는 컴포넌트는 반드시 Provider 내부에 있어야 합니다. Provider 밖에서 사용하면 기본값이 반환되거나 오류가 발생할 수 있습니다.
  • 실수 7. Provider의 value를 전달하지 않음 <AuthContext.Provider>만 작성하면 공유할 값이 없습니다. 반드시 value={{ state, dispatch }}를 전달해야 합니다.
  • 실수 8. useContext 결과가 null일 수 있다는 점을 고려하지 않음 기본값을 null로 만들었다면 Provider 밖에서 사용할 때 오류가 날 수 있습니다. Custom Hook에서 Provider 내부 사용 여부를 검사하면 좋습니다.
  • 실수 9. Context에 너무 많은 값을 넣음 모든 상태를 하나의 거대한 Context에 넣기보다 AuthContext, ThemeContext, ToastContext처럼 역할별로 나누는 것이 좋습니다.
  • 실수 10. props drilling을 무조건 나쁘다고 생각함 1~2단계의 명확한 전달은 props가 더 단순할 수 있습니다. 여러 단계 깊이 전달되거나 여러 컴포넌트가 공통으로 사용하는 값일 때 Context를 고려합니다.
  • 실수 11. Context가 상태 변경 규칙까지 자동으로 정리한다고 생각함 Context는 값을 공유하는 기능입니다. 상태 변경 규칙을 정리하려면 useReducer를 함께 사용하는 것이 좋습니다.
  • 실수 12. reducer가 너무 많은 일을 함 reducer는 새 state를 계산하는 함수로 유지하고, API 요청, localStorage 저장, alert 같은 side effect는 이벤트 핸들러나 useEffect에서 처리하는 것이 좋습니다.
Interview

8. 면접 포인트

Q1. useReducer란 무엇인가요?

useReducer는 useState처럼 상태를 관리하는 Hook이지만, 상태 변경 로직을 reducer 함수로 분리할 수 있다는 점이 다릅니다.
상태가 객체처럼 복잡하거나 변경 action이 여러 개인 경우에 유용합니다.

Q2. useReducer의 기본 형식은 무엇인가요?

const [state, dispatch] = useReducer(reducer, initialState)입니다.
state는 현재 상태값, dispatch는 상태 변경 요청 함수, reducer는 현재 state와 action을 받아 새 state를 반환하는 함수, initialState는 초기 상태입니다.

Q3. action이란 무엇인가요?

action은 reducer에게 전달하는 상태 변경 요청 객체입니다. 일반적으로 { type: "LOGIN", payload: user }처럼 type과 payload 형태로 작성합니다.

Q4. dispatch란 무엇인가요?

dispatch는 상태 변경을 요청하는 함수입니다. 직접 state를 바꾸는 것이 아니라 action을 dispatch하면 reducer가 그 action에 따라 새로운 state를 계산합니다.

Q5. reducer란 무엇인가요?

reducer는 상태 변경 규칙을 모아 둔 함수입니다. 현재 state와 action을 받아 action type에 따라 새 state를 반환합니다. 기존 state를 직접 수정하지 않는 것이 중요합니다.

Q6. useState와 useReducer는 언제 구분해서 사용하나요?

단순한 값 하나를 관리할 때는 useState가 간단합니다.
하지만 로그인 상태처럼 user, error, message 등 여러 값을 함께 관리하고 action 종류가 많다면 useReducer로 변경 로직을 분리하는 것이 좋습니다.

Q7. Context API란 무엇인가요?

Context API는 전역적으로 또는 특정 컴포넌트 트리 범위에서 데이터를 공유하기 위한 기능입니다.
Provider가 값을 제공하고 하위 컴포넌트는 useContext로 그 값을 꺼내 사용할 수 있습니다.

Q8. Provider는 무엇인가요?

Provider는 Context의 값을 제공하는 역할을 합니다. Provider로 감싼 하위 컴포넌트들은 useContext를 통해 value에 들어 있는 값을 사용할 수 있습니다.

Q9. useContext는 무엇인가요?

useContext는 Context 값을 읽기 위한 Hook입니다. Provider가 제공한 값을 props 없이 하위 컴포넌트에서 바로 사용할 수 있게 해 줍니다.

Q10. props drilling이란 무엇인가요?

props drilling은 깊은 컴포넌트 구조에서 상위 데이터를 하위 컴포넌트까지 전달하기 위해 중간 컴포넌트들이 props를 계속 넘겨야 하는 문제입니다.
Context API를 사용하면 이런 전달 과정을 줄일 수 있습니다.

Q11. useReducer와 useContext를 함께 쓰면 무엇이 좋은가요?

useReducer는 상태 변경 로직을 정리하고, useContext는 그 state와 dispatch를 여러 컴포넌트에서 공유하게 해 줍니다.
로그인 상태나 테마처럼 여러 컴포넌트가 사용하는 상태를 관리할 때 유용합니다.

Q12. Provider 범위 밖에서 useContext를 사용하면 어떤 문제가 생기나요?

Context를 사용하는 컴포넌트는 해당 Provider 내부에 있어야 합니다. Provider 밖에서 useContext를 호출하면 기본값이 반환되며, 기본값이 null이면 오류가 날 수 있습니다.

Q13. Context를 사용하면 props를 아예 쓰지 않아도 되나요?

아닙니다. Context는 props를 완전히 대체하는 것이 아니라 props drilling을 줄이기 위한 도구입니다.
가까운 컴포넌트 사이의 데이터 전달은 props가 더 명확할 수 있고, 여러 컴포넌트가 공통으로 쓰는 값은 Context가 적합합니다.

Q14. reducer 안에서 API 요청을 해도 되나요?

일반적으로 reducer는 현재 state와 action을 받아 새 state를 반환하는 상태 변경 함수로 유지하는 것이 좋습니다.
API 요청, localStorage 저장, alert 같은 side effect는 이벤트 핸들러나 useEffect에서 처리하는 것이 좋습니다.

Summary

9. 이번 단원 정리

반드시 기억할 구분

useReducer
→ 복잡한 상태 관리
→ 상태 변경 로직 분리
→ const [state, dispatch] = useReducer(reducer, initialState)

state
→ 현재 상태

dispatch
→ 상태 변경 요청 함수
→ action을 reducer에 전달

action
→ 상태 변경 요청 객체
→ 보통 { type, payload } 형태

reducer
→ 현재 state와 action을 받아 새 state 반환
→ 기존 state 직접 수정 금지

useContext
→ Context 값을 꺼내 쓰는 Hook

Context
→ 공유 데이터를 담는 컨테이너

Provider
→ 하위 컴포넌트에게 Context 값을 제공

props drilling
→ 중간 컴포넌트가 전달만을 위해 props를 계속 넘기는 상황

useReducer + useContext
→ 복잡한 상태 변경 로직은 reducer로 정리
→ state와 dispatch는 Provider로 제공
→ 하위 컴포넌트는 useContext로 필요한 값만 사용

핵심 체크리스트

  • useReducer는 복잡한 상태 변경 로직을 정리할 때 사용합니다.
  • state는 현재 상태값입니다.
  • action은 상태 변경 요청 객체입니다.
  • dispatch는 action을 reducer에게 전달하는 함수입니다.
  • reducer는 현재 state와 action을 받아 새 state를 반환합니다.
  • reducer는 기존 state를 직접 수정하지 않고 새 객체를 반환해야 합니다.
  • 단순한 상태는 useState, 복잡한 상태 변경은 useReducer를 고려합니다.
  • Context는 공유 데이터를 담는 컨테이너입니다.
  • Provider는 하위 컴포넌트에게 Context 값을 제공합니다.
  • useContext는 Provider가 제공한 값을 꺼내 쓰는 Hook입니다.
  • Context를 사용하는 컴포넌트는 Provider 내부에 있어야 합니다.
  • useReducer + useContext 조합은 props drilling을 줄이는 데 도움이 됩니다.

학습 메모

이번 단원은 상태 관리가 단순한 컴포넌트 내부를 넘어, 여러 컴포넌트가 함께 사용하는 구조로 확장되는 단계입니다.
useReducer는 상태 변경 규칙을 정리하고, useContext는 Provider가 제공한 공통 데이터를 하위 컴포넌트가 바로 사용하게 해 줍니다.

오늘의 핵심 문법

const [state, dispatch] = useReducer(reducer, initialState);

dispatch({
    type: "LOGIN",
    payload: user
});

<AuthContext.Provider value={{ state, dispatch }}>
    {children}
</AuthContext.Provider>

const { state, dispatch } = useContext(AuthContext);

가장 많이 틀리는 부분

reducer 안에서 기존 state를 직접 수정하거나, Provider 범위 밖에서 useContext를 사용하는 실수가 많습니다.
reducer는 새 state를 반환해야 하고, Context를 사용하는 컴포넌트는 반드시 Provider 내부에 있어야 합니다.
