props, children, state

컴포넌트를 나누었다면 이제 컴포넌트 사이에 데이터를 어떻게 전달하고, 변하는 데이터는 어디에서 관리할지 이해해야 합니다.
이번 단원에서는 부모가 자식에게 내려주는 읽기 전용 데이터인 props, 컴포넌트 태그 사이의 내용을 전달하는 children,

1. 이번 단원은 무엇을 배우는가?

앞 단원에서는 React 화면을 Header, Footer, CourseCard 같은 작은 컴포넌트로 나누는 방법을 배웠습니다.
하지만 컴포넌트를 나누는 것만으로는 충분하지 않습니다.
각 컴포넌트가 어떤 데이터를 보여줄지, 어떤 값이 변할지, 자식 컴포넌트가 부모에게 어떻게 변경을 요청할지까지 이해해야 합니다.
React의 데이터 흐름은 기본적으로 부모에서 자식으로 내려갑니다.
부모가 자식에게 데이터를 내려줄 때는 props를 사용하고, 컴포넌트 태그 사이에 넣은 내용은 children으로 전달됩니다.
반대로 화면에서 변하는 값은 state로 관리하고, state는 직접 대입하지 않고 변경 함수로 바꿔야 합니다.

props 부모가 자식에게 전달하는 읽기 전용 데이터입니다.
children 컴포넌트 태그 사이의 내용을 전달하는 기본 props입니다.
state 컴포넌트가 직접 관리하는 변하는 데이터입니다.

이번 단원의 한 줄 정리

props는 부모가 자식에게 내려주는 읽기 전용 도시락이고, children은 컴포넌트 태그 사이에 넣는 내용이며, state는 컴포넌트가 직접 관리하는 변하는 데이터입니다.

4. 핵심 개념 설명

4-1. props는 부모가 자식에게 내려주는 데이터다

props는 부모 컴포넌트가 자식 컴포넌트에게 전달하는 데이터입니다. HTML 태그에 속성을 쓰듯이, React 컴포넌트에도 속성을 줄 수 있습니다.

부모에서 props 전달

<LunchBoxCard
    owner="가길동"
    lunch="김밥 도시락"
    drink="물"
/>

owner, lunch, drink가 자식 컴포넌트로 전달됩니다.

자식에서 props 사용

export default function LunchBoxCard({ owner, lunch, drink }) {
    return (
        <article>
            <h2>{owner}의 도시락</h2>
            <p>메뉴: {lunch}</p>
            <p>음료: {drink}</p>
        </article>
    );
}

자식은 전달받은 props를 JSX 안에서 읽어 화면에 출력합니다.

4-2. props는 읽기 전용이다

props는 부모가 소유한 데이터를 자식에게 내려준 값입니다. 자식이 props를 직접 바꾸면 부모는 그 변경을 알 수 없고, 상태 관리 흐름이 깨질 수 있습니다.

잘못된 사고방식

function LunchBoxCard({ lunch }) {
    lunch = "샌드위치 도시락";

    return <p>{lunch}</p>;
}

자식이 props 값을 직접 바꾸려고 합니다. React의 단방향 데이터 흐름과 맞지 않습니다.

올바른 사고방식

function LunchBoxCard({ lunch, onChangeLunch }) {
    return (
        <>
            <p>{lunch}</p>
            <button onClick={onChangeLunch}>
                도시락 변경 요청
            </button>
        </>
    );
}

자식은 부모가 내려준 변경 요청 함수를 이벤트에서 호출합니다.

props 변경 요청 흐름

자식이 props를 직접 변경하지 않는다.
↓
자식이 이벤트 함수를 호출한다.
↓
부모가 state를 변경한다.
↓
새 state가 다시 props로 자식에게 내려간다.

4-3. children은 컴포넌트 태그 사이의 내용이다

children은 React에서 기본 제공하는 특별한 props입니다. 컴포넌트 태그 사이에 넣은 JSX 내용이 children으로 전달됩니다.

사용하는 쪽 
<Panel>
    <h2>오늘의 학습</h2>
    <p>props, children, state를 공부합니다.</p>
</Panel>
→
받는 쪽 
function Panel({ children }) {
    return (
        <section className="panel">
            {children}
        </section>
    );
}

children은 Layout, Panel, Card, Modal처럼 내부 내용을 감싸는 컴포넌트를 만들 때 자주 사용합니다.

4-4. children도 조건부 렌더링할 수 있다

children도 JSX 값이므로 조건에 따라 보여주거나 숨길 수 있습니다. 

export default function NoticePanel({ title, isVisible, children }) {
    return (
        <section className="notice-panel">
            <h2>{title}</h2>

            {isVisible ? (
                <div className="notice-content">
                    {children}
                </div>
            ) : (
                <p className="empty-message">
                    내용이 없습니다.
                </p>
            )}
        </section>
    );
}

4-5. state는 컴포넌트가 관리하는 변하는 데이터다

state는 컴포넌트 안에서 관리하는 변하는 데이터입니다. 버튼 클릭, 입력값 변경, 모달 열림/닫힘, 선택된 탭처럼 화면에서 변하는 값은 state로 관리합니다.

state 생성 useState 
const [count, setCount] = useState(0);

초기값이 0인 count state를 만듭니다.

현재 상태값 count 
<p>현재 카운트: {count}</p>

JSX에서 현재 state 값을 출력합니다.

상태 변경 함수 setCount 
setCount(count + 1);

state는 직접 대입하지 않고 변경 함수로 바꿉니다.

리렌더링 화면 갱신 
state 변경
→ 컴포넌트 리렌더링
→ 화면 갱신

state가 바뀌면 React가 컴포넌트를 다시 렌더링합니다.

4-6. state는 직접 대입하지 않는다

state를 대입 연산자로 직접 바꾸면 React가 상태 변경을 알 수 없습니다. 반드시 state 변경 함수를 사용해야 합니다.

잘못된 예

count = count + 1;

JavaScript 변수만 몰래 바꾸는 것이므로 React가 리렌더링해야 한다는 사실을 알 수 없습니다.

올바른 예

setCount(count + 1);

React에게 state가 바뀌었음을 알리고, 컴포넌트 리렌더링이 일어나도록 합니다.

4-7. props와 state의 차이

구분 props state
의미 부모가 자식에게 전달하는 데이터 컴포넌트가 관리하는 변하는 데이터
소유자 부모 컴포넌트 해당 state를 가진 컴포넌트
변경 가능 여부 자식이 직접 변경 불가 setState 함수로 변경 가능
전달 방향 부모 → 자식 컴포넌트 내부 또는 상위 컴포넌트에서 관리
사용 예 제목, 설명, 표시 옵션, 이벤트 함수 카운트, 입력값, 열림/닫힘, 선택 상태
리렌더링 부모가 새 props를 주면 자식 갱신 state 변경 시 컴포넌트 리렌더링

4-8. local state와 application state

특정 컴포넌트 안에서만 필요한 state는 local state입니다.
여러 컴포넌트가 함께 알아야 하는 state는 공통 부모나 Top 레벨 컴포넌트에 두고 props로 내려보내야 하며, 이런 상태를 application state라고 볼 수 있습니다.

local state

function CounterCard() {
    const [count, setCount] = useState(0);

    return (
        <button onClick={() => setCount(count + 1)}>
            {count}
        </button>
    );
}

count가 CounterCard 안에서만 필요하다면 local state입니다.

application state

function App() {
    const [userName, setUserName] = useState("가길동");

    return (
        <>
            <Header userName={userName} />
            <ProfileCard userName={userName} />
        </>
    );
}

여러 컴포넌트가 함께 사용하는 값은 공통 부모에서 관리합니다.

6. 자식이 부모 데이터를 바꾸고 싶을 때

자식은 props를 직접 바꾸지 않습니다. 부모가 state와 변경 함수를 가지고 있고, 자식은 그 변경 함수를 props로 받아 이벤트에서 호출합니다.

부모 App

function App() {
    const [lunch, setLunch] = useState("김밥 도시락");

    function changeLunch() {
        setLunch("샌드위치 도시락");
    }

    return (
        <LunchBoxCard
            lunch={lunch}
            onChangeLunch={changeLunch}
        />
    );
}

부모는 state와 state 변경 함수를 가지고 있습니다.

자식 LunchBoxCard

function LunchBoxCard({ lunch, onChangeLunch }) {
    return (
        <article>
            <p>오늘의 도시락: {lunch}</p>

            <button onClick={onChangeLunch}>
                도시락 변경 요청
            </button>
        </article>
    );
}

자식은 props를 직접 수정하지 않고, 부모가 내려준 함수를 호출합니다.

실행 흐름

App이 lunch state를 가진다.
↓
App이 lunch를 LunchBoxCard에 props로 전달한다.
↓
LunchBoxCard는 lunch를 화면에 보여준다.
↓
버튼을 누르면 onChangeLunch를 호출한다.
↓
onChangeLunch는 App의 changeLunch 함수다.
↓
App이 setLunch로 state를 변경한다.
↓
새 lunch가 다시 props로 내려간다.
↓
LunchBoxCard 화면이 갱신된다.

Mistakes

8. 자주 하는 실수

  • 실수 1. props와 state를 같은 것으로 생각함 props는 부모에게서 받은 읽기 전용 데이터이고, state는 컴포넌트가 직접 관리하는 변하는 데이터입니다.
  • 실수 2. 자식 컴포넌트에서 props를 직접 수정하려고 함 자식은 props를 직접 바꾸지 않고, 부모가 내려준 이벤트 함수를 호출해 변경을 요청해야 합니다.
  • 실수 3. state를 대입 연산자로 바꾸려고 함 count = count + 1처럼 직접 대입하지 말고 setCount(count + 1)처럼 변경 함수를 사용해야 합니다.
  • 실수 4. children을 일반 props와 완전히 별개로 생각함 children도 props의 한 종류입니다. 다만 React가 기본 제공하는 특별한 props입니다.
  • 실수 5. children을 받았는데 렌더링하지 않음 children을 받았다면 JSX 안에서 {children}으로 출력해야 태그 사이 내용이 화면에 표시됩니다.
  • 실수 6. local state와 application state를 구분하지 않음 한 컴포넌트만 쓰는 상태는 local state로 두고, 여러 컴포넌트가 함께 쓰는 상태는 공통 부모나 Top 레벨 컴포넌트에서 관리해야 합니다.
  • 실수 7. 부모의 state 변경 함수를 호출하지 않고 자식에서 자체 state로 해결하려고 함 같은 데이터를 부모와 자식이 각각 state로 관리하면 어떤 값이 진짜인지 헷갈립니다. 데이터의 소유자를 하나로 정해야 합니다.
  • 실수 8. 객체나 배열 state를 직접 수정함 객체와 배열은 기존 값을 직접 바꾸지 말고, 새 객체나 새 배열을 만들어 state를 업데이트해야 합니다.
  • 실수 9. useState를 조건문이나 반복문 안에서 호출함 Hook은 함수형 컴포넌트의 최상위에서 호출해야 합니다. 조건문이나 반복문 안에서 호출하면 안 됩니다.
  • 실수 10. 이벤트 함수를 실행 결과로 전달함 onClick={setCount(count + 1)}가 아니라 onClick={() => setCount(count + 1)}처럼 나중에 실행할 함수를 전달해야 합니다.
Interview

9. 면접 포인트

Q1. props란 무엇인가요?

props는 부모 컴포넌트가 자식 컴포넌트에게 전달하는 읽기 전용 데이터입니다. 컴포넌트 태그의 속성처럼 전달하고, 자식 컴포넌트는 props를 받아 JSX에서 사용합니다.

Q2. props는 왜 읽기 전용인가요?

props는 부모에서 자식으로 내려오는 단방향 데이터입니다.
자식이 직접 수정하면 부모의 상태와 화면이 어긋날 수 있으므로 읽기 전용으로 다루고, 변경이 필요하면 부모에게 이벤트로 요청해야 합니다.

Q3. props는 어떤 방향으로 전달되나요?

props는 부모 컴포넌트에서 자식 컴포넌트로 내려가는 Top-Down 방식으로 전달됩니다.

Q4. 자식 컴포넌트가 부모의 값을 바꾸고 싶으면 어떻게 해야 하나요?

자식은 props를 직접 수정하지 않고, 부모가 내려준 이벤트 핸들러 함수를 호출해야 합니다. 부모가 그 요청을 받아 state를 변경하고, 변경된 값을 다시 props로 내려주는 방식입니다.

Q5. children이란 무엇인가요?

children은 컴포넌트 태그 사이에 들어간 내용을 전달하는 기본 props입니다. Layout, Panel, Modal처럼 내부 내용을 감싸는 컴포넌트에서 자주 사용합니다.

Q6. children은 언제 사용하나요?

공통 틀은 컴포넌트가 담당하고, 안쪽 내용은 사용하는 곳에서 자유롭게 넣고 싶을 때 사용합니다.
예를 들어 Layout 컴포넌트는 공통 헤더와 푸터를 제공하고, 본문은 children으로 받아 렌더링할 수 있습니다.

Q7. state란 무엇인가요?

state는 컴포넌트 내부에서 관리하는 변하는 데이터입니다. state가 변경되면 React가 컴포넌트를 다시 렌더링해 화면을 갱신합니다.

Q8. useState는 무엇인가요?

useState는 함수형 컴포넌트에서 상태를 만들기 위한 Hook입니다. 현재 상태값과 상태 변경 함수를 배열 형태로 반환합니다.

Q9. state를 직접 대입하지 않고 setState 함수로 변경해야 하는 이유는 무엇인가요?

state를 직접 수정하면 React가 상태 변경을 알 수 없어 리렌더링이 일어나지 않을 수 있습니다. setState 함수를 호출해야 React가 상태 변경을 감지하고 리렌더링을 수행합니다.

Q10. props와 state의 차이는 무엇인가요?

props는 부모가 자식에게 전달하는 값이고, 자식은 직접 수정할 수 없습니다. state는 컴포넌트 내부에서 관리하는 값이며 setState 함수를 통해 변경할 수 있습니다.

Q11. local state와 application state의 차이는 무엇인가요?

local state는 특정 컴포넌트 안에서만 필요한 상태이고, application state는 여러 컴포넌트가 공유해야 해서 상위 컴포넌트에서 관리하는 상태입니다.

Q12. 객체나 배열 state를 직접 수정하면 왜 문제가 되나요?

객체나 배열 state는 직접 수정하지 않고 새 객체나 새 배열을 만들어 업데이트해야 합니다. 그래야 React가 변경된 참조를 보고 리렌더링할 수 있습니다.

Summary

10. 이번 단원 정리

반드시 기억할 구분

props
→ 부모가 자식에게 전달
→ 읽기 전용
→ 자식이 직접 수정하지 않음
→ 변경이 필요하면 이벤트로 부모에게 요청

children
→ 컴포넌트 태그 사이의 내용
→ props.children으로 전달
→ Layout, Panel, Card 같은 wrapper 컴포넌트에서 자주 사용

state
→ 변하는 데이터
→ useState로 생성
→ setState 함수로 변경
→ 변경 시 컴포넌트 리렌더링

local state
→ 한 컴포넌트 안에서만 필요한 상태

application state
→ 여러 컴포넌트가 공유해야 해서 상위 컴포넌트에서 관리하는 상태

부모-자식 데이터 흐름
→ 부모 state
→ props로 자식에게 전달
→ 자식은 이벤트로 부모에게 변경 요청
→ 부모 state 변경
→ 새 props 전달

핵심 체크리스트

  • props는 부모가 자식에게 전달하는 읽기 전용 데이터입니다.
  • props는 부모에서 자식으로 Top-Down 방식으로 전달됩니다.
  • 자식은 props를 직접 수정하지 않습니다.
  • 자식이 부모 데이터를 바꾸고 싶으면 부모가 내려준 이벤트 함수를 호출합니다.
  • children은 컴포넌트 태그 사이의 내용입니다.
  • children도 props의 한 종류입니다.
  • state는 컴포넌트가 직접 관리하는 변하는 데이터입니다.
  • state는 대입 연산자가 아니라 setState 함수로 변경합니다.
  • local state는 한 컴포넌트 안에서만 필요한 상태입니다.
  • application state는 여러 컴포넌트가 공유해야 하는 상태입니다.
  • 상태를 어디에 둘지는 “누가 이 값을 필요로 하는가?”를 기준으로 결정합니다.

학습 메모

이번 단원은 React 데이터 흐름의 핵심입니다. React에서는 자식이 부모 데이터를 직접 바꾸지 않습니다. 부모가 state를 가지고, 자식에게 props로 내려주고, 자식은 이벤트를 통해 부모에게 변경을 요청하는 흐름을 기억해야 합니다.

오늘의 핵심 문법

const [lunch, setLunch] = useState("김밥 도시락");

<LunchBoxCard
    lunch={lunch}
    onChangeLunch={changeLunch}
/>

function LunchBoxCard({ lunch, onChangeLunch }) {
    return (
        <button onClick={onChangeLunch}>
            도시락 변경 요청
        </button>
    );
}

가장 많이 틀리는 부분

props를 자식에서 직접 바꾸거나, state를 count = count + 1처럼 직접 대입하는 실수가 많습니다. props는 읽기 전용이고, state는 반드시 setState 함수로 변경해야 합니다.
