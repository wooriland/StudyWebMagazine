3. REST API 연동과 배포 한눈에 보기

REST API 연동은 React가 서버와 데이터를 주고받는 흐름이고, 배포는 개발한 React 앱을 브라우저가 실행할 수 있는 정적 파일로 만드는 흐름입니다.

개념 역할 대표 예 주의점
REST API React와 서버가 HTTP로 JSON 데이터를 주고받는 방식 GET /products URL, Method, 응답 JSON 구조를 맞춰야 합니다.
Ajax 페이지 새로고침 없이 서버와 비동기 통신 fetch, axios 응답은 나중에 도착하므로 비동기 흐름을 이해해야 합니다.
fetch 브라우저 내장 HTTP 요청 기능 await fetch(url) response.json()으로 JSON 변환이 필요합니다.
axios 설치해서 사용하는 HTTP 요청 라이브러리 axios.get(url) 응답 데이터는 보통 response.data에 있습니다.
Promise 비동기 작업의 성공/실패를 나중에 처리 then, catch, await 요청 직후에는 아직 결과가 없을 수 있습니다.
json-server JSON 파일로 빠르게 REST API 서버 생성 npx json-server --watch ... React 개발 서버와 별도로 실행해야 합니다.
npm run build React 앱을 배포용 정적 파일로 변환 dist 폴더 생성 배포할 때는 src가 아니라 dist 결과물을 사용합니다.
Spring static 배포 React 빌드 결과물을 Spring Boot 정적 리소스로 제공 static/react/index.html React Router 사용 시 새로고침 fallback 문제를 고려해야 합니다.

4-1. React는 서버와 JSON으로 대화한다

React는 화면을 담당합니다. 하지만 회원 정보, 게시글, 상품 목록, 좋아요 수 같은 실제 데이터는 보통 서버와 DB에 있습니다.
그래서 React는 API 서버에 요청을 보내고, 서버가 응답한 JSON 데이터를 state에 저장합니다.

4-2. REST API와 HTTP Method

REST API는 클라이언트와 서버가 HTTP를 통해 데이터를 주고받는 방식입니다. URL은 자원의 위치를 나타내고, HTTP Method는 어떤 작업인지 나타냅니다.

Method 의미 React 프로젝트 예
GET 데이터 조회 게시글 목록 조회, 상품 상세 조회
POST 데이터 생성 게시글 작성, 회원가입
PUT 데이터 전체 수정 회원 정보 전체 수정
PATCH 데이터 일부 수정 좋아요 증가, 닉네임 수정
DELETE 데이터 삭제 게시글 삭제, 댓글 삭제

4-3. fetch와 axios

React에서 REST API를 호출할 때 자주 사용하는 도구는 fetch와 axios입니다. 둘 다 Promise 기반이지만 응답을 다루는 방식이 다릅니다.

fetch

const response = await fetch("http://localhost:3002/products");

if (!response.ok) {
    throw new Error("요청 실패");
}

const data = await response.json();

브라우저 내장 기능입니다.
응답 JSON을 사용하려면 response.json() 변환이 필요합니다.

axios

import axios from "axios";

const response = await axios.get("http://localhost:3002/products");
const data = response.data;

설치해서 사용하는 HTTP 요청 라이브러리입니다.
응답 데이터는 보통 response.data에 들어 있습니다.

4-4. Promise와 async / await

서버 요청은 바로 끝나지 않습니다. Promise는 비동기 작업의 성공 또는 실패를 나중에 처리하기 위한 객체입니다. async / await는 Promise를 읽기 쉽게 작성하는 문법입니다.

then / catch 방식

fetch("http://localhost:3002/products")
    .then((response) => response.json())
    .then((data) => {
        console.log(data);
    })
    .catch((error) => {
        console.error(error);
    });

async / await 방식

async function loadProducts() {
    const response = await fetch("http://localhost:3002/products");
    const data = await response.json();

    console.log(data);
}

4-5. useEffect에서 API를 호출하는 이유

API 요청은 렌더링 이후 실행되는 부가 작업입니다. 그래서 컴포넌트가 처음 화면에 나타난 뒤 useEffect에서 서버 데이터를 조회하는 구조가 자주 사용됩니다. 

useEffect(() => {
    async function loadProducts() {
        const response = await fetch("http://localhost:3002/products");
        const data = await response.json();

        setProducts(data);
    }

    loadProducts();
}, []);

의존성 배열 []는 컴포넌트가 처음 렌더링된 뒤 한 번만 API 요청을 실행하겠다는 의미입니다.

4-6. loading / error / data 상태 관리

서버 요청은 시간이 걸릴 수도 있고 실패할 수도 있습니다. 그래서 요청 중, 실패, 성공 상태를 나누어 화면을 처리해야 합니다. 

const [products, setProducts] = useState([]);
const [loading, setLoading] = useState(false);
const [error, setError] = useState(null);

네트워크 오류나 서버 오류가 발생했을 때
메시지를 보여줍니다.

4-7. json-server

json-server는 실제 백엔드 서버가 준비되기 전, JSON 파일 하나로 빠르게 REST API 서버를 만들어 주는 도구입니다. 

npx json-server --watch ./src/db/data.json --port 3002

위 명령을 실행하면 data.json의 최상위 키를 기준으로 API 주소가 만들어집니다. 예를 들어 products 배열이 있으면 http://localhost:3002/products로 목록을 조회할 수 있습니다.

6-4. src/App.jsx - fetch 사용

import { useEffect, useState } from "react";
import "./App.css";

export default function App() {
    const [products, setProducts] = useState([]);
    const [loading, setLoading] = useState(false);
    const [error, setError] = useState(null);

    useEffect(() => {
        async function loadProducts() {
            try {
                setLoading(true);
                setError(null);

                const response = await fetch("http://localhost:3002/products");

                if (!response.ok) {
                    throw new Error("상품 목록을 불러오지 못했습니다.");
                }

                const data = await response.json();
                setProducts(data);
            } catch (error) {
                setError(error.message);
            } finally {
                setLoading(false);
            }
        }

        loadProducts();
    }, []);

    if (loading) {
        return (
            <main className="app-page">
                <section className="card">
                    <h1>로딩 중</h1>
                    <p>상품 목록을 불러오는 중입니다...</p>
                </section>
            </main>
        );
    }

    if (error) {
        return (
            <main className="app-page">
                <section className="card error-card">
                    <h1>오류 발생</h1>
                    <p>{error}</p>
                    <p>json-server가 실행 중인지 확인해 주세요.</p>
                </section>
            </main>
        );
    }

    return (
        <main className="app-page">
            <header className="page-hero">
                <p>React REST API</p>
                <h1>상품 목록 조회</h1>

            </header>

                                <span>{product.price.toLocaleString()}원</span>

                        ))}
                    </ul>
                )}
            </section>
        </main>
    );
}

Mistakes

7. 자주 하는 실수

  • 실수 1. 비동기 요청 결과를 바로 사용할 수 있다고 생각함 fetch 요청을 보낸 직후에는 아직 데이터가 없을 수 있습니다. 데이터는 then 내부 또는 await 이후에 사용해야 합니다.
  • 실수 2. fetch에서 response.json()을 빼먹음 fetch의 응답 객체는 실제 JSON 데이터가 아닙니다. await response.json()으로 변환해야 합니다.
  • 실수 3. fetch와 axios 응답 구조를 혼동함 fetch는 response.json(), axios는 response.data를 사용합니다.
  • 실수 4. useEffect 의존성 배열을 빼먹음 목록을 처음 한 번만 가져오려면 보통 []를 사용합니다. 의존성 배열이 없으면 렌더링마다 요청이 반복될 수 있습니다.
  • 실수 5. useEffect 콜백을 직접 async로 작성함 useEffect(async () => {})보다, effect 내부에 async function을 만들고 호출하는 구조가 권장됩니다.
  • 실수 6. API 서버를 켜지 않고 React만 실행함 React 개발 서버와 API 서버는 별개입니다. localhost:3002/products를 호출하려면 json-server가 실행 중이어야 합니다.
  • 실수 7. URL 주소나 포트를 잘못 씀 json-server를 3002 포트로 실행했다면 React에서도 http://localhost:3002/products로 요청해야 합니다.
  • 실수 8. loading과 error 처리를 하지 않음 서버 요청은 시간이 걸리고 실패할 수 있습니다. 요청 중 화면과 실패 화면을 나누어 처리해야 사용자 경험이 좋아집니다.
  • 실수 9. CORS 오류를 React 코드 문제로만 생각함 React 코드가 맞아도 Spring Boot 서버에서 다른 출처 요청을 허용하지 않으면 브라우저가 요청을 막을 수 있습니다.
  • 실수 10. build 결과물이 src라고 생각함 src는 개발 소스이고, dist는 빌드 후 생성되는 배포용 결과물입니다.
  • 실수 11. dist 폴더 전체 위치를 잘못 복사함 static/react/dist/index.html 구조가 아니라, 보통 dist 안의 내용물을 static/react 아래에 복사합니다.
  • 실수 12. React Router 사용 시 배포 후 새로고침 404를 놓침 SPA 라우팅은 서버 fallback 설정이 필요할 수 있습니다. 새로고침 시 서버가 해당 경로를 index.html로 돌려주지 않으면 404가 발생할 수 있습니다.
Interview

8. 면접 포인트

Q1. REST API란 무엇인가요?

REST API는 HTTP Method와 URL을 이용해 클라이언트와 서버가 데이터를 주고받는 방식입니다.
React에서는 fetch나 axios로 API를 호출하고, 서버가 응답한 JSON 데이터를 state에 저장해서 화면에 렌더링합니다.

Q2. Ajax란 무엇인가요?

Ajax는 브라우저가 페이지 전체를 새로고침하지 않고 서버와 데이터를 주고받을 수 있게 하는 비동기 통신 방식입니다.
React에서는 fetch나 axios를 사용해 Ajax 요청을 보낼 수 있습니다.

Q3. fetch와 axios의 차이는 무엇인가요?

fetch는 브라우저 기본 API이고,
axios는 별도 설치가 필요한 HTTP 클라이언트 라이브러리입니다.
fetch는 응답을 JSON으로 변환하는 과정이 필요하고,
axios는 보통 response.data로 데이터를 다룹니다.

Q4. Promise란 무엇인가요?

Promise는 비동기 작업의 결과를 표현하는 객체입니다. 성공하면 resolve, 실패하면 reject 흐름으로 이어지며, then, catch 또는 async/await로 처리할 수 있습니다.

Q5. async / await는 무엇인가요?

async/await는 Promise를 더 읽기 쉽게 작성하는 문법입니다. await는 Promise가 처리될 때까지 기다린 뒤 결과를 반환하므로, then 체인보다 코드 흐름을 이해하기 쉽습니다.

Q6. React에서 API 호출을 useEffect에서 하는 이유는 무엇인가요?

서버 데이터 조회는 컴포넌트 렌더링 이후에 수행되는 side effect입니다. 그래서 useEffect 안에서 API를 호출하고, 응답 데이터를 state에 저장해 다시 렌더링합니다.

Q7. loading, error, data 상태를 나누는 이유는 무엇인가요?

API 요청은 비동기 작업이므로 응답 전에는 loading 상태를 보여주고, 실패하면 error 상태를 보여줘야 합니다. 성공 시에는 data를 화면에 렌더링합니다.

Q8. json-server는 무엇인가요?

json-server는 백엔드 서버가 준비되지 않았을 때 JSON 파일만으로 REST API를 빠르게 만들 수 있는 개발용 도구입니다.
프론트엔드에서 목록 조회, 상세 조회, 등록, 수정, 삭제 연습을 할 때 유용합니다.

Q9. 비동기 처리에서 가장 많이 하는 실수는 무엇인가요?

비동기 요청은 즉시 결과가 반환되지 않습니다. 따라서 fetch 다음 줄에서 바로 데이터를 사용하면 아직 값이 없을 수 있습니다.
응답이 도착한 뒤 then 또는 await 이후에 처리해야 합니다.

Q10. npm run build는 무엇인가요?

npm run build는 개발용 React 소스를 브라우저가 실행할 수 있는 최적화된 정적 파일로 묶는 과정입니다. Vite에서는 dist 폴더가 생성되고, 이 결과물을 배포 서버에 올립니다.

Q11. dist 폴더는 무엇인가요?

dist는 Vite가 npm run build를 통해 생성한 배포용 결과물입니다. 개발 소스인 src가 아니라 dist 안의 index.html과 assets 파일들을 실제 서버에 배포합니다.

Q12. Spring Boot에 React를 배포하는 기본 흐름은 무엇인가요?

React 프로젝트를 빌드한 뒤 dist 폴더의 내용을 Spring Boot의 static 폴더에 복사하면 Spring Boot가 React 정적 파일을 제공할 수 있습니다.
API는 Spring Controller에서 제공하고, 화면은 static에 배포된 React 파일이 담당할 수 있습니다.

Q13. CORS 오류는 왜 발생하나요?

CORS는 브라우저의 보안 정책과 관련된 문제입니다.
React가 localhost:5173에서 실행되고 Spring Boot가 localhost:8080에서 실행되면 출처가 다르기 때문에,
서버에서 해당 출처를 허용해야 요청이 성공합니다.

Q14. React Router를 사용하는 SPA를 배포할 때 새로고침 404가 나는 이유는 무엇인가요?

SPA 라우팅은 클라이언트에서 처리되지만, 새로고침 시에는 서버에 직접 해당 URL 요청이 갑니다.
서버가 모든 프론트엔드 라우트를 index.html로 fallback하도록 설정하지 않으면 404가 발생할 수 있습니다.

  • Promise는 비동기 작업의 성공/실패를 나중에 처리합니다.
  • async/await는 Promise 흐름을 읽기 쉽게 만듭니다.
