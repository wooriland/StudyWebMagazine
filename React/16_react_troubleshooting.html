개발 환경 오류 해결 노트

React를 공부하다 보면 코드가 맞는 것 같은데도 실행이 안 되거나,
터미널에 빨간 오류가 나오거나, 브라우저 화면이 하얗게 비는 상황을 자주 만나게 됩니다.

Overview

1. 이번 단원은 무엇을 배우는가?

개발 중 오류는 피할 수 없습니다. 중요한 것은 오류가 발생하지 않는 것이 아니라, 오류가 발생했을 때 어디서부터 확인해야 하는지 아는 것입니다.

환경 오류 Environment

PowerShell, Node.js, npm,
포트, 패키지 설치 문제입니다.

코드 오류 Code

import/export, JSX, Hook, Router
설정 문제입니다.

서버 오류 Server

json-server, Spring Boot,
API URL, CORS 문제입니다.

배포 오류 Deploy

build 결과물, dist 경로,
static 배포, 새로고침 404 문제입니다.

오류를 해결할 때는 빨간 글씨 전체를 외우려고 하지 말고, 먼저 Error 이름, 파일명, 줄 번호, 핵심 단어를 찾아야 합니다.

이번 단원의 한 줄 정리

React 오류 해결은 “빨간 글씨를 무서워하는 것”이 아니라, 오류 메시지를 읽고 환경·문법·라우터·서버·배포 문제로 분류해 하나씩 확인하는 과정입니다.

오늘의 핵심 질문 “React 코드가 맞는 것 같은데 왜 실행이 안 될까? 오류 메시지를 어디부터 읽고 어떻게 해결해야 할까?”
교안 핵심 요약 React 개발 환경은 Node.js와 NPM 위에서 동작함 → Vite 프로젝트는 npm install, npm run dev, npm run build 흐름을 가짐 →
Vite는 개발 서버와 build 설정을 vite.config.js에서 다룰 수 있음 → build 결과물은 Vite 기준 보통 dist 폴더에 생성됨 →
JSX에서는 class가 아니라 className, onclick이 아니라 onClick, 인라인 스타일은 객체 형태로 작성해야 함 →
Hook은 함수형 컴포넌트 최상위에서 호출해야 하며 조건문/반복문 안에서 호출하면 안 됨 →
React Router는 BrowserRouter, Routes, Route, Outlet 규칙을 지켜야 함 →
REST API 연동은 fetch 또는 axios를 사용하고 json-server는 빠른 API 테스트용으로 실행 가능함 →
Windows PowerShell 실행 정책 오류가 발생할 수 있음 →
교안 부록에서는 PowerShell 관리자 권한 실행 후 get-ExecutionPolicy, Set-ExecutionPolicy RemoteSigned로 실행 정책을 조정하는 방법을 제시함
실습 목표 터미널에서 Node/NPM 버전을 확인하고, Vite 개발 서버를 실행하며,
일부러 import/export 오류와 JSX 오류를 만들어 보고, 오류 메시지를 읽어 수정합니다.
또한 PowerShell UnauthorizedAccess가 발생했을 때 ExecutionPolicy를 확인하는 절차를 이해하고,
npm run build 후 dist 폴더 생성 여부를 확인합니다.

3. 핵심 개념 설명

3-1. 오류 해결은 외우기가 아니라 분류하기다

오류를 모두 외우는 것은 불가능합니다. 대신 오류가 어느 범주에 속하는지 분류하면 해결 방향이 보입니다.

오류 종류 대표 원인 먼저 확인할 것
환경 오류 PowerShell 실행 정책, Node/npm 설치, 포트 충돌 터미널 위치, 실행 명령, 권한, 버전
코드 오류 import 경로, JSX 문법, Hook 규칙, Router 설정 파일명, 줄 번호, 최근 수정 파일
서버 오류 API 서버 미실행, URL/포트 오류, CORS API 서버 실행 여부, 브라우저 직접 접속
배포 오류 dist 경로, static 복사 위치, Router 새로고침 404 build 결과물, 배포 폴더, 서버 fallback

3-2. 오류 해결의 기본 순서

1. 오류 메시지 첫 줄 읽기
2. 파일명과 줄 번호 확인
3. 어떤 명령을 실행했는지 확인
4. 최근 수정한 파일 확인
5. 환경 오류인지 코드 오류인지 분류
6. 핵심 단어 확인
7. 한 번에 하나씩 수정
8. 저장 후 다시 실행

3-3. 오류 메시지에서 봐야 하는 것

Error 이름

Module not found, ReferenceError, UnauthorizedAccess 등

파일명

src/App.jsx, src/main.jsx처럼 오류가 발생한 파일입니다.

줄 번호

src/App.jsx:12:5라면 12번째 줄 5번째 칸 근처를 확인합니다.

핵심 단어

not defined, Cannot read, CORS, Unexpected token 등입니다.

3-4. PowerShell UnauthorizedAccess

Windows 환경에서 VS Code 터미널로 npm 또는 npx 명령을 실행할 때 UnauthorizedAccess가 발생할 수 있습니다.
이 경우는 React 코드 오류가 아니라 PowerShell의 스크립트 실행 정책 문제일 수 있습니다. 

get-ExecutionPolicy
Set-ExecutionPolicy RemoteSigned

Restricted는 스크립트 실행이 제한된 상태이고, RemoteSigned는 로컬에서 만든 스크립트는 실행할 수 있게 하는 정책입니다.

3-5. Vite 실행과 build 확인

node --version
npm --version
npm install
npm run dev
npm run build

npm run dev는 개발 서버 실행, npm run build는 배포용 정적 파일 생성 명령입니다. Vite에서는 보통 dist 폴더에 build 결과물이 생성됩니다.

React Errors

5. import·JSX·Hook·Router·API 오류 예제

5-1. import/export 오류

named export인데 default import로 가져오면 오류가 납니다.

// src/utils/math.js
export function add(a, b) {
    return a + b;
}
// 잘못된 App.jsx
import add from "./utils/math.js";
// 올바른 App.jsx
import { add } from "./utils/math.js";

5-2. JSX 오류

JSX는 HTML과 비슷하지만 속성 이름이 다릅니다.

// 잘못된 코드
export default function App() {
    return (
        <div class="card">
            <button onclick={() => alert("클릭")}>
                클릭
            </button>
        </div>
    );
}
// 올바른 코드
export default function App() {
    return (
        <div className="card">
            <button onClick={() => alert("클릭")}>
                클릭
            </button>
        </div>
    );
}

5-3. 함수 레퍼런스를 JSX에 출력하는 오류

컴포넌트를 함수 이름만 넣어 출력하면 Functions are not valid as a React child 오류가 발생할 수 있습니다. 

// 잘못된 코드
function Message() {
    return <p>안녕하세요</p>;
}

export default function App() {
    return (
        <main>
            {Message}
        </main>
    );
}
// 올바른 코드
export default function App() {
    return (
        <main>
            <Message />
        </main>
    );
}

5-4. Hook 오류

Hook은 조건문 안이 아니라 컴포넌트 최상위에서 호출해야 합니다.

// 잘못된 코드
export default function App({ isLogin }) {
    if (isLogin) {
        const [user, setUser] = useState(null);
    }

    return <main>App</main>;
}
// 올바른 코드
export default function App({ isLogin }) {
    const [user, setUser] = useState(null);

    if (isLogin) {
        return <main>로그인 상태</main>;
    }

    return <main>로그아웃 상태</main>;
}

5-5. useEffect 무한 반복

의존성 배열에 넣은 state를 effect 안에서 계속 변경하면 반복될 수 있습니다.

// 위험한 코드
useEffect(() => {
    setCount(count + 1);
}, [count]);
// 처음 한 번만 실행할 목적이라면
useEffect(() => {
    setCount((prevCount) => prevCount + 1);
}, []);

5-6. Router 오류

BrowserRouter로 감싸지 않으면 Router 관련 기능이 동작하지 않을 수 있습니다.

// 잘못된 main.jsx
ReactDOM.createRoot(document.getElementById("root")).render(
    <App />
);
// 올바른 main.jsx
import { BrowserRouter } from "react-router-dom";

ReactDOM.createRoot(document.getElementById("root")).render(
    <BrowserRouter>
        <App />
    </BrowserRouter>
);

5-7. Route 구조 오류

Route는 Routes 내부에 작성해야 하고, element에는 JSX를 넣어야 합니다.

// 잘못된 코드
<Route path="/about" element={About} />
// 올바른 코드
<Routes>
    <Route path="/about" element={<About />} />
</Routes>

5-8. API 오류

fetch 응답 객체는 바로 데이터가 아니므로 response.json()이 필요합니다.

// 잘못된 코드
const response = await fetch("http://localhost:3002/products");
setProducts(response);
// 올바른 코드
const response = await fetch("http://localhost:3002/products");
const data = await response.json();

setProducts(data);

5-9. API 서버 확인

React 개발 서버와 API 서버는 별개입니다. json-server를 따로 실행해야 합니다.

npx json-server --watch ./src/db/data.json --port 3002
http://localhost:3002/products

spring-project/
└─ src/
   └─ main/
      └─ resources/

            └─ react/
               ├─ index.html
               └─ assets/

                  └─ index-xxxx.css

보통은 dist 폴더 자체를 넣는 것이 아니라, dist 안의 파일들을 static/react 아래에 복사합니다.

6-4. React Router 배포 후 새로고침 404

브라우저에서 /react/about 새로고침
↓
서버에 /react/about 직접 요청
↓
서버가 해당 경로를 모름
↓
404 발생 가능

이 문제는 React 코드만의 문제가 아니라 서버 fallback 설정 문제일 수 있습니다. SPA 경로 요청이 들어오면 index.html을 반환하도록 설정해야 할 수 있습니다.

Mistakes

8. 자주 하는 실수

  • 실수 1. 오류 메시지를 끝까지 읽지 않음 오류 메시지가 길어도 위쪽의 Error 이름, 파일명, 줄 번호, 핵심 단어를 먼저 확인해야 합니다.
  • 실수 2. 터미널 위치를 확인하지 않음 npm run dev는 package.json이 있는 프로젝트 루트에서 실행해야 합니다.
  • 실수 3. npm install을 빼먹음 새로 받은 프로젝트는 node_modules가 없을 수 있으므로 npm install이 필요합니다.
  • 실수 4. 서버 재시작이 필요한데 저장만 함 vite.config.js, .env, 포트 설정, 패키지 설치 후에는 서버 재시작이 필요할 수 있습니다.
  • 실수 5. import/export의 {}를 반대로 씀 named export는 import { name }, default export는 import name입니다.
  • 실수 6. JSX를 HTML처럼 작성함 class는 className, onclick은 onClick, style="color:red"는 style={{ color: "red" }}로 작성합니다.
  • 실수 7. Hook을 조건문 안에서 호출함 Hook은 조건문, 반복문, 중첩 함수 안이 아니라 컴포넌트 최상위에서 호출해야 합니다.
  • 실수 8. API 서버를 켜지 않고 React 오류라고 생각함 React 개발 서버와 API 서버는 별개입니다. API 요청 실패 시 서버 실행 여부와 URL/포트를 확인해야 합니다.
  • 실수 9. CORS를 프론트 코드 문제로만 봄 CORS는 브라우저 보안 정책과 서버 허용 설정 문제입니다. Spring Boot CORS 설정을 확인해야 할 수 있습니다.
  • 실수 10. build 결과물을 잘못 배포함 배포할 때는 src가 아니라 npm run build로 생성된 dist 결과물을 사용합니다.
Interview

9. 면접 포인트

Q1. React 개발 중 오류가 발생하면 가장 먼저 무엇을 확인하나요?

먼저 오류 메시지에서 파일명과 줄 번호를 확인하고, 환경 문제인지 코드 문제인지 분류합니다. 그다음 최근 수정한 코드와 오류 메시지의 핵심 단어를 비교해 원인을 좁힙니다.

Q2. PowerShell UnauthorizedAccess는 어떤 문제인가요?

React 코드 오류라기보다 PowerShell의 스크립트 실행 정책 문제일 수 있습니다.
get-ExecutionPolicy로 현재 정책을 확인하고, 필요한 경우 Set-ExecutionPolicy RemoteSigned로 변경해 해결할 수 있습니다.

Q3. npm install은 언제 필요한가요?

npm install은 package.json에 정의된 의존성 패키지를 설치하는 명령입니다. 프로젝트를 새로 받았거나 node_modules가 없을 때 먼저 실행해야 합니다.

Q4. npm run dev가 실패하면 무엇을 확인해야 하나요?

먼저 package.json이 있는 폴더에서 실행했는지 확인합니다. 그다음 node_modules 설치 여부, 포트 충돌, PowerShell 실행 정책 문제를 순서대로 확인합니다.

Q5. import/export 오류에서 가장 많이 하는 실수는 무엇인가요?

named export는 {}로 가져오고, default export는 {} 없이 가져와야 합니다. 이 둘을 반대로 쓰거나 import 경로를 잘못 작성하는 경우가 많습니다.

Q6. JSX에서 HTML과 다른 대표 문법은 무엇인가요?

JSX는 HTML처럼 보이지만 JavaScript 확장 문법입니다. 그래서 className, onClick처럼 camelCase 속성을 사용하고, style은 style={{ color: "red" }}처럼 객체로 작성합니다.

Q7. Hook을 조건문 안에서 호출하면 안 되는 이유는 무엇인가요?

Hook은 매 렌더링마다 같은 순서로 호출되어야 합니다. 조건문 안에서 호출하면 렌더링 조건에 따라 호출 순서가 바뀔 수 있어 React의 상태 관리가 깨질 수 있습니다.

Q8. React Router에서 자주 하는 설정 실수는 무엇인가요?

BrowserRouter 누락, Routes/Route 구조 오류, element에 JSX를 넣지 않은 경우, 중첩 라우팅에서 Outlet을 빼먹은 경우가 많습니다.

Q9. API 요청 실패 시 무엇을 확인해야 하나요?

프론트 코드만 보지 않고 API 서버 실행 여부, 요청 URL, 포트, 응답 상태, CORS 설정을 함께 확인해야 합니다.

Q10. Vite에서 build 결과물은 어디에 생성되나요?

Vite에서 npm run build를 실행하면 기본적으로 dist 폴더에 배포용 정적 파일이 생성됩니다. 배포할 때는 src가 아니라 dist 결과물을 사용합니다.

Q11. React Router를 배포한 뒤 새로고침 404가 나는 이유는 무엇인가요?

React Router는 클라이언트에서 경로를 해석합니다.
하지만 사용자가 특정 경로에서 새로고침하면 서버에 직접 그 URL을 요청하므로, 서버가 index.html로 fallback하지 않으면 404가 발생할 수 있습니다.
