1. 이번 단원은 무엇을 배우는가?

앞 단원에서는 JSX가 HTML이 아니라 React가 화면을 만들기 위한 UI 설계도라는 것을 배웠습니다.
이번 단원에서는 그 JSX로 표현한 UI가 실제 브라우저 화면에 반영될 때 어떤 일이 일어나는지, 그리고 React가 왜 Virtual DOM이라는 중간 단계를 사용하는지 이해합니다.
브라우저는 HTML을 읽어 DOM을 만들고, CSS를 읽어 CSSOM을 만든 뒤, 요소의 위치와 크기를 계산하고, 색상과 배경을 그리고, 여러 레이어를 합성해 최종 화면을 출력합니다.
이 과정에서 DOM 변경이 자주 발생하면 Reflow, Repaint, Composite 비용이 반복될 수 있습니다.

3. 브라우저 렌더링과 Virtual DOM 한눈에 보기

브라우저는 HTML과 CSS를 읽어 화면을 계산하고 그립니다. React는 이 실제 DOM 변경을 줄이기 위해 Virtual DOM에서 먼저 변경점을 계산한 뒤, 꼭 필요한 변경만 실제 DOM에 반영합니다.

개념 의미 핵심 포인트
DOM HTML 문서를 브라우저가 객체 트리로 표현한 것 JavaScript가 DOM API로 Node를 찾고 수정할 수 있습니다.
CSSOM CSS 규칙을 브라우저가 객체 트리로 표현한 것 DOM과 CSSOM을 결합해 레이아웃 계산을 준비합니다.
Reflow 요소의 위치와 크기를 다시 계산하는 단계 width, height, margin, font-size 변경과 관련됩니다.
Repaint 색상, 배경, 텍스트, 그림자 같은 픽셀을 다시 그리는 단계 color, background-color, box-shadow 변경과 관련됩니다.
Composite 여러 레이어를 합성해 최종 화면을 출력하는 단계 배경, 텍스트, 이미지, 애니메이션 레이어를 합칩니다.
Virtual DOM 실제 DOM의 가벼운 JavaScript 객체 표현 실제 DOM 변경 전에 메모리에서 UI 구조를 비교합니다.
Diffing 이전 Virtual DOM과 새 Virtual DOM을 비교하는 과정 어디가 바뀌었는지 찾습니다.
Reconciliation Diffing으로 찾은 변경점을 실제 DOM에 반영하는 과정 전체 DOM이 아니라 바뀐 부분만 반영합니다.
Batch Update 여러 변경을 묶어서 처리하는 방식 실제 DOM 조작 횟수를 줄이는 데 도움을 줍니다.

4. 핵심 개념 설명

4-1. DOM은 브라우저가 만든 HTML 객체 트리다

DOM은 Document Object Model의 약자입니다. HTML 문서의 태그들을 브라우저가 읽고, JavaScript가 다룰 수 있는 트리 형태의 객체 구조로 만든 것입니다.

HTML 문서

<body>
    <h1>Hello React</h1>
    <p>Virtual DOM을 공부합니다.</p>
</body>

브라우저가 읽는 마크업 문서입니다.

DOM 트리

document
└─ html
   └─ body
      ├─ h1
      │  └─ "Hello React"
      └─ p
         └─ "Virtual DOM을 공부합니다."

HTML 태그가 JavaScript가 접근할 수 있는 객체 트리로 표현됩니다.

4-2. DOM API로 실제 DOM을 직접 조작할 수 있다

JavaScript는 DOM API를 사용해 실제 DOM 요소를 찾고 수정할 수 있습니다. 하지만 DOM 변경이 자주 발생하면 브라우저는 화면을 다시 계산하고 다시 그릴 수 있습니다. 

const title = document.querySelector("h1");
title.textContent = "Hello Virtual DOM";

중요한 이해

  • DOM API 호출 자체가 항상 큰 문제인 것은 아닙니다.
  • 문제는 DOM 변경 이후 브라우저가 Reflow, Repaint, Composite를 다시 수행할 수 있다는 점입니다.
  • DOM 변경이 많고 자주 발생하면 화면 갱신 비용이 커질 수 있습니다.

4-3. 브라우저 렌더링 과정

브라우저가 화면을 그리는 과정을 단순화하면 DOM 생성, CSSOM 생성, Reflow, Repaint, Composite 순서로 볼 수 있습니다.

요소의 너비, 높이, 여백, 위치가 바뀌면 브라우저가 레이아웃을 다시 계산합니다.

4-5. Virtual DOM은 실제 DOM의 가벼운 JavaScript 객체 표현이다

Virtual DOM은 실제 DOM을 바로 직접 조작하는 대신, React가 메모리 안에 만들어 두는 가벼운 UI 구조 객체입니다.
실제 DOM의 완전한 브라우저 객체 복사본이 아니라, UI 구조를 비교하기 위한 JavaScript 객체 표현으로 이해하는 것이 좋습니다.

구분 실제 DOM Virtual DOM
관리 주체 브라우저 React
형태 브라우저 DOM 객체 트리 JavaScript 객체 구조
역할 실제 화면과 연결 변경점 계산과 비교
변경 비용 Reflow/Repaint 발생 가능 메모리 계산 중심
개발자 관점 직접 조작하면 복잡해질 수 있음 state/props 기반 UI 계산에 적합

4-6. Virtual DOM 작동 원리

React는 컴포넌트가 렌더링되면 Virtual DOM을 만들고, state나 props가 바뀌면 새로운 Virtual DOM을 만듭니다.
그 다음 이전 Virtual DOM과 새로운 Virtual DOM을 비교해 변경점을 찾고, 변경된 부분만 실제 DOM에 반영합니다.

1 Virtual DOM 생성

컴포넌트의 UI 구조를
JavaScript 객체로 표현합니다.

2 state / props 변경

데이터가 바뀌면
컴포넌트가 다시 렌더링됩니다.

3 새 Virtual DOM 생성

변경된 데이터 기준으로
새 UI 설계도를 만듭니다.

4 Diffing

이전 설계도와 새 설계도를
비교합니다.

5 Reconciliation

바뀐 부분만
실제 DOM에 반영합니다.

4-7. Diffing과 Reconciliation

Diffing은 이전 Virtual DOM과 새로운 Virtual DOM을 비교해 어디가 바뀌었는지 찾는 과정입니다. Reconciliation은 Diffing으로 찾은 변경점을 실제 DOM에 반영하는 과정입니다.

이전 Virtual DOM 
section
├─ h1: "Count: 0"
└─ button: "증가"
→
새 Virtual DOM 
section
├─ h1: "Count: 1"
└─ button: "증가"

비교 결과

button은 그대로이고, h1의 텍스트만 Count: 0에서 Count: 1로 바뀌었습니다. React는 전체 DOM을 다시 만드는 대신 바뀐 텍스트 부분만 실제 DOM에 반영합니다.

4-8. Batch Update

Batch Update는 여러 변경을 실제 DOM에 여러 번 나누어 반영하지 않고, 가능한 한 묶어서 처리하는 방식입니다. 이를 통해 실제 DOM 조작 횟수를 줄일 수 있습니다. 

setCount(count + 1);
setTheme("dark");
setMessage("업데이트 완료");

위처럼 한 이벤트 안에서 여러 상태가 바뀌어도 React는 내부적으로 변경을 모아 업데이트를 처리하려고 합니다.

6-2. src/App.jsx

import { useState } from "react";
import "./App.css";

export default function App() {
    const [count, setCount] = useState(0);
    const [theme, setTheme] = useState("light");
    const [message, setMessage] = useState("아직 업데이트 전입니다.");

    const isDark = theme === "dark";

    function handleIncrease() {
        setCount(count + 1);
        setMessage("count state가 변경되었습니다.");
    }

    function handleThemeChange() {
        setTheme(isDark ? "light" : "dark");
        setMessage("theme state가 변경되었습니다.");
    }

    return (
        <main className={isDark ? "app-page dark" : "app-page"}>
            <section className="hero-card">
                <p className="lesson-label">React.js With Vite</p>
                <h1>브라우저 렌더링과 Virtual DOM</h1>
                <p>
                    state가 바뀌면 React는 새로운 Virtual DOM을 만들고,
                    이전 Virtual DOM과 비교한 뒤 필요한 부분만 실제 DOM에 반영합니다.
                </p>
            </section>

            <section className="counter-card">
                <h2>Count: {count}</h2>
                <p>
                    버튼을 누르면 count state가 바뀌고 화면의 숫자가 갱신됩니다.
                </p>

                <button onClick={handleIncrease}>
                    count 증가
                </button>
            </section>

            <section className="counter-card">
                <h2>Theme: {theme}</h2>
                <p>
                    테마 버튼을 누르면 className이 바뀌고 배경색이 달라집니다.
                </p>

                <button onClick={handleThemeChange}>
                    테마 변경
                </button>
            </section>

            <section className="counter-card">
                <h2>업데이트 메시지</h2>
                <p>{message}</p>
            </section>
        </main>
    );
}

Direct DOM

7. 직접 DOM 조작 방식과 React 방식 비교

React 앱에서도 document.querySelector()로 DOM을 직접 찾고 바꿀 수는 있습니다.
하지만 React는 state와 props를 기준으로 UI를 계산하므로, 직접 DOM 조작을 남발하면 React가 알고 있는 상태와 실제 화면이 어긋날 수 있습니다.

직접 DOM 조작 방식

<h1 id="count">0</h1>
<button id="increase">증가</button>
let count = 0;

const countText = document.querySelector("#count");
const increaseButton = document.querySelector("#increase");

increaseButton.addEventListener("click", () => {
    count += 1;
    countText.textContent = count;
});

개발자가 직접 DOM을 찾고, 이벤트를 연결하고, 텍스트를 수정합니다.

React 방식

import { useState } from "react";

export default function App() {
    const [count, setCount] = useState(0);

    return (
        <>
            <h1>Count: {count}</h1>
            <button onClick={() => setCount(count + 1)}>
                증가
            </button>
        </>
    );
}

개발자는 state를 바꾸고, React가 화면 반영을 관리합니다.

React 사고방식

직접 DOM 조작 방식
→ 이 DOM을 찾아서 글자를 바꾸자.

React 방식
→ count state를 바꾸자.
→ 화면은 count 값을 기준으로 다시 계산된다.

직접 DOM 조작을 남발하면 생길 수 있는 문제

  • React가 알고 있는 state와 실제 DOM 화면이 어긋날 수 있습니다.
  • 다음 렌더링 때 React가 직접 바꾼 DOM을 다시 덮어쓸 수 있습니다.
  • DOM 변경 이후 발생하는 Reflow/Repaint 비용을 개발자가 직접 관리해야 합니다.
  • 화면이 커질수록 어떤 DOM을 언제 바꿔야 하는지 추적하기 어려워집니다.
State Mutation

8. 참조 타입 state와 변경 감지

교안에서는 Primitive Type은 값 비교이고, Reference Type인 배열이나 객체는 주소 비교라고 설명합니다.
객체나 배열 state의 내부 값만 직접 수정하고 같은 객체를 다시 전달하면 React가 변경을 제대로 감지하지 못할 수 있습니다.

잘못된 객체 수정

const [user, setUser] = useState({
    name: "Kim",
    age: 20
});

function changeName() {
    user.name = "Lee";
    setUser(user);
}

객체 내부 값은 바뀌었지만 객체 참조가 그대로라
변경 감지가 명확하지 않을 수 있습니다.

새 객체로 업데이트

function changeName() {
    setUser({
        ...user,
        name: "Lee"
    });
}

기존 객체를 복사하고 바뀐 속성만 덮어쓴 새 객체를 전달합니다.

잘못된 배열 수정

items.push("새 항목");
setItems(items);

기존 배열을 직접 수정하면 참조가 그대로일 수 있습니다.

새 배열로 업데이트

setItems([
    ...items,
    "새 항목"
]);

기존 배열을 복사하고 새 항목을 추가한 새 배열을 전달합니다.

이번 단원에서는 이렇게만 기억하기

React는 state와 props의 변화를 기준으로 새로운 UI를 계산합니다.
객체와 배열처럼 참조 타입을 다룰 때는 기존 값을 직접 수정하기보다 새 객체나 새 배열을 만들어 전달하는 습관이 중요합니다.

Mistakes

9. 자주 하는 실수

  • 실수 1. React에서 document.querySelector()로 화면을 계속 바꾸려고 함 React에서는 DOM을 직접 찾고 수정하기보다 state를 변경하고, 화면은 state의 결과로 표현하는 방식이 기본입니다.
  • 실수 2. DOM API 호출 자체가 무조건 느리다고 외움 정확히는 DOM 변경 이후 발생할 수 있는 Reflow, Repaint, Composite 과정이 비용을 만들 수 있습니다.
  • 실수 3. Reflow와 Repaint를 구분하지 못함 Reflow는 위치와 크기를 다시 계산하는 단계이고, Repaint는 색상, 배경, 그림자 같은 시각 요소를 다시 그리는 단계입니다.
  • 실수 4. Virtual DOM을 실제 DOM의 완전한 복제라고 생각함 Virtual DOM은 브라우저 DOM 객체를 그대로 하나 더 만든 것이 아니라, React가 메모리에서 비교하기 위한 가벼운 JavaScript 객체 표현입니다.
  • 실수 5. Virtual DOM은 항상 직접 DOM보다 빠르다고 외움 Virtual DOM은 무조건 빠른 기술이라기보다, 변경점을 계산하고 실제 DOM 조작을 최소화하는 업데이트 구조입니다.
  • 실수 6. 객체나 배열 state를 직접 수정함 객체와 배열은 참조 타입이므로 직접 수정하지 말고, 새 객체나 새 배열을 만들어 state를 업데이트하는 것이 좋습니다.
  • 실수 7. Diffing과 Reconciliation을 같은 말로 생각함 Diffing은 차이를 찾는 과정이고, Reconciliation은 찾은 변경점을 실제 DOM에 반영하는 과정입니다.
  • 실수 8. React가 브라우저 렌더링 과정을 없앤다고 생각함 React는 Reflow, Repaint, Composite를 없애는 것이 아니라, 불필요한 실제 DOM 조작을 줄여 브라우저 렌더링 부담을 줄이도록 돕습니다.
Interview

10. 면접 포인트

Q1. DOM이란 무엇인가요?

DOM은 Document Object Model의 약자로, HTML 문서를 JavaScript가 접근할 수 있도록 객체 트리로 표현한 모델입니다.
브라우저는 HTML을 읽어 DOM 트리를 만들고, JavaScript는 DOM API로 이를 조작할 수 있습니다.

Q2. DOM을 직접 자주 변경하면 왜 성능 문제가 생길 수 있나요?

DOM 변경 이후 브라우저가 요소의 위치와 크기를 다시 계산하는 Reflow, 픽셀을 다시 그리는 Repaint, 레이어를 합성하는 Composite를 수행할 수 있기 때문입니다.
DOM Node가 많고 변경이 잦을수록 이 비용이 커질 수 있습니다.

Q3. Reflow와 Repaint의 차이는 무엇인가요?

Reflow는 요소의 위치와 크기를 다시 계산하는 과정이고, Repaint는 계산된 레이아웃을 바탕으로 색상, 배경, 텍스트, 그림자 같은 시각 요소를 다시 그리는 과정입니다.

Q4. Composite는 무엇인가요?

Composite는 브라우저가 배경, 텍스트, 이미지, 애니메이션 같은 여러 렌더링 레이어를 합성해서 최종 화면으로 출력하는 단계입니다.

Q5. Virtual DOM은 무엇인가요?

Virtual DOM은 실제 DOM의 구조를 메모리 안에 가볍게 표현한 JavaScript 객체입니다. React는 이를 이용해 변경점을 계산하고 실제 DOM 조작을 최소화합니다.

Q6. Virtual DOM은 왜 사용하나요?

실제 DOM을 직접 여러 번 조작하면 Reflow, Repaint 같은 브라우저 렌더링 비용이 반복될 수 있습니다.
Virtual DOM을 사용하면 변경 내용을 메모리에서 먼저 계산하고, 실제로 바뀐 부분만 DOM에 반영할 수 있습니다.

Q7. state나 props가 바뀌면 React에서는 어떤 일이 일어나나요?

state나 props가 변경되면 React는 컴포넌트를 다시 렌더링하고 새로운 Virtual DOM을 만듭니다. 이후 이전 Virtual DOM과 비교해 변경된 부분만 실제 DOM에 반영합니다.

Q8. Diffing이란 무엇인가요?

Diffing은 이전 Virtual DOM과 새로운 Virtual DOM을 비교해 어떤 부분이 바뀌었는지 찾는 과정입니다.

Q9. Reconciliation이란 무엇인가요?

Reconciliation은 Diffing 결과를 바탕으로 실제 DOM을 업데이트하는 재조정 과정입니다.
React는 전체 DOM을 다시 만드는 것이 아니라 변경된 부분만 반영해 DOM 조작을 최소화합니다.

Q10. Batch Update란 무엇인가요?

Batch Update는 여러 상태 변경이나 노드 변경을 모아 한 번에 처리하는 방식입니다. 이를 통해 반복적인 실제 DOM 조작을 줄이고 성능을 개선할 수 있습니다.

Q11. Virtual DOM이 항상 실제 DOM보다 빠른가요?

항상 그렇다고 말하면 정확하지 않습니다. Virtual DOM은 변경점을 계산해 실제 DOM 조작을 최소화하는 데 장점이 있습니다.
특히 상태 변화가 많고 UI가 복잡한 애플리케이션에서 효율적인 업데이트 구조를 제공합니다.

Q12. React에서 직접 DOM 조작을 남발하면 왜 좋지 않나요?

React는 state와 props를 기준으로 UI를 계산합니다.
개발자가 직접 DOM을 바꾸면 React가 알고 있는 상태와 실제 화면이 어긋날 수 있고, 다음 렌더링 때 React가 다시 자신의 계산 결과로 화면을 덮어쓸 수 있습니다.

Q13. 객체나 배열 state를 직접 수정하면 왜 문제가 될 수 있나요?

객체와 배열은 참조 타입이라 내부 값만 직접 수정하면 참조가 그대로일 수 있습니다. React는 변경 감지를 위해 새 객체나 새 배열을 만들어 state를 업데이트하는 방식을 권장합니다.

Summary

11. 이번 단원 정리

반드시 기억할 흐름

DOM
→ HTML 문서를 객체 트리로 표현한 것

Reflow
→ 위치와 크기 다시 계산

Repaint
→ 색상, 배경, 텍스트 등 시각 요소 다시 그리기

Composite
→ 여러 레이어를 합성해 최종 화면 출력

Virtual DOM
→ 실제 DOM의 가벼운 JavaScript 객체 표현

state / props 변경
→ 새로운 Virtual DOM 생성

Diffing
→ 이전 Virtual DOM과 새 Virtual DOM 비교

Reconciliation
→ 변경된 부분만 실제 DOM에 반영

Batch Update
→ 여러 변경을 묶어서 실제 DOM 조작 최소화

핵심 체크리스트

  • DOM은 HTML 문서를 브라우저가 객체 트리로 표현한 것입니다.
  • DOM 변경 이후 브라우저는 Reflow, Repaint, Composite를 다시 수행할 수 있습니다.
  • Reflow는 요소의 위치와 크기를 다시 계산하는 단계입니다.
  • Repaint는 색상, 배경, 텍스트, 그림자 같은 시각 요소를 다시 그리는 단계입니다.
  • Composite는 여러 레이어를 합성해 최종 화면을 출력하는 단계입니다.
  • Virtual DOM은 실제 DOM을 추상화한 가벼운 JavaScript 객체 표현입니다.
  • state나 props가 바뀌면 React는 새로운 Virtual DOM을 만듭니다.
  • Diffing은 이전 Virtual DOM과 새 Virtual DOM을 비교하는 과정입니다.
  • Reconciliation은 변경된 부분만 실제 DOM에 반영하는 과정입니다.
  • Batch Update는 여러 변경을 묶어 실제 DOM 조작을 줄이는 방식입니다.
  • React에서는 DOM을 직접 수정하기보다 state를 바꾸고 UI를 state의 결과로 표현하는 것이 좋습니다.

학습 메모

이번 단원은 React의 성능 최적화 마법을 외우는 단원이 아니라, 브라우저가 화면을 그리는 비용과 React가 그 비용을 줄이기 위해 어떤 구조를 사용하는지 이해하는 단원입니다.
Virtual DOM은 실제 DOM을 없애는 것이 아니라, 실제 DOM 조작을 줄이기 위한 메모리 속 설계도라고 생각하면 됩니다.

Virtual DOM을 “항상 실제 DOM보다 빠른 복사본”이라고 단순하게 외우면 안 됩니다.
핵심은 실제 DOM 조작을 최소화하고, state와 props 중심으로 UI 업데이트 흐름을 일관되게 관리하는 데 있습니다.
