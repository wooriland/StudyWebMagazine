Git/GitHub - Chapter 04

로컬 저장소 만들기

로컬 저장소 만들기는 평범한 Eclipse 프로젝트를 Git이 변경 이력을 관리할 수 있는 프로젝트로 바꾸는 단원입니다.
프로젝트를 Git 저장소로 초기화하면 저장소의 핵심 정보가 들어가는 .git 디렉터리가 생성되고, 실제 파일이 있는 프로젝트 폴더는 Working Directory가 됩니다.
이번 단원의 핵심은 저장소 생성, 파일 추적, Staging, Commit, GitHub Push가 모두 서로 다른 단계라는 점을 이해하는 것입니다.

01

로컬 저장소를 만든다는 것

Eclipse에서 프로젝트를 만들었다고 해서 자동으로 Git 저장소가 되는 것은 아닙니다.
처음 만든 프로젝트는 파일을 작성하고 실행할 수 있는 일반 프로젝트 폴더일 뿐입니다. 여기에 Git 저장소 구조를 추가해야 Git이 파일의 변경 상태와 커밋 이력을 관리할 수 있습니다.

Project

일반 프로젝트

파일을 만들고 수정할 수 있지만,
Git 커밋 이력이나 브랜치 정보는 아직 없습니다.

Working Tree

.git 디렉터리

Git의 핵심 정보를 관리합니다.

Repository

Local Repository

내 컴퓨터에서 프로젝트의 버전 이력을
독립적으로 관리할 수 있는 Git 저장소입니다.

핵심 문장

로컬 저장소를 만든다는 것은 일반 프로젝트 폴더에 .git이라는 Git 관리 구조를 추가해, 프로젝트의 변경 이력을 기록할 준비를 하는 것입니다.

02

저장소 생성 전과 생성 후 비교

저장소를 만들기 전과 후를 비교하면 Create Repository가 실제로 무엇을 하는지 쉽게 이해할 수 있습니다.
저장소를 만든다고 기존 소스 코드가 바뀌는 것은 아닙니다. Git이 이력을 관리할 수 있는 구조가 추가되는 것입니다.

Before

저장소 생성 전

eclipse-git-study/
├─ .project
├─ .classpath
├─ .settings/
└─ src/
  • 일반 Eclipse 프로젝트입니다.
  • .git 폴더가 없습니다.
  • Git 커밋 이력이 없습니다.
  • 파일 상태를 Git이 추적하지 않습니다.
After

저장소 생성 후

eclipse-git-study/
├─ .git/
├─ .project
├─ .classpath
├─ .settings/
└─ src/
  • Git 저장소 구조가 준비됩니다.
  • .git 폴더가 생성됩니다.
  • 커밋을 저장할 수 있게 됩니다.
  • 기존 파일은 아직 Untracked일 수 있습니다.
저장소 생성

Git이 버전을 기록할 장소를 준비합니다.

파일 추적

어떤 파일을 버전 관리할지 등록합니다.

Commit

Staged 변경을 첫 번째 버전으로 확정합니다.

Push

로컬 커밋을 GitHub 원격 저장소에 올립니다.

서로 다른 단계

Create Repository를 실행했다고 파일이 자동으로 Add되거나 Commit되는 것은 아닙니다. GitHub에 업로드되는 것도 아닙니다.

.git 폴더를 함부로 삭제하면 안 되는 이유

.git을 삭제해도 현재 HTML이나 Java 파일은 남을 수 있지만, 커밋 이력, 브랜치, HEAD, 저장소 설정 같은 Git 정보는 사라질 수 있습니다.
저장소를 제거하려는 명확한 목적이 없다면 내부 파일을 직접 수정하거나 삭제하지 않습니다.

04

Repository Root와 저장소 범위 이해하기

Repository Root는 Git이 하나의 버전 이력으로 관리하는 최상위 경계입니다.
저장소를 만들 때는 단순히 .git이 생긴다는 사실보다 “어느 폴더 아래의 파일까지 같은 이력으로 관리할 것인가?”를 먼저 생각해야 합니다.

Single Project

프로젝트 하나당 저장소 하나

study-api/
├─ .git/
├─ pom.xml
└─ src/

구조가 단순하고 이력이 분리되어
처음 배우는 사람에게 이해하기 쉽습니다.

Multi Project

관련 프로젝트를 하나의 저장소로 관리

worship-flow/
├─ .git/
├─ frontend/
├─ backend/
├─ ai-service/
└─ docs/

여러 모듈이 하나의 기능과 배포 흐름을 공유한다면
함께 관리할 수 있습니다.

Workspace Risk

Workspace 전체 저장소 주의

eclipse-workspace/
├─ .git/
├─ .metadata/
├─ ProjectA/
├─ ProjectB/
└─ ProjectC/

관련 없는 여러 프로젝트와 Eclipse 내부 정보가
하나의 저장소 범위에 들어갈 수 있습니다.

저장소 범위 결정 질문

  • 이 파일들은 같은 제품을 구성하는가?
  • 같은 팀이 관리하는가?
  • 같은 기능 변경에 함께 수정되는가?
  • 같은 배포 흐름을 가지는가?

저장소 생성 전 경로 확인

  • 프로젝트의 실제 디스크 위치
  • Repository Root로 사용할 위치
  • .git이 생성될 위치
  • 상위 폴더에 기존 .git이 있는지
05

중첩 저장소를 조심해야 하는 이유

상위 폴더가 이미 Git 저장소인데 하위 프로젝트에 새 .git을 만들면 중첩 저장소가 될 수 있습니다. 서브모듈 등 의도적인 구조가 아니라면 초보자에게 큰 혼란을 줍니다.

상위 저장소

parent-repository/

parent-repository/
├─ .git/
└─ child-project/
   ├─ .git/
   └─ src/

혼란이 생기는 질문

  • 현재 어느 저장소에서 작업 중인가?
  • 어느 .git이 명령을 처리하는가?
  • 어느 저장소에 커밋되는가?
  • Eclipse는 어느 저장소와 연결되었는가?

안전한 확인 순서

새 저장소를 만들기 전에 프로젝트 폴더뿐 아니라 상위 폴더에도 기존 .git이 있는지 확인합니다. 기존 저장소에 포함해야 하는 프로젝트라면 새 저장소를 만들지 않고 기존 저장소와 연결합니다.

06

git init과 git clone 비교

내 컴퓨터에 Git 저장소를 준비하는 대표 방법은 git init과 git clone입니다. 이번 단원에서는 기존 Eclipse 프로젝트에 새 저장소를 만드는 git init 흐름을 다룹니다.

git init

기존 폴더를 새 저장소로 만들기

기존 프로젝트 폴더
  ↓ git init
새 Local Repository
  • 새로운 .git이 생성됩니다.
  • 기존 파일은 그대로 유지됩니다.
  • 커밋 이력은 아직 없습니다.
  • 기존 파일은 Untracked일 수 있습니다.
git clone

기존 저장소를 복제하기

GitHub Repository
  ↓ git clone
Local Repository
  • 프로젝트 파일을 가져옵니다.
  • 기존 커밋 이력도 가져옵니다.
  • 원격 저장소 연결도 함께 설정됩니다.
  • 나중 Clone 단원에서 자세히 배웁니다.
07

Eclipse에서 로컬 저장소 만드는 전체 흐름

Eclipse에서는 Git Bash에서 git init을 입력하는 대신, EGit의 Share Project Wizard를 사용해 프로젝트와 저장소를 연결할 수 있습니다.

1 프로젝트 생성

Git으로 관리할 Eclipse 프로젝트를 준비합니다.

2 프로젝트 우클릭

Package Explorer 또는
Project Explorer에서 프로젝트를 선택합니다.

3 Team → Share Project

프로젝트를 Git Team Provider와
연결하는 Wizard를 엽니다.

4 Git 선택

사용할 버전 관리 시스템으로 Git을 선택합니다.

5 Repository 경로 확인

.git이 만들어질 저장소 위치를 확인합니다.

6 Create Repository

로컬 Git 저장소 구조를 초기화합니다.

7 Finish

프로젝트와 저장소 연결을 완료합니다.

8 상태 확인

브랜치 장식과 파일의 ? 표시를 확인합니다.

Share Project의 정확한 의미

Team → Share Project는 프로젝트를 GitHub에 공개하는 메뉴가 아닙니다. Eclipse 프로젝트를 EGit의 Git Team Provider와 연결하는 메뉴입니다.

09

저장소 생성 후 화면에서 확인할 것

Create Repository와 Finish를 완료하면 Eclipse 프로젝트에 Git 관련 장식이 나타날 수 있습니다. 장식 모양은 Eclipse와 EGit 버전에 따라 조금 다를 수 있지만, 표시가 의미하는 개념은 같습니다.

파일이 존재하지만 아직 Git이 추적하지 않는
Untracked 상태임을 뜻합니다.

EGit이 현재 로컬 저장소를 인식하고 있는지 확인할 수 있습니다.

main과 master

첨부 교안에서는 초기 브랜치가 master로 표시되지만,
실제 환경에서는 main 또는 사용자가 설정한 다른 이름이 나타날 수 있습니다.
브랜치 이름이 다르다고 저장소 생성이 실패한 것은 아닙니다.

첫 커밋 전 브랜치

저장소를 만들었지만 아직 첫 Commit을 하지 않았다면
브랜치 이름은 준비되어 있어도 브랜치가 가리킬 실제 커밋은 아직 없습니다.
따라서 git log에 이력이 없을 수 있습니다.

10

Eclipse 조작과 CLI 명령어 연결

Eclipse에서 버튼으로 수행한 작업을 Git CLI와 연결해보면 저장소 생성의 내부 의미를 더 정확하게 이해할 수 있습니다.

Eclipse 작업 CLI 명령어 또는 관점 의미
Eclipse 프로젝트 생성 프로젝트 폴더 생성 Working Directory 후보를 준비합니다.
Team → Share Project EGit Team Provider 연결 Eclipse 프로젝트와 Git 저장소 연결을 시작합니다.
Create Repository git init 현재 경로에 로컬 Git 저장소를 초기화합니다.
프로젝트 장식 확인 git status 현재 저장소와 파일 상태를 확인합니다.
Repository Root 확인 git rev-parse --show-toplevel Working Tree의 최상위 경로를 확인합니다.
.git 위치 확인 git rev-parse --git-dir 현재 Git Directory의 위치를 확인합니다.
현재 브랜치 확인 git branch --show-current 현재 작업 중인 브랜치 이름을 확인합니다.
커밋 이력 확인 git log 첫 커밋 전에는 표시할 이력이 없을 수 있습니다.

CLI로 로컬 저장소 만들기

cd C:/dev/eclipse-git-study

git init
git status

저장소 위치 확인하기

git rev-parse --show-toplevel
git rev-parse --git-dir
git branch --show-current

아직 하지 않는 명령어

이번 단원에서는 저장소 구조만 준비합니다. git add, git commit, git remote add, git push는 다음 단원과 원격 저장소 단원에서 이어집니다.

11

첫 Add 전에 추적 대상 확인하기

저장소를 만들었다고 모든 파일을 곧바로 git add .로 올리는 것은 좋은 습관이 아닙니다. 먼저 어떤 파일을 Git으로 관리하고, 어떤 파일을 제외할지 확인해야 합니다.

Java

자동 생성 파일

bin/
target/
build/
*.class

빌드하면 다시 생성할 수 있는 결과물은
보통 Git 추적에서 제외합니다.

Node.js

의존성과 빌드 결과

node_modules/
dist/
build/

크기가 크거나 설치 명령으로
다시 만들 수 있는 파일은
보통 저장소에 올리지 않습니다.

Security

민감 정보

.env
*.key
secret/
credentials.json

비밀번호, API Key, Token 같은 민감 정보는
Git 저장소에 포함하지 않아야 합니다.

Temporary

로그와 임시 파일

*.log
*.tmp
.cache/
temp/

실행 과정에서 자동 생성되는 임시 파일은
프로젝트 이력을 불필요하게
복잡하게 만듭니다.

Git으로 관리할 가능성이 높은 파일

  • 팀에 필요한 설정 파일

자주 하는 실수

로컬 저장소 생성은 간단한 작업처럼 보이지만, 저장소 범위와 각 단계의 의미를 잘못 이해하면 프로젝트 전체 관리 구조가 복잡해질 수 있습니다.

1. Share Project를
GitHub 공개로 생각

Share Project는 EGit과
프로젝트를 연결하는 메뉴입니다.
GitHub 업로드는
Remote 연결과 Push가 필요합니다.

2. 저장소 생성과 Commit을 혼동

Create Repository는
.git 구조를 만들 뿐입니다.
첫 버전은 Add 후 Commit해야 생성됩니다.

3. 저장소 생성과 파일 추적을 혼동

저장소를 만들었어도
기존 파일은 Untracked일 수 있습니다.
Add해야 추적 흐름이 시작됩니다.

4. ? 표시를 오류라고 생각

물음표는 보통 저장소 생성 실패가 아니라
아직 추적되지 않은 파일을 나타냅니다.

5. Workspace 전체를 저장소로 만듦

.metadata와 여러 독립 프로젝트가
하나의 저장소에 들어갈 수 있습니다.

6. 상위 저장소 안에 새 .git 생성

상위 폴더에 이미 .git이 있으면
의도하지 않은 중첩 저장소가 될 수 있습니다.

7. .git 내부를 직접 수정

Git 이력과 참조 정보가 손상될 수 있으므로
명확한 목적 없이
직접 수정하거나 삭제하지 않습니다.

8. 반드시 master가
보여야 한다고 생각

실제 환경에서는 main 또는
다른 초기 브랜치명이 표시될 수 있습니다.

9. GitHub Token이 필요하다고 생각

로컬 저장소 생성과 로컬 Commit에는
GitHub 인증이 필요하지 않습니다.

10. 곧바로 git add . 실행

민감 정보와 빌드 결과물이
포함되지 않았는지 먼저 확인해야 합니다.

11. 프로젝트 폴더와
Repository Root를 항상 같다고 생각

저장소 루트가
프로젝트의 상위 폴더일 수도 있으므로
실제 경로를 확인해야 합니다.

12. 기존 저장소를 삭제하고 다시 만듦

기존 .git에 중요한 이력이 있을 수 있으므로,
삭제 전에 현재 저장소의 목적과
이력을 확인해야 합니다.

면접 포인트

Q1. 로컬 저장소란 무엇인가요?

로컬 저장소는 개발자의 컴퓨터에 존재하는 Git 저장소입니다. 파일 변경 상태, Staging Area, 커밋 이력과 브랜치를 로컬에서 관리할 수 있습니다.
GitHub에 연결하지 않아도 Commit을 만들고 이력을 확인할 수 있습니다.

Q2. git init은 무엇을 하나요?

git init은 현재 디렉터리를 Git 저장소로 초기화합니다. 일반적으로 .git 디렉터리를 만들고 커밋, 브랜치, HEAD, Index 등을 관리할 구조를 준비합니다. 기존 파일을 자동으로 Commit하지는 않습니다.

Q3. .git 폴더는 무엇인가요?

.git 폴더는 Git 저장소의 핵심 메타데이터 폴더입니다. 커밋 정보, 브랜치 참조, HEAD, Index, 저장소 설정 등이 들어 있습니다. 이를 삭제하면 현재 파일은 남아 있어도 Git 이력은 사라질 수 있습니다.

Q4. Working Directory와 Local Repository의 차이는 무엇인가요?

Working Directory는 개발자가 실제 파일을 만들고 수정하는 공간입니다. Local Repository는 Commit으로 확정된 버전 이력을 저장하는 공간입니다.
Working Directory의 변경은 Add와 Commit을 거쳐 Local Repository에 기록됩니다.

Q5. 저장소를 만들면 기존 파일이 자동으로 추적되나요?

아닙니다. git init은 저장소 구조만 준비합니다. 기존 파일은 Untracked일 수 있고, git add로 Staging Area에 올린 뒤 git commit해야 첫 버전에 포함됩니다.

Q6. Eclipse의 Share Project는 무엇을 하나요?

Share Project는 Eclipse 프로젝트를 EGit 같은 Team Provider와 연결하는 과정입니다. Git을 선택하고 새 저장소를 만들거나 기존 저장소와 연결할 수 있습니다.
이 작업 자체가 GitHub 업로드를 의미하지는 않습니다.

Q7. Create Repository와 Commit의 차이는 무엇인가요?

Create Repository는 .git 저장소 구조를 만드는 작업입니다. Commit은 Staging Area에 올라간 변경을 하나의 버전으로 Local Repository에 기록하는 작업입니다.

Q8. Repository Root란 무엇인가요?

Repository Root는 Git이 하나의 버전 이력으로 관리하는 최상위 작업 경로입니다. 그 아래의 파일들이 해당 저장소의 관리 범위에 포함됩니다.

Q9. git init과 git clone의 차이는 무엇인가요?

git init은 현재 로컬 폴더에 새로운 Git 저장소를 만듭니다. git clone은 이미 존재하는 Git 저장소의 프로젝트 파일과 커밋 이력을 내 컴퓨터로 복제합니다.

Q10. Eclipse Workspace 전체를 저장소로 만들면 왜 문제가 될 수 있나요?

Workspace 전체를 저장소로 만들면 .metadata와 여러 독립 프로젝트가 같은 Git 관리 범위에 들어갈 수 있습니다. 불필요한 파일과 관련 없는 프로젝트가 같은 이력에 섞일 수 있습니다.

Q11. 저장소 생성 직후 파일의 ? 표시는 무엇인가요?

해당 파일이 아직 Git 추적 대상이 아닌 Untracked 상태라는 의미입니다. 오류가 아니라 정상적인 초기 상태이며, 다음 단계에서 Add할 파일과 Ignore할 파일을 구분해야 합니다.

Q12. 초기 브랜치가 main이 아니라 master여도 정상인가요?

네. 초기 브랜치 이름은 Git 설정과 도구 버전에 따라 main, master 또는 다른 이름일 수 있습니다.

Q13. 하나의 저장소에 여러 Eclipse 프로젝트를 넣을 수 있나요?

가능합니다. 서로 밀접하게 관련된 프로젝트라면 하나의 Repository Root 아래에 둘 수 있습니다. 다만 독립 프로젝트를 무분별하게 함께 관리하면 이력과 배포 관리가 복잡해질 수 있습니다.

Q14. 로컬 저장소 생성에 GitHub 인증이 필요한가요?

아닙니다. git init과 로컬 Commit에는 GitHub 인증이 필요하지 않습니다. 인증은 GitHub 원격 저장소에 Push하거나 Pull할 때 필요할 수 있습니다.

헷갈리기 쉬운 개념

프로젝트 생성 vs 저장소 생성

Eclipse 프로젝트 생성은 파일 작업 공간을 만드는 것이고,
Git 저장소 생성은 이력 관리 구조를 만드는 것입니다.

Share Project vs GitHub 공유

Share Project는 EGit 연결이고,
GitHub 공유는 Remote 설정과 Push가 필요합니다.

Create Repository vs Commit

Create Repository는 .git 생성,
Commit은 하나의 버전 생성입니다.

파일 존재 vs Git 추적

파일이 프로젝트에 존재해도 아직 Untracked 상태일 수 있습니다.

Project Root vs Repository Root

프로젝트 폴더가 저장소 루트일 수도 있지만,
저장소 루트가 상위 폴더일 수도 있습니다.

main vs master

초기 브랜치 이름은 환경에 따라 다를 수 있으며
둘 다 정상일 수 있습니다.
