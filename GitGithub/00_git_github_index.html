Git/GitHub - Chapter 00

Git/GitHub 학습 로드맵

Git/GitHub 학습 로드맵은 앞으로 Git과 GitHub를 공부하기 전에 전체 지도를 먼저 그리는 단원입니다.
Git은 내 컴퓨터에서 프로젝트의 변경 이력을 관리하는 도구이고, GitHub는 그 Git 저장소를 원격 저장소에 올려 백업하고 협업하게 해주는 서비스입니다.
이번 단원에서는 세부 명령어를 외우기보다 Working Directory, Staging Area, Local Repository, Remote Repository가 어떻게 연결되는지 먼저 이해합니다.

01

Git/GitHub를 공부한다는 것

Git/GitHub를 공부한다는 것은 단순히 명령어를 외우는 것이 아닙니다.
개발 프로젝트에서 파일이 어떻게 바뀌었는지 기록하고, 필요할 때 과거 상태를 확인하거나 되돌리며, 여러 사람이 같은 프로젝트를 안전하게 수정할 수 있는 흐름을 배우는 것입니다.

Why

왜 필요한가?

어제는 잘 되던 코드가 오늘 깨졌을 때,
어떤 변경 때문에 문제가 생겼는지
확인할 수 있습니다.

History

무엇을 기록하는가?

언제, 누가, 어떤 파일을, 왜 수정했는지
의미 있는 변경 단위로 기록합니다.

Recovery

어떻게 되돌리는가?

커밋 이력을 기준으로
특정 시점의 코드를 확인하거나
reset, revert, restore로 되돌릴 수 있습니다.

Collaboration

어떻게 협업하는가?

GitHub 원격 저장소를 기준으로
push, pull, branch, pull request
흐름을 사용합니다.

핵심 문장

Git은 파일 저장 도구가 아니라 변경 이력 관리 도구이고, GitHub는 그 이력을 원격으로 공유하는 협업 플랫폼입니다.

03

Git과 GitHub는 같은 것이 아니다

초보자가 가장 먼저 헷갈리는 부분은 Git과 GitHub를 같은 것으로 생각하는 것입니다. Git은 내 컴퓨터에서 변경 이력을 관리하는 도구이고, GitHub는 그 Git 저장소를 인터넷에 올려 공유하고 협업하는 서비스입니다.

구분 의미 초보자식 이해 대표 작업
Git 버전 관리 도구 내 컴퓨터에서 변경 기록을 남기는 도구 add, commit, log, branch
GitHub 원격 저장소 서비스 Git 기록을 인터넷에 올려 공유하는 공간 repository, push, pull, PR
Local Repository 내 컴퓨터의 Git 저장소 내 PC 안의 버전 창고 commit 저장
Remote Repository GitHub의 원격 저장소 인터넷에 있는 버전 창고 push, pull, clone

04

Git은 파일이 단순히 “있다 / 없다”만 보는 것이 아니라, 현재 파일이 어떤 상태인지 추적합니다. 이 상태 변화를 이해하면 add, commit, push의 의미가 훨씬 분명해집니다.

아직 이번 커밋에 포함할지
결정하지 않은 상태입니다.

Committed

첨부 교안은 Eclipse 기준으로 Git을 설명합니다. 하지만 Eclipse의 버튼이 실제 Git 명령어로 무엇에 해당하는지 함께 이해해야 VS Code, Git Bash, IntelliJ에서도 같은 원리로 사용할 수 있습니다.

06

처음에 반드시 구분해야 하는 개념

Git을 어렵게 만드는 이유는 비슷해 보이는 명령어가 많기 때문입니다. 아래 비교는 앞으로 모든 단원에서 계속 기준이 됩니다.

Commit vs Push

Commit은 내 컴퓨터의 로컬 저장소에 버전을 남기는 작업입니다.
Push는 그 커밋을 GitHub 원격 저장소로 업로드하는 작업입니다.

Clone vs Pull

Clone은 원격 저장소 전체를 처음 내 컴퓨터로 복사하는 작업입니다.
Pull은 이미 받은 프로젝트에서 최신 변경만 가져와 합치는 작업입니다.

Fetch vs Pull

Fetch는 원격 변경 정보를 가져오기만 합니다.
Pull은 Fetch 후 Merge까지 수행합니다.

Reset vs Revert

Reset은 특정 시점 이후 기록을 지울 수 있어 위험합니다.
Revert는 취소 커밋을 새로 만들어 기록을 보존합니다.

Revert vs Restore

Revert는 이미 커밋된 내용을 취소하는 새 커밋을 만듭니다.
Restore는 주로 커밋 전 작업 파일의 변경을 되돌릴 때 사용합니다.

Branch vs Merge

Branch는 새 기능을 안전하게 작업하는 독립 흐름입니다.
Merge는 나누어진 작업 흐름을 현재 브랜치에 합치는 작업입니다.

07

오류 메시지 미리보기

Git 오류는 대부분 내 로컬 저장소와 GitHub 원격 저장소의 상태가 맞지 않거나, 같은 파일을 서로 다르게 수정했을 때 발생합니다. 이번 로드맵에서는 대표 오류의 의미만 먼저 잡아둡니다.

오류 / 상황
의미
해결 방향
push rejected
원격 저장소에 내가 아직 가져오지 않은 커밋이 있습니다.
먼저 pull한 뒤 다시 push합니다.
Conflict
나와 팀원이 같은 파일의 같은 부분을 다르게 수정했습니다.
충돌 표시를 직접 수정한 뒤 add, commit, push합니다.

충돌 파일에서 내 코드 영역을 나타냅니다.
남길 코드만 정리하고 충돌 표시를 삭제합니다.

GitHub 계정 권한, 토큰, Windows 자격 증명 문제가 있을 수 있습니다.

현재 폴더가 Git 저장소가 아닙니다.
프로젝트 위치와 .git 폴더 존재 여부를 확인합니다.

Git 오류를 읽는 순서

  1. 현재 내가 어느 폴더에서 작업 중인지 확인합니다.
  2. 현재 브랜치가 무엇인지 확인합니다.
  3. 수정 파일이 Staging Area에 올라갔는지 확인합니다.
  4. 커밋이 로컬에만 있는지, GitHub에도 올라갔는지 구분합니다.
  5. 원격 저장소에 내가 아직 받지 않은 변경이 있는지 확인합니다.
  6. 충돌 표시가 있다면 사람이 직접 남길 코드를 결정합니다.

팀 프로젝트에서 GitHub가 필요한 이유

팀 프로젝트에서는 여러 사람이 같은 프로젝트를 동시에 수정합니다. GitHub는 이때 공통 기준이 되는 원격 저장소 역할을 합니다.

작업 시작 전 Pull

원격 저장소의 최신 변경을 먼저 받아와
내 로컬 프로젝트를 최신 상태로 맞춥니다.

기능별 Branch 작업

로그인, 회원가입, 게시판처럼 기능별 브랜치에서
안전하게 작업할 수 있습니다.

Commit 후 Push

내 변경을 의미 있는 단위로 커밋하고
GitHub 원격 저장소에 업로드합니다.

Conflict 해결

같은 파일의 같은 부분을 수정했다면
충돌 표시를 직접 정리한 뒤 다시 커밋합니다.

Pull Request

Fork 또는 브랜치 작업 후 원본 프로젝트에
병합 요청을 보낼 수 있습니다.

저장소 소유자가 팀원을 초대하면
같은 원격 저장소에서 함께 작업할 수 있습니다.

팀 작업 안전 규칙

작업이 끝났다고 바로 push하지 말고, 먼저 pull로 원격의 새로운 변경을 가져온 뒤 push하는 습관을 가져야 합니다. 특히 같은 파일을 수정했다면 충돌 가능성을 항상 생각해야 합니다.

10

Branch와 Merge의 큰 그림

Branch는 안정적인 기본 코드에 직접 영향을 주지 않고 새 기능을 실험하기 위한 독립된 작업 흐름입니다. 작업이 끝나면 Merge를 통해 기본 브랜치에 합칩니다.

main 또는 master 안정적인 기본 코드
feature/login 로그인 기능 작업
feature/nav-update 네비게이션 수정 작업
merge 검증된 기능을 기본 브랜치에 병합
핵심 기준

브랜치는 새 기능을 안전하게 만들기 위한 작업 공간이고, Merge는 그 작업 결과를 현재 브랜치에 합치는 작업입니다.

면접 포인트 미리보기

Q1. Git과 GitHub의 차이는 무엇인가요?

Git은 로컬에서 프로젝트의 변경 이력을 관리하는 버전 관리 도구입니다. GitHub는 Git 저장소를 원격으로 올려 공유하고 협업할 수 있게 해주는 서비스입니다.

Q2. Commit과 Push의 차이는 무엇인가요?

Commit은 변경 사항을 로컬 저장소에 하나의 버전으로 저장하는 작업입니다. Push는 로컬 저장소에 저장된 커밋을 GitHub 같은 원격 저장소로 업로드하는 작업입니다.

Q3. Add 또는 Staging Area는 왜 필요한가요?

Add는 이번 커밋에 포함할 변경 사항을 고르는 단계입니다. 작업 디렉토리에 여러 변경 파일이 있을 수 있기 때문에 의미 있는 변경만 골라 Staging Area에 올린 뒤 커밋할 수 있습니다.

Q4. Pull과 Fetch의 차이는 무엇인가요?

Fetch는 원격 저장소의 최신 변경 정보를 가져오지만 현재 작업 파일에는 바로 합치지 않습니다. Pull은 fetch 후 merge까지 수행해서 원격 변경을 현재 브랜치에 바로 반영합니다.

Q5. Reset과 Revert의 차이는 무엇인가요?

Reset은 특정 커밋 시점으로 돌아가면서 이후 커밋 기록을 삭제할 수 있습니다. Revert는 기존 기록을 삭제하지 않고 특정 커밋의 내용을 취소하는 새 커밋을 생성합니다.

Q6. Conflict는 언제 발생하나요?

두 브랜치 또는 로컬과 원격에서 같은 파일의 같은 부분을 다르게 수정했을 때 발생합니다. Git이 자동으로 병합할 수 없으므로 사람이 직접 코드를 정리해야 합니다.

학습 메모

오늘의 암기 포인트

  • Git은 파일 저장 도구가 아니라 변경 이력 관리 도구입니다.
  • GitHub는 Git 저장소를 원격으로 공유하고 협업하기 위한 서비스입니다.
  • Working Directory는 실제 파일을 수정하는 공간입니다.
  • Staging Area는 이번 커밋에 포함할 변경을 올려두는 대기실입니다.
  • Commit은 로컬 저장소에 버전을 남기는 작업입니다.
  • Push는 로컬 커밋을 GitHub 원격 저장소에 올리는 작업입니다.
  • Pull은 원격 저장소의 변경을 가져와 현재 작업에 합치는 작업입니다.

헷갈리기 쉬운 개념

Commit vs Push

Commit은 내 컴퓨터에 저장,
Push는 GitHub에 업로드입니다.

Clone vs Pull

Clone은 처음 전체 복사,
Pull은 이후 최신 변경 가져오기입니다.

Fetch vs Pull

Fetch는 가져오기만 하고,
Pull은 가져온 뒤 merge까지 합니다.

Reset vs Revert

Reset은 기록을 지울 수 있고,
Revert는 취소 커밋으로 기록을 보존합니다.

Branch vs Merge

Branch는 독립 작업 흐름,
Merge는 그 흐름을 합치는 작업입니다.

Conflict

오류라기보다 Git이 자동으로 선택할 수 없어
사람이 직접 결정해야 하는 상황입니다.
