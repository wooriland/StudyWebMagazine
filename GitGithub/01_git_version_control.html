Git과 버전 관리 개념은 Git 명령어를 외우기 전에 “왜 개발자가 Git을 사용하는지”를 이해하는 단원입니다.

이번 단원에서는 버전 관리, 형상 관리, Git, GitHub의 차이를 처음 공부하는 사람도 이해할 수 있도록 정리합니다.

01

Git을 공부한다는 것은 무엇일까?

Git을 공부한다는 것은 단순히 git add, git commit, git push 같은 명령어를 외우는 것이 아닙니다.
Git을 공부한다는 것은 프로젝트가 변해 가는 과정을 기록하고, 문제가 생겼을 때 원인을 추적하며, 여러 사람이 같은 프로젝트를 안전하게 수정할 수 있는 기준을 배우는 것입니다.

Record

변경 기록

어떤 파일을 언제, 누가, 왜 수정했는지
커밋 이력으로 남길 수 있습니다.

Compare

변경 비교

수정 전 코드와 수정 후 코드를 비교해서
어떤 부분이 바뀌었는지 확인할 수 있습니다.

Recover

문제 복구

코드가 깨졌을 때 이전 커밋을 확인하거나
되돌리기 기능을 사용할 수 있습니다.

Collaboration

협업 기준

GitHub 원격 저장소를 기준으로
팀원이 같은 프로젝트를
함께 관리할 수 있습니다.

핵심 문장

Git은 파일 저장 도구가 아니라 변경 이력 관리 도구입니다. Git을 잘 사용한다는 것은 코드를 저장하는 것이 아니라 프로젝트가 변화한 과정을 이해 가능한 기록으로 남기는 것입니다.

02

파일 저장, 파일 백업, 버전 관리의 차이

Git을 처음 배울 때 가장 먼저 구분해야 하는 것은 저장, 백업, 버전 관리입니다. 셋은 비슷해 보이지만 실제 의미는 다릅니다.

구분 의미 예시 한계 / 장점
파일 저장 현재 파일 내용을 저장 Ctrl + S 현재 상태만 남고 이전 변경 흐름은 알기 어렵습니다.
파일 백업 파일을 복사해서 따로 보관 final.html, final2.html 최신 파일이 무엇인지 헷갈리고 변경 이유를 알기 어렵습니다.
버전 관리 변경 이력을 시간순으로 기록 Commit: Add login page 변경 이유, 작성자, 수정 파일, 시점을 함께 추적할 수 있습니다.
형상 관리 프로젝트 전체 구성 상태 관리 소스 코드, 설정, 문서, DB 스크립트 특정 시점의 프로젝트 전체 상태를 관리할 수 있습니다.

초보자가 자주 하는 방식

project_final.html, project_final2.html, project_final_real.html처럼 파일 이름으로 버전을 관리하는 방식은 프로젝트가 커질수록 위험합니다.
어떤 파일이 진짜 최신인지 알기 어렵고, 여러 파일이 함께 바뀐 경우 서로 맞는 조합을 찾기도 어렵습니다.

03

파일 이름 백업 방식과 Git 방식 비교

파일 이름으로 백업하는 방식은 “복사본”을 늘리는 방식입니다. 반면 Git은 “프로젝트가 어떻게 변했는지”를 커밋 이력으로 관리합니다.

파일 이름 백업 방식 복사본이 계속 늘어나는 방식 
project.html
project_final.html
project_final2.html
project_final_real.html
project_final_real_last.html

최신 파일을 구분하기 어렵고, 변경 이유와 변경 범위를 알기 어렵습니다.

Git 버전 관리 방식 변경 이력이 커밋으로 남는 방식 
Commit 1: Add basic HTML structure
Commit 2: Add navigation menu
Commit 3: Add Git/GitHub section
Commit 4: Fix responsive layout

언제 어떤 작업을 했는지 기록되므로 문제 추적과 복구가 훨씬 쉬워집니다.

버전 관리 흐름

파일 수정
  ↓
변경 내용 확인
  ↓
이번 버전에 포함할 파일 선택
  ↓
커밋 메시지 작성
  ↓
로컬 저장소에 버전 생성
  ↓
필요하면 GitHub 원격 저장소에 업로드
04

Git과 GitHub의 차이

Git과 GitHub는 이름이 비슷해서 처음에는 같은 것처럼 느껴질 수 있습니다. 하지만 둘은 역할이 다릅니다.
Git은 내 컴퓨터에서 변경 이력을 관리하는 도구이고,
GitHub는 그 Git 저장소를 인터넷에 올려 공유하고 협업하게 해주는 서비스입니다.

Git

내 컴퓨터의 기록 도구

프로젝트의 변경 이력을 내 컴퓨터에서
커밋 단위로 관리합니다.

GitHub

온라인 공유 공간

Git 저장소를 원격에 올려
백업, 협업, 포트폴리오로 활용합니다.

Local

로컬 저장소

내 컴퓨터 안에 있는 저장소입니다.
Commit은 이곳에 먼저 기록됩니다.

Remote

원격 저장소

GitHub에 있는 저장소입니다.
Push와 Pull로 로컬 저장소와 연결됩니다.

구분 Git GitHub
정체 버전 관리 도구 원격 저장소 서비스
위치 내 컴퓨터에서 동작 인터넷 서비스로 동작
핵심 역할 변경 이력 관리 백업, 공유, 협업
대표 기능 add, commit, log, branch repository, push, pull, PR
05

버전 관리와 형상 관리

버전 관리는 파일과 코드가 어떻게 바뀌었는지 관리하는 것입니다. 형상 관리는 여기서 더 넓게, 프로젝트를 구성하는 전체 요소의 상태를 관리하는 개념입니다.

Version

버전 관리

파일과 코드의 변경 이력을
시간순으로 관리합니다.
“언제 무엇이 바뀌었는가?”에 초점이 있습니다.

Configuration

형상 관리

소스 코드, 설정 파일, 문서, DB 스크립트 등
프로젝트 전체 상태를 관리합니다.

SCM

소스 코드 관리

개발 소스의 변경 이력을 관리합니다.
Git은 대표적인 소스 코드 관리 도구로
사용됩니다.

Team

협업 관리

여러 사람이 같은 프로젝트를 수정할 때
변경 내용을 안전하게 합칠 수 있도록 돕습니다.

쉽게 구분하기

버전 관리는 이 파일이 언제 어떻게 바뀌었는가?에 가깝고, 형상 관리는 이 프로젝트가 이 시점에 어떤 구성으로 동작했는가?에 가깝습니다.

06

Git이 버전을 만드는 기본 흐름

Git은 파일을 수정하자마자 바로 버전으로 만들지 않습니다. 실제 파일을 수정한 뒤, 이번 커밋에 포함할 파일을 고르고, 커밋 메시지를 작성한 다음 로컬 저장소에 하나의 버전으로 기록합니다.

1 Working Directory

실제 파일을 수정하는
작업 공간입니다.

2 Add

이번 버전에 포함할
변경 파일을 선택합니다.

3 Staging Area

커밋 대기실입니다.
Index라고도 부릅니다.

4 Commit

로컬 저장소에
하나의 버전을 생성합니다.

5 Local Repository

내 컴퓨터 안의
Git 저장소입니다.

6 Push

필요하면 GitHub
원격 저장소로 업로드합니다.

Git 용어로 표현한 흐름

Working Directory
  ↓ git add

Staging Area / Index
  ↓ git commit -m "메시지"

Local Repository
  ↓ git push

Remote Repository / GitHub

Eclipse 조작 흐름과 개념 연결

첨부 교안은 Eclipse 기준으로 Git을 설명합니다. Eclipse 메뉴를 외우는 것보다 중요한 것은 그 메뉴가 Git의 어떤 개념과 연결되는지 이해하는 것입니다.

Eclipse 조작 Git 개념 의미
Git 사용자 정보 설정 작성자 정보 설정 커밋에 누가 작성했는지 남기기 위한 준비입니다.
프로젝트 생성 Working Directory 준비 실제 파일을 수정할 작업 공간을 만듭니다.
Team → Share Project Git 관리 시작 현재 프로젝트를 Git이 관리하도록 등록합니다.
Create Repository 로컬 저장소 생성 .git 폴더가 생기고 Git 이력을 관리할 준비가 됩니다.
Team → Add to Index Staging 이번 커밋에 포함할 변경 파일을 선택합니다.
Commit Message 작성 버전 설명 작성 이 커밋이 어떤 의미인지 기록합니다.
Commit 버전 생성 로컬 저장소에 하나의 버전을 만듭니다.
Eclipse 실습 기준

Eclipse에서 버튼을 누를 때마다 “지금 변경이 Working Directory에 있는가, Staging Area에 있는가, Local Repository에 기록되었는가?”를 함께 생각해야 합니다.

08

CLI 명령어와 연결하기

Eclipse에서는 버튼으로 Git을 조작하지만, 실제 개념은 Git CLI 명령어와 연결됩니다. 이 연결을 알아두면 VS Code, IntelliJ, Git Bash에서도 같은 원리로 사용할 수 있습니다.

Eclipse 작업 CLI 명령어 의미
사용자 이름 설정 git config --global user.name "이름" 커밋 작성자 이름을 설정합니다.
사용자 이메일 설정 git config --global user.email "이메일" 커밋 작성자 이메일을 설정합니다.
Share Project / Create Repository git init 현재 폴더를 Git 저장소로 초기화합니다.
Add to Index git add 파일명 변경 파일을 스테이지에 올립니다.
전체 Add to Index git add . 현재 폴더의 변경 파일을 한 번에 스테이지에 올립니다.
Commit git commit -m "메시지" 스테이지된 변경을 로컬 저장소에 기록합니다.
상태 확인 git status 현재 파일 상태를 확인합니다.
History 확인 git log 커밋 이력을 확인합니다.
Push git push origin main 로컬 커밋을 원격 저장소로 업로드합니다.
Pull git pull origin main 원격 변경을 가져와 병합합니다.

처음 기억할 기본 명령어

git init
git add .
git commit -m "Add Git version control lesson"
git status
git log
git push
git pull
09

자주 하는 실수

Git을 처음 배울 때는 명령어보다 개념을 헷갈리는 경우가 많습니다. 아래 실수들은 앞으로 Git을 공부하면서 계속 조심해야 합니다.

1. 저장과 커밋을 같은 것으로 생각

Ctrl + S는 파일 저장이고,
commit은 Git 이력에 하나의 버전을 남기는 작업입니다.

2. Commit과 Push를 같은 것으로 생각

Commit은 내 컴퓨터 로컬 저장소에 기록하는 것이고,
Push는 GitHub 원격 저장소로 업로드하는 것입니다.

3. 파일 이름으로 버전 관리

final, final2, real_final 같은 방식은
최신 파일을 구분하기 어렵고 변경 이유도 남지 않습니다.

4. 모든 변경을 한 커밋에 몰아넣음

로그인, 게시판, README 수정을 한 커밋에 넣으면
나중에 문제 원인을 찾기 어렵습니다.

5. Git과 GitHub를 같은 것으로 이해

Git은 로컬 버전 관리 도구이고,
GitHub는 원격 저장소와 협업 서비스입니다.

6. .git 폴더를 직접 삭제

.git 폴더는 Git 이력을 관리하는 핵심 폴더입니다.
함부로 삭제하면 저장소 정보가 사라질 수 있습니다.

이번 단원은 오류 해결을 깊게 다루는 단계는 아니지만, Git의 기본 개념과 관련된 대표 오류를 미리 알아두면 좋습니다.

Git 오류를 읽는 순서

  1. 현재 내가 올바른 프로젝트 폴더에서 작업 중인지 확인합니다.
  2. 현재 폴더에 .git 폴더가 있는지 확인합니다.
  3. git status로 변경 파일 상태를 확인합니다.
  4. 커밋이 로컬에만 있는지, GitHub까지 올라갔는지 구분합니다.
  5. 원격 저장소에 내가 아직 받지 않은 변경이 있는지 확인합니다.
  6. 충돌 표시가 있다면 사람이 직접 최종 코드를 선택합니다.

12

면접 포인트

Q1. Git은 무엇인가요?

Git은 프로젝트의 변경 이력을 관리하는 분산 버전 관리 도구입니다. 파일을 단순 저장하는 것이 아니라 언제, 누가, 어떤 내용을 수정했는지 커밋 단위로 기록할 수 있습니다.

Q2. GitHub는 무엇인가요?

GitHub는 Git 저장소를 원격 서버에 올려 관리할 수 있게 해주는 서비스입니다. 로컬에서 만든 커밋을 push해서 백업하거나, 팀원이 clone 또는 pull해서 같은 프로젝트를 작업할 수 있습니다.

Q3. Git과 GitHub의 차이는 무엇인가요?

Git은 내 컴퓨터에서 변경 이력을 관리하는 도구이고, GitHub는 그 Git 저장소를 인터넷에 올려 공유하고 협업하게 해주는 플랫폼입니다.

Q4. 버전 관리가 필요한 이유는 무엇인가요?

프로젝트가 커지면 어떤 파일이 언제, 왜 수정되었는지 기억하기 어렵습니다. 버전 관리를 하면 커밋 이력으로 변경 내용을 추적하고, 문제가 생겼을 때 이전 상태를 확인하거나 되돌릴 수 있습니다.

Q5. 파일 백업과 버전 관리는 어떻게 다른가요?

파일 백업은 단순히 복사본을 만드는 방식이고, 버전 관리는 변경 시점, 작성자, 변경 이유와 내용을 이력으로 관리하는 방식입니다.

Q6. Commit과 Push의 차이는 무엇인가요?

Commit은 변경 내용을 로컬 저장소에 하나의 버전으로 기록하는 작업이고, Push는 그 커밋을 GitHub 같은 원격 저장소에 업로드하는 작업입니다.

Q7. 형상 관리란 무엇인가요?

형상 관리는 소스 코드뿐 아니라 설정 파일, 문서, DB 스크립트, 배포 설정 등 프로젝트를 구성하는 전체 요소의 상태를 관리하는 것입니다.

Q8. 좋은 커밋 메시지는 어떤 메시지인가요?

좋은 커밋 메시지는 변경 내용을 짧고 명확하게 설명해야 합니다. update보다 Add Git version control lesson처럼 작업 목적이 드러나는 메시지가 좋습니다.

학습 메모

오늘의 암기 포인트

  • Git은 파일 저장 도구가 아니라 변경 이력 관리 도구입니다.
  • GitHub는 Git 저장소를 원격으로 공유하고 협업하기 위한 서비스입니다.
  • 파일 저장과 커밋은 다릅니다. 저장은 현재 파일을 저장하는 것이고, 커밋은 변경 이력을 하나의 버전으로 기록하는 것입니다.
  • Commit은 로컬 저장소에 기록하는 작업이고, Push는 그 커밋을 GitHub 원격 저장소로 업로드하는 작업입니다.
  • 버전 관리는 언제, 누가, 왜, 무엇을 바꿨는지 기록하는 것입니다.
  • 형상 관리는 소스 코드뿐 아니라 프로젝트 전체 구성 상태를 관리하는 더 넓은 개념입니다.

헷갈리기 쉬운 개념

저장 vs 커밋

저장은 현재 파일을 보존하는 것이고,
커밋은 변경 이력을 Git에 기록하는 것입니다.

백업 vs 버전 관리

백업은 복사본을 만드는 것이고,
버전 관리는 변경 흐름을 기록하는 것입니다.

Git vs GitHub

Git은 로컬 버전 관리 도구이고,
GitHub는 원격 저장소 서비스입니다.

Commit vs Push

Commit은 내 컴퓨터에 기록,
Push는 GitHub에 업로드입니다.

버전 관리 vs 형상 관리

버전 관리는 변경 이력 중심,
형상 관리는 프로젝트 전체 상태 중심입니다.

.git 폴더

Git 이력을 관리하는 핵심 폴더이므로
직접 삭제하지 않는 것이 좋습니다.
