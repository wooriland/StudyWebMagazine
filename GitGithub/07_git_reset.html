Git/GitHub - Chapter 07

Reset으로 이전 커밋 상태로 되돌리기

Git Reset은 현재 브랜치와 HEAD를 과거 Commit으로 이동시키는 기능입니다.
Reset 모드에 따라 Staging Area와 Working Directory까지 함께 바뀔 수 있기 때문에,
실행 전에 어떤 변경을 남기고 어떤 변경을 버릴지 정확하게 판단해야 합니다.
이번 단원의 핵심 질문은 “잘못 만든 Commit을 취소하면서 작업한 코드는 어디까지 남겨야 하는가?” 입니다.

Reset Soft Reset Mixed Reset Hard Reset HEAD Branch Commit ID HEAD~1 Staging Area Index Working Directory History Rewrite Backup Branch reflog ORIG_HEAD Restore Revert non-fast-forward push rejected Force Push
학습 목표 1

Reset이 HEAD, Index,
Working Directory 중 어디까지
변경하는지 설명할 수 있습니다.

학습 목표 2

Eclipse와 CLI에서
soft, mixed, hard Reset을
구분하여 실행할 수 있습니다.

학습 목표 3

Reset 전 안전 점검과
reflog를 이용한 복구 흐름,
공유 History의 위험성을 이해합니다.

01

Reset을 공부한다는 것

Reset은 단순히 이전 파일을 보는 기능이 아닙니다.
현재 브랜치가 가리키는 Commit을 과거로 이동시키고, 선택한 모드에 따라 다음 Commit을 준비하는 Index와 실제 파일이 있는 Working Directory까지 바꾸는 기능입니다.

Move

HEAD와 Branch 이동

현재 브랜치가 가리키는 최신 Commit을
지정한 과거 Commit으로 옮깁니다.

Keep

변경 보존 범위 결정

취소한 Commit의 코드를
Stage 상태, 파일 상태 또는
삭제 상태로 결정합니다.

Rewrite

Local History 재구성

잘못 만든 최근 Commit을 해제하고
더 의미 있는 단위로 다시 만들 수 있습니다.

Risk

데이터 손실 위험

Hard Reset은 실제 파일까지
덮어쓸 수 있으므로 실행 전 점검과
백업이 필요합니다.

핵심 문장

Reset은 취소할 Commit을 지우는 버튼이라기보다, 현재 브랜치가 가리킬 Commit을 다시 선택하는 기능입니다.

02

Reset은 언제 필요할까?

Commit을 만든 뒤 Message가 잘못되었거나, 여러 목적의 파일이 하나의 Commit에 섞였다는 사실을 발견할 수 있습니다.
아직 다른 사람과 공유하지 않은 Local History라면 Reset을 이용해 Commit을 다시 구성할 수 있습니다.

Message

Commit Message 수정

의미 없는 Message를 취소하고
변경 목적이 보이는 Message로
다시 Commit합니다.

Split

큰 Commit 나누기

한 Commit에 섞인 기능, 문서,
스타일 변경을 여러 Commit으로 분리합니다.

Squash

여러 Commit 합치기

최근의 작은 WIP Commit들을
하나의 의미 있는 Commit으로 다시 만듭니다.

Restage

Stage 대상 다시 선택

Commit은 취소하되 파일은 남겨서
Add 과정부터 다시 진행합니다.

Discard

로컬 실험 완전 폐기

필요 없는 실험 Commit과 파일 변경을
모두 대상 Commit 상태로 되돌립니다.

Recover

깔끔한 History 만들기

Push 전 Local History를 정리해
팀원이 읽기 좋은 Commit 흐름을 만듭니다.

잘못 만든 Commit C

update

포함된 변경:
- 07_git_reset.html
- 07_git_reset.css
- README.md
- temporary-note.txt
- 관계없는 네비게이션 CSS
Reset 후 재구성
Commit C-1

Add Git reset lesson

Reset 단원의 HTML과 CSS만 포함합니다.

Commit C-2

Update Git learning roadmap

README 변경은 별도 Commit으로 분리합니다.

05

Soft / Mixed / Hard 전체 비교

세 Reset 모드는 모두 HEAD와 현재 Branch를 대상 Commit으로 이동합니다. 차이는 Index와 Working Directory를 어디까지 함께 이동시키는가에 있습니다.

SOFT

Commit만 취소

  • HEAD 이동
  • Index 유지
  • Working Directory 유지
  • 변경은 Staged 상태
git reset --soft
MIXED

Commit과 Stage 취소

  • HEAD 이동
  • Index를 대상 상태로 변경
  • Working Directory 유지
  • 변경은 Unstaged 상태
git reset --mixed
HARD

Commit과 파일까지 되돌림

  • HEAD 이동
  • Index를 대상 상태로 변경
  • Working Directory도 변경
  • 변경 내용 손실 가능
git reset --hard

06

Soft Reset — Commit만 취소하기

Soft Reset은 HEAD와 현재 Branch만 과거 Commit으로 이동합니다. Index와 Working Directory는 유지되므로 취소된 Commit의 변경은 Staged Changes에 남습니다.

History 
Reset 전

A ── B ── C
              ↑
         HEAD, main

git reset --soft B

Reset 후

A ── B
       ↑
  HEAD, main
Git Staging

Staged Changes

  • 07_git_reset.html 수정
  • 07_git_reset.css 수정

Add를 다시 하지 않고 바로 새로운 Commit을 만들 수 있습니다.

Commit Message 다시 작성

Commit 내용은 유지하면서 Message를 더 명확하게 작성합니다.

빠뜨린 파일 추가

Commit을 취소한 뒤 누락된 파일을 Stage하고
다시 Commit합니다.

여러 Commit 합치기

최근 Commit들의 변경을 Stage 상태로 모아
하나의 Commit으로 만듭니다.

CLI 흐름

git reset --soft HEAD~1

git status

git commit -m "새로운 Commit Message"
Soft Reset 암기

Commit 기록만 취소하고, 취소된 Commit의 내용은 Staged Changes에 남깁니다.

07

Mixed Reset — Commit과 Stage 취소하기

Mixed Reset은 HEAD와 Index를 대상 Commit 상태로 이동시키지만 Working Directory의 실제 파일은 유지합니다. 따라서 취소된 Commit의 변경은 Unstaged Changes에 남습니다.

History 
Reset 전

A ── B ── C
              ↑
         HEAD, main

git reset --mixed B

Reset 후

A ── B
       ↑
  HEAD, main
Git Staging

Unstaged Changes

  • 07_git_reset.html 수정
  • 07_git_reset.css 수정
  • README.md 수정

필요한 파일을 다시 골라 Add to Index부터 진행해야 합니다.

큰 Commit 나누기

하나의 Commit에 섞인 변경을 목적별로 다시 Stage합니다.

관계없는 파일 제외

잘못 포함된 README나 임시 파일을
다음 Commit에서 제외합니다.

Add 과정부터 재검토

어떤 파일이 다음 Commit에 들어갈지 처음부터 다시 선택합니다.

CLI 흐름

git reset --mixed HEAD~1

# --mixed는 기본값이므로 다음과 같음
git reset HEAD~1

git status

git add GitGithub/07_git_reset.html
git add css/GitGithub/07_git_reset.css

git diff --staged
git commit -m "Add Git reset lesson"
Mixed Reset 암기

Commit과 Stage 상태를 취소하지만, 실제 파일 변경은 Unstaged Changes로 남깁니다.

08

Hard Reset — 실제 파일까지 되돌리기

Hard Reset은 HEAD, Index, Working Directory를 모두 대상 Commit 상태로 맞춥니다.
취소된 Commit의 변경과 아직 Commit하지 않은 추적 파일 변경까지 잃을 수 있으므로 가장 주의해야 하는 Reset 모드입니다.

History 
Reset 전

A ── B ── C
              ↑
         HEAD, main

git reset --hard B

Reset 후

A ── B
       ↑
  HEAD, main
Working Directory

Commit B 당시 상태

  • Staged Changes 없음
  • Unstaged Changes 없음
  • 파일 내용이 B 시점으로 변경

Commit C에서 작성한 추적 파일 변경은 Working Directory에서도 사라질 수 있습니다.

Tracked

미Commit 추적 파일 변경

아직 Commit하지 않은 수정 내용이
대상 Commit 상태로 덮어써질 수 있습니다.

Staged

Staged Changes

Index에만 올려둔 변경도
대상 Commit 상태로 되돌아갑니다.

Untracked

방해되는 Untracked 경로

대상 파일을 복원하는 데 방해되는 경로는
제거되거나 덮어써질 수 있습니다.

Commit

대상 이후 Commit

현재 Branch에서 더 이상
도달할 수 없는 상태가 될 수 있습니다.

CLI 흐름

git status
git log --oneline --decorate -5

git branch backup-before-hard-reset

git reset --hard HEAD~1

git status
git log --oneline --decorate -5

Hard Reset은 Ctrl + Z가 아니다

Commit되어 있던 과거 위치는 reflog에서 찾을 가능성이 있지만, 한 번도 Commit하지 않은 Working Directory 변경은 Git History에 존재하지 않습니다.
Hard Reset 전에 반드시 상태 확인과 백업을 진행해야 합니다.

10

CLI 명령어와 연결하기

Eclipse의 Reset 메뉴는 CLI의 git reset 명령과 연결됩니다. 실행 전 상태 확인, 백업 Branch, 실행 후 검증 명령까지 함께 익혀야 합니다.

목적 명령어 의미
현재 상태 확인 git status Staged, Unstaged, Untracked 변경을 확인합니다.
현재 Branch 확인 git branch --show-current Reset할 Branch가 맞는지 확인합니다.
최근 History 확인 git log --oneline --graph --decorate -10 현재 HEAD와 대상 Commit을 확인합니다.
Soft Reset git reset --soft CommitID 변경을 Staged 상태로 유지합니다.
Mixed Reset git reset --mixed CommitID 변경을 Unstaged 상태로 유지합니다.
기본 Mixed Reset git reset CommitID 옵션 생략 시 기본적으로 Mixed입니다.
Hard Reset git reset --hard CommitID Index와 실제 파일까지 대상 Commit 상태로 맞춥니다.
한 Commit 이전 HEAD~1 첫 번째 Parent 방향으로 한 단계 이전입니다.
Backup Branch git branch backup-before-reset Reset 전 Commit 위치를 별도 Branch로 보존합니다.
과거 HEAD 기록 git reflog Reset 전 HEAD 이동 기록을 확인합니다.
Reset 전 위치 git show ORIG_HEAD Reset 전 Branch 끝을 확인합니다.
파일 Unstage git restore --staged 파일명 HEAD 이동 없이 Stage만 해제합니다.
파일 변경 취소 git restore 파일명 특정 파일의 Working Directory 변경을 복원합니다.
공유 Commit 취소 git revert CommitID 취소 내용을 새 Commit으로 기록합니다.

잘못 Reset했을 때 복구하기

Reset 전 Commit은 즉시 완전히 사라지는 것이 아니라, 로컬 reflog에 이전 HEAD 위치가 남아 있을 수 있습니다. 복구가 필요하다면 추가 작업을 멈추고 먼저 reflog를 확인합니다.

git reflog 결과 예시

b71d205 HEAD@{0}: reset: moving to HEAD~2
d92f810 HEAD@{1}: commit: Add hard reset warning
a41c2e7 HEAD@{2}: commit: Add mixed reset example

현재 위치는 b71d205이고, Reset 전 위치는 d92f810임을 확인할 수 있습니다.

Pull을 반복하지 않습니다.

Commit을 찾습니다.

필요 시 원래 위치 복구

검토 후 main을 원래
Commit으로 이동시킵니다.

reflog에서 Commit 찾기

git reflog

복구 Branch 만들기

복구 Branch 확인

복구 가능성이 있음
  • Reset으로 Branch에서 떨어진 Commit
  • 과거 HEAD가 가리키던 Commit
  • Rebase 전 Commit 위치
복구가 보장되지 않음
  • 한 번도 Commit하지 않은 코드
  • 저장하지 않은 편집기 내용
  • Hard Reset으로 버린 미Commit 변경

reflog는 영구 백업이 아니다

reflog는 로컬 Reference의 이동 기록이며 시간이 지나면 정리될 수 있습니다. 중요한 작업은 Backup Branch와 정상적인 Commit으로 보존해야 합니다.

자주 하는 실수

C를 취소하고 B까지 남기려면
C가 아니라 B를 선택해야 합니다.

Commit 단위 Reset은

하나만 없애려고 Reset

대상 이후 모든 Commit이
현재 Branch에서 제외될 수 있습니다.

면접 포인트

Q1. Git Reset이란 무엇인가요?

현재 Branch와 HEAD를 특정 Commit으로 이동시키는 명령입니다. 모드에 따라 Staging Area와 Working Directory까지 대상 Commit 상태로 바꿀 수 있습니다.

Q2. Reset 대상 Commit은 어떻게 선택하나요?

취소할 Commit이 아니라 Reset 후 남길 마지막 Commit을 선택합니다. C를 취소하고 B까지 유지하려면 B를 대상으로 선택합니다.

Q3. Soft Reset은 무엇인가요?

HEAD와 현재 Branch만 이동시키고 Index와 Working Directory는 유지합니다. 취소된 Commit의 변경은 Staged 상태로 남습니다.

Q4. Mixed Reset은 무엇인가요?

HEAD와 Index를 대상 Commit 상태로 이동시키지만 Working Directory는 유지합니다. 변경은 Unstaged 상태로 남습니다.

Q5. Hard Reset은 무엇인가요?

HEAD, Index, Working Directory를 모두 대상 Commit 상태로 맞춥니다. 미Commit 변경까지 사라질 수 있어 위험합니다.

Q6. git reset의 기본 모드는 무엇인가요?

Mixed입니다. 따라서 git reset HEAD~1은 일반적으로 git reset --mixed HEAD~1과 같은 의미입니다.

Q7. Soft, Mixed, Hard를 한 문장씩 설명해보세요.

Soft는 Commit만 취소하고 Stage를 유지합니다. Mixed는 Commit과 Stage를 취소하고 파일을 유지합니다. Hard는 Commit, Stage와 파일 변경까지 대상 상태로 되돌립니다.

Q8. Reset 이후 Commit은 즉시 완전히 삭제되나요?

현재 Branch에서 도달할 수 없게 되어 일반 History에서 보이지 않을 수 있지만, reflog나 다른 Branch가 해당 Commit을 가리킬 수 있습니다.

Q9. HEAD~1은 무엇인가요?

현재 HEAD에서 첫 번째 Parent 방향으로 한 단계 이전 Commit을 의미합니다.

Q10. Merge Commit에서 HEAD~1은 무엇인가요?

Merge Commit의 첫 번째 Parent 방향으로 한 단계 이전 Commit입니다. 복잡한 History에서는 정확한 Commit ID를 사용하는 편이 안전합니다.

Q11. Reset과 Restore의 차이는 무엇인가요?

Reset은 Commit 단위로 사용하면 Branch와 HEAD를 이동시킵니다. Restore는 파일이나 Index를 복원하며 Branch History를 이동시키지 않습니다.

Q12. Reset과 Revert의 차이는 무엇인가요?

Reset은 Branch를 과거로 이동하여 History를 재구성합니다. Revert는 특정 Commit의 변경을 취소하는 새 Commit을 만듭니다.

Q13. Push한 Commit에는 왜 Revert가 안전한가요?

기존 History를 삭제하지 않고 취소용 새 Commit을 추가하므로 Remote와 팀원이 가진 History 연결을 유지할 수 있습니다.

Q14. Reset 후 Push가 rejected되는 이유는 무엇인가요?

Remote는 Reset 전의 최신 Commit을 가리키지만 Local은 과거 Commit으로 이동하여 non-fast-forward 상태가 되었기 때문입니다.

Q15. 바로 Force Push하면 안 되는 이유는 무엇인가요?

Remote History를 덮어써 팀원의 Commit이나 그 Commit을 기반으로 한 작업을 제거할 수 있기 때문입니다.

Q16. 잘못 Reset했을 때 어떻게 복구하나요?

git reflog에서 Reset 전 HEAD 위치를 찾고, 먼저 해당 Commit을 가리키는 복구 Branch를 만듭니다.

Q17. reflog는 무엇인가요?

로컬에서 HEAD와 Branch 같은 Reference가 언제 어느 Commit으로 이동했는지 기록한 로그입니다.

Q18. reflog가 모든 파일을 복구하나요?

아닙니다. reflog는 Commit 위치를 기록합니다. 한 번도 Commit하지 않은 Working Directory 변경은 복구되지 않을 수 있습니다.

Q19. ORIG_HEAD란 무엇인가요?

Reset이나 Merge 같은 일부 작업 전에 기존 Branch 끝을 임시로 기록하는 Reference입니다.

Q20. Backup Branch를 만드는 이유는 무엇인가요?

Reset 전 최신 Commit을 계속 가리키게 하여 Reset 이후에도 기존 History에 쉽게 접근하기 위해서입니다.

Q21. Commit 단위 Reset은 파일 하나에만 적용되나요?

아닙니다. 현재 Repository의 Branch, Index와 Working Directory에 영향을 줄 수 있습니다. 특정 파일만 되돌리려면 Restore를 사용합니다.

Q22. Mixed Reset은 언제 유용한가요?

잘못 구성된 Commit을 취소하고 파일은 유지하면서 어떤 파일을 다시 Stage할지 처음부터 선택하고 싶을 때 유용합니다.

Q23. Soft Reset은 언제 유용한가요?

Stage 구성을 유지한 채 Message를 다시 작성하거나 최근 여러 Commit을 하나로 합쳐 다시 Commit할 때 유용합니다.

Q24. Hard Reset 전 무엇을 확인해야 하나요?

현재 Repository와 Branch, Staged·Unstaged 변경, 대상 Commit, Push 여부와 Backup Branch를 확인해야 합니다.

학습 메모

오늘의 암기 포인트

  • Reset은 현재 Branch와 HEAD를 특정 Commit으로 이동합니다.
  • Reset에서는 취소할 Commit이 아니라 남길 마지막 Commit을 선택합니다.
  • Soft는 HEAD만 이동하고 변경을 Staged 상태로 남깁니다.
  • Mixed는 HEAD와 Index를 이동하고 변경을 Unstaged 상태로 남깁니다.
  • Mixed는 git reset의 기본 모드입니다.
  • Hard는 HEAD, Index, Working Directory를 대상 Commit 상태로 맞춥니다.
  • Hard Reset은 미Commit 변경을 잃게 할 수 있습니다.
  • Commit 단위 Reset은 Repository 전체에 영향을 줄 수 있습니다.
  • Reset 전 Commit은 reflog에서 찾을 가능성이 있습니다.
  • reflog는 미Commit 파일 변경을 보존하는 백업 기능이 아닙니다.
  • 공유 전 Local History는 Reset으로 정리할 수 있습니다.
  • Push된 공유 History에는 Revert를 우선 고려합니다.

헷갈리기 쉬운 개념

취소할 Commit vs 남길 Commit

Reset에서는 취소할 Commit이 아니라
Reset 후 Branch가 가리킬 Commit을 선택합니다.

Soft vs Mixed

Soft는 Stage 유지, Mixed는 Stage 해제입니다.

Mixed vs Hard

Mixed는 파일을 유지하고,
Hard는 파일까지 대상 상태로 바꿉니다.

Reset vs Restore

Reset은 History를 이동하고, Restore는 파일을 복원합니다.

Reset vs Revert

Reset은 Branch를 과거로 이동하고,
Revert는 취소용 새 Commit을 만듭니다.

reflog vs 파일 백업

reflog는 Commit 위치 기록이며
미Commit 파일 백업은 아닙니다.

main vs origin/main

Reset 후 Local main과 Remote origin/main이
다른 Commit을 가리킬 수 있습니다.

rejected vs Commit 실패

Reset 후 Push rejected는
Local Commit 생성 실패가 아니라 History 차이 문제입니다.

Hard Reset vs git clean

Hard Reset은 모든 Untracked 파일을 정리하는
전용 명령이 아닙니다.

Hard Reset 전 최종 체크

복구용 빠른 명령어

git reflog

git branch recovery-before-reset CommitID

git switch recovery-before-reset

git show ORIG_HEAD

잘못 Reset했다면 추가 작업을 멈추고 먼저 이전 Commit을 가리키는 복구 Branch를 만드는 것이 안전합니다.
