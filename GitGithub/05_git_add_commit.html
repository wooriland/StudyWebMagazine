Git/GitHub - Chapter 05

Add / Staging / Commit 기본 흐름

Add / Staging / Commit 기본 흐름은 Working Directory에서 수정한 파일을 골라 Staging Area에 등록하고,
하나의 의미 있는 Commit으로 Local Repository에 기록하는 방법을 배우는 단원입니다.
Git은 작업 중인 모든 변경을 무조건 한 번에 저장하지 않습니다.
현재 변경 가운데 다음 버전에 포함할 내용을 Add로 직접 선택하고, 그 결과를 Commit으로 확정합니다.
이번 단원의 핵심 질문은 “현재 수정한 내용 중 이번 버전에 무엇을 포함할 것인가?”입니다.

Working Directory Untracked Modified Add Add to Index Staging Area Index Staged Changes Unstaged Changes Commit Commit Message Local Repository Snapshot git status git diff git diff --staged Unstage Commit and Push
학습 목표 1

파일 저장, Add, Commit,
Push가 서로 다른 작업임을 구분할 수 있습니다.

학습 목표 2

Untracked, Modified, Staged,
Committed 상태의 변화 흐름을 설명할 수 있습니다.

학습 목표 3

Eclipse Git Staging 탭에서
필요한 변경만 선택해 의미 있는 Commit을 만들 수 있습니다.

01

이번 단원은 단순히 git add와 git commit 명령어를 외우는 단원이 아닙니다. 작업 중인 여러 변경 가운데 서로 관련된 변경만 골라 하나의 설명 가능한 버전으로 만드는 방법을 배우는 단원입니다.

02

파일 저장부터 GitHub까지의 전체 흐름

파일 저장, Add, Commit, Push는 각각 다른 공간에 영향을 주는 별개의 작업입니다.
이 네 단계를 구분하지 못하면 “Commit했는데 왜 GitHub에 안 보이지?”, “Add했는데 왜 History가 없지?” 같은 혼란이 생깁니다.

1 파일 저장

편집기에서 변경한 내용을
Working Directory에 저장합니다.

Ctrl + S
2 Add

다음 Commit에 포함할 내용을
Staging Area에 등록합니다.

git add
3 Commit

Stage된 내용을 Local Repository에
하나의 버전으로 기록합니다.

git commit
4 Push

로컬에 만들어진 Commit을
GitHub 원격 저장소로 전송합니다.

git push

  ↓ git add

  ↓ git commit

  ↓ git push

Remote Repository / GitHub

git add Staging Area 아니요 아니요

03

Staging Area는 왜 필요할까?

개발자는 한 번에 하나의 파일만 수정하지 않습니다. 여러 파일을 동시에 수정하더라도 각각의 변경 목적은 서로 다를 수 있습니다.

05

Add는 실행한 시점의 내용을 Stage한다

git add는 파일 이름만 등록하는 것이 아닙니다. Add를 실행한 시점의 파일 내용을 Staging Area에 저장합니다.
따라서 Add한 뒤 파일을 다시 수정하면 같은 파일의 두 상태가 동시에 존재할 수 있습니다.

1

파일 작성

2

git add hello.txt
3

파일 다시 수정

Hello
Git
4

Commit 실행

Commit 포함:
Hello

남은 변경:
Git

Staging Area의 hello.txt

Hello

Add를 실행한 시점의 내용입니다. 현재 Commit에 포함될 내용입니다.

Working Directory의 hello.txt

Hello
Git

Add 이후 다시 수정한 최신 내용입니다. 추가된 Git 줄은 아직 Unstaged입니다.

중요한 원칙

한 번 Add한 파일을 이후 다시 수정해도 최신 내용이 자동으로 Stage되지는 않습니다. 최신 수정까지 Commit하려면 파일을 다시 Add해야 합니다.

07

Eclipse에서 첫 Commit 만드는 전체 흐름

저장소 생성 직후 파일에 ? 표시가 있다면 아직 Git이 추적하지 않는 Untracked 상태입니다. 필요한 파일을 Add to Index한 뒤 Staged Changes를 확인하고 Commit합니다.

1 파일 작성·수정

프로젝트 파일을 만들거나 수정한 뒤 저장합니다.

2 상태 확인

파일의 ? 또는 변경 장식과 Unstaged Changes를 확인합니다.

3 Add 대상 결정

이번 Commit에 넣을 파일과 제외할 파일을 구분합니다.

4 Add to Index

프로젝트 또는 특정 파일을 Stage합니다.

5 Staged 확인

Staged Changes에 원하는 파일만 있는지 확인합니다.

6 Diff 검토

파일을 열어 실제 변경 줄과 내용을 확인합니다.

7 Message 작성

이번 변경 목적이 드러나는 Commit Message를 작성합니다.

8 Commit

Staged Changes를 Local Repository에 기록합니다.

9 결과 확인

Git Staging과 History에서 새 Commit을 확인합니다.

  → Team
  → Add to Index

프로젝트 내 변경이 모두 같은 목적일 때 사용합니다.

특정 파일 Stage

특정 파일 우클릭
  → Team
  → Add to Index

필요한 파일만 정확히 Commit하고 싶을 때 사용합니다.

프로젝트 우클릭
  → Team
  → Show In History

08

CLI 명령어와 연결하기

Eclipse의 Add to Index와 Commit은 Git CLI 명령어와 직접 연결됩니다. 명령어를 함께 이해하면 VS Code, IntelliJ, Git Bash에서도 같은 원리로 작업할 수 있습니다.

목적 CLI 명령어 의미
현재 상태 확인 git status Untracked, Modified, Staged 상태를 확인합니다.
특정 파일 Stage git add 파일명 특정 파일의 현재 내용을 Index에 등록합니다.
특정 폴더 Stage git add 폴더명 지정한 폴더 범위의 변경을 Stage합니다.
현재 경로 Stage git add . 현재 경로와 그 하위의 변경을 Stage합니다.
저장소 전체 변경 Stage git add -A 추가, 수정, 삭제 변경을 저장소 전체 범위에 반영합니다.
Unstaged 변경 확인 git diff 아직 Stage하지 않은 실제 변경 내용을 확인합니다.
Staged 변경 확인 git diff --staged 다음 Commit에 실제로 포함될 변경을 확인합니다.
Unstage git restore --staged 파일명 파일 내용은 유지하고 Commit 대상에서 제외합니다.
Commit 생성 git commit -m "메시지" Stage된 내용을 Local Repository에 기록합니다.
간단한 History git log --oneline Commit ID와 메시지를 한 줄씩 확인합니다.

이번 단원 파일만 Commit

git status

git add GitGithub/05_git_add_commit.html
git add css/GitGithub/05_git_add_commit.css

git diff --staged

git commit -m "Add Git add and commit lesson"

Commit 결과 확인

git status
git log --oneline

Stage하지 않은 README나 임시 파일은 Working Directory에 그대로 남을 수 있습니다.

git status 결과 읽기

  new file:   05_git_add_commit.html
  new file:   05_git_add_commit.css

Unstage, 부분 Stage, 삭제와 이름 변경

Commit 전에 잘못 Stage한 파일을 뺄 수도 있고, 한 파일의 일부 변경만 선택하거나 파일 삭제와 이름 변경도 하나의 Git 변경으로 기록할 수 있습니다.

Unstage

잘못 Stage한 파일 빼기

git restore --staged README.md

실제 파일은 유지하면서
다음 Commit 대상에서만 제외합니다.

Partial Stage

일부 변경만 Stage

git add -p App.java

한 파일 안의 여러 변경을
서로 다른 Commit으로 분리할 수 있습니다.

Delete

파일 삭제 기록

git rm old-file.txt

git commit -m "Remove obsolete file"

추적 중인 파일의 삭제도
Stage한 뒤 Commit할 수 있습니다.

Rename

파일 이름 변경

git mv old-name.html new-name.html

파일 이동이나 이름 변경도
하나의 변경으로 기록할 수 있습니다.

Unstage와 Delete의 차이

Unstage

실제 파일은 유지하고
Commit 대상에서만 제외합니다.

Delete

Working Directory의
실제 파일을 삭제합니다.

첫 Commit 전 주의

첫 Commit 전에는 기준이 되는 HEAD가 아직 없기 때문에
git restore --staged가 상황에 따라 제한될 수 있습니다.
Eclipse Git Staging에서 Unstage하거나
git rm --cached 파일명을 사용할 수 있습니다.

좋은 Commit 단위와 Commit Message

Git은 Commit을 자동으로 의미 있게 만들어주지 않습니다. 개발자가 관련 변경만 Stage하고 목적이 드러나는 메시지를 작성해야 나중에 읽기 쉬운 History가 만들어집니다.

fix
수정
최종
진짜최종
test

메시지만 보고는 어떤 작업인지 알 수 없습니다.

Good

목적이 보이는 메시지

Remove unused test files

메시지만 읽어도 변경 목적을 파악할 수 있습니다.

Add Git add and commit lesson

- Explain file state lifecycle
- Add Eclipse staging workflow
- Add CLI command examples

첫 줄은 요약하고, 필요하면 빈 줄 아래에 상세 내용을 작성합니다.

좋은 Commit 단위

  • 하나의 논리적인 작업 목적을 가집니다.
  • 관련 없는 변경이 섞이지 않습니다.
  • 독립적으로 검토할 수 있습니다.
  • 필요하면 그 Commit만 되돌릴 수 있습니다.

Commit 전 확인 질문

  • 이번 Commit의 목적을 한 문장으로 말할 수 있는가?
  • 원하는 파일만 Stage되었는가?
  • 코드가 의미 있는 상태로 동작하는가?
  • 민감 정보가 포함되지 않았는가?
좋은 History의 시작

좋은 Commit은 많은 파일을 한꺼번에 저장하는 Commit이 아니라, 하나의 변경 목적을 분명하게 설명할 수 있는 Commit입니다.

12

Commit과 Commit and Push 구분하기

Eclipse Git Staging에는 Commit과 Commit and Push가 함께 보일 수 있습니다. 두 버튼은 최종 도착 위치가 다릅니다.

Local Only

Commit

Staging Area
  ↓ Commit
Local Repository
  • 내 컴퓨터에 버전을 만듭니다.
  • 인터넷 연결 없이도 가능합니다.
  • GitHub에는 아직 나타나지 않습니다.
  • 작성자 정보와 Commit Message가 기록됩니다.
Local + Remote

Commit and Push

Staging Area
  ↓ Commit
Local Repository
  ↓ Push
Remote Repository
  • 먼저 로컬 Commit을 만듭니다.
  • 그 Commit을 GitHub에 전송합니다.
  • 원격 저장소와 인증 설정이 필요합니다.
  • 원격 상태에 따라 Push가 거절될 수 있습니다.

초보 학습 기준

처음에는 Commit과 Push를 분리해서 수행하는 것이 좋습니다.
먼저 로컬 Commit이 정상적으로 생성됐는지 확인한 뒤 Push하면, 오류가 로컬 단계인지 원격 단계인지 더 쉽게 구분할 수 있습니다.

자주 하는 실수

Add와 Commit의 역할을 구분하지 못하거나 Staged Changes를 검토하지 않으면 불필요한 파일과 관련 없는 변경이 하나의 Commit에 섞일 수 있습니다.

1. 파일 저장을 Commit으로 착각

Ctrl + S는 Working Directory 저장이고,
Commit은 Local Repository
버전 생성입니다.

2. Add를 Commit으로 착각

Add는 Staging Area 등록일 뿐
아직 Local Repository에
버전이 생기지 않습니다.

3. Commit을 Push로 착각

Commit은 로컬에 기록하고
GitHub 업로드는 Push가 담당합니다.

4. Add 후 재수정하고
다시 Add하지 않음

Add 이후의 최신 수정은
Unstaged로 남으므로
최신 상태까지 넣으려면 다시 Add해야 합니다.

5. 같은 파일이
두 영역에 보여 오류라고 생각

Staged 버전과 Add 이후 추가 수정이
함께 존재하면 두 영역에
동시에 표시될 수 있습니다.

6. 항상 git add . 사용

임시 파일, 로그, 빌드 결과와
민감 정보가 함께 Stage될 수 있습니다.

7. Staged Diff를 확인하지 않음

의도하지 않은 코드와 파일 삭제가
Commit에 포함될 수 있습니다.

8. Unstaged Changes도
Commit된다고 생각

기본 Commit은 Staged Changes만 기록하고
Unstaged 변경은
Working Directory에 남습니다.

9. 잘못 Stage한 파일을 삭제함

Commit 대상에서 빼려면
Delete가 아니라 Unstage를 해야 합니다.

10. 관련 없는 변경을
한 Commit에 포함

기능 추가, 문서 수정, 오류 수정이 섞이면
이력 추적과 되돌리기가 어려워집니다.

11. Commit Message를
update로 작성

메시지만 보고 작업 내용을 알 수 있도록
구체적인 목적을 작성해야 합니다.

12. .gitignore를 확인하지 않음

node_modules, target, .env 같은 파일이
실수로 Commit될 수 있습니다.

13. 이미 추적 중인 파일이
자동 제외된다고 생각

이미 Tracked인 파일은 .gitignore만으로
자동 추적 해제되지 않습니다.

14. Commit Message에
민감 정보 작성

Commit Message도 History에 남으므로
Password와 Token을 작성하면 안 됩니다.

15. 테스트하지 않고 Commit

문법 오류, 실행 오류, 디버그 코드가
그대로 이력에 남을 수 있습니다.

16. 잘못된 브랜치에서 Commit

Commit은
현재 체크아웃된 브랜치에 기록되므로
Commit 전에 브랜치를 확인해야 합니다.

면접 포인트

Q1. git add는 무엇을 하나요?

git add는 Working Directory의 현재 변경 내용을 Staging Area 또는 Index에 등록합니다. 즉 다음 Commit에 포함할 변경을 선택하는 작업입니다.

Q2. Staging Area는 왜 필요한가요?

작업 중인 모든 변경이 항상 같은 목적을 가지는 것은 아니기 때문입니다. 여러 변경 중 이번 Commit에 포함할 변경만 선택해 기능 단위의 명확한 Commit을 만들 수 있습니다.

Q3. Index와 Staging Area는 같은 뜻인가요?

기본 Git 학습에서는 같은 개념으로 이해할 수 있습니다. 다음 Commit에 포함될 파일 내용의 상태를 준비하고 관리하는 영역입니다.

Q4. Add와 Commit의 차이는 무엇인가요?

Add는 Working Directory의 변경을 Staging Area에 등록하는 작업이고, Commit은 Stage된 내용을 Local Repository에 하나의 버전으로 기록하는 작업입니다.

Q5. Commit과 Push의 차이는 무엇인가요?

Commit은 Local Repository에 변경 이력을 기록하고, Push는 로컬에 만들어진 Commit을 GitHub 같은 Remote Repository로 전송합니다.

Q6. 파일을 Add한 뒤 다시 수정하면 어떻게 되나요?

Add 당시 내용은 Staging Area에 남고, 이후 추가 수정은 Working Directory의 Unstaged 변경으로 남습니다. 최신 수정까지 Commit하려면 다시 Add해야 합니다.

Q7. 같은 파일이 Staged와 Unstaged에 동시에 표시될 수 있나요?

가능합니다. Add한 후 파일을 다시 수정하면 Add 당시 내용은 Staged Changes에, 이후 변경은 Unstaged Changes에 나타납니다.

Q8. Commit은 Working Directory의 모든 변경을 기록하나요?

아닙니다. 기본 Commit은 Staging Area에 준비된 변경만 기록합니다. Unstaged Changes와 Untracked 파일은 Add하지 않으면 포함되지 않습니다.

Q9. Commit은 무엇을 저장하나요?

개념적으로 현재 Staging Area가 나타내는 프로젝트 상태의 Snapshot을 저장합니다. 여기에 Commit Message, 작성자, 날짜, 부모 Commit 정보가 연결됩니다.

Q10. git status는 왜 사용하나요?

파일들이 Untracked, Modified, Staged 중 어떤 상태인지 확인하고, 다음 Commit에 무엇이 들어가는지 판단하기 위해 사용합니다.

Q11. git diff와 git diff --staged의 차이는 무엇인가요?

git diff는 아직 Stage하지 않은 변경을 보여주고, git diff --staged는 다음 Commit에 실제로 포함될 Stage된 변경을 보여줍니다.

Q12. 잘못 Stage한 파일은 어떻게 빼나요?

Eclipse에서는 Remove from Git Index 또는 Unstaged Changes로 이동할 수 있습니다. CLI에서는 일반적으로 git restore --staged 파일명을 사용합니다.

Q13. 좋은 Commit 단위란 무엇인가요?

하나의 논리적인 작업 목적을 가지며, 관련 없는 변경이 섞이지 않고 독립적으로 검토하거나 되돌릴 수 있는 Commit입니다.

Q14. 좋은 Commit Message의 기준은 무엇인가요?

메시지만 읽어도 무엇을 변경했는지 알 수 있어야 합니다. update보다 Add Git staging workflow lesson처럼 구체적인 메시지가 좋습니다.

Q15. Untracked 파일은 Commit에 자동으로 포함되나요?

아닙니다. Untracked 파일은 먼저 git add를 통해 Staging Area에 올려야 Commit에 포함됩니다.

Q16. .gitignore에 추가하면 이미 Commit된 파일도 자동 제외되나요?

아닙니다. .gitignore는 주로 아직 추적되지 않는 파일에 적용됩니다. 이미 추적 중인 파일은 별도의 추적 해제 작업이 필요합니다.

Q17. git commit -am은 어떤 명령인가요?

이미 추적 중인 파일의 수정과 삭제를 Stage한 뒤 Commit하는 단축 명령입니다. 새 Untracked 파일은 포함하지 않으며, 처음에는 Add와 Commit을 분리해서 익히는 것이 좋습니다.

Q18. Commit 후에도 git status에 변경이 남을 수 있나요?

가능합니다. Commit은 Staged Changes만 기록하므로, Stage하지 않은 Modified 파일이나 Untracked 파일은 Working Directory에 남을 수 있습니다.

오늘의 암기 포인트

  • 파일 저장, Add, Commit, Push는 모두 서로 다른 작업입니다.
  • Add는 Working Directory의 현재 내용을 Staging Area에 등록합니다.
  • Staging Area 또는 Index는 다음 Commit에 들어갈 내용을 준비하는 공간입니다.
  • Commit은 Staging Area의 내용을 Local Repository에 하나의 버전으로 기록합니다.
  • Commit은 Working Directory의 모든 변경을 자동으로 기록하지 않습니다.
  • Untracked 파일은 Add하지 않으면 Commit에 포함되지 않습니다.
  • Add 후 파일을 다시 수정하면 최신 수정까지 포함하기 위해 다시 Add해야 합니다.
  • 같은 파일이 Staged Changes와 Unstaged Changes에 동시에 나타날 수 있습니다.
  • git diff --staged는 다음 Commit에 실제로 들어갈 변경을 보여줍니다.
  • Commit은 로컬 저장소에 기록하고 Push는 원격 저장소에 전송합니다.

헷갈리기 쉬운 개념

파일 저장 vs Commit

파일 저장은 Working Directory에 반영하고,
Commit은 Local Repository에 버전을 만듭니다.

Add vs Commit

Add는 Commit 대상 선택이고,
Commit은 선택한 내용을 버전으로 확정하는 작업입니다.

Commit vs Push

Commit은 내 컴퓨터에 기록하고,
Push는 GitHub에 전송합니다.

Staged vs Unstaged

Staged는 다음 Commit 대상이고,
Unstaged는 아직 선택하지 않은 변경입니다.

Unstage vs Delete

Unstage는 Commit 대상에서만 제외하고,
Delete는 실제 파일을 삭제합니다.

.gitignore vs Tracked

.gitignore는 이미 추적 중인 파일을
자동으로 추적 해제하지 않습니다.
