Git/GitHub - Chapter 06

커밋 히스토리와 커밋 메시지

커밋 히스토리와 커밋 메시지는 지금까지 만든 Commit을 단순히 쌓아두는 데서 끝내지 않고,
프로젝트가 언제, 누구에 의해, 어떤 이유로 변화했는지 읽는 단원입니다.
Git History에서는 Commit ID, Parent Commit, HEAD, Branch, Author, Commit Message, Changed Files, Diff를 확인할 수 있습니다.
이번 단원의 핵심 질문은 “나와 팀원이 나중에 다시 보아도 이 변경의 목적과 흐름을 이해할 수 있는가?” 입니다.

Eclipse Show In History와

01

Git History를 공부한다는 것

Commit을 만드는 것만으로는 충분하지 않습니다.
프로젝트가 커질수록 과거의 Commit을 읽고, 문제가 생긴 시점과 변경 목적을 찾는 능력이 중요해집니다.
Git History는 프로젝트가 변화해 온 과정을 보여주는 기록입니다.

When

언제 바뀌었는가?

Commit 날짜와 순서를 통해
변경이 만들어진 시점을 확인합니다.

Who

누가 바꾸었는가?

Author와 Committer 정보를 통해
변경 작성자와 기록자를 확인합니다.

Why

왜 바꾸었는가?

Commit Message를 통해
변경 목적과 배경을 파악합니다.

What

무엇이 바뀌었는가?

Changed Files와 Diff를 통해
실제 추가·수정·삭제 내용을 확인합니다.

핵심 문장

Git History는 단순한 백업 목록이 아니라, 프로젝트가 왜, 어떻게 변화했는지 보여주는 작업 기록입니다.

02

History는 언제 필요할까?

현재 코드만 보면 지금 어떤 상태인지는 알 수 있지만, 그 코드가 왜 만들어졌는지, 언제부터 문제가 생겼는지는 알기 어렵습니다.
History는 현재 코드가 만들어진 과정을 역순으로 추적할 수 있게 해줍니다.

Bug

오류가 시작된 시점 찾기

최근 Commit을 비교하여
어느 변경 이후 문제가 시작되었는지 찾습니다.

Reason

코드 변경 이유 확인

Commit Message와 Diff를 함께 보며
특정 코드가 추가된 이유를 확인합니다.

Owner

변경 담당자 확인

Author 정보를 통해 변경 의도를 물어볼 사람을 찾습니다.

Recovery

되돌릴 버전 선택

Commit ID와 변경 내용을 확인한 뒤
Reset 또는 Revert 대상을 선택합니다.

Review

코드 리뷰 범위 파악

Commit별 변경 파일과 Diff를 확인하여
검토 범위를 명확하게 만듭니다.

Story

프로젝트 발전 과정 설명

Commit 순서를 통해 기능이 어떤 단계로
추가되었는지 설명할 수 있습니다.

문제 상황

네비게이션 메뉴가 두 개씩 활성화된다

현재 코드만 보아서는 언제부터
경로 비교 방식이 바뀌었는지 알기 어렵습니다.

History 확인 
7a201fc Update Git lesson navigation
1fa922e Add Git history lesson
f21d80a Add Git staging lesson
Diff 확인 
- if (currentPath === item.path) {
+ if (currentPath.includes(item.path)) {

가장 최근 Commit에서 경로 비교 방식이 바뀐 것을 확인할 수 있습니다.

Commit ID와 짧은 해시

Commit Message는 중복될 수 있지만 Commit ID는 특정 Commit을 정확하게 구분하는 식별자입니다. Reset, Revert, Show, Diff 등에서 대상 Commit을 지정할 때 사용합니다.

06

Parent, Root, Merge Commit과 HEAD

History는 Commit들이 Parent 관계로 연결된 구조입니다. 일반 Commit, 첫 Commit, Merge Commit은 Parent 개수가 다를 수 있습니다.

Root Commit

Parent가 없는 첫 Commit

A
↑
Initial Commit

저장소의 첫 Commit이므로 이전 Commit이 없습니다.

Normal Commit

Parent가 하나인 일반 Commit

A → B → C

C의 Parent는 B이고, B의 Parent는 A입니다.

Merge Commit

Parent가 둘 이상인 Commit

A ─ B ─ C ─ M
     \       /
      D ─ E

M은 C와 E 두 작업 흐름을 합친 Commit입니다.

HEAD ↓ main ↓ Commit C

HEAD와 Branch를 구분하기

  • Commit은 특정 시점의 Snapshot입니다.
  • Branch는 특정 Commit을 가리키는 움직이는 이름입니다.
  • HEAD는 현재 체크아웃한 작업 위치를 나타냅니다.
  • 새 Commit이 생기면 현재 Branch가 새 Commit으로 이동합니다.
HEAD → main

현재 main Branch에서 작업 중입니다.

feature/login

로컬 기능 개발 Branch입니다.

origin/main

마지막 Fetch를 기준으로 로컬이 기억하는
원격 main의 상태입니다.

tag: v1.0.0

특정 Commit에 붙인 버전 이름입니다.

09

CLI에서 History 확인하기 — git log

Eclipse의 Show In History는 CLI의 git log와 연결됩니다. 옵션을 조합하면 Commit 목록, Graph, Branch와 변경 통계를 원하는 형태로 볼 수 있습니다.

목적 명령어 확인할 내용
기본 History git log 전체 ID, Author, Date, Message
한 줄 History git log --oneline 축약 ID와 Message
Graph 포함 git log --oneline --graph Parent와 Branch 관계
Branch 장식 git log --decorate HEAD, Branch, Tag

실제 Patch git log -p Commit별 실제 코드 Diff

11

Local History와 Remote History 비교

Commit은 먼저 Local Repository에 만들어집니다.
Push해야 GitHub의 Remote History에 반영됩니다.
반대로 다른 사람이 Push한 Commit은 Fetch 또는 Pull해야 로컬에서 확인할 수 있습니다.

Same

Local과 Remote가 같은 상태

Local
A → B → C

Remote
A → B → C

main과 origin/main이
같은 Commit을 가리킵니다.

Ahead

Local이 앞선 상태

Local
A → B → C → D

Remote
A → B → C

D는 아직 Push되지 않은
Local Commit입니다.

Behind

Remote가 앞선 상태

Local
A → B → C

Remote
A → B → C → D

원격의 D를 아직 Fetch 또는
Pull하지 않았습니다.

Diverged

History가 갈라진 상태

          D → E  Local
         /
A → B → C
         \
          F → G  Remote

Local과 Remote 양쪽에
서로 다른 고유 Commit이 있습니다.

아직 Push하지 않은 Local Commit

git log --oneline origin/main..main

Local에 없는 Remote Commit

git log --oneline main..origin/main

origin/main은 GitHub의 실시간 화면이 아니다

origin/main은 마지막 Fetch 또는 Pull을 기준으로 로컬 Git이 기억하고 있는 원격 main의 상태입니다. 최신 원격 상태를 확인하려면 먼저 Fetch가 필요할 수 있습니다.

docs: 인증 설정 설명 보완

의미 있는 단위로 Commit하기

하나의 논리적인 작업 목적을 가진 Commit을 흔히 Atomic Commit이라고 합니다.
파일 개수가 아니라 변경 목적, Review 가능성, 되돌리기 가능성을 기준으로 판단합니다.

하나의 목적

Commit 하나를 한 문장으로

필요하다면 해당 Commit만
취소할 수 있어야 합니다.

Works

의미 있는 상태

가능하면 빌드, 실행 또는

부분 Staging을 사용해 버그 수정과 리팩터링을 서로 다른 Commit으로 분리할 수 있습니다.

Too Large

좋은 Version History를 만드는 습관

좋은 History는 Commit 버튼을 누르는 순간만 관리하는 것이 아닙니다. 현재 Branch 확인, 테스트, 선택적 Stage, Staged Diff 검토와 민감 정보 확인까지 하나의 반복 루틴으로 만들어야 합니다.

현재 코드가 의미 있게 동작하는지 확인합니다.

실제 Commit에 들어갈 코드를 검토합니다.

Token, 비밀번호, .env 포함 여부를 확인합니다.

8 Commit Message

변경 목적이 보이는 Message를 작성합니다.

9 History 재확인

Commit 후 status, log와 Diff를 확인합니다.

git diff

Commit 후

git status

git show HEAD

Amend와 공유된 History 주의사항

마지막 Commit의 Message를 잘못 작성했다면 git commit --amend로 수정할 수 있습니다. 하지만 Amend는 기존 Commit을 직접 고치는 것이 아니라 새 Commit으로 대체합니다.

Before

기존 Commit

a41c2e7

Add page

→ Amend →
After

새 Commit

d92f810

Add Git history and commit message lesson

git commit --amend -m "Add Git history and commit message lesson"
Local Only

아직 Push하지 않은 마지막 Commit

혼자 사용하는 Local History라면
Amend를 이용해 Message나 포함 내용을 정리할 수 있습니다.

Shared

이미 Push해 공유된 Commit

ID가 바뀌면 Remote와 팀원의 History가 달라질 수 있으므로
함부로 Amend하지 않습니다.

History를 다시 작성할 수 있는 작업

commit --amend reset rebase force push

안전 기준

아직 공유하지 않은 Local History는 필요에 따라 정리할 수 있지만, 다른 팀원이 이미 사용 중인 Remote History는 함부로 재작성하지 않습니다.
공유된 변경을 취소할 때는 일반적으로 Revert가 더 안전합니다.

자주 하는 실수

History를 제대로 읽지 않거나 Commit 습관이 불명확하면 문제 추적과 협업이 어려워집니다. 다음은 초보자가 특히 자주 하는 실수입니다.

1. 현재 수정 내용도
History에 보인다고 생각

History에는 Commit된 내용만 보이며,
현재 변경은 Git Staging과
status에서 확인합니다.

2. Message만 보고
Diff를 확인하지 않음

Message와 실제 변경이 다를 수 있으므로
Changed Files와 Diff도 확인해야 합니다.

3. Commit ID를 순차 번호로 생각

Commit ID는 1, 2, 3처럼
증가하는 번호가 아니라
Commit을 식별하는 해시 기반 값입니다.

4. 짧은 ID 7자리는
항상 충분하다고 생각

저장소 안에서 고유하게 구분될 만큼
충분히 긴 앞부분을 사용해야 합니다.

5. Message를 바꿔도
ID가 같다고 생각

Amend로 Message를 바꾸면
새 Commit 객체와 새 ID가 만들어집니다.

6. History가 항상 일직선이라고 생각

Branch와 Merge가 있으면
History는 갈라지고 다시 합쳐질 수 있습니다.

7. 최신 Commit이 목록 아래라고 생각

기본 git log와 History View는
최신 Commit부터 보여줍니다.

8. git log가 멈췄다고 생각

Pager가 열린 것일 수 있으며
q를 누르면 종료할 수 있습니다.

9. 모든 Branch가
자동으로 보인다고 생각

여러 Branch 흐름은
--all --graph --decorate
옵션으로 확인합니다.

10. 파일 하나당
Commit 하나라고 생각

Commit의 기준은 파일 개수가 아니라
하나의 논리적인 변경 목적입니다.

11. 하루 작업을
하나의 Commit에 몰아넣음

여러 기능과 문서 변경이 섞이면
Review와 부분 되돌리기가 어려워집니다.

12. 지나치게 작은 Commit을 남발

의미 없는 중간 상태가 많아지면
History가 노이즈로 가득 찹니다.

13. 기능 변경과 전체 포맷을 섞음

실제 기능 변경이 대량 포맷 Diff에 묻혀
Review하기 어려워집니다.

14. Message를 update로만 작성

변경 목적이 보이지 않아
검색과 문제 추적이 어려워집니다.

15. Conventional Commits를
필수라고 생각

feat, fix 형식은 선택적인 팀 규칙이며
Git 자체의 필수 문법이 아닙니다.

16. 민감 정보를 다음 Commit에서
지우면 끝이라고 생각

삭제 전 Commit의 Snapshot에는
민감 정보가 계속 남아 있을 수 있습니다.

17. Local과 GitHub History가
항상 같다고 생각

Push하지 않은 Local Commit이나
Fetch하지 않은 Remote Commit이
있을 수 있습니다.

18. Push한 Commit을 쉽게 Amend

새 ID가 만들어져 Remote와
팀원의 History가 달라질 수 있습니다.

오류 메시지와 상태 읽는 법

History 관련 오류는 Commit이 없는지, Commit ID가 잘못되었는지, 현재 Branch가 다른지, Local과 Remote가 갈라졌는지를 구분해서 읽어야 합니다.

오류 / 상태
의미
확인 및 해결 방향
not a git repository
현재 위치가 Git 저장소 안이 아닙니다.
프로젝트 경로와 .git 존재 여부를 확인합니다.

저장소는 있지만 아직 첫 Commit이 없습니다.
파일을 Add하고 Initial Commit을 만듭니다.

요청한 이전 Commit이 존재하지 않습니다.
Commit 개수와 현재 HEAD 위치를 확인합니다.

ID, Branch, 경로와 `--` 구분자를 확인합니다.

해당 Object ID를 현재 저장소에서 찾을 수 없습니다.
ID 오타, 다른 저장소 ID, Fetch 여부를 확인합니다.
ambiguous argument
Revision과 파일 경로가 모호하거나 존재하지 않습니다.

git log 화면이 끝나지 않음
Pager에서 로그가 열린 상태입니다.
키보드에서 q를 눌러 종료합니다.
ahead of origin/main
Local에만 존재하는 Commit이 있습니다.
History를 확인하고 필요하면 Push합니다.
behind origin/main
Remote에만 존재하는 Commit이 있습니다.
Fetch 후 History를 확인하고 Pull 또는 Merge를 검토합니다.
have diverged
Local과 Remote 양쪽에 서로 다른 Commit이 있습니다.
Fetch 후 Graph를 확인하고 Merge 또는 Rebase 정책을 따릅니다.

Remote History에 Local에 없는 Commit이 있을 수 있습니다.
무조건 Force Push하지 말고 Remote 변경을 먼저 통합합니다.
Conflict
두 History에서 같은 부분을 다르게 수정했습니다.
충돌 해결 후 Add하고 Commit 또는 Continue합니다.
403 Permission
GitHub 저장소 권한 또는 인증 문제입니다.
Token, 계정, Remote URL과 저장소 권한을 확인합니다.
HEAD detached at ...
HEAD가 Branch가 아니라 특정 Commit을 직접 가리킵니다.
작업을 보존하려면 새 Branch를 만들고 그 Branch에서 Commit합니다.

History 문제를 확인하는 순서

  1. 현재 올바른 Git 저장소인지 확인합니다.
  2. 첫 Commit이 존재하는지 확인합니다.
  3. git branch --show-current로 현재 Branch를 확인합니다.
  4. git log --oneline --all로 Commit 존재 여부를 확인합니다.
  5. 축약 ID가 너무 짧거나 오타가 없는지 확인합니다.
  6. 필요하면 git fetch로 원격 참조를 갱신합니다.
  7. Local과 Remote가 Ahead, Behind, Diverged 중 어느 상태인지 확인합니다.
  8. 공유된 History에는 Force Push나 Reset을 먼저 사용하지 않습니다.

면접 포인트

Q1. Git History란 무엇인가요?

지금까지 생성한 Commit들이 Parent 관계로 연결된 버전 기록입니다. Commit ID, 작성자, 날짜, Message, Changed Files와 Diff를 통해 프로젝트가 어떻게 변화했는지 확인할 수 있습니다.

Q2. History에 현재 수정 중인 파일도 나타나나요?

아닙니다. History에는 Commit으로 확정된 내용만 나타납니다. 현재 변경은 git status, git diff, git diff --staged와 Eclipse Git Staging에서 확인합니다.

Q3. Commit에는 어떤 정보가 들어 있나요?

프로젝트 Snapshot, Parent Commit, Author, Committer, 날짜, Commit Message와 고유한 Commit ID가 연결됩니다.

Q4. Commit ID란 무엇인가요?

특정 Commit을 정확히 구분하는 해시 기반 식별자입니다. 명령어에서는 저장소 안에서 고유하게 구분 가능한 앞부분을 축약해 사용할 수 있습니다.

Q5. 축약 Commit ID는 왜 사용할 수 있나요?

전체 ID가 길기 때문에 현재 저장소에서 해당 Commit을 고유하게 식별할 수 있는 앞부분만 사용할 수 있습니다. 7자리를 자주 사용하지만 항상 7자리면 충분한 것은 아닙니다.

Q6. Commit Message를 수정하면 ID도 바뀌나요?

네. Message는 Commit 객체를 구성하는 정보이므로 Amend하면 새 Commit이 생성되고 ID도 달라집니다.

Q7. Parent Commit이란 무엇인가요?

현재 Commit의 바로 이전 버전을 가리키는 Commit입니다. 첫 Root Commit은 Parent가 없고, Merge Commit은 둘 이상의 Parent를 가질 수 있습니다.

Q8. HEAD란 무엇인가요?

현재 체크아웃해 작업 중인 위치입니다. 일반적으로 현재 Branch를 가리키고, 그 Branch가 최신 Commit을 가리킵니다.

Q9. Branch와 Commit의 차이는 무엇인가요?

Commit은 특정 시점의 Snapshot이고, Branch는 특정 Commit을 가리키는 움직이는 이름입니다. 새 Commit이 만들어지면 현재 Branch가 새 Commit으로 이동합니다.

Q10. origin/main은 무엇인가요?

마지막 Fetch 또는 Pull을 기준으로 로컬 Git이 기억하는 원격 main의 상태인 Remote-tracking Reference입니다. GitHub 서버의 실시간 상태 자체는 아닙니다.

Q11. git log와 git show의 차이는 무엇인가요?

git log는 여러 Commit의 History를 확인하고, git show는 특정 Commit 하나의 정보와 Diff를 자세히 확인합니다.

Q12. git show와 git diff의 차이는 무엇인가요?

git show는 특정 Commit이 Parent에 비해 만든 변경을 보여주고, git diff CommitA CommitB는 지정한 두 Snapshot 사이의 차이를 보여줍니다.

Q13. git log --oneline --graph --decorate --all은 무엇인가요?

Commit을 한 줄로 표시하고 Parent 관계를 그래프로 나타내며, HEAD, Branch, Tag와 여러 참조의 History를 함께 보여줍니다.

Q14. Author와 Committer의 차이는 무엇인가요?

Author는 원래 변경을 작성한 사람이고, Committer는 그 Commit을 현재 History에 기록한 사람입니다.

Q15. git blame은 무엇에 사용하나요?

파일의 각 줄을 마지막으로 변경한 Commit을 찾는 데 사용합니다. 관련 Commit과 변경 문맥을 찾기 위한 출발점으로 활용합니다.

Q16. 좋은 Commit Message의 기준은 무엇인가요?

Message만 읽어도 무엇을 변경했는지 알 수 있어야 하며, 필요한 경우 본문에서 왜 변경했는지 설명해야 합니다.

Q17. Atomic Commit이란 무엇인가요?

하나의 논리적인 목적을 가진 독립적인 Commit입니다. 관련 없는 변경이 섞이지 않고 독립적으로 검토하거나 되돌릴 수 있어야 합니다.

Q18. Atomic Commit은 파일 하나만 포함해야 하나요?

아닙니다. 파일 개수가 아니라 변경 목적이 기준입니다. 하나의 기능을 완성하는 여러 파일은 같은 Commit에 포함할 수 있습니다.

Q19. Commit은 작을수록 무조건 좋은가요?

아닙니다. 너무 큰 Commit은 검토하기 어렵고, 지나치게 작은 Commit은 History를 노이즈로 만듭니다. 하나의 설명 가능한 작업 단위가 기준입니다.

Q20. Conventional Commits는 Git 필수 규칙인가요?

아닙니다. feat, fix, docs 같은 형식은 팀에서 선택해 사용하는 Message 규칙입니다.

Q21. Local History와 Remote History의 차이는 무엇인가요?

Commit은 먼저 Local Repository에 생성되고, Push해야 Remote Repository에 반영됩니다.

Q22. Push한 Commit을 Amend하면 왜 위험한가요?

Amend는 새 Commit ID를 만듭니다. Remote와 팀원의 History가 기존 Commit을 기준으로 하고 있다면 서로 다른 History가 되어 충돌할 수 있습니다.

Q23. 공유된 History에서 Revert가 안전한 이유는 무엇인가요?

기존 History를 삭제하지 않고 특정 변경을 취소하는 새로운 Commit을 만들기 때문에 팀원이 가진 History와 연결을 유지할 수 있습니다.

학습 메모

오늘의 암기 포인트

  • Git History는 Commit들이 Parent 관계로 연결된 버전 기록입니다.
  • History에는 Commit된 내용만 나타납니다.
  • 현재 변경은 git status와 Git Staging에서 확인합니다.
  • Commit은 특정 시점의 프로젝트 Snapshot입니다.
  • Diff는 두 Snapshot 사이의 차이입니다.
  • Commit ID는 특정 Commit을 정확하게 지정하는 식별자입니다.
  • 축약 ID는 저장소에서 고유하게 구분될 만큼 충분히 길어야 합니다.
  • Root Commit은 Parent가 없고 일반 Commit은 보통 Parent가 하나입니다.
  • HEAD는 현재 체크아웃된 작업 위치를 나타냅니다.
  • Branch는 특정 Commit을 가리키는 움직이는 포인터입니다.
  • git log는 History를 보고 git show는 특정 Commit을 자세히 봅니다.
  • Atomic Commit은 파일 개수가 아니라 변경 목적을 기준으로 판단합니다.
  • Commit은 Local에 먼저 생기며 Push해야 Remote에 반영됩니다.

헷갈리기 쉬운 개념

현재 변경 vs History

현재 변경은 Git Staging에서,
확정된 Commit은 History에서 확인합니다.

Commit vs Diff

Commit은 Snapshot이고, Diff는 두 Snapshot의 차이입니다.

Commit vs Branch

Commit은 버전이고, Branch는 Commit을 가리키는 이름입니다.

HEAD vs main

HEAD는 현재 위치이고, main은 로컬 Branch 이름입니다.

main vs origin/main

main은 로컬 Branch,
origin/main은 원격 상태를 기억하는 로컬 참조입니다.

Author vs Committer

Author는 변경 작성자,
Committer는 현재 History에 기록한 사람입니다.

Amend vs 단순 수정

Amend는 기존 ID를 유지하지 않고 새 Commit을 만듭니다.

Atomic vs 파일 하나

Atomic Commit은 파일 수가 아니라
하나의 논리적인 목적이 기준입니다.

Local vs Remote History

Push 또는 Fetch 여부에 따라
두 History는 서로 다를 수 있습니다.
