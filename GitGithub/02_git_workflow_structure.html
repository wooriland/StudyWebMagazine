Git/GitHub - Chapter 02

Git 작업 공간 구조 이해

Git 작업 공간 구조 이해는 Git을 본격적으로 사용하기 전에 반드시 알아야 하는 핵심 단원입니다.
Git은 파일을 수정하자마자 바로 버전으로 저장하지 않습니다.
실제 파일을 수정하는 Working Directory, 커밋할 파일을 고르는 Staging Area / Index,
내 컴퓨터에 커밋 이력을 저장하는 Local Repository, GitHub에 있는 Remote Repository를 구분해서 이해해야 합니다.
이번 단원의 핵심 질문은 “내가 수정한 파일은 지금 Git의 어느 공간에 있는가?”입니다.

01

Git 작업 공간 구조를 공부한다는 것

Git을 처음 배울 때 어려운 이유는 명령어가 어려워서가 아니라, 파일이 지금 어떤 상태에 있는지 눈에 잘 보이지 않기 때문입니다.
같은 파일이라도 Git 관점에서는 아직 추적하지 않는 상태, 수정만 된 상태, 커밋 대기 상태, 로컬 저장소에 기록된 상태, GitHub까지 올라간 상태로 나뉩니다.

Work

Working Directory

실제 파일을 만들고 수정하는 작업 공간입니다.
Eclipse나 VS Code에서
보고 있는 프로젝트 폴더입니다.

Stage

Staging Area / Index

이번 커밋에 포함할 변경 파일을
올려두는 대기실입니다.
Eclipse에서는 Add to Index로 표현됩니다.

Local

Local Repository

내 컴퓨터 안에 있는 Git 저장소입니다.
Commit을 하면 이곳에 버전이 저장됩니다.

Remote

Remote Repository

GitHub에 있는 원격 저장소입니다.
Push와 Pull을 통해 로컬과 연결됩니다.

핵심 문장

Git 작업 공간 구조를 이해한다는 것은 파일 수정, add, commit, push가 각각 어느 공간을 바꾸는지 이해하는 것입니다.

02

03

Working Directory와 .git 폴더 이해하기

Working Directory는 실제 프로젝트 파일이 존재하는 공간입니다.
HTML, CSS, JavaScript, Java, SQL 파일을 만들고 수정하는 폴더가 바로 Working Directory입니다.
이 프로젝트가 Git 저장소로 등록되면 내부에 .git이라는 숨김 폴더가 생깁니다.

프로젝트 폴더 구조 예시

STUDYWEBMAGAZINE/
├─ .git/
├─ 00_index.html
├─ GitGithub/
│  ├─ 00_git_github_index.html
│  ├─ 01_git_version_control.html
│  └─ 02_git_workflow_structure.html
├─ css/
│  └─ GitGithub/
│     └─ 02_git_workflow_structure.css
└─ js/
   └─ 00_nav.js

.git 폴더의 의미

.git 폴더는 Git의 기억 창고입니다.
커밋 이력, 브랜치 정보, HEAD 정보, 파일 추적 정보가 들어 있습니다.
이 폴더를 삭제하면 프로젝트 파일은 남아 있어도 Git 이력은 사라질 수 있습니다.

  • 커밋 기록 관리
  • 브랜치 정보 관리
  • 파일 추적 정보 관리
  • 현재 HEAD 위치 관리

.git 폴더 주의사항

.git 폴더는 숨김 폴더라서 불필요한 파일처럼 보일 수 있지만, 실제로는 Git 저장소의 핵심입니다. 직접 삭제하거나 내부 파일을 임의로 수정하지 않는 것이 좋습니다.

04

Staging Area / Index는 왜 필요할까?

Staging Area는 이번 커밋에 포함할 변경 사항을 고르는 공간입니다. 모든 수정 파일을 한 번에 커밋하는 것이 아니라, 의미 있는 작업 단위로 나누기 위해 Staging Area가 필요합니다.

수정된 파일들

02_git_workflow_structure.html 수정
02_git_workflow_structure.css 수정
README.md 수정
test.txt 생성

커밋 1에 포함할 파일

git add GitGithub/02_git_workflow_structure.html
git add css/GitGithub/02_git_workflow_structure.css

git commit -m "Add Git workflow structure lesson"

커밋 2에 따로 남길 파일

git add README.md

git commit -m "Update README learning section"
Staging Area를 쓰는 이유

Staging Area는 수정된 모든 파일 중에서 이번 커밋에 넣을 파일만 고를 수 있게 해줍니다. 그래서 커밋 이력을 기능 단위로 깔끔하게 관리할 수 있습니다.

06

Fetch, Merge, Pull을 공간 구조로 이해하기

지금까지는 내 컴퓨터에서 GitHub로 올라가는 흐름을 봤습니다. 반대로 GitHub의 변경을 내 컴퓨터로 가져오는 흐름도 있습니다. 여기서 fetch, merge, pull의 차이를 이해해야 합니다.

Fetch

가져오기만 함

원격 저장소의 최신 커밋 정보를
Local Repository로 가져오지만,
Working Directory의 실제 파일은 바로 바뀌지 않습니다. 

Remote Repository
  ↓ fetch
Local Repository
Merge

합치기

가져온 변경이나 다른 브랜치의 변경을
현재 작업 브랜치에 합칩니다.
같은 파일의 같은 부분을 수정했다면 충돌이 발생할 수 있습니다. 

Local Repository
  ↓ merge
Working Directory
Pull

가져오고 합치기

Pull은 Fetch와 Merge를 한 번에 수행합니다.
즉 원격 변경을 가져와 현재 작업 브랜치에 바로 반영합니다. 

git pull
=
git fetch
+
git merge
암기 기준

fetch는 가져오기만, merge는 합치기, pull은 가져오고 합치기입니다.

07

HEAD는 무엇을 가리킬까?

HEAD는 현재 내가 작업 기준으로 삼고 있는 위치입니다. 조금 더 정확히 말하면, 현재 체크아웃된 브랜치의 최신 커밋을 가리키는 포인터입니다.
지금은 깊게 외우기보다 “내가 지금 보고 있는 현재 커밋 위치”라고 이해하면 됩니다.

Commit A

→
Commit B

→
Commit C

Git 작업 공간 구조 단원 추가

HEAD를 알아야 하는 이유

HEAD는 나중에 reset, revert, branch, checkout을 배울 때 매우 중요합니다. 현재 내가 어느 커밋을 기준으로 작업하는지 알려주는 기준점입니다.

08

Eclipse 조작 흐름과 Git 내부 구조 연결

첨부 교안은 Eclipse 기준으로 Git을 설명합니다. Eclipse 메뉴를 외우는 것보다 중요한 것은, 각 메뉴가 Git의 어떤 공간을 바꾸는지 이해하는 것입니다.

Eclipse 조작 Git 내부 구조 의미
프로젝트 생성 Working Directory 준비 실제 파일을 만들고 수정할 작업 폴더를 준비합니다.
Team → Share Project Git 관리 시작 현재 프로젝트를 Git이 추적할 수 있도록 등록합니다.
Create Repository .git / Local Repository 생성 프로젝트 안에 Git 이력을 관리할 저장소를 만듭니다.
파일 생성 또는 수정 Working Directory 변경 아직 커밋된 것이 아니라 실제 파일만 바뀐 상태입니다.
Team → Add to Index Staging Area로 이동 이번 커밋에 포함할 변경 파일을 선택합니다.
Git Staging 탭 Staged 상태 확인 어떤 파일이 커밋 대기 상태인지 확인합니다.
Commit Local Repository에 저장 Staging Area의 변경을 하나의 버전으로 저장합니다.
Commit and Push Local + Remote에 반영 로컬 커밋 후 GitHub 원격 저장소에도 업로드합니다.
Team → Pull Remote → Local → Working Directory GitHub의 최신 변경을 가져와 현재 작업에 병합합니다.
Team → Fetch From Remote → Local 원격 변경 정보만 가져오고 실제 작업 파일에는 바로 합치지 않습니다.
Team → Merge Local → Working Directory 가져온 변경이나 다른 브랜치를 현재 작업에 합칩니다.
09

CLI 명령어와 공간 이동 연결

Eclipse 버튼은 Git CLI 명령어와 연결됩니다. CLI 명령어를 함께 이해하면 VS Code, Git Bash, IntelliJ에서도 같은 원리로 사용할 수 있습니다.

명령어 공간 이동 의미
git init Working Directory → Git 저장소 초기화 현재 폴더를 Git이 관리할 수 있도록 만듭니다.
git status 상태 확인 파일이 Untracked, Modified, Staged 상태인지 확인합니다.
git add 파일명 Working Directory → Staging Area 특정 변경 파일을 커밋 후보로 올립니다.
git add . Working Directory → Staging Area 현재 폴더의 변경 파일을 한 번에 올립니다.
git commit -m "메시지" Staging Area → Local Repository 스테이지된 변경을 하나의 버전으로 저장합니다.
git log Local Repository 조회 커밋 이력을 확인합니다.
git push origin main Local Repository → Remote Repository 로컬 커밋을 GitHub에 업로드합니다.
git fetch origin Remote Repository → Local Repository 원격 변경 정보를 가져오지만 바로 합치지는 않습니다.
git merge 브랜치명 Local Repository → Working Directory 가져온 변경이나 다른 브랜치를 현재 작업에 합칩니다.
git pull origin main Remote Repository → Local + Working Directory 원격 변경을 가져와 병합까지 수행합니다.

이번 단원 작성 후 커밋 예시

git status
git add GitGithub/02_git_workflow_structure.html
git add css/GitGithub/02_git_workflow_structure.css
git commit -m "Add Git workflow structure lesson"
git push
10

자주 하는 실수

Git 작업 공간 구조를 이해하지 못하면 대부분의 Git 실수가 발생합니다. 아래 항목은 초보자가 특히 자주 헷갈리는 부분입니다.

1. 파일 저장을 커밋으로 착각

저장은 Working Directory의 파일만 바꾼 것입니다.
Local Repository에 기록하려면 commit이 필요합니다.

2. Add를 커밋으로 착각

Add는 Staging Area에 올리는 작업입니다.
아직 로컬 저장소에 버전이 생긴 것은 아닙니다.

3. Commit을 Push로 착각

Commit은 내 컴퓨터에 저장하는 작업이고,
GitHub에 올리려면 Push가 필요합니다.

4. .git 폴더 삭제

.git 폴더는 Git 이력과 브랜치 정보를 담은 핵심 폴더입니다.
삭제하면 저장소 정보가 사라질 수 있습니다.

5. 무조건 git add . 사용

편리하지만 불필요한 파일까지 올라갈 수 있습니다.
commit 전에는 git status로 확인하는 습관이 필요합니다.

6. GitHub가 최신이면 로컬도 최신이라고 생각

원격 저장소가 바뀌어도 내 컴퓨터는 자동으로 바뀌지 않습니다.
최신 변경을 가져오려면 pull이 필요합니다.

11

오류 메시지 읽는 법

Git 오류는 대부분 파일이나 커밋이 지금 어느 공간에 있는지 몰라서 발생합니다.
오류 메시지를 볼 때도 Working Directory, Staging Area, Local Repository, Remote Repository 중 어디에서 문제가 생겼는지 생각해야 합니다.

오류 / 상황
공간 관점 의미
해결 방향
nothing to commit
Staging Area에 커밋할 내용이 없거나 Working Directory에 변경이 없습니다.
git status로 파일 상태를 확인합니다.
not a git repository
현재 폴더에 .git 폴더가 없거나 Git 저장소 내부가 아닙니다.
올바른 프로젝트 폴더로 이동하거나 git init을 실행합니다.
push rejected
Remote Repository에는 있는데
Local Repository에는 없는 커밋이 있습니다.
먼저 git pull 후 다시 push합니다.
<<<<<<< HEAD
Working Directory에서 병합 충돌이 발생했습니다.
최종 코드를 직접 선택하고 충돌 표시를 삭제한 뒤 add, commit합니다.
403 Permission
Remote Repository에 접근할 권한
또는 인증 정보가 올바르지 않습니다.
GitHub 권한, 토큰, Windows 자격 증명, Git 사용자 정보를 확인합니다.

Git 오류를 읽는 순서

  1. 현재 폴더가 Git 저장소인지 확인합니다.
  2. .git 폴더가 있는지 확인합니다.
  3. git status로 파일 상태를 확인합니다.
  4. 변경 파일이 Staging Area에 올라갔는지 확인합니다.
  5. 커밋이 Local Repository에 있는지 확인합니다.
  6. Remote Repository에 내가 아직 받지 않은 커밋이 있는지 확인합니다.
  7. 충돌 표시가 있다면 직접 정리 후 다시 add, commit합니다.

면접 포인트

Q1. Working Directory란 무엇인가요?

Working Directory는 실제 파일이 존재하고 개발자가 코드를 수정하는 작업 공간입니다. 아직 커밋으로 확정되지 않은 변경이 발생하는 공간입니다.

Q2. Staging Area는 왜 필요한가요?

Staging Area는 이번 커밋에 포함할 변경 사항을 선택하는 공간입니다. 여러 변경 중 의미 있는 변경만 골라 커밋할 수 있게 해줍니다.

Q3. Index와 Staging Area는 같은 뜻인가요?

Git에서 Index는 Staging Area와 거의 같은 의미로 사용됩니다. 커밋할 준비가 된 변경 사항의 스냅샷을 의미하며, Eclipse에서는 Add to Index라는 메뉴로 표현됩니다.

Q4. Local Repository란 무엇인가요?

Local Repository는 내 컴퓨터 안에 있는 Git 저장소입니다. commit을 하면 변경 이력이 이곳에 저장됩니다.

Q5. Remote Repository란 무엇인가요?

Remote Repository는 GitHub 같은 원격 서버에 있는 저장소입니다. Local Repository의 커밋을 push로 올리고, 원격 저장소의 변경을 pull로 가져올 수 있습니다.

Q6. Add와 Commit의 차이는 무엇인가요?

Add는 Working Directory의 변경을 Staging Area에 올리는 작업이고, Commit은 Staging Area의 변경을 Local Repository에 하나의 버전으로 저장하는 작업입니다.

Q7. Commit과 Push의 차이는 무엇인가요?

Commit은 로컬 저장소에 변경 이력을 기록하는 작업이고, Push는 그 커밋을 GitHub 같은 원격 저장소에 업로드하는 작업입니다.

Q8. Fetch와 Pull의 차이는 무엇인가요?

Fetch는 원격 저장소의 최신 변경 정보를 로컬 저장소로 가져오지만 현재 작업 파일에는 바로 병합하지 않습니다. Pull은 fetch 후 merge까지 수행합니다.

Q9. .git 폴더는 무엇인가요?

.git 폴더는 Git 저장소의 핵심 폴더입니다. 커밋 이력, 브랜치 정보, HEAD 정보, 추적 정보가 들어 있으므로 직접 삭제하면 안 됩니다.

Q10. HEAD란 무엇인가요?

HEAD는 현재 체크아웃된 브랜치의 최신 커밋을 가리키는 포인터입니다. 쉽게 말하면 지금 내가 작업 기준으로 삼고 있는 현재 위치입니다.

학습 메모

오늘의 암기 포인트

  • Working Directory는 실제 파일을 수정하는 작업 공간입니다.
  • Staging Area 또는 Index는 이번 커밋에 포함할 변경을 올려두는 커밋 대기실입니다.
  • Local Repository는 내 컴퓨터 안에 있는 Git 저장소입니다.
  • Remote Repository는 GitHub에 있는 원격 저장소입니다.
  • add는 Working Directory의 변경을 Staging Area로 올리는 작업입니다.
  • commit은 Staging Area의 변경을 Local Repository에 버전으로 저장하는 작업입니다.
  • push는 Local Repository의 커밋을 Remote Repository로 업로드하는 작업입니다.
  • pull은 fetch와 merge를 한 번에 수행하는 작업입니다.

헷갈리기 쉬운 개념

파일 저장 vs Commit

파일 저장은 Working Directory의 내용만 바꾸는 것이고,
Commit은 Local Repository에 버전을 기록하는 것입니다.

Add vs Commit

Add는 Staging Area에 올리는 작업이고,
Commit은 Local Repository에 저장하는 작업입니다.

Commit vs Push

Commit은 내 컴퓨터에 저장,
Push는 GitHub에 업로드입니다.

Fetch vs Pull

Fetch는 가져오기만 하고,
Pull은 가져온 뒤 merge까지 합니다.

.git 폴더

Git 이력을 관리하는 핵심 폴더입니다.
직접 삭제하지 않는 것이 좋습니다.
