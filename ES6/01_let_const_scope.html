Scope란 무엇인가

Scope

변수, 함수, 클래스 등의 이름에 접근할 수 있는 코드의 범위입니다.

변수는 선언한 모든 위치에서 무조건 사용할 수 있는 것이 아닙니다. 어느 영역에 선언했는지에 따라 접근 가능한 범위가 달라집니다.

함수 밖에서는 지역 변수에 접근할 수 없다
function printUser() {
    const username = "가길동";

    console.log(username);
}

printUser();

console.log(username);
// ReferenceError: username is not defined

렉시컬 스코프

JavaScript의 함수는 어디에서 호출되었는지가 아니라 어디에서 정의되었는지를 기준으로 바깥쪽 스코프를 결정합니다.

호출 위치가 아니라 정의 위치가 기준이다
const message = "전역 메시지";

function printMessage() {
    console.log(message);
}

function run() {
    const message = "run 내부 메시지";

    printMessage();
}

run();
// 전역 메시지
호출 위치 run() 함수 내부
정의 위치 전역 스코프

Shadowing

안쪽 스코프에서 바깥쪽 변수와 같은 이름을 선언하면 가장 가까운 안쪽 변수가 바깥 변수를 가립니다. 이를 Shadowing이라고 합니다.

같은 이름, 서로 다른 스코프
const username = "전역 사용자";

function printUser() {
    const username = "함수 사용자";

    if (true) {
        const username = "블록 사용자";
        console.log(username);
    }

    console.log(username);
}

printUser();
console.log(username);
출력 
블록 사용자
함수 사용자
전역 사용자
Shadowing 자체는 문법 오류가 아닙니다.

하지만 같은 이름이 여러 겹으로 반복되면 어떤 변수를 사용하는지 파악하기 어려워질 수 있으므로 의미가 다른 값에는 구체적인 이름을 사용하는 것이 좋습니다.

06

호이스팅과 TDZ

JavaScript는 코드를 실행하기 전에 현재 스코프의 선언 정보를 먼저 준비합니다. 이 동작을 일반적으로 호이스팅이라고 합니다.

var의 실행 흐름

  1. 1
    스코프 시작

    value 선언 정보가 등록됩니다.

  2. 2
    undefined로 초기화

    선언문 전에도 접근할 수 있습니다.

  3. 3
    할당문 실행

    value = 100이 실행됩니다.

let, const의 실행 흐름

  1. 1
    스코프 시작

    value 바인딩이 만들어집니다.

  2. 2
    TDZ 상태

    초기화 전이므로 접근하면 오류가 발생합니다.

  3. 3
    선언문 실행

    초기화가 끝난 뒤부터 사용할 수 있습니다.

var

console.log(score);

var score = 100;

// undefined

선언 정보가 등록되면서 undefined로 초기화됩니다.

let, const

console.log(score);

const score = 100;

// ReferenceError

선언문이 실행되기 전까지 TDZ에 있으므로 접근할 수 없습니다.

TDZ란 무엇인가

블록 시작
Temporal Dead Zone

이 구간에서 변수에 접근하면 ReferenceError

선언과 초기화 const count = 10;
사용 가능
TDZ 예제
{
    // count의 TDZ 시작

    console.log(count);
    // ReferenceError

    const count = 10;
    // TDZ 종료

    console.log(count);
    // 10
}

TDZ와 Shadowing이 함께 나타나는 경우

바깥 변수가 있어도 안쪽 TDZ가 우선한다
const count = 10;

{
    console.log(count);

    const count = 20;
}
실행 결과 ReferenceError: Cannot access 'count' before initialization

블록 안에 const count 선언이 있기 때문에 블록 시작부터 안쪽 count가 바깥 변수를 가립니다. 하지만 아직 선언문 전이므로 TDZ에 있습니다.

정확한 표현

“let과 const는 호이스팅되지 않는다”보다 “호이스팅되지만 선언문 전까지 초기화되지 않아 접근할 수 없다”가 더 정확한 설명입니다.

07

const는 객체를 얼리지 않는다

가장 중요한 구분

const는 변수 바인딩의 재할당을 막지만, 객체와 배열 내부의 변경까지 막지는 않습니다.

불가능

변수 재할당

const user = {
    name: "가길동"
};

user = {
    name: "나길동"
};
// TypeError
가능

객체 내부 변경

const user = {
    name: "가길동"
};

user.name = "나길동";
user.age = 20;

기본 자료형

다른 값으로 재할당할 수 없다
const score = 100;

score = 200;
// TypeError: Assignment to constant variable.

배열

배열 내부 요소는 변경할 수 있다
const numbers = [1, 2, 3, 4, 5];

numbers[0] = 100;
numbers.push(6);

console.log(numbers);
// [100, 2, 3, 4, 5, 6]
코드 가능 여부 이유
numbers[0] = 100 가능 기존 배열의 요소 변경
numbers.push(6) 가능 기존 배열 내부 변경
numbers = [] 불가능 다른 배열을 변수에 재할당

객체

객체 프로퍼티 변경, 추가, 삭제
const user = {
    name: "KOSMO",
    age: 10
};

user.name = "코스모";
user.address = "서울";
delete user.age;

console.log(user);
// { name: "코스모", address: "서울" }

Object.freeze()와의 차이

const

변수의 재할당을 막습니다.

const user = {
    name: "가길동"
};

user.name = "나길동";
// 가능

Object.freeze()

객체의 바로 아래 프로퍼티 변경을 제한합니다.

"use strict";

const user = Object.freeze({
    name: "가길동"
});

user.name = "나길동";
// TypeError
Object.freeze()도 기본적으로 얕게 동작합니다. 
const settings = Object.freeze({
    options: {
        fontSize: 16
    }
});

settings.options.fontSize = 20;

console.log(settings.options.fontSize);
// 20

바깥 객체는 동결되었지만 중첩된 options 객체는 동결되지 않았습니다.

변수 이름은 모두 대문자로 작성해야 할까

일반적인 const 변수 
const user = {};
const userList = [];
const submitButton =
    document.querySelector("#submit");

일반 지역 변수는 보통 camelCase를 사용합니다.

프로그램 상수 
const MAX_RETRY_COUNT = 3;
const API_BASE_URL = "/api";
const DEFAULT_PAGE_SIZE = 20;

고정 상수의 의미를 강조할 때 대문자와 밑줄을 사용합니다.

반복문과 전역 변수에서의 차이

반복문과 비동기 콜백

var는 반복 전체에서 하나의 변수를 공유하지만, let은 각 반복마다 새로운 바인딩을 만듭니다.

var

반복 1 반복 2 반복 3
→
하나의 i 공유

let

반복 1 → i = 0 반복 2 → i = 1 반복 3 → i = 2

var 사용

for (var i = 0; i < 3; i++) {
    setTimeout(function () {
        console.log(i);
    }, 0);
}
출력 3, 3, 3

let 사용

for (let i = 0; i < 3; i++) {
    setTimeout(function () {
        console.log(i);
    }, 0);
}
출력 0, 1, 2

for...of에서는 const를 사용할 수 있다

각 반복마다 새로운 바인딩이 만들어진다
const users = ["가길동", "나길동", "다길동"];

for (const user of users) {
    console.log(user);
}

각 반복에서 새로운 user 바인딩이 만들어지므로 반복 변수 자체를 재할당하지 않는다면 const를 사용할 수 있습니다.

일반 증가 반복문에서는 const를 사용할 수 없습니다. 
for (const index = 0; index < 3; index++) {
    console.log(index);
}

index++는 변수 재할당이므로 TypeError가 발생합니다.

전역 변수와 window

브라우저의 일반 script
<script>
    var legacyName = "var";
    let modernName = "let";
    const fixedName = "const";

    console.log(window.legacyName);
    // "var"

    console.log(window.modernName);
    // undefined

    console.log(window.fixedName);
    // undefined
</script>

브라우저의 일반 비모듈 스크립트에서 최상위 var는 window의 프로퍼티가 될 수 있습니다. 최상위 let과 const는 window 프로퍼티로 등록되지 않습니다.

ES Module
<script type="module">
    var moduleValue = 100;

    console.log(window.moduleValue);
    // undefined
</script>
React와 Vite는 모듈 환경을 사용합니다.

React 프로젝트의 JavaScript 파일은 일반적으로 모듈로 처리되므로 최상위 변수도 일반 비모듈 스크립트처럼 window에 등록되지 않습니다.

JavaScript 실전 예제

변수 선언 키워드는 “값의 타입”이 아니라 “이 변수에 다른 값을 다시 할당할 것인가”를 기준으로 선택합니다.

상품 주문 금액 계산
const TAX_RATE = 0.1;

function calculateOrder(items) {
    let subtotal = 0;

    for (const item of items) {
        const lineTotal = item.price * item.quantity;
        subtotal += lineTotal;
    }

    const tax = subtotal * TAX_RATE;
    const total = subtotal + tax;

    return {
        subtotal,
        tax,
        total
    };
}

const items = [
    {
        name: "키보드",
        price: 50000,
        quantity: 1
    },
    {
        name: "마우스",
        price: 30000,
        quantity: 2
    }
];

const result = calculateOrder(items);

console.log(result);
변수 선언 선택 이유
TAX_RATE const 재할당하지 않는 고정 비율
subtotal let 반복하면서 합계를 계속 재할당
item const 각 반복마다 새 바인딩이 생성되고 직접 재할당하지 않음
lineTotal const 한 반복 안에서 계산 후 변경하지 않음
tax const 계산 후 재할당하지 않음
total const 최종 계산 후 재할당하지 않음
items const 다른 배열로 재할당하지 않음
result const 함수 결과를 다른 값으로 재할당하지 않음

변수는 사용하는 위치와 가깝게 선언한다

권장 방식

if (user.role === "admin") {
    const adminMessage =
        `${user.name}님은 관리자입니다.`;

    console.log(adminMessage);
}

관리자 메시지가 필요한 블록 안에서만 변수를 선언합니다.

불필요하게 넓은 범위

let adminMessage;

if (user.role === "admin") {
    adminMessage =
        `${user.name}님은 관리자입니다.`;
}

실제 필요 범위보다 바깥에 변수를 선언하면 접근과 변경 가능 범위가 넓어집니다.

실무 판단 순서
  1. 이 변수에 다른 값을 다시 할당하는지 확인합니다.
  2. 재할당이 없다면 const를 사용합니다.
  3. 실제 재할당이 필요하면 let을 사용합니다.
  4. 변수의 선언 위치를 실제 사용 범위에 최대한 가깝게 둡니다.

React와 연결되는 지점

상태 변수는 왜 const로 선언할까

React useState
import { useState } from "react";

function Counter() {
    const [count, setCount] = useState(0);

    const increase = () => {
        setCount(
            (previousCount) => previousCount + 1
        );
    };

    return (
        <button onClick={increase}>
            {count}
        </button>
    );
}

React에서는 현재 렌더링의 count 변수에 직접 값을 대입하지 않습니다. 상태 변경 함수인 setCount()를 호출합니다.

첫 번째 렌더링 const count = 0
→
상태 변경 요청 setCount(1)
→
두 번째 렌더링 const count = 1
현재 변수의 재할당이 아닙니다.

상태 변경 후 React가 컴포넌트 함수를 다시 실행하고, 새로운 렌더링에서 새로운 상태값을 전달합니다. 그래서 상태 변수는 const로 받는 것이 자연스럽습니다.

const와 React 불변성은 서로 다른 규칙이다

직접 변경 
const [user, setUser] = useState({
    name: "가길동",
    age: 20
});

const changeName = () => {
    user.name = "나길동";
    setUser(user);
};

JavaScript 문법상 객체 내부 변경은 가능하지만, React 상태를 직접 변경한 뒤 같은 객체를 전달하는 방식은 피해야 합니다.

새 객체 생성 
const changeName = () => {
    setUser((previousUser) => ({
        ...previousUser,
        name: "나길동"
    }));
};

기존 객체는 유지하고 변경된 값을 가진 새로운 객체를 상태로 전달합니다.

JavaScript의 const

변수 자체의 재할당을 방지합니다.

React의 불변성

기존 상태 객체와 배열을 직접 변경하지 않습니다.

React 배열 상태

기존 배열 직접 변경 
todos.push(newTodo);
setTodos(todos);
새 배열 생성 
setTodos((previousTodos) => [
    ...previousTodos,
    newTodo
]);
배열 상태에서 항목 삭제
setTodos((previousTodos) =>
    previousTodos.filter(
        (todo) => todo.id !== targetId
    )
);

컴포넌트 내부의 스코프

이벤트 핸들러는 바깥 컴포넌트 스코프에 접근할 수 있다
function UserCard({ user }) {
    const isAdult = user.age >= 19;

    function handleClick() {
        const message =
            `${user.name}님을 선택했습니다.`;

        console.log(message);
        console.log(isAdult);
    }

    return (
        <button onClick={handleClick}>
            사용자 선택
        </button>
    );
}
UserCard 함수 스코프 user isAdult handleClick
handleClick 함수 스코프 message

React에서도 let이 필요한 경우

렌더링 중 계산되는 지역 변수
function StatusMessage({ status }) {
    let message = "대기 중입니다.";

    if (status === "success") {
        message = "완료되었습니다.";
    } else if (status === "error") {
        message = "오류가 발생했습니다.";
    }

    return <p>{message}</p>;
}
지역 let은 렌더링 사이에 값을 보존하지 않습니다.

컴포넌트가 다시 실행되면 지역 변수는 다시 만들어집니다. 렌더링 사이에 값을 유지해야 한다면 useState나 useRef 같은 React 기능을 사용해야 합니다.

오류 메시지 읽는 법

변수 관련 오류는 오류 종류와 핵심 문구를 함께 읽으면 원인을 빠르게 좁힐 수 있습니다.

SyntaxError

코드 문법 자체를 해석할 수 없는 경우

ReferenceError

현재 스코프에서 이름을 찾지 못하거나 TDZ에서 접근한 경우

TypeError

현재 값이나 변수 상태에서 허용되지 않는 연산을 한 경우

TypeError

Assignment to constant variable.

const count = 1;
count = 2;
의미
const 변수에 새로운 값을 재할당했습니다.
해결
재할당이 실제로 필요하다면 let으로 선언합니다.
SyntaxError

Missing initializer in const declaration

const result;
의미
const 선언에 초기값이 없습니다.
해결
초기값을 지정하거나 나중에 할당해야 한다면 let을 사용합니다.
SyntaxError

Identifier 'count' has already been declared

let count = 1;
let count = 2;
의미
같은 스코프에서 같은 이름을 다시 선언했습니다.
해결
재할당이면 두 번째 선언 키워드를 제거하고, 다른 의미라면 이름을 변경합니다.
ReferenceError

Cannot access 'count' before initialization

console.log(count);

let count = 10;
의미
let 또는 const의 TDZ에서 접근했습니다.
해결
선언과 초기화를 사용 코드보다 먼저 실행합니다.
ReferenceError

count is not defined

if (true) {
    const count = 10;
}

console.log(count);
의미
현재 스코프 체인에서 해당 이름을 찾지 못했습니다.
해결
변수의 올바른 사용 범위를 확인하고 해당 범위 안에서 사용합니다.
TypeError

Cannot assign to read only property

"use strict";

const config = Object.freeze({
    theme: "dark"
});

config.theme = "light";
의미
동결되었거나 읽기 전용인 객체 프로퍼티를 변경했습니다.
구분
const 재할당 오류가 아니라 객체 프로퍼티 변경 오류입니다.
오류가 없지만 undefined가 나오는 경우도 주의해야 합니다. 
console.log(score);

var score = 100;

// undefined

오류가 발생하지 않았다고 정상적인 코드라는 뜻은 아닙니다. var의 초기화 방식 때문에 잘못된 접근이 조용히 통과한 것일 수 있습니다.

디버깅 순서

  1. 1

    선언 키워드가 var, let, const 중 무엇인지 확인합니다.

  2. 2

    오류가 발생한 위치가 변수 선언문보다 앞인지 확인합니다.

  3. 3

    같은 스코프에서 동일한 이름을 중복 선언했는지 확인합니다.

  4. 4

    변수 선언과 사용 코드가 같은 스코프에 있는지 확인합니다.

  5. 5

    const 변수 자체를 재할당했는지 확인합니다.

  6. 6

    객체 내부 변경과 변수 재할당을 혼동했는지 확인합니다.

  7. 7

    React 상태 객체나 배열을 직접 변경했는지 확인합니다.

13

면접 포인트

Q1. var, let, const의 차이는 무엇인가요?

var는 함수 스코프이며 재선언과 재할당이 가능합니다.
let과 const는 블록 스코프이며 같은 스코프에서 재선언할 수 없습니다.
let은 재할당 가능, const는 재할당 불가능입니다.

Q2. 실무에서는 무엇을 기본으로 사용하나요?

재할당이 없다면 const를 기본으로 사용하고,
실제로 값이 바뀌는 경우에만 let을 사용합니다.
새 코드에서는 특별한 이유가 없다면 var를 지양합니다.

Q3. const 객체의 프로퍼티를 수정할 수 있나요?

수정할 수 있습니다.
const는 객체 자체를 불변으로 만드는 것이 아니라 변수의 재할당을 막기 때문입니다.

Q4. 함수 스코프와 블록 스코프의 차이는 무엇인가요?

함수 스코프는 함수 전체에서 유효한 범위이고,
블록 스코프는 if, for 등의 중괄호 안에서만 유효한 범위입니다.

Q5. 호이스팅이란 무엇인가요?

JavaScript가 코드를 실행하기 전에 현재 스코프의 선언 정보를 먼저 준비하는 동작입니다.
var는 undefined로 초기화되지만, let과 const는 TDZ 상태가 됩니다.

Q6. TDZ란 무엇인가요?

스코프 시작부터 let 또는 const 선언문이 실행되기 전까지의 구간입니다.
이 구간에서 접근하면 ReferenceError가 발생합니다.

Q7. 스코프 체인이란 무엇인가요?

현재 스코프에서 변수를 찾지 못하면 바깥쪽 스코프로 이동하며 이름을 검색하는 구조입니다.

Q8. 렉시컬 스코프란 무엇인가요?

함수의 바깥 스코프가 함수의 호출 위치가 아니라 정의 위치에 의해 결정되는 방식입니다.

Q9. 반복문에서 var와 let의 차이는 무엇인가요?

var는 반복 전체에서 하나의 변수를 공유하지만,
let은 반복마다 새로운 바인딩을 만듭니다.
비동기 콜백에서 결과 차이가 명확하게 나타납니다.

Q10. React 상태 변수는 왜 const인가요?

현재 렌더링의 상태 변수를 직접 재할당하지 않고 상태 변경 함수를 호출하기 때문입니다.
상태가 변경되면 새로운 렌더링에서 새로운 상태값을 받습니다.

Q11. React 상태 객체는 왜 직접 수정하면 안 되나요?

기존 상태 참조를 직접 변경하면 React가 변경을 올바르게 판단하기 어렵고
이전 상태도 함께 변경될 수 있습니다.
따라서 새로운 객체나 배열을 생성해 전달해야 합니다.

Q12. Object.freeze()와 const의 차이는 무엇인가요?

const는 변수의 재할당을 막고,
Object.freeze()는 객체 프로퍼티 변경을 제한합니다.
다만 기본적으로 중첩 객체까지 동결하지는 않습니다.

30초 면접 답변

var는 함수 스코프이며 재선언과 재할당이 가능하고, 선언 전에 접근하면 undefined가 나옵니다.
let과 const는 블록 스코프이며 같은 스코프에서 재선언할 수 없고, 선언 전에는 TDZ 때문에 ReferenceError가 발생합니다.
let은 재할당할 수 있지만 const는 재할당할 수 없습니다.
다만 const 객체와 배열의 내부는 변경할 수 있습니다.
실무에서는 재할당이 없다면 const를 기본으로 사용하고, 실제 값 변경이 필요할 때만 let을 사용합니다.

코드 예측 문제

문제 1 
function example() {
    if (true) {
        var value = 10;
    }

    console.log(value);
}

example();
결과: 10

var는 함수 스코프입니다.

문제 2 
function example() {
    if (true) {
        const value = 10;
    }

    console.log(value);
}

example();
결과: ReferenceError

const는 블록 스코프입니다.

문제 3 
const user = {
    name: "가길동"
};

user.name = "나길동";

console.log(user.name);
결과: 나길동

객체 내부 프로퍼티 변경은 재할당이 아닙니다.

문제 4 
const value = 10;

{
    console.log(value);
    const value = 20;
}
결과: ReferenceError

안쪽 value가 TDZ 상태에서 바깥 변수를 가립니다.

문제 5 
const numbers = [1, 2, 3];

numbers.push(4);
numbers[0] = 100;

console.log(numbers);
결과: [100, 2, 3, 4]

기존 배열의 내부 변경은 가능합니다.

문제 6 
for (const value of [10, 20, 30]) {
    console.log(value);
}
결과: 10, 20, 30

각 반복마다 새로운 value 바인딩이 생성됩니다.

14

이번 단원 최종 정리

Scope

변수와 함수 이름에 접근할 수 있는 코드의 범위

var

함수 스코프, 재선언 가능, 선언 전 접근 시 undefined

let

블록 스코프, 재할당 가능, 같은 스코프 재선언 불가

const

블록 스코프, 재할당 불가, 선언 시 초기값 필요

TDZ

let과 const 선언 전 접근이 금지되는 구간

스코프 체인

현재 스코프에서 바깥 스코프로 변수를 검색하는 구조

렉시컬 스코프

호출 위치가 아니라 함수 정의 위치로 바깥 스코프가 결정됨

Shadowing

안쪽의 같은 이름 변수가 바깥 변수를 가리는 현상

const 객체

변수 재할당은 불가능하지만 객체와 배열 내부 변경은 가능

실무 원칙

기본은 const, 실제 재할당이 필요할 때만 let

React 상태

상태 변수는 setter로 변경하며
객체와 배열을 직접 수정하지 않음

모듈 환경

React와 Vite의 최상위 변수는 일반적으로 모듈 스코프에 속함

한 문장으로 마무리

let과 const는 변수를 필요한 블록 안에 제한하여 코드의 영향을 예측하기 쉽게 만들며, const는 객체를 얼리는 문법이 아니라 변수의 재할당을 막는 문법입니다.

학습 메모

이번 단원에서는 변수를 선언하는 문법보다 변수의 범위와 변경 가능성을 설계하는 방법이 더 중요합니다.
코드를 작성할 때마다 “이 값은 다시 할당되는가?”, “이 변수는 어디까지 보여야 하는가?”를 먼저 생각하는 습관을 들이면 좋습니다.

변수 선언 기본 규칙

재할당이 없다면 const, 실제 재할당이 필요하다면 let, 기존 코드 분석이 아니라면 var는 가급적 사용하지 않습니다.

const 핵심 구분

const는 변수와 값의 연결을 고정합니다. 객체 내부 변경을 막는 불변성 문법은 아닙니다.

React에서 기억할 것

JavaScript에서 객체 내부 변경이 가능하더라도, React 상태 객체와 배열은 직접 변경하지 않고 새로운 객체와 배열을 만들어 setter에 전달합니다.
