Oracle SQL Developer - Chapter 23

사용자 생성 및 권한

Oracle에서 사용자는 단순한 로그인 ID가 아니라, 데이터베이스에 접속하고,
테이블·VIEW·시퀀스 같은 객체를 소유하며, 권한에 따라 작업을 수행하는 주체입니다.
이번 단원에서는 CDB와 PDB 사용자, CREATE USER, GRANT, REVOKE, ROLE, 시스템 권한, 객체 권한, QUOTA, 계정 잠금 해제, 사용자 삭제,
그리고 Java·Spring 프로젝트의 DB 계정 분리까지 학습합니다.

학습 목표 사용자를 만들고 필요한 권한을
안전하게 부여하는 흐름을 이해한다.
실습 방향 SYSTEM 접속 → PDB 확인 → 사용자 생성 →
권한 부여 → 일반 사용자 접속 확인
프로젝트 연결 Spring Boot datasource 계정,
읽기 전용 계정, 쓰기 계정 분리와 연결한다.
Overview

1. 사용자 생성과 권한이란?

Oracle에서 사용자는 데이터베이스에 접속하는 계정이면서, 테이블이나 VIEW 같은 객체를 소유하는 스키마 역할도 합니다.
하지만 사용자를 만들었다고 해서 바로 접속하거나 테이블을 만들 수 있는 것은 아닙니다.

CREATE USER 사용자 계정 생성
GRANT 사용자에게 권한 또는 역할 부여
REVOKE 부여된 권한 또는 역할 회수

한 줄 정리

사용자 생성은 계정을 만드는 것이고, 권한 부여는 그 계정이 무엇을 할 수 있는지 정하는 과정입니다.

2. 단원 개념 카드

2-1. Oracle 사용자란?

Oracle에서 사용자는 단순한 로그인 ID가 아닙니다. 사용자는 데이터베이스에 접속하는 계정이면서, 객체를 소유하는 스키마이며, 권한을 부여받아 작업을 수행하는 주체입니다.

접속 계정 Login Account

SQL Developer, SQL*Plus, Spring Boot에서 DB에 접속할 때 사용하는 계정입니다.

스키마 Schema Owner

테이블, VIEW, 시퀀스, 인덱스 같은 객체를 소유하는 공간입니다.

권한 주체 Privilege Target

GRANT와 REVOKE의 대상이 되는 사용자 또는 역할입니다.

프로젝트 계정 Application User

Spring Boot 애플리케이션이 접속하는 전용 DB 계정으로 사용됩니다.

2-2. 사용자는 생성 직후 아무 권한도 없다

사용자를 생성해도 바로 접속할 수 없습니다.
접속하려면 CREATE SESSION 권한이 필요하고, 테이블을 만들려면 CREATE TABLE 권한이 필요하며, VIEW를 만들려면 CREATE VIEW 권한이 필요합니다.

사용자 생성

CREATE USER study_user IDENTIFIED BY Study1234;

계정만 생성된 상태입니다.

접속 권한 부여

GRANT CREATE SESSION TO study_user;

이제 DB에 접속할 수 있습니다.

테이블 생성 권한

GRANT CREATE TABLE TO study_user;
ALTER USER study_user QUOTA UNLIMITED ON users;

테이블 생성 권한과 테이블스페이스 할당량이 필요할 수 있습니다.

VIEW 생성 권한

GRANT CREATE VIEW TO study_user;

VIEW를 만들 수 있습니다.

2-3. CDB 사용자와 PDB 사용자

구분 CDB 사용자 PDB 사용자
의미 Container Database 전체에서 사용할 수 있는 공통 사용자 특정 PDB 안에서 사용하는 로컬 사용자
계정명 C## 접두사를 사용하는 경우가 있음 보통 C##을 붙이지 않음
사용 목적 관리자 또는 공통 관리용 실습, 프로젝트, 애플리케이션 작업용
예시 C##ICT study_user, myhouse_app

5. SQL Developer 실습 준비

사용자 생성과 권한 부여는 일반적으로 관리자 계정에서 수행합니다. SQL Developer에서는 접속 설정에서 서비스 이름을 XEPDB1로 지정하고 SYSTEM 계정으로 접속합니다.

5-1. SYSTEM으로 PDB 접속

sqlplus system/비밀번호@localhost:1521/XEPDB1

SQL Developer에서는 호스트 이름 localhost, 포트 1521, 서비스 이름 XEPDB1을 사용합니다.

5-2. 현재 컨테이너 확인

SHOW CON_NAME;

결과가 XEPDB1인지 확인합니다.

5-3. PDB 목록 확인

SELECT name
FROM v$pdbs;

5-4. CDB에서 PDB로 이동

ALTER SESSION SET CONTAINER = XEPDB1;

SHOW CON_NAME;

실습 전 주의

사용자 생성, 권한 부여, 사용자 삭제는 관리자 계정에서 실행해야 합니다.
실습 계정을 삭제할 때 DROP USER ... CASCADE는 해당 사용자의 객체까지 삭제할 수 있으므로 주의해야 합니다.

6. SQL 예제 카드 - 기본 사용자 생성

6-1. 기존 실습 사용자 정리

DROP USER study_user CASCADE;
DROP USER app_owner CASCADE;
DROP USER app_reader CASCADE;
DROP ROLE staff;

처음 실행하면 사용자가 없어서 오류가 날 수 있습니다.
첫 실습에서는 괜찮습니다.

6-2. 사용자 생성

CREATE USER study_user IDENTIFIED BY Study1234
DEFAULT TABLESPACE users
TEMPORARY TABLESPACE temp;

6-3. 권한 부여

GRANT CREATE SESSION TO study_user;
GRANT CREATE TABLE TO study_user;
GRANT CREATE VIEW TO study_user;
ALTER USER study_user QUOTA UNLIMITED ON users;

6-4. 일반 사용자 접속

CONN study_user/Study1234@localhost:1521/XEPDB1;

SHOW USER;

6-5. 테이블 생성

CREATE TABLE user_priv_test (
    id NUMBER,
    name VARCHAR2(30),
    CONSTRAINT pk_user_priv_test PRIMARY KEY (id)
);

6-6. 데이터 입력과 조회

INSERT INTO user_priv_test VALUES (1, '권한 실습');
COMMIT;

SELECT *
FROM user_priv_test;

6-7. VIEW 생성

CREATE OR REPLACE VIEW v_user_priv_test AS
SELECT id,
       name
FROM user_priv_test;

SELECT *
FROM v_user_priv_test;

6-8. VIEW 목록 확인

SELECT view_name
FROM user_views
WHERE view_name = 'V_USER_PRIV_TEST';
ROLE

7. ROLE 실습 카드

ROLE은 여러 권한을 묶어놓은 권한 꾸러미입니다. 여러 사용자에게 같은 권한 세트를 부여해야 할 때 역할을 만들어 관리하면 편리합니다.

7-1. 관리자 계정으로 접속

CONN system/비밀번호@localhost:1521/XEPDB1;

7-2. ROLE 생성

CREATE ROLE staff;

7-3. ROLE에 권한 부여

GRANT CREATE TABLE, CREATE VIEW TO staff;

7-4. 사용자에게 ROLE 부여

GRANT staff TO study_user;

7-5. ROLE 확인

SELECT granted_role
FROM dba_role_privs
WHERE grantee = 'STUDY_USER';

7-6. 대표 역할

CONNECT
→ DB에 접속할 수 있는 역할

RESOURCE
→ 객체를 생성할 수 있는 역할

DBA
→ 최고 권한 역할

8. 객체 권한 실습 카드

다른 사용자의 테이블을 조회하려면 객체 권한이 필요합니다. 권한이 없으면 실제 테이블이 존재해도 ORA-00942 오류가 발생할 수 있습니다.

APP_OWNER 테이블 소유자
→
GRANT SELECT 조회 권한 부여
→
APP_READER 조회 전용 사용자

8-1. 사용자 생성

CREATE USER app_owner IDENTIFIED BY Owner1234
DEFAULT TABLESPACE users
TEMPORARY TABLESPACE temp;

CREATE USER app_reader IDENTIFIED BY Reader1234
DEFAULT TABLESPACE users
TEMPORARY TABLESPACE temp;

8-2. 권한 부여

GRANT CREATE SESSION, CREATE TABLE TO app_owner;
ALTER USER app_owner QUOTA UNLIMITED ON users;

GRANT CREATE SESSION TO app_reader;

8-3. APP_OWNER로 테이블 생성

CONN app_owner/Owner1234@localhost:1521/XEPDB1;

CREATE TABLE owner_notice (
    notice_id NUMBER,
    title VARCHAR2(100),
    CONSTRAINT pk_owner_notice PRIMARY KEY (notice_id)
);

INSERT INTO owner_notice VALUES (1, '첫 번째 공지');
INSERT INTO owner_notice VALUES (2, '두 번째 공지');
COMMIT;

8-4. APP_READER로 권한 없이 조회

CONN app_reader/Reader1234@localhost:1521/XEPDB1;

SELECT *
FROM app_owner.owner_notice;

권한이 없으면 ORA-00942 오류가 발생할 수 있습니다.

8-5. SELECT 권한 부여

CONN app_owner/Owner1234@localhost:1521/XEPDB1;

GRANT SELECT ON owner_notice TO app_reader;

8-6. APP_READER로 다시 조회

CONN app_reader/Reader1234@localhost:1521/XEPDB1;

SELECT *
FROM app_owner.owner_notice;

8-7. 권한 회수

CONN app_owner/Owner1234@localhost:1521/XEPDB1;

REVOKE SELECT ON owner_notice FROM app_reader;

Admin Option

10. WITH ADMIN OPTION

WITH ADMIN OPTION은 권한을 받은 사용자가 그 시스템 권한을 다른 사용자에게 다시 부여할 수 있도록 하는 옵션입니다.
권한이 확산될 수 있으므로 실무에서는 매우 신중하게 사용해야 합니다.

10-1. WITH ADMIN OPTION 부여

GRANT CREATE TABLE TO study_user WITH ADMIN OPTION;

10-2. 의미

SYSTEM
→ study_user에게 CREATE TABLE 권한 부여
→ study_user가 다른 사용자에게 CREATE TABLE 권한 부여 가능

주의

WITH ADMIN OPTION은 권한 위임을 가능하게 하므로, 일반 개발자 계정이나 애플리케이션 계정에는 보통 부여하지 않는 것이 안전합니다.

12. 실습 문제 카드

실습 문제의 SCOTT.EMP, HR.EMPLOYEES, DEPARTMENTS, LOCATIONS 조회는 객체 권한과 연결해서 이해할 수 있습니다.
다른 사용자의 테이블을 조회하려면 해당 객체에 대한 SELECT 권한이 필요합니다.

문제 1. 실습 사용자 생성

CREATE USER study_user IDENTIFIED BY Study1234
DEFAULT TABLESPACE users
TEMPORARY TABLESPACE temp;

GRANT CREATE SESSION, CREATE TABLE, CREATE VIEW TO study_user;
ALTER USER study_user QUOTA UNLIMITED ON users;

문제 2. 일반 사용자 접속 확인

CONN study_user/Study1234@localhost:1521/XEPDB1;

SHOW USER;

문제 3. 테이블과 VIEW 생성

CREATE TABLE user_priv_test (
    id NUMBER,
    name VARCHAR2(30),
    CONSTRAINT pk_user_priv_test PRIMARY KEY (id)
);

CREATE OR REPLACE VIEW v_user_priv_test AS

문제 4. ROLE 생성과 부여

CREATE ROLE staff;
GRANT CREATE TABLE, CREATE VIEW TO staff;
GRANT staff TO study_user;

문제 5. SCOTT.EMP 조회 권한

GRANT SELECT ON SCOTT.EMP TO CHOI;

문제 6. HR 테이블 조회 권한

GRANT SELECT ON HR.EMPLOYEES TO APP_READER;
GRANT SELECT ON HR.DEPARTMENTS TO APP_READER;
GRANT SELECT ON HR.LOCATIONS TO APP_READER;

  ON e.department_id = d.department_id

Errors

13. 자주 만나는 오류

ORA-01045

ORA-01045: user lacks CREATE SESSION privilege; logon denied

사용자는 생성되었지만 CREATE SESSION 권한이 없는 경우입니다. 

GRANT CREATE SESSION TO 사용자;

ORA-00942

ORA-00942: table or view does not exist

테이블이 없거나, 다른 사용자 객체에 대한 SELECT 권한이 없거나,
소유자명을 붙이지 않은 경우 발생할 수 있습니다. 

GRANT SELECT ON app_owner.property TO app_user;

SELECT *
FROM app_owner.property;

ORA-01950

ORA-01950: no privileges on tablespace 'USERS'

CREATE TABLE 권한은 있지만 USERS 테이블스페이스 할당량이 없는 경우입니다. 

ALTER USER 사용자 QUOTA UNLIMITED ON users;
Mistake

14. 자주 하는 실수

  • 실수 1. 사용자만 만들면 접속할 수 있다고 생각함 CREATE SESSION 권한을 별도로 부여해야 접속할 수 있습니다.
  • 실수 2. CREATE TABLE 권한만 주고 QUOTA를 주지 않음 테이블 생성에는 테이블스페이스 할당량이 필요할 수 있습니다.
  • 실수 3. PDB 사용자에 C##을 붙임 PDB 실습용 로컬 사용자는 보통 C## 접두사를 사용하지 않습니다.
  • 실수 4. 현재 컨테이너를 확인하지 않음 SHOW CON_NAME으로 XEPDB1에 접속했는지 확인해야 합니다.
  • 실수 5. 다른 사용자 테이블을 소유자명 없이 조회함 다른 사용자 테이블은 app_owner.owner_notice처럼 소유자명을 붙여야 합니다.
  • 실수 6. 객체 권한 없이 조회하려고 함 다른 사용자 객체를 조회하려면 SELECT 객체 권한이 필요합니다.
  • 실수 7. CONNECT, RESOURCE만 주면 VIEW도 만들 수 있다고 생각함 VIEW 생성에는 CREATE VIEW 권한을 따로 부여해야 합니다.
  • 실수 8. ANY 권한을 쉽게 부여함 CREATE ANY TABLE 같은 권한은 매우 강력하므로 일반 사용자에게는 신중해야 합니다.
  • 실수 9. DROP USER CASCADE를 가볍게 사용함 해당 사용자가 소유한 객체까지 삭제될 수 있으므로 실무에서는 매우 주의해야 합니다.
  • 실수 10. Spring에서 SYSTEM 계정을 사용함 애플리케이션은 전용 DB 계정을 만들고 최소 권한만 부여하는 것이 안전합니다.
Interview

15. 면접 질문 카드

Q1. Oracle에서 사용자란 무엇인가요?

Oracle의 사용자는 데이터베이스에 접속하는 계정이면서, 테이블이나 VIEW 같은 객체를 소유하는 스키마 역할도 합니다.
사용자는 생성 직후 권한이 없기 때문에 필요한 권한을 별도로 부여해야 합니다.

Q2. CREATE USER만 하면 바로 접속할 수 있나요?

아닙니다. 사용자를 생성한 후 CREATE SESSION 권한을 부여해야 접속할 수 있습니다.

Q3. CDB 사용자와 PDB 사용자의 차이는 무엇인가요?

CDB 사용자는 컨테이너 데이터베이스 전체에서 사용할 수 있는 공통 사용자이며, 보통 C## 접두사를 사용합니다. PDB 사용자는 특정 PDB 안에서 사용하는 로컬 사용자입니다.

Q4. 시스템 권한과 객체 권한의 차이는 무엇인가요?

시스템 권한은 CREATE SESSION, CREATE TABLE처럼 데이터베이스 차원의 작업 권한입니다.
객체 권한은 특정 테이블이나 VIEW에 대한 SELECT, INSERT, UPDATE, DELETE 같은 권한입니다.

Q5. CREATE TABLE 권한만 있으면 테이블을 만들 수 있나요?

항상 그렇지는 않습니다. CREATE TABLE 권한과 함께 테이블스페이스 할당량이 필요할 수 있습니다.

Q6. CONNECT와 RESOURCE 역할은 무엇인가요?

CONNECT는 DB 접속과 관련된 역할이고, RESOURCE는 객체 생성과 관련된 역할입니다. 다만 VIEW 생성은 CREATE VIEW 권한을 따로 부여해야 할 수 있습니다.

Q7. 다른 사용자의 테이블을 조회하려면 어떻게 해야 하나요?

테이블 소유자 또는 관리자가 SELECT 객체 권한을 부여해야 합니다. 조회할 때는 SCOTT.EMP처럼 소유자명을 붙여 조회합니다.

Q8. REVOKE는 무엇인가요?

REVOKE는 사용자에게 부여된 권한이나 역할을 회수하는 명령입니다.

Q9. ROLE은 무엇인가요?

ROLE은 여러 권한을 묶어놓은 권한 꾸러미입니다. 여러 사용자에게 동일한 권한 세트를 부여할 때 관리가 편리합니다.

Q10. WITH ADMIN OPTION은 무엇인가요?

권한을 받은 사용자가 그 시스템 권한을 다른 사용자에게 다시 부여할 수 있도록 하는 옵션입니다. 권한이 확산될 수 있으므로 실무에서는 신중하게 사용해야 합니다.

Q11. Spring 프로젝트에서 DB 계정을 왜 분리해야 하나요?

관리자 계정으로 애플리케이션을 운영하면 보안상 위험합니다. Spring 프로젝트에서는 애플리케이션 전용 계정을 만들고 필요한 권한만 부여하는 것이 좋습니다.

Q12. ORA-00942가 발생하면 무엇을 확인해야 하나요?

실제 테이블 존재 여부, 현재 접속 계정, 테이블 소유자, 객체 권한, 소유자명 prefix 사용 여부를 확인해야 합니다.

Project

16. Java·Spring·Oracle 프로젝트 연결

실제 프로젝트에서는 SYSTEM 계정으로 Spring Boot 애플리케이션을 운영하지 않습니다. 애플리케이션 전용 계정을 만들고, 필요한 권한만 부여하는 것이 안전합니다.

Spring Boot용 계정 생성

CREATE USER myhouse_app IDENTIFIED BY Myhouse1234
DEFAULT TABLESPACE users
TEMPORARY TABLESPACE temp;

GRANT CREATE SESSION TO myhouse_app;
GRANT CREATE TABLE TO myhouse_app;
GRANT CREATE VIEW TO myhouse_app;
ALTER USER myhouse_app QUOTA UNLIMITED ON users;

application.yml 연결

spring:
  datasource:
    url: jdbc:oracle:thin:@localhost:1521/XEPDB1
    username: myhouse_app
    password: Myhouse1234

읽기 전용 계정

CREATE USER app_read IDENTIFIED BY Read1234;

GRANT CREATE SESSION TO app_read;

GRANT SELECT ON app_owner.property TO app_read;
GRANT SELECT ON app_owner.notice TO app_read;

쓰기 계정

CREATE USER app_write IDENTIFIED BY Write1234;

GRANT CREATE SESSION TO app_write;

GRANT SELECT, INSERT, UPDATE, DELETE
ON app_owner.property
TO app_write;

프로젝트 권한 분리

SYSTEM
→ 사용자 생성, 권한 부여, DB 관리

APP_OWNER
→ 테이블 소유

APP_READ
→ SELECT 전용

APP_WRITE
→ SELECT, INSERT, UPDATE, DELETE

SPRING_APP
→ Spring Boot가 접속하는 애플리케이션 계정

최소 권한 원칙

필요한 권한만 부여하고,
관리자 권한이나 ANY 권한은
일반 애플리케이션 계정에 주지 않는 것이 안전합니다.

학습 메모

오늘의 핵심

Oracle 사용자는 생성만으로는 아무 작업도 할 수 없습니다. 접속, 테이블 생성, VIEW 생성, 다른 사용자 객체 조회는 모두 권한이 필요합니다.

암기 문법

CREATE USER 사용자 IDENTIFIED BY 암호;

GRANT CREATE SESSION TO 사용자;
GRANT CREATE TABLE TO 사용자;
GRANT CREATE VIEW TO 사용자;

ALTER USER 사용자 QUOTA UNLIMITED ON users;

REVOKE 권한 FROM 사용자;

DROP USER 사용자 CASCADE;

객체 권한

GRANT SELECT ON 소유자.테이블 TO 사용자;

SELECT *
FROM 소유자.테이블;

REVOKE SELECT ON 소유자.테이블 FROM 사용자;

ROLE

CREATE ROLE staff;

GRANT CREATE TABLE, CREATE VIEW TO staff;

GRANT staff TO study_user;

주의할 점

DROP USER ... CASCADE는 해당 사용자가 소유한 객체까지 삭제할 수 있습니다. 실무에서는 반드시 영향 범위를 확인한 뒤 실행해야 합니다.

프로젝트 연결

Spring Boot 프로젝트에서는 SYSTEM이 아니라 애플리케이션 전용 DB 계정을 사용하고, 필요한 권한만 부여하는 최소 권한 원칙을 지키는 것이 좋습니다.
