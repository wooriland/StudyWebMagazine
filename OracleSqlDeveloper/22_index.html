INDEX

인덱스는 테이블에서 원하는 행을 빠르게 찾기 위해 사용하는 검색 보조 객체입니다.
책의 색인처럼 원하는 값을 빠르게 찾도록 도와주지만,
INSERT, UPDATE, DELETE가 발생할 때는 인덱스도 함께 관리되어야 하므로 무조건 많이 만드는 것이 좋은 것은 아닙니다.
이번 단원에서는 CREATE INDEX, DROP INDEX, 자동 생성 인덱스, 복합 인덱스, USER_INDEXES, USER_IND_COLUMNS, 실행 계획,
Java·Spring 프로젝트의 검색 성능 연결까지 학습합니다.

학습 목표 인덱스가 무엇인지 이해하고
어떤 컬럼에 적용해야 하는지 판단한다.
실습 방향 실습 테이블 생성 → 자동 인덱스 확인 →
명시적 인덱스 생성 → 조회 → 삭제
프로젝트 연결 로그인, 게시판 검색, 매물 검색,
좋아요 중복 방지, 페이징 성능과 연결한다.
Overview

1. 인덱스란?

인덱스는 테이블에서 원하는 행을 빠르게 찾기 위해 사용하는 데이터베이스 객체입니다.
테이블의 모든 행을 처음부터 끝까지 확인하는 대신, 특정 컬럼 기준으로 만들어진 찾아보기 구조를 이용해 원하는 데이터의 위치를 빠르게 찾을 수 있도록 도와줍니다.

INDEX 행 검색 속도를 높이는 검색 보조 객체
SELECT 성능 WHERE, JOIN, ORDER BY, 페이징 조건에서 도움 가능
DML 부담 INSERT, UPDATE, DELETE 시 인덱스도 함께 갱신

한 줄 정리

인덱스는 책의 색인처럼 데이터를 빠르게 찾게 해주는 객체이지만, 데이터 변경이 많으면 관리 비용이 증가하므로 필요한 곳에만 설계해야 합니다.

Concept Card

2. 단원 개념 카드

2-1. 인덱스를 왜 사용하는가?

테이블에 데이터가 적을 때는 전체를 검색해도 큰 문제가 없습니다.
하지만 데이터가 수십만 건, 수백만 건으로 늘어나면 조건에 맞는 행 하나를 찾기 위해 테이블 전체를 읽는 것은 부담이 됩니다. 이때 인덱스가 검색 범위를 줄여줄 수 있습니다.

인덱스 없음

조건에 맞는 행을 찾기 위해 테이블 전체를 확인할 수 있습니다.

SELECT *
FROM index_member
WHERE email = 'hong@test.com';

-- 개념적으로 TABLE ACCESS FULL 가능

인덱스 있음

인덱스에서 조건값을 먼저 찾고 실제 행으로 이동할 수 있습니다.

CREATE INDEX idx_index_member_email
ON index_member(email);

SELECT *
FROM index_member
WHERE email = 'hong@test.com';

2-2. 인덱스 생성 방식

인덱스는 개발자가 직접 CREATE INDEX로 만들 수도 있고, PRIMARY KEY 또는 UNIQUE 제약조건에 의해 자동으로 생성될 수도 있습니다.

명시적 인덱스 CREATE INDEX

개발자가 특정 컬럼의 검색 성능을 위해 직접 생성합니다.

자동 인덱스 PRIMARY KEY

기본키 제약조건을 지원하기 위해 자동 생성될 수 있습니다.

자동 인덱스 UNIQUE

중복 방지를 빠르게 검사하기 위해 자동 생성될 수 있습니다.

복합 인덱스 Multi Column

두 개 이상의 컬럼을 묶어 자주 쓰는 조건 패턴에 맞게 생성합니다.

2-3. 인덱스의 장점과 단점

구분 내용 예시
장점 WHERE 조건 검색 속도를 높일 수 있음 login_id, email 검색
장점 JOIN 조건 컬럼 검색에 도움 가능 board.member_id, property.publisher_id
장점 정렬, 페이징 쿼리에 도움 가능 created_at DESC, board_id DESC
단점 INSERT 시 테이블과 인덱스를 함께 갱신해야 함 회원가입, 게시글 작성
단점 UPDATE 시 인덱스 컬럼 값이 바뀌면 인덱스도 갱신 email 변경, 상태 변경
단점 DELETE 시 인덱스 정보도 함께 정리해야 함 데이터 삭제, 로그 정리

3. 실행 순서 그림 카드

3-1. 인덱스 없는 검색

인덱스가 없으면 Oracle은 조건에 맞는 데이터를 찾기 위해 테이블의 여러 행을 직접 확인할 수 있습니다. 데이터가 적으면 문제가 작지만, 데이터가 많아질수록 부담이 커집니다.

INDEX_MEMBER

1 | kim | 김철수 2 | lee | 이영희 3 | hong | 홍길동 4 | park | 박민수 5 | choi | 최민수
Full Table Scan 개념

kim 확인 → lee 확인 → hong 확인

데이터가 많으면 많은 행을 확인할 수 있음

3-2. 인덱스 있는 검색

인덱스가 있으면 인덱스에서 조건값을 먼저 찾고, 해당 행 위치로 이동할 수 있습니다.

IDX_MEMBER_LOGIN

choi → 5번 행 hong → 3번 행 kim → 1번 행 lee → 2번 행 park → 4번 행
→

INDEX_MEMBER

3 | hong | 홍길동

3-3. 인덱스와 DML 부담

인덱스는 SELECT에는 도움을 줄 수 있지만, 데이터가 바뀔 때는 인덱스도 같이 바뀌어야 합니다.

INSERT 테이블 행 추가

새 데이터 저장

+ 인덱스 추가

인덱스 구조에도 새 값 반영

UPDATE 테이블 값 변경

인덱스 컬럼이면 인덱스도 변경

DELETE 테이블 행 삭제

인덱스 정보도 정리

중요

인덱스는 많이 만드는 것이 아니라, 자주 쓰는 조회 패턴에 맞게 필요한 만큼 설계하는 것이 중요합니다.

4. 인덱스 기본 문법

인덱스는 CREATE INDEX로 생성하고 DROP INDEX로 삭제합니다. 교안에서는 인덱스는 수정이 불가능하므로 수정이 필요할 때는 삭제 후 다시 생성해야 한다고 정리합니다.

4-1. 단일 컬럼 인덱스 생성

CREATE INDEX 인덱스명
ON 테이블명(컬럼명);

4-2. 예시

CREATE INDEX idx_index_member_email
ON index_member(email);

4-3. 복합 인덱스 생성

CREATE INDEX idx_index_board_member_created
ON index_board(member_id, created_at);

4-4. 인덱스 삭제

DROP INDEX idx_index_member_email;

4-5. 인덱스 수정 방식

DROP INDEX idx_index_board_member;

CREATE INDEX idx_index_board_member
ON index_board(member_id, created_at);

인덱스는 직접 수정하지 않고 삭제 후 다시 생성합니다.

4-6. 인덱스 컬럼 확인

SELECT index_name,
       table_name,
       column_name,
       column_position
FROM user_ind_columns
WHERE table_name = 'INDEX_MEMBER'
ORDER BY index_name, column_position;

5. SQL Developer 실습 준비

인덱스 실습은 자동 생성 인덱스와 직접 생성 인덱스를 구분해서 확인하는 것이 중요합니다. 회원 테이블과 게시글 테이블을 만들고, PK와 UNIQUE로 자동 생성된 인덱스를 확인한 뒤 직접 인덱스를 생성해봅니다.

5-1. 기존 실습 객체 정리

DROP TABLE index_board PURGE;
DROP TABLE index_member PURGE;
DROP SEQUENCE index_member_seq;
DROP SEQUENCE index_board_seq;
DROP INDEX idx_index_member_email;
DROP INDEX idx_index_board_member;
DROP INDEX idx_index_board_title;
DROP INDEX idx_index_board_member_created;

처음 실행하면 ORA-00942, ORA-02289, ORA-01418 오류가 날 수 있습니다. 삭제할 객체가 없다는 뜻이므로 첫 실습에서는 괜찮습니다.

5-2. 회원 테이블 생성

CREATE TABLE index_member (
    member_id NUMBER,
    login_id VARCHAR2(30) NOT NULL,
    member_name VARCHAR2(50) NOT NULL,
    email VARCHAR2(100),
    phone VARCHAR2(20),
    active CHAR(1) DEFAULT 'Y',
    created_at DATE DEFAULT SYSDATE,
    CONSTRAINT pk_index_member PRIMARY KEY (member_id),
    CONSTRAINT uk_index_member_login UNIQUE (login_id),
    CONSTRAINT chk_index_member_active CHECK (active IN ('Y', 'N'))
);

5-3. 게시글 테이블 생성

CREATE TABLE index_board (
    board_id NUMBER,
    title VARCHAR2(200) NOT NULL,
    content VARCHAR2(1000),
    member_id NUMBER NOT NULL,
    view_count NUMBER DEFAULT 0,
    deleted_yn CHAR(1) DEFAULT 'N',
    created_at DATE DEFAULT SYSDATE,
    CONSTRAINT pk_index_board PRIMARY KEY (board_id),
    CONSTRAINT fk_index_board_member
        FOREIGN KEY (member_id)
        REFERENCES index_member(member_id),
    CONSTRAINT chk_index_board_deleted CHECK (deleted_yn IN ('Y', 'N'))
);

5-4. 시퀀스 생성

CREATE SEQUENCE index_member_seq
START WITH 1
INCREMENT BY 1
NOCACHE
NOCYCLE;

CREATE SEQUENCE index_board_seq
START WITH 1
INCREMENT BY 1
NOCACHE
NOCYCLE;

6. 자동 생성 인덱스 확인

PRIMARY KEY와 UNIQUE 제약조건은 중복 검사와 빠른 검색을 위해 자동으로 인덱스를 생성할 수 있습니다.
회원 테이블의 member_id 기본키와 login_id UNIQUE 제약조건이 어떤 인덱스와 연결되는지 확인합니다.

6-1. 제약조건 확인

SELECT table_name,
       constraint_name,
       constraint_type
FROM user_constraints
WHERE table_name IN ('INDEX_MEMBER', 'INDEX_BOARD')
ORDER BY table_name, constraint_name;

6-2. 인덱스 목록 확인

SELECT index_name,
       table_name,
       uniqueness
FROM user_indexes
WHERE table_name IN ('INDEX_MEMBER', 'INDEX_BOARD')
ORDER BY table_name, index_name;

6-3. 인덱스 컬럼 확인

SELECT index_name,
       table_name,
       column_name,
       column_position
FROM user_ind_columns
WHERE table_name IN ('INDEX_MEMBER', 'INDEX_BOARD')
ORDER BY table_name, index_name, column_position;

6-4. 확인 포인트

7. SQL 예제 카드 - 명시적 인덱스 생성

WHERE 조건이나 JOIN 조건에서 자주 사용하는 컬럼은 인덱스 후보가 됩니다. 단, 데이터 양과 분포, 실행 계획을 함께 확인해야 합니다.

7-1. 이메일 검색용 인덱스

SELECT *
FROM index_member
WHERE email = 'hong@test.com';

CREATE INDEX idx_index_member_email
ON index_member(email);

7-2. 게시글 작성자 검색용 인덱스

SELECT *
FROM index_board
WHERE member_id = 1
ORDER BY created_at DESC;

CREATE INDEX idx_index_board_member
ON index_board(member_id);

7-3. 작성자별 최신 게시글 복합 인덱스

CREATE INDEX idx_index_board_member_created
ON index_board(member_id, created_at);

SELECT *
FROM index_board
WHERE member_id = 1
ORDER BY created_at DESC;

복합 인덱스는 컬럼 순서가 중요합니다.

7-4. 제목 검색용 인덱스

CREATE INDEX idx_index_board_title
ON index_board(title);

SELECT *
FROM index_board
WHERE title = '게시판 검색 기능';

7-5. LIKE 검색 주의

SELECT *
FROM index_board
WHERE title LIKE '%검색%';

앞에 %가 붙는 검색은 일반 B-tree 인덱스 활용이 어려울 수 있습니다.

7-6. 생성 후 확인

9. 실행 계획으로 인덱스 확인

인덱스를 만들었다고 해서 Oracle이 반드시 인덱스를 사용하는 것은 아닙니다.
Oracle Optimizer가 비용을 계산해서 전체 테이블 검색이 더 유리하다고 판단하면 인덱스가 있어도 사용하지 않을 수 있습니다.

인덱스 사용 전후 비교 쿼리

SELECT *
FROM index_member
WHERE email = 'hong@test.com';

CREATE INDEX idx_index_member_email
ON index_member(email);

SELECT *
FROM index_member
WHERE email = 'hong@test.com';

실행 계획에서 볼 수 있는 표현

TABLE ACCESS FULL
→ 전체 테이블 검색

INDEX RANGE SCAN
→ 인덱스 범위 검색

INDEX UNIQUE SCAN
→ UNIQUE / PK 인덱스 검색

TABLE ACCESS BY INDEX ROWID
→ 인덱스로 찾은 뒤 테이블 행 접근

실습 시 주의

실습 테이블에 데이터가 적으면 인덱스를 만들어도 실행 계획에서 Full Table Scan이 나올 수 있습니다.
이는 실패가 아니라 Optimizer가 전체를 읽는 것이 더 저렴하다고 판단했기 때문일 수 있습니다.

10. 어떤 컬럼에 인덱스를 설정해야 할까?

인덱스는 자주 쓰는 조회 패턴에 맞게 설계해야 합니다.
교안에서는 WHERE 조건이나 JOIN 조건에서 자주 사용되는 컬럼, 광범위한 값을 포함하는 컬럼, 많은 NULL 값을 포함하는 컬럼을 인덱스 후보로 제시합니다.

인덱스 후보 이유 예시 컬럼
WHERE 조건에 자주 사용 조건 검색 범위를 줄일 수 있음 login_id, email, region_code
JOIN 조건에 자주 사용 테이블 연결 시 검색 비용 감소 가능 member_id, department_id, publisher_id
값의 종류가 많음 특정 값으로 데이터 범위를 좁히기 좋음 email, phone, order_id
정렬 / 페이징에 자주 사용 목록 조회와 페이지 처리에 도움 가능 created_at, board_id, viewed_at
NULL이 많은 컬럼 조회 조건과 데이터 분포에 따라 후보가 될 수 있음 deleted_at, canceled_at

10-1. 인덱스를 피해야 하는 경우

데이터가 적은 테이블

10건, 20건 정도의 코드 테이블은 전체 검색해도 부담이 작을 수 있습니다.

자주 UPDATE되는 컬럼

조회수, 상태값처럼 자주 바뀌는 컬럼은 인덱스 갱신 부담이 생깁니다.

값 종류가 적은 컬럼

Y/N, M/F처럼 값 종류가 적은 컬럼은 단독 인덱스 효과가 낮을 수 있습니다.

검색 조건에 거의 안 쓰는 컬럼

WHERE나 JOIN에 거의 쓰이지 않으면 인덱스 유지 비용만 생길 수 있습니다.

Practice

11. 실습 문제 카드

첨부된 실습 문제에는 WHERE 조건 검색, JOIN, ORDER BY, TOP-N, 페이징 문제가 많이 포함되어 있습니다. 이런 쿼리들은 데이터가 많아질수록 인덱스 설계와 연결됩니다.

문제 1. 이메일 검색용 인덱스 만들기

CREATE INDEX idx_index_member_email
ON index_member(email);

SELECT index_name,
       table_name,
       column_name
FROM user_ind_columns
WHERE table_name = 'INDEX_MEMBER';

문제 2. 회원 ID로 게시글 조회 인덱스

CREATE INDEX idx_index_board_member
ON index_board(member_id);

SELECT index_name,
       column_name
FROM user_ind_columns
WHERE table_name = 'INDEX_BOARD'
ORDER BY index_name, column_position;

문제 3. 작성자별 최신 게시글 복합 인덱스

CREATE INDEX idx_index_board_member_created
ON index_board(member_id, created_at);

SELECT *
FROM index_board
WHERE member_id = 1
ORDER BY created_at DESC;

문제 4. 인덱스 삭제

DROP INDEX idx_index_board_member;

SELECT index_name
FROM user_indexes
WHERE index_name = 'IDX_INDEX_BOARD_MEMBER';

문제 5. 인덱스 수정 방식 확인

DROP INDEX idx_index_board_member_created;

CREATE INDEX idx_index_board_member_created
ON index_board(member_id, deleted_yn, created_at);

문제 6. PK / UNIQUE 자동 인덱스 확인

SELECT index_name,
       table_name,
       uniqueness
FROM user_indexes
WHERE table_name = 'INDEX_MEMBER'
ORDER BY index_name;

문제 7. WHERE 조건 문제와 인덱스 연결

SELECT *
FROM emp
WHERE deptno = 30;

CREATE INDEX idx_emp_deptno
ON emp(deptno);

문제 8. JOIN 조건 문제와 인덱스 연결

SELECT e.last_name,
       e.salary,
       d.department_name
FROM employees e
JOIN departments d
  ON e.department_id = d.department_id;

CREATE INDEX idx_employees_department_id
ON employees(department_id);

문제 9. 페이징과 인덱스 연결

SELECT *
FROM board
WHERE deleted_yn = 'N'
ORDER BY board_id DESC
OFFSET 0 ROWS FETCH NEXT 10 ROWS ONLY;

CREATE INDEX idx_board_deleted_id
ON board(deleted_yn, board_id);
Mistake

12. 자주 하는 실수

  • 실수 1. 인덱스가 많을수록 무조건 좋다고 생각함 인덱스가 많으면 SELECT에는 도움이 될 수 있지만 DML 성능은 떨어질 수 있습니다.
  • 실수 2. 자주 UPDATE되는 컬럼에 인덱스를 생성함 조회수나 상태값처럼 자주 바뀌는 컬럼은 인덱스 갱신 부담이 큽니다.
  • 실수 3. 데이터가 적은 테이블에 인덱스를 많이 만듦 데이터가 적으면 전체 테이블 검색이 더 효율적일 수 있습니다.
  • 실수 4. 값 종류가 적은 컬럼에 단독 인덱스를 만듦 Y/N 같은 컬럼은 단독 인덱스 효과가 낮을 수 있습니다.
  • 실수 5. FK 컬럼은 자동 인덱스가 생긴다고 착각함 PK와 UNIQUE는 자동 인덱스가 생길 수 있지만 FK는 직접 고려해야 합니다.
  • 실수 6. 인덱스를 만들면 항상 사용된다고 생각함 Oracle Optimizer가 전체 스캔이 더 낫다고 판단하면 인덱스를 사용하지 않을 수 있습니다.
  • 실수 7. 인덱스를 수정할 수 있다고 생각함 인덱스는 수정이 아니라 삭제 후 재생성하는 방식으로 변경합니다.
  • 실수 8. 복합 인덱스 컬럼 순서를 무시함 복합 인덱스는 앞쪽 컬럼부터 어떤 조건에 쓰이는지가 중요합니다.
  • 실수 9. LIKE '%검색어%'에 일반 인덱스가 잘 먹을 거라고 생각함 앞에 %가 붙는 포함 검색은 일반 B-tree 인덱스 활용이 어려울 수 있습니다.
Interview

13. 면접 질문 카드

Q1. 인덱스란 무엇인가요?

인덱스는 테이블에서 원하는 행을 빠르게 찾기 위해 사용하는 데이터베이스 객체입니다.
책의 색인처럼 특정 컬럼 값을 기준으로 데이터 위치를 빠르게 찾을 수 있게 해주며, 주로 SELECT 조회 성능을 향상시키기 위해 사용합니다.

Q2. 인덱스를 사용하면 왜 조회가 빨라지나요?

인덱스가 없으면 조건에 맞는 데이터를 찾기 위해 테이블 전체를 검색해야 할 수 있습니다.
인덱스가 있으면 Oracle이 인덱스 구조에서 조건값을 먼저 찾고 해당 행 위치로 이동할 수 있기 때문에 검색 범위를 줄일 수 있습니다.

Q3. 인덱스는 어떻게 생성하나요?

CREATE INDEX 문을 사용합니다. 예를 들어 CREATE INDEX idx_member_email ON member(email); 과 같이 작성합니다.
두 개 이상의 컬럼을 묶어 복합 인덱스도 만들 수 있습니다.

Q4. 인덱스는 어떻게 삭제하나요?

DROP INDEX 인덱스명 문을 사용합니다. 예를 들어 DROP INDEX idx_member_email; 과 같이 작성합니다.

Q5. 인덱스는 수정할 수 있나요?

일반적으로 인덱스는 직접 수정하지 않고 삭제 후 다시 생성합니다. 컬럼 구성을 바꾸려면 DROP INDEX 후 CREATE INDEX를 다시 실행합니다.

Q6. PK와 UNIQUE는 인덱스와 어떤 관계가 있나요?

PRIMARY KEY와 UNIQUE 제약조건은 중복 방지와 빠른 검사를 위해 내부적으로 인덱스를 자동 생성할 수 있습니다.
PK와 UNIQUE는 제약조건이고, 인덱스는 그 제약조건을 효율적으로 지원하는 구조로 함께 만들어질 수 있습니다.

Q7. 인덱스를 많이 만들면 좋은가요?

무조건 좋지 않습니다.
인덱스는 SELECT 성능에는 도움이 될 수 있지만, INSERT, UPDATE, DELETE가 발생할 때 인덱스도 함께 갱신되어야 하므로 DML 성능이 저하될 수 있습니다.

Q8. 어떤 컬럼에 인덱스를 설정하는 것이 좋나요?

WHERE 조건에 자주 사용되는 컬럼, JOIN 조건에 자주 사용되는 컬럼, 값의 종류가 다양한 컬럼, 데이터가 많은 테이블에서 자주 검색되는 컬럼에 인덱스를 고려합니다.

Q9. 어떤 컬럼에는 인덱스를 피해야 하나요?

데이터가 적은 테이블, 자주 갱신되는 컬럼, 값의 종류가 적은 컬럼, 검색 조건으로 거의 사용되지 않는 컬럼에는 인덱스를 신중하게 적용해야 합니다.

Q10. 인덱스를 만들었는데도 사용되지 않을 수 있나요?

네. Oracle Optimizer는 비용을 계산해서 인덱스를 사용할지 전체 테이블을 읽을지 결정합니다.
테이블이 작거나 조건 결과가 너무 많으면 인덱스가 있어도 전체 테이블 검색을 선택할 수 있습니다.

Q11. 복합 인덱스란 무엇인가요?

복합 인덱스는 두 개 이상의 컬럼을 묶어서 만든 인덱스입니다. 자주 사용하는 WHERE 조건과 ORDER BY 패턴을 기준으로 컬럼 순서를 설계해야 합니다.

Q12. LIKE 검색에서 인덱스가 항상 사용되나요?

아닙니다. LIKE '검색%'처럼 앞부분이 고정된 검색은 인덱스 활용 가능성이 있지만,
LIKE '%검색%'처럼 앞에 %가 붙는 검색은 일반 B-tree 인덱스를 활용하기 어려울 수 있습니다.

Project

14. Java·Spring·Oracle 프로젝트 연결

인덱스는 Java/Spring 프로젝트에서 로그인, 게시판 목록, 매물 검색, 좋아요 중복 방지, 조회 로그 분석, 관리자 통계, 페이징 성능과 직접 연결됩니다.

로그인 기능

SELECT *
FROM users
WHERE login_id = :loginId;

ALTER TABLE users
ADD CONSTRAINT uk_users_login UNIQUE (login_id);

UNIQUE 제약조건으로 중복 아이디를 막고 인덱스 효과도 기대할 수 있습니다.

게시판 목록

SELECT board_id,
       title,
       writer_id,
       created_at
FROM board
WHERE deleted_yn = 'N'
ORDER BY board_id DESC
OFFSET :offset ROWS FETCH NEXT :size ROWS ONLY;

CREATE INDEX idx_board_deleted_id
ON board(deleted_yn, board_id);

작성자별 게시글

SELECT *
FROM board
WHERE writer_id = :writerId
ORDER BY created_at DESC;

CREATE INDEX idx_board_writer_created
ON board(writer_id, created_at);

매물 검색

SELECT *
FROM property
WHERE region_code = :regionCode
  AND property_type = :propertyType
  AND listing_status = 'ACTIVE'
ORDER BY created_at DESC
OFFSET :offset ROWS FETCH NEXT :size ROWS ONLY;

CREATE INDEX idx_property_search
ON property(region_code, property_type, listing_status, created_at);

좋아요 중복 방지

SELECT *
FROM property_like
WHERE user_id = :userId
  AND property_id = :propertyId;

ALTER TABLE property_like
ADD CONSTRAINT uk_property_like_user_property
UNIQUE (user_id, property_id);

같은 사용자가 같은 매물에 중복 좋아요를 누르는 것을 막고
조회 속도도 기대할 수 있습니다.

조회 로그

SELECT *
FROM property_view_log
WHERE user_id = :userId
ORDER BY viewed_at DESC;

CREATE INDEX idx_view_log_user_time
ON property_view_log(user_id, viewed_at);

CREATE INDEX idx_view_log_property_time
ON property_view_log(property_id, viewed_at);

관리자 통계

SELECT property_id,
       COUNT(*) AS view_count
FROM property_view_log
GROUP BY property_id;

통계 쿼리는 인덱스뿐 아니라
집계 테이블, 배치 처리, 캐싱까지 함께 고려할 수 있습니다.

Summary

15. 최종 정리

INDEX
→ 행의 검색 속도를 향상시키는 데이터베이스 객체

목적
→ SELECT 조회 성능 향상

생성 방식
→ CREATE INDEX로 직접 생성
→ PRIMARY KEY / UNIQUE KEY로 자동 생성 가능

장점
→ WHERE 조건 검색 속도 향상
→ JOIN 조건 검색 속도 향상
→ 페이징 / 정렬 쿼리에 도움 가능

단점
→ INSERT / UPDATE / DELETE 시 인덱스도 갱신되어야 함
→ 인덱스가 많으면 DML 성능 저하 가능
→ 저장 공간과 관리 비용 증가

좋은 후보
→ WHERE 조건에 자주 사용되는 컬럼
→ JOIN 조건에 자주 사용되는 컬럼
→ 값의 종류가 많은 컬럼
→ 데이터가 많은 테이블의 검색 컬럼

피해야 할 후보
→ 데이터가 적은 테이블
→ 자주 UPDATE되는 컬럼
→ 값의 종류가 너무 적은 컬럼
→ 거의 검색 조건으로 쓰지 않는 컬럼

확인
→ USER_INDEXES
→ USER_IND_COLUMNS

학습 메모

오늘의 핵심

인덱스는 테이블에서 원하는 행을 빠르게 찾기 위한 검색 보조 객체입니다. SELECT에는 도움이 될 수 있지만 DML에는 추가 비용이 생길 수 있습니다.

암기 문법

CREATE INDEX 인덱스명
ON 테이블명(컬럼명);

CREATE INDEX 인덱스명
ON 테이블명(컬럼1, 컬럼2);

DROP INDEX 인덱스명;

인덱스 확인

SELECT index_name,
       table_name,
       uniqueness
FROM user_indexes;

SELECT index_name,
       table_name,
       column_name,
       column_position
FROM user_ind_columns;

좋은 인덱스 후보

  • WHERE 조건에 자주 쓰는 컬럼
  • JOIN 조건에 자주 쓰는 컬럼
  • 값의 종류가 많은 컬럼
  • 데이터가 많은 테이블의 검색 컬럼
  • 정렬과 페이징에 자주 쓰는 컬럼

주의할 점

인덱스는 많을수록 좋은 것이 아닙니다. 데이터 변경 시 인덱스도 갱신되어야 하므로, 자주 UPDATE되는 컬럼이나 데이터가 적은 테이블에는 신중하게 적용해야 합니다.

프로젝트 연결

인덱스는 로그인 조회, 게시판 페이징, 매물 검색, 좋아요 중복 방지, 조회 로그 분석, 관리자 통계 성능과 연결됩니다.
