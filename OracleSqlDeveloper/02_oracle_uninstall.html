Oracle 삭제와 재설치 준비

Oracle Database를 삭제하고 재설치할 때 확인해야 할
Windows 서비스, 레지스트리, PATH 환경 변수, 설치 폴더의 의미를 학습합니다.
이 단원은 Oracle을 자주 삭제하기 위한 내용이 아니라,
설치가 꼬였을 때 어떤 흔적을 정리해야 다시 안정적으로 실습 환경을 만들 수 있는지 이해하는 복구 매뉴얼입니다.

1. 이 단원에서 배우는 것

이번 단원은 Oracle SQL 문법을 배우는 단원이 아니라, Oracle 설치 환경이 꼬였을 때 어떻게 삭제하고 재설치를 준비해야 하는지 이해하는 단원입니다.
Oracle은 일반 프로그램처럼 제어판에서 삭제하면 항상 완전히 정리되는 프로그램이 아닙니다.
Oracle Database는 서버 프로그램이기 때문에 설치 과정에서 Windows 서비스, Listener, 레지스트리, PATH 환경 변수, 설치 폴더에 여러 흔적을 남깁니다.
이 흔적이 남아 있으면 재설치할 때 기존 Oracle Home이나 서비스, 포트 정보와 충돌할 수 있습니다.

삭제 전 확인 정말 삭제가 필요한 문제인지 먼저 판단합니다.
흔적 정리 서비스, 레지스트리, PATH, 설치 폴더를 확인합니다.
재설치 준비 재부팅 후 깨끗한 상태에서 다시 설치를 준비합니다.

2. Oracle 삭제가 복잡한 이유

Oracle Database는 단순한 실행 프로그램이 아니라 데이터베이스 서버입니다.
설치하면 데이터베이스 엔진, 리스너, 관리자 계정, 서비스, 실행 경로, 데이터 파일 등이 함께 구성됩니다.

일반 프로그램 삭제

일반적인 프로그램은 제어판에서 제거하면 대부분의 실행 파일과 바로가기,
관련 설정이 정리됩니다.
남은 파일이 있더라도 다음 설치에 큰 영향을 주지 않는 경우가 많습니다.

  • 프로그램 파일 제거
  • 바로가기 제거
  • 간단한 설정 파일 제거

Oracle 삭제

Oracle은 Windows 서비스, Listener, 레지스트리, PATH 환경 변수, Oracle Home,
데이터베이스 파일이 함께 연결되어 있어 삭제 후에도 흔적이 남을 수 있습니다.

  • Oracle Database 프로그램
  • OracleServiceXE 같은 Windows 서비스
  • Oracle Listener
  • 레지스트리 설정
  • PATH 환경 변수
  • 설치 폴더와 데이터 파일

중요한 관점

Oracle 삭제는 “프로그램 하나를 지우는 작업”이 아니라 Windows에 남아 있는 Oracle 관련 연결 정보를 정리하는 과정입니다.
그래서 삭제 후 재설치가 목적이라면 제어판 삭제만 보고 끝났다고 판단하면 안 됩니다.

3. 삭제하기 전에 먼저 판단할 것

SQL Developer 접속이 실패한다고 해서 바로 Oracle을 삭제하면 안 됩니다. 접속 실패의 원인은 설치 자체가 아니라 접속 정보, 계정 잠금, 서비스 중지, 포트 오류일 수 있습니다.

삭제하지 않아도 될 가능성이 높은 경우

  • 비밀번호를 잘못 입력한 경우
  • 포트 번호를 잘못 입력한 경우
  • SID와 Service Name을 헷갈린 경우
  • 서비스 이름에 XEPDB1이 아닌 다른 값을 입력한 경우
  • OracleServiceXE가 중지된 경우
  • Listener가 중지된 경우
  • HR 계정이 잠겨 있는 경우
  • SCOTT 계정이 아직 활성화되지 않은 경우

삭제와 재설치를 고려할 수 있는 경우

  • 설치 도중 중단되어 서비스가 일부만 등록된 경우
  • 제어판 삭제 후에도 Oracle 서비스가 계속 남아 있는 경우
  • 재설치 시 기존 Oracle Home 충돌이 발생하는 경우
  • PATH에 예전 Oracle 경로가 남아 계속 충돌하는 경우
  • 여러 Oracle 버전을 설치했다가 경로와 포트가 꼬인 경우
  • 학습 환경을 완전히 초기화해야 하는 경우

4. Oracle 삭제와 재설치 준비 흐름

Oracle XE 계열을 삭제하고 재설치를 준비할 때는 보통 아래 순서로 생각합니다. 실제 작업에서는 각 단계마다 Oracle 관련 항목이 맞는지 반드시 확인해야 합니다.

01 제어판 삭제

제어판의 프로그램 및 기능에서 Oracle Database XE를 제거합니다. 이 단계는 Oracle 프로그램의 기본 제거 과정입니다.

02 서비스 확인

Windows 서비스에 Oracle로 시작하는 항목이 남아 있는지 확인합니다. 남은 서비스는 재설치 충돌의 원인이 될 수 있습니다.

03 레지스트리 확인

regedit에서 Oracle 관련 항목이 남아 있는지 확인합니다. 레지스트리는 위험하므로 정확한 항목만 신중하게 다룹니다.

04 PATH 정리

시스템 환경 변수 PATH에 이전 Oracle 실행 경로가 남아 있는지 확인합니다. 예전 sqlplus 경로가 남으면 새 설치와 충돌할 수 있습니다.

05 재부팅

서비스, 레지스트리, 환경 변수 변경 사항이 반영되도록 재부팅합니다. 남은 Oracle 프로세스나 잠긴 파일도 정리됩니다.

06 설치 폴더 삭제

마지막으로 실제 설치 폴더가 남아 있다면 확인 후 삭제합니다. 설치 경로를 잘못 판단하면 다른 파일을 지울 수 있으므로 주의합니다.

5. 실행 순서 그림으로 이해하기

Oracle 삭제와 재설치 준비는 “삭제 버튼 클릭”이 아니라, Windows에 남은 Oracle 관련 흔적을 순서대로 정리하는 과정입니다.

접속 오류 발생 비밀번호, 포트, 서비스 이름, 계정 상태 먼저 확인
→
삭제 필요 판단 단순 접속 오류인지 설치 꼬임인지 구분
→
Oracle 제거 제어판에서 Oracle Database XE 제거
→
흔적 정리 서비스, 레지스트리, PATH, 설치 폴더 확인
→
재설치 준비 재부팅 후 관리자 권한으로 새 설치 준비

비유로 이해하기

Oracle 프로그램은 사무실이고, Windows 서비스는 출입 시스템, 레지스트리는 건물 관리대장, PATH는 사무실 위치를 알려주는 길 안내,
설치 폴더는 사무실에 남은 집기와 문서라고 볼 수 있습니다. 사무실만 비웠다고 끝이 아니라 관리대장과 길 안내까지 정리해야 새 사무실을 깔끔하게 만들 수 있습니다.

6. 삭제 전 SQL Developer에서 확인할 SQL

이번 단원에서 SQL은 Oracle을 삭제하는 명령이 아니라, 삭제가 정말 필요한지 판단하기 위한 상태 확인용 명령입니다.
아래 SQL이 정상 실행된다면 Oracle 자체가 완전히 망가진 것은 아닐 수 있습니다.

현재 접속 계정 확인

SHOW USER;

현재 어떤 계정으로 접속했는지 확인합니다. SYSTEM, SCOTT, HR 중 어느 계정인지 먼저 확인해야 테이블이 보이지 않는 원인을 판단할 수 있습니다.

PDB 목록 확인

SELECT name
FROM v$pdbs;

Oracle XE의 기본 PDB인 XEPDB1이 정상적으로 보이는지 확인합니다. XEPDB1이 보이면 PDB 자체는 존재하는 상태입니다.

서비스와 PDB 연결 확인

SELECT name, pdb
FROM v$services
ORDER BY name;

SQL Developer 또는 Spring Boot JDBC URL에서 사용하는 서비스 이름이 실제 PDB와 연결되어 있는지 확인합니다.

CDB 이름 확인

SELECT name
FROM v$database;

현재 Oracle Database의 CDB 이름을 확인합니다. 이 SQL이 정상 실행된다면 Oracle 서버에 접속은 가능한 상태입니다.

현재 계정의 테이블 목록 확인

SELECT table_name
FROM user_tables;

현재 계정이 소유한 테이블을 확인합니다. SCOTT 계정에서 EMP, DEPT가 보이지 않는다면 Oracle 전체 삭제보다 SCOTT 활성화 여부를 먼저 확인해야 합니다.

7. Oracle 프로그램 삭제와 SQL 삭제 명령 구분

이번 단원의 삭제는 Windows에서 Oracle Database 프로그램을 제거하는 것입니다. SQL에서 사용하는 DELETE, DROP, TRUNCATE와는 완전히 다른 개념입니다.

Oracle 프로그램 삭제

제어판, 레지스트리, PATH, 설치 폴더를 정리하여
Oracle Database 서버 프로그램 자체를 제거합니다. 

제어판 → 프로그램 및 기능 → Oracle 제거
regedit에서 Oracle 관련 항목 확인
PATH에서 Oracle 경로 확인
재부팅 후 설치 폴더 정리

SQL 삭제 명령

데이터베이스 안의 데이터나 객체를 삭제하는 SQL 명령입니다.
Oracle 프로그램을 삭제하는 작업이 아닙니다. 

DELETE FROM emp
WHERE empno = 7788;

DROP TABLE emp;

TRUNCATE TABLE emp;

8. 실습 문제 연결

이번 단원은 SQL 문제를 직접 푸는 단원이 아니라, SQL 실습 문제를 풀 수 있는 환경이 망가졌을 때 복구 방향을 이해하는 단원입니다.

SCOTT 계정 실습과 연결

실습 문제의 초반부는 SCOTT 계정의
EMP, DEPT 테이블을 사용합니다.
Oracle 설치나 SCOTT 계정이 꼬여 있으면
WHERE, ORDER BY, GROUP BY
문제를 풀 수 없습니다.

HR 계정 실습과 연결

HR 계정의 EMPLOYEES, DEPARTMENTS, LOCATIONS 테이블은 JOIN과 함수 실습에
많이 사용됩니다. HR 계정이 잠겨 있거나
샘플 스키마가 없으면
실습 문제를 진행할 수 없습니다.

Spring 프로젝트와 연결

Spring Boot 프로젝트에서
Oracle 접속이 실패할 때도
바로 코드를 고치는 것이 아니라,
SQL Developer 접속, 서비스 상태, XEPDB1, 계정,
권한, PATH 상태를 순서대로 확인해야 합니다.

9. 자주 하는 실수

  • 접속 실패만 보고 바로 Oracle을 삭제함 비밀번호, 포트, 서비스 이름, 계정 잠금, Listener 상태 문제일 수 있습니다. 삭제는 마지막 선택지입니다.
  • 제어판에서 삭제하면 완전히 삭제됐다고 생각함 Oracle은 Windows 서비스, 레지스트리, PATH, 설치 폴더에 흔적이 남을 수 있습니다.
  • 레지스트리를 무작정 삭제함 레지스트리는 Windows 핵심 설정입니다. Oracle 관련 항목인지 정확히 확인하고, 가능하면 백업 후 작업해야 합니다.
  • PATH 정리를 생략함 이전 Oracle 실행 경로가 PATH에 남아 있으면 새 설치 후에도 예전 sqlplus 경로를 참조할 수 있습니다.
  • 재부팅을 생략함 서비스, 레지스트리, 환경 변수 변경 사항은 재부팅 후 정상 반영되는 경우가 많습니다.
  • 설치 폴더를 먼저 삭제함 폴더를 먼저 삭제하면 제거 프로그램이 정상 동작하지 않을 수 있습니다. 제어판 삭제 후 마지막 단계에서 폴더를 확인하는 것이 안전합니다.

10. 면접 포인트

Q1. Oracle 삭제가 제어판 삭제만으로 끝나지 않을 수 있는 이유는 무엇인가요?

Oracle Database는 서버 프로그램이기 때문에 설치 시 Windows 서비스, Listener, 레지스트리, PATH 환경 변수, Oracle Home 같은 여러 요소가 함께 등록됩니다.
제어판에서 프로그램을 제거해도 일부 정보가 남아 재설치 시 충돌할 수 있습니다.

Q2. SQL Developer 접속 실패 시 바로 Oracle을 삭제해야 하나요?

아닙니다. 먼저 비밀번호, 포트, 서비스 이름, SID와 Service Name 선택, OracleServiceXE, Listener 실행 여부, 계정 잠금 상태를 확인해야 합니다.
단순 접속 설정 문제라면 삭제 없이 해결할 수 있습니다.

Q3. PATH 환경 변수는 Oracle 재설치와 어떤 관련이 있나요?

PATH는 Windows가 실행 파일을 찾는 경로 목록입니다.
Oracle 삭제 후에도 예전 Oracle bin 경로가 남아 있으면 새 Oracle이 아닌 이전 실행 파일을 참조하거나 버전 충돌이 발생할 수 있습니다.

Q4. Oracle 프로그램 삭제와 SQL의 DELETE, DROP은 어떻게 다른가요?

Oracle 프로그램 삭제는 Windows에서 Oracle Database 서버 자체를 제거하는 작업입니다.
DELETE는 테이블 안의 데이터를 삭제하는 DML이고, DROP은 테이블 같은 데이터베이스 객체를 삭제하는 DDL입니다.

Q5. Spring Boot에서 Oracle 연결이 실패하면 어떤 순서로 점검하겠습니까?

먼저 SQL Developer에서 같은 host, port, service name, username, password로 접속되는지 확인합니다.
그 다음 Oracle 서비스와 Listener, XEPDB1 존재 여부, 계정 잠금, 권한 문제를 확인하고, 그래도 해결되지 않으면 설치 환경 문제를 검토합니다.

11. 이번 단원 핵심 정리

Oracle 삭제는 프로그램 하나를 지우는 작업이 아니라, Windows 서비스, 레지스트리, PATH 환경 변수, 설치 폴더에 남은 Oracle 흔적을 정리하는 과정입니다.
SQL Developer 접속 오류가 발생했다고 바로 삭제하면 안 됩니다. 먼저 계정, 비밀번호, 포트, 서비스 이름, Listener, PDB 상태를 확인해야 합니다.
Oracle XE 재설치 준비는 보통 제어판 삭제 → 서비스 확인 → 레지스트리 확인 → PATH 정리 → 재부팅 → 설치 폴더 삭제 흐름으로 이해하면 됩니다.

학습 메모

이 페이지는 Oracle Database가 정상적으로 설치되지 않았거나 재설치가 필요할 때, 어떤 순서로 삭제와 환경 정리를 준비해야 하는지 공부하는 단원입니다.
Oracle은 일반 프로그램과 달리 Windows 서비스, 레지스트리, PATH 환경 변수, 설치 폴더에 흔적을 남길 수 있으므로 제어판 삭제만으로 끝났다고 판단하면 안 됩니다.
접속 오류가 생겼을 때는 먼저 계정, 비밀번호, 포트, XEPDB1 서비스 이름, Listener와 PDB 상태를 확인하고, 정말 설치 환경이 꼬였을 때만 삭제와 재설치를 검토해야 합니다.
