Oracle SQL Developer - Chapter 27

PL/SQL 예외 처리와 프로시저

예외 처리는 PL/SQL 실행 중 발생하는 오류를 안전하게 처리하는 구조이고,
저장 프로시저는 자주 사용하는 PL/SQL 로직을 데이터베이스에 저장해 반복 호출하는 객체입니다.
이번 단원에서는 PREDEFINED, NON-PREDEFINED, USER DEFINE 예외와
SQLCODE, SQLERRM, PRAGMA EXCEPTION_INIT, RAISE, RAISE_APPLICATION_ERROR,
그리고 CREATE OR REPLACE PROCEDURE, IN, OUT, IN OUT 매개변수까지 학습합니다.

학습 목표 오류 발생 흐름을 제어하고
PL/SQL 로직을 프로시저로 저장한다.
실습 방향 예외 실습 → 복사 테이블 생성 →
프로시저 생성 → OUT 값 확인 → 오류 확인
프로젝트 연결 Spring, MyBatis, JDBC에서
Oracle 저장 프로시저를 호출하는 구조로 연결한다.
Overview

1. 이번 단원은 무엇을 배우는가?

앞 단원에서는 PL/SQL의 기본 블록, 변수, 조건문, 반복문, SELECT INTO, DML 결과 확인을 배웠습니다. 이번 단원은 그 내용을 실무형으로 확장합니다.
실행 중 오류가 발생했을 때 안전하게 처리하는 예외 처리를 배우고, 자주 쓰는 PL/SQL 로직을 데이터베이스에 저장해 반복 호출하는 저장 프로시저를 배웁니다.

익명 PL/SQL 블록 
DECLARE
    v_empno EMP.EMPNO%TYPE := 7369;
BEGIN
    UPDATE EMP
    SET SAL = 1200
    WHERE EMPNO = v_empno;
END;
/

한 번 실행하고 끝나는 일회성 블록입니다.

→
저장 프로시저 
CREATE OR REPLACE PROCEDURE emp_sal_proc
IS
BEGIN
    UPDATE EMP
    SET SAL = 1200
    WHERE EMPNO = 7369;
END;
/

DB에 저장해두고 필요할 때마다 호출할 수 있습니다.

예외 처리 실패 상황을 그냥 중단하지 않고
메시지, ROLLBACK, 로그, OUT 값으로 제어한다.
저장 프로시저 반복되는 PL/SQL 업무 로직을
이름 있는 DB 객체로 저장한다.
Spring 연결 MyBatis CALLABLE, JDBC CallableStatement,
SimpleJdbcCall로 호출한다.

한 줄 정리

예외 처리는 실패를 제어하는 문법이고, 프로시저는 성공과 실패를 포함한 업무 흐름을 DB에 저장해 재사용하는 객체입니다.

Concept Card

2. 단원 개념 카드

2-1. 예외 처리는 왜 필요한가?

PL/SQL 실행 중 오류가 발생하면 블록은 중단됩니다.
예외 처리를 사용하면 오류가 발생했을 때 메시지를 출력하거나, 처리 결과를 OUT 매개변수에 담거나, ROLLBACK을 수행하거나, 호출자에게 업무 오류를 전달할 수 있습니다.

상황 발생 가능 예외 실무 의미
SELECT INTO 결과가 없음 NO_DATA_FOUND 조회하려는 회원, 주문, 게시글이 존재하지 않음
SELECT INTO 결과가 여러 행 TOO_MANY_ROWS 조건이 유일하지 않음, PK 기준 조회가 아님
0으로 나누기 ZERO_DIVIDE 비율, 평균, 정산 계산에서 분모가 0
PK 또는 UNIQUE 중복 DUP_VAL_ON_INDEX 아이디, 코드, 주문번호 중복
업무 규칙 위반 USER DEFINE EXCEPTION 재고 부족, 음수 급여, 이미 취소된 주문

2-2. 예외의 세 가지 종류

PREDEFINED EXCEPTION 미리 정의된 예외

Oracle이 자주 발생하는 오류에 이름을 붙여 둔 예외입니다.

NO_DATA_FOUND
TOO_MANY_ROWS
ZERO_DIVIDE
DUP_VAL_ON_INDEX
NON-PREDEFINED EXCEPTION 비미리정의 예외

오류 번호는 있지만 이름이 없는 예외를 사용자가 이름과 연결합니다.

e_null_deptno EXCEPTION;
PRAGMA EXCEPTION_INIT(e_null_deptno, -1400);
USER DEFINE EXCEPTION 사용자 정의 예외

Oracle 오류가 아니어도 업무 조건에 따라 직접 예외를 발생시킵니다.

e_invalid_salary EXCEPTION;

IF p_sal < 0 THEN
    RAISE e_invalid_salary;
END IF;
RAISE_APPLICATION_ERROR 애플리케이션 오류 전달

사용자 정의 오류 번호와 메시지를 Oracle 오류처럼 호출자에게 전달합니다.

RAISE_APPLICATION_ERROR(-20001, '급여는 0 이상이어야 합니다.');

2-3. 프로시저란?

저장 프로시저는 PL/SQL 블록에 이름을 붙여 데이터베이스에 저장한 객체입니다.
익명 블록은 한 번 실행하고 끝나지만, 프로시저는 DB에 저장되어 필요할 때마다 호출할 수 있습니다.

익명 블록

DECLARE
BEGIN
    실행문;
END;
/

한 번 실행하고 끝납니다. 이름이 없고 DB 객체로 저장되지 않습니다.

저장 프로시저

CREATE OR REPLACE PROCEDURE 프로시저명
IS
BEGIN
    실행문;
END;
/

이름을 가진 DB 객체로 저장되고, EXECUTE로 반복 호출합니다.

7. SQL 예제 카드 - 미리 정의된 예외

미리 정의된 예외는 Oracle이 자주 발생하는 오류에 미리 이름을 붙여둔 예외입니다.
처음에는 NO_DATA_FOUND, TOO_MANY_ROWS, ZERO_DIVIDE, DUP_VAL_ON_INDEX 네 가지를 확실히 익히면 됩니다.

NO_DATA_FOUND SELECT INTO 0행
TOO_MANY_ROWS SELECT INTO 여러 행
ZERO_DIVIDE 0으로 나누기
DUP_VAL_ON_INDEX PK / UNIQUE 중복

7-1. ZERO_DIVIDE

DECLARE
    v_result NUMBER;
BEGIN
    v_result := 10 / 0;

    DBMS_OUTPUT.PUT_LINE('결과: ' || v_result);
EXCEPTION
    WHEN ZERO_DIVIDE THEN
        DBMS_OUTPUT.PUT_LINE('0으로 나눌 수 없습니다.');
END;
/
결과 확인 0으로 나눌 수 없습니다.

7-2. NO_DATA_FOUND

DECLARE
    v_ename EMP.ENAME%TYPE;
BEGIN
    SELECT ENAME
    INTO v_ename
    FROM EMP
    WHERE EMPNO = -1;

    DBMS_OUTPUT.PUT_LINE('사원명: ' || v_ename);
EXCEPTION
    WHEN NO_DATA_FOUND THEN
        DBMS_OUTPUT.PUT_LINE('조회된 사원이 없습니다.');
END;
/

7-3. TOO_MANY_ROWS

DECLARE
    v_ename EMP.ENAME%TYPE;
BEGIN
    SELECT ENAME
    INTO v_ename
    FROM EMP
    WHERE DEPTNO = 20;

    DBMS_OUTPUT.PUT_LINE('사원명: ' || v_ename);
EXCEPTION
    WHEN TOO_MANY_ROWS THEN
        DBMS_OUTPUT.PUT_LINE('조회 결과가 2행 이상입니다.');
END;
/

7-4. DUP_VAL_ON_INDEX

BEGIN
    INSERT INTO dept_proc_copy(DEPTNO, DNAME, LOC)
    VALUES(10, '중복부서', 'SEOUL');

    DBMS_OUTPUT.PUT_LINE('입력 성공');
EXCEPTION
    WHEN DUP_VAL_ON_INDEX THEN
        DBMS_OUTPUT.PUT_LINE('이미 존재하는 부서번호입니다.');
END;
/

7-5. OTHERS와 SQLCODE, SQLERRM

DECLARE
    v_name VARCHAR2(3);
BEGIN
    v_name := 'ORACLE';

    DBMS_OUTPUT.PUT_LINE(v_name);
EXCEPTION
    WHEN OTHERS THEN
        DBMS_OUTPUT.PUT_LINE('알 수 없는 오류가 발생했습니다.');
        DBMS_OUTPUT.PUT_LINE('오류 코드: ' || SQLCODE);
        DBMS_OUTPUT.PUT_LINE('오류 메시지: ' || SQLERRM);
END;
/

VARCHAR2(3)에 ORACLE을 넣으려 했기 때문에 VALUE_ERROR 계열 오류가 발생합니다.

8. SQL 예제 카드 - 비미리정의 예외와 사용자 정의 예외

8-1. PRAGMA EXCEPTION_INIT

DECLARE
    e_null_deptno EXCEPTION;
    PRAGMA EXCEPTION_INIT(e_null_deptno, -1400);
BEGIN
    INSERT INTO dept_proc_copy(DEPTNO, DNAME, LOC)
    VALUES(NULL, 'NULL부서', 'SEOUL');

    DBMS_OUTPUT.PUT_LINE('입력 성공');
EXCEPTION
    WHEN e_null_deptno THEN
        DBMS_OUTPUT.PUT_LINE('부서번호는 NULL일 수 없습니다.');
    WHEN OTHERS THEN
        DBMS_OUTPUT.PUT_LINE('기타 오류: ' || SQLCODE || ' / ' || SQLERRM);
END;
/

ORA-01400을 e_null_deptno라는 예외명과 연결합니다.

8-2. 사용자 정의 예외

DECLARE
    v_sal EMP.SAL%TYPE := -100;
    e_invalid_salary EXCEPTION;
BEGIN
    IF v_sal < 0 THEN
        RAISE e_invalid_salary;
    END IF;

    DBMS_OUTPUT.PUT_LINE('급여: ' || v_sal);
EXCEPTION
    WHEN e_invalid_salary THEN
        DBMS_OUTPUT.PUT_LINE('급여는 0 이상이어야 합니다.');
END;
/

Oracle 문법 오류가 아니라 업무 규칙 위반을 직접 예외로 처리합니다.

8-3. RAISE_APPLICATION_ERROR

BEGIN
    RAISE_APPLICATION_ERROR(-20001, '급여는 0 이상이어야 합니다.');
END;
/
결과 확인 ORA-20001: 급여는 0 이상이어야 합니다.

사용자 정의 오류 번호와 메시지를 호출자에게 Oracle 오류처럼 전달합니다. Spring에서는 SQLException 또는 DataAccessException 계열로 받을 수 있습니다.

RAISE와 RAISE_APPLICATION_ERROR 차이

RAISE 사용자정의예외
→ PL/SQL 블록 안에서 선언한 예외를 발생시킨다.

RAISE_APPLICATION_ERROR
→ -20000 ~ -20999 범위의 사용자 오류 번호와 메시지를 호출자에게 전달한다.

9. SQL 예제 카드 - 트랜잭션과 예외 처리

예외 처리는 단순히 메시지를 출력하는 것에서 끝나지 않습니다. 실무에서는 오류 발생 시 이미 수행된 DML을 되돌릴지, 일부만 되돌릴지 결정해야 합니다.

COMMIT / ROLLBACK 개념

COMMIT;
-- 변경 사항 확정

ROLLBACK;
-- 마지막 COMMIT 이후 변경 취소

SAVEPOINT 개념

SAVEPOINT before_update;

ROLLBACK TO before_update;

전체가 아니라 특정 저장 지점까지만 되돌릴 수 있습니다.

9-1. SAVEPOINT 실습

BEGIN
    SAVEPOINT before_update;

    UPDATE emp_proc_copy
    SET SAL = SAL + 100
    WHERE DEPTNO = 10;

    DBMS_OUTPUT.PUT_LINE(SQL%ROWCOUNT || '개의 행을 수정했습니다.');

    ROLLBACK TO before_update;

    DBMS_OUTPUT.PUT_LINE('SAVEPOINT 지점으로 ROLLBACK 했습니다.');
END;
/
결과 확인 수정 메시지는 출력되지만 ROLLBACK TO 때문에 급여 변경은 취소됩니다.

9-2. 확인 SELECT

SELECT EMPNO,
       ENAME,
       SAL,
       DEPTNO
FROM emp_proc_copy
WHERE DEPTNO = 10
ORDER BY EMPNO;

Spring 프로젝트 주의

  • SQL Developer 단독 실습에서는 프로시저 안에서 COMMIT을 해도 흐름을 보기 쉽습니다.
  • Spring Service에서 호출하는 프로시저라면 내부 COMMIT이 @Transactional 롤백과 충돌할 수 있습니다.
  • 실무에서는 트랜잭션 책임을 Java Service에 둘지, DB 프로시저에 둘지 명확히 정해야 합니다.

CREATE OR REPLACE PROCEDURE print_emp_proc
(
    p_empno IN EMP.EMPNO%TYPE
)
IS
    v_ename EMP.ENAME%TYPE;
    v_sal   EMP.SAL%TYPE;
BEGIN
    SELECT ENAME, SAL
    INTO v_ename, v_sal
    FROM EMP
    WHERE EMPNO = p_empno;

    WHEN TOO_MANY_ROWS THEN
        DBMS_OUTPUT.PUT_LINE('조회 결과가 2행 이상입니다.');
    WHEN OTHERS THEN
        DBMS_OUTPUT.PUT_LINE('오류: ' || SQLCODE || ' / ' || SQLERRM);
END;
/

11-2. OUT 매개변수 프로시저

CREATE OR REPLACE PROCEDURE emp_sal_proc
(
    p_empno  IN  emp_proc_copy.empno%TYPE,
    p_sal    IN  emp_proc_copy.sal%TYPE,
    p_count  OUT NUMBER,
    p_msg    OUT VARCHAR2
)
IS
    e_invalid_salary EXCEPTION;
BEGIN
    p_count := 0;
    p_msg := NULL;

    IF p_sal < 0 THEN
        RAISE e_invalid_salary;
    END IF;

    UPDATE emp_proc_copy
    SET SAL = p_sal
    WHERE EMPNO = p_empno;

    p_count := SQL%ROWCOUNT;

    IF SQL%NOTFOUND THEN
        p_msg := TO_CHAR(p_empno) || '번 사원번호가 존재하지 않습니다.';
    ELSE
        p_msg := TO_CHAR(p_count) || '개의 행이 수정되었습니다.';
    END IF;

EXCEPTION
    WHEN e_invalid_salary THEN
        p_count := 0;
        p_msg := '급여는 0 이상이어야 합니다.';
    WHEN OTHERS THEN
        p_count := -1;
        p_msg := '오류: ' || SQLCODE || ' / ' || SQLERRM;
END;
/

CREATE OR REPLACE PROCEDURE update_emp_info_proc

    p_ename IN  emp_proc_copy.ename%TYPE,
    p_sal   IN  emp_proc_copy.sal%TYPE,

END;

CREATE OR REPLACE PROCEDURE sp_check_member
(
    i_empno IN  EMP.EMPNO%TYPE,
    i_ename IN  EMP.ENAME%TYPE,
    flag    OUT NUMBER
)
IS
    cnt NUMBER := 0;
BEGIN
    SELECT COUNT(*)
    INTO cnt
    FROM EMP
    WHERE EMPNO = i_empno;

    IF cnt = 0 THEN
        flag := 0;
    ELSE
        cnt := 0;

        SELECT COUNT(*)
        INTO cnt
        FROM EMP
        WHERE EMPNO = i_empno
          AND ENAME = UPPER(i_ename);

        IF cnt = 1 THEN
            flag := 2;
        ELSE
            flag := 1;
        END IF;
    END IF;
END;
/
VARIABLE v_flag NUMBER;

EXEC sp_check_member(7369, 'SMITH', :v_flag);
PRINT v_flag;

EXEC sp_check_member(7369, 'KING', :v_flag);
PRINT v_flag;

EXEC sp_check_member(9999, 'SMITH', :v_flag);
PRINT v_flag;

0은 사원번호 없음, 1은 이름 불일치, 2는 사원번호와 이름 일치입니다.

CREATE OR REPLACE PROCEDURE even_sum_proc
(
    p_endnum IN  NUMBER,
    p_sum    OUT NUMBER
)
IS
BEGIN
    p_sum := 0;

    FOR i IN 1..p_endnum LOOP
        IF MOD(i, 2) = 0 THEN
            p_sum := p_sum + i;
        END IF;
    END LOOP;
END;
/
VARIABLE v_sum NUMBER;

PRINT v_sum;

15. 실행 권한 카드

다른 계정이 내 프로시저를 실행하려면 EXECUTE 권한이 필요합니다. 테이블 직접 권한을 주지 않고 프로시저 실행 권한만 주면 보안성을 높일 수 있습니다.

GRANT EXECUTE ON emp_sal_proc TO HR;

15-2. 다른 계정에서 실행

EXEC SCOTT.emp_sal_proc(7369, 1200, :v_count, :v_msg);

15-3. 실행 권한 회수

REVOKE EXECUTE ON emp_sal_proc FROM HR;

15-4. 권한 설계 포인트

테이블 직접 UPDATE 권한
→ 위험할 수 있음

프로시저 EXECUTE 권한
→ 정해진 업무 로직만 실행 가능

Mistake

17. 자주 하는 실수

  • 실수 1. 예외 처리를 오류가 안 나게 하는 문법으로 생각함 예외 처리는 오류를 없애는 문법이 아니라, 오류가 발생했을 때 처리 흐름을 정하는 문법입니다.
  • 실수 2. SELECT INTO가 여러 행을 처리할 수 있다고 생각함 SELECT INTO는 정확히 1행만 처리합니다. 여러 행은 커서를 사용해야 합니다.
  • 실수 3. WHEN OTHERS THEN NULL로 오류를 숨김 오류를 숨기면 장애 원인을 찾기 어렵습니다. SQLCODE와 SQLERRM을 기록해야 합니다.
  • 실수 4. PRAGMA EXCEPTION_INIT 오류 번호를 양수로 씀 ORA-01400은 -1400처럼 음수로 연결해야 합니다.
  • 실수 5. OUT 매개변수에 리터럴을 전달함 OUT 값은 바인드 변수로 받아야 합니다. :v_count, :v_msg처럼 전달합니다.
  • 실수 6. IN OUT 매개변수에 숫자 리터럴을 전달함 IN OUT은 변경된 값을 돌려받아야 하므로 반드시 변수를 전달해야 합니다.
  • 실수 7. 프로시저 매개변수 자료형에 크기를 씀 p_msg OUT VARCHAR2(200)가 아니라 p_msg OUT VARCHAR2처럼 씁니다.
  • 실수 8. 프로시저에 함수처럼 RETURN 값을 쓰려고 함 프로시저는 OUT 매개변수로 값을 돌려주고, 함수는 RETURN 값으로 돌려줍니다.
  • 실수 9. 프로시저 생성 후 마지막 /를 빼먹음 END; 다음 새 줄의 /가 있어야 SQL Developer가 프로시저를 컴파일합니다.
  • 실수 10. 프로시저가 INVALID인데 실행함 USER_OBJECTS의 STATUS와 SHOW ERRORS로 컴파일 오류를 먼저 확인합니다.
  • 실수 11. Spring에서 호출할 프로시저에 무조건 COMMIT을 넣음 @Transactional과 충돌할 수 있으므로 트랜잭션 책임을 명확히 정해야 합니다.
  • 실수 12. DBMS_OUTPUT을 API 응답으로 착각함 Spring API 응답은 OUT 매개변수, 결과 셋, 예외, 로그 테이블 등을 사용해야 합니다.
Interview

18. 면접 질문 카드

Q1. PL/SQL 예외 처리란 무엇인가요?

PL/SQL 실행 중 발생하는 오류를 EXCEPTION 영역에서 처리하는 구조입니다.
오류가 발생하면 실행 흐름이 EXCEPTION 영역으로 이동하고, 해당 예외명과 일치하는 WHEN 절이 실행됩니다.

Q2. 예외의 종류를 설명해보세요.

PREDEFINED EXCEPTION은 Oracle이 미리 이름을 붙여둔 예외입니다.
NON-PREDEFINED EXCEPTION은 오류 번호는 있지만 이름이 없는 오류를 PRAGMA EXCEPTION_INIT으로 연결한 예외입니다.
USER DEFINE EXCEPTION은 사용자가 직접 선언하고 RAISE로 발생시키는 예외입니다.

Q3. NO_DATA_FOUND와 TOO_MANY_ROWS는 언제 발생하나요?

둘 다 SELECT INTO와 관련됩니다. NO_DATA_FOUND는 SELECT INTO 결과가 0행일 때 발생하고, TOO_MANY_ROWS는 결과가 2행 이상일 때 발생합니다.
SELECT INTO는 정확히 한 행만 처리할 수 있습니다.

Q4. WHEN OTHERS는 무엇인가요?

명시적으로 처리하지 않은 모든 예외를 잡는 구문입니다.
실무에서는 오류를 숨기지 말고 SQLCODE, SQLERRM을 로그로 남기거나 필요하면 RAISE로 다시 예외를 던져야 합니다.

Q5. SQLCODE와 SQLERRM은 무엇인가요?

SQLCODE는 현재 발생한 Oracle 오류 코드를 숫자로 반환하고, SQLERRM은 해당 오류 메시지를 반환합니다. 예외 처리에서 원인 파악과 로그 기록에 사용합니다.

Q6. PRAGMA EXCEPTION_INIT은 언제 사용하나요?

Oracle 오류 번호는 있지만 미리 정의된 예외명이 없는 오류를 사용자가 선언한 예외명과 연결할 때 사용합니다. 예를 들어 ORA-01400은 -1400으로 연결합니다.

Q7. 사용자 정의 예외는 언제 사용하나요?

Oracle 문법 오류는 아니지만 업무 규칙을 위반했을 때 사용합니다.
예를 들어 급여가 음수이거나, 재고가 부족하거나, 이미 취소된 주문을 다시 취소하려는 경우 직접 예외를 발생시킬 수 있습니다.

Q8. RAISE와 RAISE_APPLICATION_ERROR의 차이는 무엇인가요?

RAISE는 PL/SQL 블록 내부에서 선언한 예외를 발생시킵니다. RAISE_APPLICATION_ERROR는 사용자 정의 오류 번호와 메시지를 Oracle 오류처럼 호출자에게 전달합니다.

Q9. 저장 프로시저란 무엇인가요?

저장 프로시저는 이름을 가진 PL/SQL 블록을 데이터베이스에 저장한 객체입니다.
자주 사용하는 업무 로직을 DB에 저장해두고 EXECUTE 또는 애플리케이션에서 반복 호출할 수 있습니다.

Q10. 프로시저와 함수의 차이는 무엇인가요?

프로시저는 작업 수행 중심이고, 함수는 값 반환 중심입니다. 함수는 RETURN 값이 필수이며 SELECT문 안에서 사용할 수 있습니다.
프로시저는 보통 EXECUTE로 실행하고, 결과가 필요하면 OUT 매개변수를 사용합니다.

Q11. IN, OUT, IN OUT 매개변수 차이는 무엇인가요?

IN은 호출자가 프로시저로 값을 전달하는 입력용 매개변수입니다.
OUT은 프로시저가 호출자에게 값을 돌려주는 출력용 매개변수입니다.
IN OUT은 값을 입력받고, 프로시저 내부에서 변경한 값을 다시 호출자에게 돌려줍니다.

Q12. OUT 매개변수는 어떻게 실행하나요?

OUT 값을 받을 바인드 변수를 먼저 선언하고, 프로시저 호출 시 콜론을 붙여 전달한 뒤 PRINT로 확인합니다. 

VARIABLE v_count NUMBER;

EXEC emp_sal_proc(7369, 1200, :v_count);

PRINT v_count;

Q13. 프로시저 컴파일 오류는 어떻게 확인하나요?

SHOW ERRORS PROCEDURE 프로시저명 또는 USER_ERRORS를 조회해서 확인합니다.

Q14. Spring에서 프로시저를 호출할 때 주의할 점은 무엇인가요?

Spring에서는 보통 @Transactional이 트랜잭션을 관리합니다.
프로시저 내부에서 무조건 COMMIT하거나 ROLLBACK하면 Java 쪽 트랜잭션 제어와 충돌할 수 있습니다.
또한 DBMS_OUTPUT은 API 응답용이 아니므로 OUT 파라미터나 예외를 통해 결과를 전달하는 것이 좋습니다.

Q15. 실무에서 프로시저는 언제 쓰나요?

대량 데이터 처리, 배치, 정산, 통계 적재, DB 내부에서 여러 DML을 묶어 처리해야 하는 경우,
레거시 시스템 연동, 테이블 직접 권한 대신 프로시저 실행 권한만 부여해야 하는 경우에 사용합니다.

학습 메모

오늘의 핵심

예외 처리는 실패를 제어하는 문법이고, 프로시저는 성공과 실패를 포함한 업무 흐름을 DB에 저장해 재사용하는 객체입니다.

예외 처리 기본형

BEGIN
    실행문;
EXCEPTION
    WHEN 예외명 THEN
        처리문;
    WHEN OTHERS THEN
        처리문;
END;
/

예외 종류

PREDEFINED
→ Oracle이 미리 이름을 붙인 예외

NON-PREDEFINED
→ 오류 번호를 예외명과 연결

USER DEFINE
→ 사용자가 직접 선언하고 RAISE

자주 쓰는 예외

NO_DATA_FOUND
TOO_MANY_ROWS
ZERO_DIVIDE
DUP_VAL_ON_INDEX
VALUE_ERROR
INVALID_NUMBER

프로시저 기본형

CREATE OR REPLACE PROCEDURE 프로시저명
(
    p_in  IN  자료형,
    p_out OUT 자료형
)
IS
BEGIN
    실행문;
END;
/

OUT 실행 공식

VARIABLE v_msg VARCHAR2(200);

EXEC 프로시저명(입력값, :v_msg);

PRINT v_msg;

상태 확인

SELECT OBJECT_NAME, STATUS
FROM USER_OBJECTS
WHERE OBJECT_TYPE = 'PROCEDURE';

SHOW ERRORS PROCEDURE 프로시저명;

주의

WHEN OTHERS THEN NULL은 오류를 숨기므로 위험합니다. SQLCODE와 SQLERRM을 남기거나 필요한 경우 RAISE로 다시 예외를 던지세요.

Spring 연결

MyBatis는 statementType="CALLABLE", JDBC는 CallableStatement, Spring JDBC는 SimpleJdbcCall, JPA는 StoredProcedureQuery로 프로시저를 호출할 수 있습니다.
