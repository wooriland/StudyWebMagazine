Oracle SQL Developer · 03

Multi Tenant, CDB, PDB 이해

Oracle 12c 이후 도입된 Multi Tenant 구조를 이해하고,
CDB와 PDB의 차이, Oracle XE에서 XEPDB1에 접속하는 이유를 학습합니다.
이 단원은 SQL 문법보다 Oracle Database의 공간 구조를 이해하는 것이 핵심이며,
이후 SCOTT, HR 계정 실습과 Java / Spring Boot Oracle 연결의 기초가 됩니다.

1. 이 단원에서 배우는 것

이번 단원은 Oracle Database 안의 구조를 이해하는 단원입니다.
Oracle을 설치하고 SQL Developer로 접속할 때 우리는 단순히 Oracle 서버에 접속하는 것이 아니라, Oracle 안의 특정 데이터베이스 공간인 XEPDB1에 접속합니다.
Oracle 12c 이후부터는 하나의 큰 데이터베이스 컨테이너 안에 여러 개의 독립된 데이터베이스 공간을 둘 수 있는 Multi Tenant Architecture가 중요해졌습니다.
이 구조를 이해해야 왜 SQL Developer에서 서비스 이름에 XEPDB1을 입력하는지, 왜 SCOTT과 HR 계정이 특정 PDB 안에서 사용되는지 이해할 수 있습니다.

Multi Tenant 하나의 Oracle 안에 여러 독립 공간을 둘 수 있는 구조
CDB 여러 PDB를 담는 상위 컨테이너 데이터베이스
PDB 사용자, 테이블, 데이터가 들어가는 독립된 실습 공간

2. Multi Tenant란 무엇인가?

Multi Tenant는 하나의 시스템 안에 여러 개의 독립된 사용 공간을 둘 수 있는 구조입니다.
Oracle에서는 이 개념이 12c 이후부터 중요해졌습니다. 하나의 CDB 안에 여러 개의 PDB를 넣어 관리할 수 있기 때문입니다.
쉽게 말하면 하나의 건물 안에 여러 사무실이 있는 구조와 비슷합니다. 건물은 하나지만, 각 사무실은 독립적으로 운영됩니다.
각 사무실마다 사용하는 사람, 문서, 보안 규칙이 다를 수 있습니다.

비유로 이해하기

CDB는 큰 건물이고, PDB는 그 건물 안의 독립된 사무실입니다.
사용자 계정은 사무실을 사용하는 사람이고, 테이블은 사무실 안의 문서함, 데이터는 문서함 안의 실제 문서라고 볼 수 있습니다.

단일 구조로만 생각했을 때

Oracle Database 하나 안에 모든 사용자, 테이블, 데이터가 단순히 들어 있다고 생각하면 처음에는 쉬워 보이지만, 12c 이후 구조와 서비스 이름, PDB 접속 개념을 이해하기 어렵습니다.

  • Oracle 전체와 실습 공간을 구분하기 어려움
  • XEPDB1의 의미를 단순 문자열로 외우게 됨
  • 사용자 생성 위치를 헷갈리기 쉬움

Multi Tenant 구조로 이해했을 때

Oracle 전체 컨테이너인 CDB와 실제 실습 공간인 PDB를 구분할 수 있습니다. SQL Developer와 Spring Boot가 어느 공간에 접속하는지 명확하게 이해할 수 있습니다.

  • CDB와 PDB 역할 구분 가능
  • XEPDB1 접속 이유 이해 가능
  • SCOTT, HR, APP_USER 위치 이해 가능

3. CDB 이해하기

CDB는 Container Database의 약자입니다. 여러 PDB를 담는 상위 데이터베이스 컨테이너입니다. Oracle 전체 구조를 관리하는 큰 틀이라고 이해하면 됩니다.

CDB의 핵심 역할

  • Oracle 전체 컨테이너 역할을 합니다.
  • 여러 개의 PDB를 담을 수 있습니다.
  • 공통 시스템 리소스를 관리합니다.
  • PDB 생성, 열기, 닫기, 삭제 같은 관리 작업의 기준이 됩니다.
  • 일반 실습보다는 관리자 관점에서 이해해야 하는 상위 구조입니다.

초보 단계에서는 CDB를 “Oracle 전체 건물”로 이해하면 됩니다. 우리가 매일 데이터를 넣고 조회하는 곳은 보통 CDB 자체가 아니라, CDB 안에 들어 있는 특정 PDB입니다.

4. PDB 이해하기

PDB는 Pluggable Database의 약자입니다. CDB 안에 존재하는 독립된 데이터베이스 공간입니다. 실제 사용자, 스키마, 테이블, 데이터는 보통 PDB 안에서 관리됩니다.

PDB의 핵심 역할

  • CDB 안에 들어가는 독립된 데이터베이스 공간입니다.
  • 자체 사용자 계정과 스키마를 가질 수 있습니다.
  • 자체 테이블, 뷰, 시퀀스, 프로시저를 가질 수 있습니다.
  • 다른 PDB와 논리적으로 분리됩니다.
  • 애플리케이션과 SQL Developer는 보통 특정 PDB에 접속합니다.

초보 단계에서는 PDB를 “Oracle 건물 안의 독립된 방”으로 이해하면 됩니다. Oracle XE 실습에서는 이 방의 이름이 보통 XEPDB1입니다.

5. Oracle Instance, CDB, PDB 구조 그림

이번 단원에서 가장 중요하게 기억해야 할 구조입니다. SCOTT, HR, 내가 만든 사용자는 보통 XEPDB1 안에서 실습과 프로젝트에 사용됩니다.

Oracle Instance Oracle 서버 실행 단위
CDB Container Database
PDB$SEED 새 PDB 생성을 위한 기본 템플릿
XEPDB1 Oracle XE 기본 실습 PDB
SYSTEM 관리자 계정
SCOTT EMP, DEPT 실습 계정
HR EMPLOYEES, DEPARTMENTS 실습 계정
APP_USER Spring 프로젝트용 사용자

핵심 기억

SQL Developer에서 서비스 이름에 XEPDB1을 입력한다는 것은 Oracle 전체에 막연히 접속하는 것이 아니라, XEPDB1이라는 PDB 공간에 접속한다는 의미입니다.

6. 왜 XEPDB1에 접속하는가?

Oracle XE를 설치하면 기본적으로 XEPDB1이라는 PDB가 제공됩니다. 그래서 SQL Developer에서 새 접속을 만들 때 서비스 이름에 XEPDB1을 입력합니다.

localhost 내 컴퓨터에 설치된 Oracle 서버
1521 Oracle Listener 기본 포트
XEPDB1 실제로 접속할 PDB
system / scott / hr PDB 안에서 사용하는 사용자 계정

따라서 XEPDB1은 단순히 외워야 하는 문자열이 아닙니다. 앞으로 SQL 실습과 Java / Spring Boot 프로젝트가 연결될 실제 데이터베이스 공간입니다.

7. SQL Developer에서 구조 확인하기

이번 단원에서 실행하는 SQL은 데이터를 조회하기 위한 SQL이 아니라, Oracle 구조가 어떻게 구성되어 있는지 확인하기 위한 환경 확인용 SQL입니다.

현재 접속 계정 확인

SHOW USER;

현재 어떤 Oracle 계정으로 접속했는지 확인합니다. SYSTEM, SCOTT, HR 중 어떤 계정으로 접속했는지 먼저 확인해야 합니다.

PDB 목록 확인

SELECT name
FROM v$pdbs;

현재 CDB 안에 어떤 PDB가 있는지 확인합니다. Oracle XE 환경에서는 보통 PDB$SEED와 XEPDB1을 확인할 수 있습니다.

CDB 이름 확인

SELECT name
FROM v$database;

현재 Oracle Database의 CDB 이름을 확인합니다. Oracle XE 환경에서는 보통 XE 계열 이름을 볼 수 있습니다.

서비스와 PDB 연결 확인

SELECT name, pdb
FROM v$services
ORDER BY name;

SQL Developer에서 사용하는 서비스 이름과 실제 PDB 연결 상태를 확인합니다. Spring Boot JDBC URL의 XEPDB1도 이 서비스 이름과 연결됩니다.

현재 계정의 테이블 목록 확인

SELECT table_name
FROM user_tables;

현재 접속한 계정이 가진 테이블 목록을 확인합니다. SCOTT 계정이라면 EMP, DEPT를 기대할 수 있고, HR 계정이라면 EMPLOYEES, DEPARTMENTS 등을 기대할 수 있습니다.

8. 결과 화면 확인 포인트

SQL Developer에서 위 SQL을 실행하면 결과 그리드가 표시됩니다. 이번 단원에서는 결과값을 통해 Oracle의 공간 구조를 확인하는 것이 중요합니다.

실행 SQL 확인할 내용 의미
SHOW USER; 현재 접속 계정 SYSTEM, SCOTT, HR 등 현재 작업 계정을 확인합니다.
SELECT name FROM v$pdbs; XEPDB1 존재 여부 Oracle XE 기본 PDB가 존재하는지 확인합니다.
SELECT name FROM v$database; CDB 이름 현재 Oracle 전체 데이터베이스 이름을 확인합니다.
SELECT name, pdb FROM v$services; 서비스와 PDB 연결 SQL Developer와 Spring Boot가 접속할 서비스 이름을 확인합니다.
SELECT table_name FROM user_tables; 현재 계정의 테이블 계정과 PDB가 맞는지 테이블 목록으로 확인합니다.

XEPDB1이 보인다는 것은 Oracle XE의 기본 PDB가 존재한다는 뜻입니다. 이후 SCOTT, HR, 내가 만든 사용자 계정은 보통 이 XEPDB1 안에서 실습과 프로젝트에 사용됩니다.

9. Java / Spring 프로젝트와의 연결

Multi Tenant, CDB, PDB 구조는 단순한 Oracle 이론이 아니라 Java와 Spring Boot 프로젝트의 데이터베이스 연결과 직접 연결됩니다.

SQL Developer 접속 정보

Host         : localhost
Port         : 1521
Service Name : XEPDB1
User         : app_user
Password     : app_password

SQL Developer에서는 위 정보로 Oracle의 특정 PDB에 접속합니다.

Spring Boot datasource 예시

spring.datasource.url=jdbc:oracle:thin:@localhost:1521/XEPDB1
spring.datasource.username=app_user
spring.datasource.password=app_password
spring.datasource.driver-class-name=oracle.jdbc.OracleDriver

Spring Boot에서도 같은 구조로 XEPDB1 PDB에 접속합니다.

JPA Entity가 실제 Oracle 테이블과 연결되려면, Spring Boot가 올바른 PDB에 접속하고, 올바른 사용자 계정으로 로그인하며, 해당 계정 안에 테이블이 존재해야 합니다.

10. 실습 문제 연결

이번 단원은 SQL 연산자나 JOIN 문제를 직접 푸는 단원이 아닙니다. 실습 문제를 풀기 전에, 실습이 이루어질 PDB 공간과 계정 구조를 확인하는 단원입니다.

SCOTT 실습 준비

SCOTT 계정의 EMP, DEPT 테이블은 WHERE, ORDER BY, GROUP BY, JOIN, SUBQUERY 문제에서 자주 사용됩니다. SCOTT 계정이 XEPDB1 안에 정상 준비되어 있어야 합니다.

HR 실습 준비

HR 계정의 EMPLOYEES, DEPARTMENTS, JOBS, LOCATIONS, COUNTRIES 테이블은 JOIN과 함수 실습에서 자주 사용됩니다. HR 계정도 XEPDB1 안에서 확인해야 합니다.

환경 확인 중심

이번 단원에서는 문제 풀이보다 v$pdbs, v$database, v$services를 통해 SQL Developer 접속 위치가 정상인지 확인하는 것이 중요합니다.

11. 자주 하는 실수

  • XEPDB1을 사용자 이름으로 착각함 XEPDB1은 계정명이 아니라 Oracle XE에서 기본으로 제공되는 PDB 또는 서비스 이름입니다. 사용자 이름에는 system, scott, hr 같은 계정을 입력합니다.
  • SID와 Service Name을 헷갈림 Oracle XE 실습 환경에서는 보통 SID가 아니라 Service Name에 XEPDB1을 입력합니다.
  • CDB와 PDB를 구분하지 않음 CDB는 상위 컨테이너이고, PDB는 실제 사용자와 테이블이 들어가는 독립된 공간입니다.
  • SYSTEM으로 모든 실습을 진행함 SYSTEM은 관리자 계정입니다. 일반 SQL 실습은 SCOTT, HR 또는 별도 사용자 계정에서 진행하는 것이 좋습니다.
  • 테이블이 안 보이면 바로 재설치를 생각함 먼저 현재 계정, 현재 PDB, user_tables 결과를 확인해야 합니다. 접속 위치가 다르면 테이블이 보이지 않을 수 있습니다.
  • Spring Boot JDBC URL에서 XEPDB1을 빼먹음 Oracle XE의 PDB에 접속하려면 JDBC URL에 localhost:1521/XEPDB1 구조가 필요합니다.

12. 면접 포인트

Q1. Oracle의 Multi Tenant 구조란 무엇인가요?

하나의 CDB 안에 여러 개의 PDB를 둘 수 있는 Oracle 12c 이후의 구조입니다. 각 PDB는 독립적인 데이터베이스처럼 동작하면서 CDB의 공통 리소스를 공유할 수 있습니다.

Q2. CDB와 PDB의 차이는 무엇인가요?

CDB는 여러 PDB를 담는 상위 컨테이너 데이터베이스입니다.
PDB는 CDB 안에 존재하는 논리적으로 분리된 독립 데이터베이스 공간이며, 실제 사용자, 스키마, 테이블, 데이터는 보통 PDB 안에서 관리됩니다.

Q3. Oracle XE에서 XEPDB1은 무엇인가요?

XEPDB1은 Oracle XE에서 기본으로 제공되는 PDB입니다. SQL Developer에서 서비스 이름으로 XEPDB1을 지정하면 해당 PDB에 접속합니다.

Q4. SQL Developer에서 Service Name에 XEPDB1을 입력하는 이유는 무엇인가요?

localhost의 1521 포트로 Oracle Listener에 접근한 뒤, XEPDB1이라는 PDB 서비스에 접속하기 위해서입니다. 즉 XEPDB1은 실제 실습이 이루어질 데이터베이스 공간입니다.

Q5. Spring Boot에서 Oracle URL의 XEPDB1은 어떤 의미인가요?

jdbc:oracle:thin:@localhost:1521/XEPDB1에서 XEPDB1은 접속할 Oracle 서비스 이름 또는 PDB를 의미합니다.
Spring Boot는 이 URL을 통해 XEPDB1 안의 사용자 계정으로 접속합니다.

Q6. 테이블이 보이지 않을 때 어떤 순서로 확인해야 하나요?

먼저 SHOW USER로 현재 계정을 확인하고, v$pdbs와 v$database로 DB 구조를 확인합니다. 이후 user_tables로 현재 계정의 테이블 목록을 확인해야 합니다.

13. 이번 단원 핵심 정리

Oracle 12c 이후에는 하나의 CDB 안에 여러 PDB를 둘 수 있는 Multi Tenant 구조를 이해해야 합니다.

CDB는 여러 PDB를 담는 상위 컨테이너이고, PDB는 실제 사용자, 스키마, 테이블, 데이터를 담는 독립된 데이터베이스 공간입니다.

Oracle XE 실습에서는 보통 XEPDB1이라는 PDB에 접속하며, SQL Developer의 Service Name과 Spring Boot의 JDBC URL에서 사용하는 XEPDB1은 같은 의미로 연결됩니다.

학습 메모

이 페이지는 Oracle 12c 이후 도입된 Multi Tenant 구조와 CDB, PDB의 차이를 이해하는 단원입니다.
CDB는 여러 PDB를 담는 상위 컨테이너이고, PDB는 사용자, 스키마, 테이블, 데이터가 들어가는 독립된 데이터베이스 공간입니다.
Oracle XE 실습에서는 보통 XEPDB1이라는 PDB에 접속하므로, SQL Developer의 Service Name과 Spring Boot의 JDBC URL에서 XEPDB1이 어떤 의미인지 이해하면 이후 SCOTT, HR 계정 실습과 Java / Spring Oracle 연결을 훨씬 안정적으로 다룰 수 있습니다.
