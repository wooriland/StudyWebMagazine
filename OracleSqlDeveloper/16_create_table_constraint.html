Oracle SQL Developer - Chapter 16

테이블 생성과 제약조건

이번 단원에서는 데이터를 저장할 테이블 구조를 직접 만들고, 잘못된 데이터가 들어오지 못하도록
PRIMARY KEY, NOT NULL, UNIQUE, DEFAULT, CHECK, FOREIGN KEY 제약조건을 설정하는 방법을 학습합니다.
이 내용은 회원가입, 게시판, 댓글, 매물 등록, 인증 토큰, 조회 로그 같은 Java·Spring·Oracle 프로젝트의 핵심 데이터 설계와 직접 연결됩니다.

학습 목표 CREATE TABLE 문법과 제약조건을 이해하고
SQL Developer에서 직접 실행한다.
실습 방향 테이블 생성 → 구조 확인 → 정상 INSERT →

Overview

1. 테이블 생성과 제약조건이란?

지금까지 배운 SELECT, JOIN, SUBQUERY, TOP-N, 페이징은 이미 존재하는 테이블에서 데이터를 조회하는 문법이었습니다.
하지만 실제 프로젝트에서는 데이터를 조회하기 전에 먼저 데이터를 저장할 구조를 만들어야 합니다.

CREATE TABLE 데이터를 저장할 테이블 구조를 만드는 DDL 문법
제약조건 잘못된 데이터가 저장되지 않도록 막는 DB의 안전장치
실무 연결 회원가입, 게시판, 매물, 인증, 통계 테이블 설계

한 줄 정리

CREATE TABLE은 데이터를 저장할 구조를 만드는 문법이고, 제약조건은 테이블에 저장되는 데이터가 반드시 지켜야 할 규칙입니다.

Concept Card

2. 단원 개념 카드

2-1. 테이블이란?

테이블은 데이터를 행과 열로 저장하는 구조입니다. 컬럼은 저장할 항목이고, 행은 실제 데이터 한 건입니다.

USER_ID LOGIN_ID EMAIL CREATED_AT
1 hong hong@test.com 2026-06-09
2 kim kim@test.com 2026-06-09

컬럼 Column

USER_ID, LOGIN_ID, EMAIL처럼 저장할 항목입니다.

행 Row

회원 1명, 게시글 1개, 매물 1개처럼
실제 데이터 한 건입니다.

테이블 Table

같은 성격의 데이터를
행과 열로 모아 저장하는 DB 객체입니다.

2-2. CREATE TABLE 기본 문법

테이블명 뒤 괄호 안에 컬럼명, 자료형, 제약조건을 작성합니다. 제약조건은 컬럼 옆에 바로 작성할 수도 있고, 테이블 마지막에 별도로 작성할 수도 있습니다. 

CREATE TABLE 테이블명 (
    컬럼명1 자료형1 [제약조건],
    컬럼명2 자료형2 [제약조건],
    컬럼명3 자료형3 [제약조건],
    CONSTRAINT 제약조건명 제약조건
);

2-3. 테이블명과 컬럼명 규칙

문자로 시작

숫자로 시작하는 이름은 피해야 합니다.

30자 이내

Oracle 기본 규칙상 너무 긴 이름은 사용하지 않습니다.

예약어 주의

SELECT, DATE, USER 같은 예약어는 피합니다.

대소문자 구별 주의

기본적으로 USERS, users, Users는 같은 이름으로 취급됩니다.

2-4. 대표 제약조건

PRIMARY KEY

행을 식별하는 기본키입니다. 중복과 NULL을 허용하지 않습니다.

NOT NULL

반드시 값이 있어야 하는 컬럼에 사용합니다.

UNIQUE

중복 값을 허용하지 않습니다. 단, Oracle에서 NULL은 여러 개 가능할 수 있습니다.

DEFAULT

값을 생략했을 때 자동으로 들어가는 기본값입니다.

CHECK

정해진 조건을 만족하는 값만 저장되도록 제한합니다.

FOREIGN KEY

다른 테이블의 PK 또는 UNIQUE 값을 참조하여 관계를 유지합니다.

3. 실행 순서 그림 카드

3-1. 테이블 생성은 “빈 표 만들기”

CREATE TABLE은 데이터를 넣는 명령이 아니라, 데이터를 넣을 수 있는 빈 구조를 만드는 명령입니다.

CREATE TABLE 실행 전 테이블 없음
→
CREATE TABLE 실행 후
USER_ID LOGIN_ID EMAIL CREATED_AT
구조만 있고 데이터는 아직 없음

3-2. 제약조건은 “입구 검사”

INSERT가 실행되면 데이터가 바로 저장되는 것이 아니라, 먼저 제약조건을 통과해야 합니다.

1 사용자 입력

아이디, 이메일, 상태값 입력

2 제약조건 검사

NOT NULL, UNIQUE, CHECK, FK 확인

3 저장 또는 거부

정상이면 저장, 위반이면 오류 발생

3-3. 부모 테이블과 자식 테이블

FOREIGN KEY는 부모 테이블에 존재하는 값만 자식 테이블에 들어갈 수 있게 합니다. 게시글은 반드시 존재하는 회원이 작성해야 한다는 규칙을 DB가 보장하는 구조입니다.

USERS 부모 테이블

USER_ID LOGIN_ID
1 hong
2 kim
REFERENCES BOARD.USER_ID → USERS.USER_ID

BOARD 자식 테이블

BOARD_ID TITLE USER_ID
100 글1 1
101 오류 글 999

핵심 이해

USERS에 USER_ID 999가 없는데 BOARD에 USER_ID 999를 넣으려고 하면 ORA-02291: parent key not found 오류가 발생합니다.

SQL Examples

4. SQL 예제 카드

4-1. 기본 테이블 생성

가장 단순한 CREATE TABLE 예제입니다.

CREATE TABLE users_basic (
    user_id NUMBER,
    login_id VARCHAR2(30),
    password VARCHAR2(100),
    email VARCHAR2(100),
    created_at DATE
);

4-2. PRIMARY KEY

PRIMARY KEY는 테이블의 한 행을 식별하는 대표 키입니다.

CREATE TABLE users_pk_test (
    user_id NUMBER,
    login_id VARCHAR2(30),
    email VARCHAR2(100),
    CONSTRAINT pk_users_pk_test PRIMARY KEY (user_id)
);

4-3. NOT NULL과 UNIQUE

필수 입력과 중복 방지를 함께 적용합니다.

CREATE TABLE users_unique_test (
    user_id NUMBER,
    login_id VARCHAR2(30) NOT NULL,
    email VARCHAR2(100) NOT NULL,
    CONSTRAINT pk_users_unique_test PRIMARY KEY (user_id),
    CONSTRAINT uk_users_unique_login UNIQUE (login_id),
    CONSTRAINT uk_users_unique_email UNIQUE (email)
);

4-4. DEFAULT

값을 생략했을 때 기본값이 자동으로 저장됩니다.

CREATE TABLE users_default_test (
    user_id NUMBER PRIMARY KEY,
    login_id VARCHAR2(30) NOT NULL,
    active CHAR(1) DEFAULT 'Y',
    user_type CHAR(1) DEFAULT 'U',
    created_at DATE DEFAULT SYSDATE
);

INSERT INTO users_default_test (
    user_id,
    login_id
) VALUES (
    1,
    'hong'
);

SELECT *
FROM users_default_test;

4-5. CHECK

허용할 값을 조건으로 제한합니다.

CREATE TABLE users_check_test (
    user_id NUMBER PRIMARY KEY,
    login_id VARCHAR2(30) NOT NULL,
    gender CHAR(1),
    active CHAR(1) DEFAULT 'Y',
    CONSTRAINT chk_users_gender CHECK (gender IN ('M', 'F')),
    CONSTRAINT chk_users_active CHECK (active IN ('Y', 'N'))
);

INSERT INTO users_check_test
VALUES (1, 'hong', 'M', 'Y');

-- 오류: gender는 M 또는 F만 가능
INSERT INTO users_check_test
VALUES (2, 'kim', 'X', 'Y');

4-6. CHECK + REGEXP_LIKE

문자열 패턴을 제약조건으로 제한합니다.

CREATE TABLE users_regex_test (
    user_id NUMBER PRIMARY KEY,
    login_id VARCHAR2(30) NOT NULL,
    CONSTRAINT chk_users_regex_login
        CHECK (REGEXP_LIKE(login_id, '^[a-z0-9]{4,12}$'))
);

INSERT INTO users_regex_test
VALUES (1, 'hong123');

-- 오류
INSERT INTO users_regex_test
VALUES (2, 'HONG!!');

4-7. FOREIGN KEY

자식 테이블의 값이 부모 테이블에 존재하는지 검사합니다.

CREATE TABLE parent_users (
    user_id NUMBER,
    login_id VARCHAR2(30) NOT NULL,
    CONSTRAINT pk_parent_users PRIMARY KEY (user_id),
    CONSTRAINT uk_parent_users_login UNIQUE (login_id)
);

CREATE TABLE child_board (
    board_id NUMBER,
    title VARCHAR2(200) NOT NULL,
    content CLOB,
    user_id NUMBER NOT NULL,
    created_at DATE DEFAULT SYSDATE,
    CONSTRAINT pk_child_board PRIMARY KEY (board_id),
    CONSTRAINT fk_child_board_user
        FOREIGN KEY (user_id)
        REFERENCES parent_users(user_id)
);

4-8. FK 정상 입력과 오류 입력

INSERT INTO parent_users
VALUES (1, 'hong');

INSERT INTO child_board (
    board_id,
    title,
    content,
    user_id
) VALUES (
    100,
    '첫 번째 게시글',
    '부모 회원이 존재하므로 정상 입력됩니다.',
    1
);

-- 오류: parent_users에 user_id 999가 없음
INSERT INTO child_board (
    board_id,
    title,
    content,
    user_id
) VALUES (
    101,
    '오류 게시글',
    '부모 회원이 존재하지 않습니다.',
    999

4-9. ON DELETE CASCADE

부모 데이터가 삭제될 때 자식 데이터도 함께 삭제됩니다.

CREATE TABLE child_board_cascade (
    board_id NUMBER,
    title VARCHAR2(200) NOT NULL,
    user_id NUMBER NOT NULL,
    CONSTRAINT pk_child_board_cascade PRIMARY KEY (board_id),
    CONSTRAINT fk_child_board_cascade_user
        FOREIGN KEY (user_id)
        REFERENCES parent_users(user_id)
        ON DELETE CASCADE
);

4-10. ON DELETE SET NULL

부모 데이터가 삭제될 때 자식 데이터는 남기고 FK 컬럼만 NULL로 바꿉니다.

CREATE TABLE child_board_set_null (
    board_id NUMBER,
    title VARCHAR2(200) NOT NULL,
    user_id NUMBER,
    CONSTRAINT pk_child_board_set_null PRIMARY KEY (board_id),
    CONSTRAINT fk_child_board_set_null_user
        FOREIGN KEY (user_id)
        REFERENCES parent_users(user_id)
        ON DELETE SET NULL
);

4-11. 통합 실습 테이블

이번 단원의 핵심 제약조건을 모두 모은 회원 테이블 예제입니다.

CREATE TABLE users_full_test (
    user_id NUMBER,
    login_id VARCHAR2(30) NOT NULL,
    password_hash VARCHAR2(100) NOT NULL,
    email VARCHAR2(100) NOT NULL,
    user_type CHAR(1) DEFAULT 'U',
    active CHAR(1) DEFAULT 'Y',
    gender CHAR(1),
    registered_at DATE DEFAULT SYSDATE,
    CONSTRAINT pk_users_full_test PRIMARY KEY (user_id),
    CONSTRAINT uk_users_full_login UNIQUE (login_id),
    CONSTRAINT uk_users_full_email UNIQUE (email),
    CONSTRAINT chk_users_full_type CHECK (user_type IN ('U', 'A')),
    CONSTRAINT chk_users_full_active CHECK (active IN ('Y', 'N')),
    CONSTRAINT chk_users_full_gender CHECK (gender IN ('M', 'F'))
);

4-12. 제약조건 확인 SQL

SELECT table_name,
       constraint_name,
       constraint_type
FROM user_constraints
WHERE table_name = 'USERS_FULL_TEST'
ORDER BY constraint_name;

SELECT constraint_name,
       column_name,
       position
FROM user_cons_columns
WHERE table_name = 'USERS_FULL_TEST'
ORDER BY constraint_name, position;

실습 성공 기준

이번 단원에서는 오류가 발생하는 것도 실습 성공입니다. PK, UNIQUE, NOT NULL, CHECK, FK가 실제로 잘못된 데이터를 막아주는지 확인해야 합니다.

Mistake

6. 자주 하는 실수

  • 실수 1. 같은 이름의 테이블을 또 생성함 이미 같은 이름의 테이블이 있으면 ORA-00955 오류가 발생합니다.
  • 실수 2. 자식 테이블을 부모 테이블보다 먼저 생성함 FK를 만들려면 참조 대상인 부모 테이블이 먼저 존재해야 합니다.
  • 실수 3. FK 부모 컬럼과 자식 컬럼 자료형이 다름 부모 컬럼이 NUMBER면 자식 FK 컬럼도 NUMBER로 맞춰야 합니다.
  • 실수 4. UNIQUE만 걸고 NOT NULL을 빼먹음 Oracle에서 UNIQUE는 NULL을 여러 개 허용할 수 있습니다.
  • 실수 5. DEFAULT와 CHECK 조건이 충돌함 DEFAULT 값이 CHECK 조건 안에 포함되는지 반드시 확인해야 합니다.
  • 실수 6. ON DELETE SET NULL인데 FK 컬럼을 NOT NULL로 만듦 부모 삭제 시 NULL로 바꿔야 하므로 FK 컬럼은 NULL 허용이어야 합니다.
  • 실수 7. DDL도 ROLLBACK될 거라고 생각함 CREATE TABLE, ALTER TABLE, DROP TABLE 같은 DDL은 Oracle에서 실행 시 자동 COMMIT됩니다.
Interview

7. 면접 질문 카드

Q1. CREATE TABLE은 무엇인가요?

CREATE TABLE은 데이터베이스에 새로운 테이블 구조를 생성하는 DDL 문장입니다. 컬럼명, 자료형, 제약조건을 정의하여 데이터를 저장할 틀을 만듭니다.

Q2. 제약조건을 사용하는 이유는 무엇인가요?

제약조건은 잘못된 데이터가 DB에 저장되는 것을 막기 위해 사용합니다. NOT NULL, UNIQUE, CHECK, FOREIGN KEY 등을 통해 데이터 무결성을 유지합니다.

Q3. PRIMARY KEY와 UNIQUE의 차이는 무엇인가요?

PRIMARY KEY는 행을 식별하는 대표 키이며 중복과 NULL을 허용하지 않습니다. UNIQUE는 중복을 막지만 Oracle에서는 NULL을 여러 개 허용할 수 있습니다.

Q4. FOREIGN KEY는 무엇인가요?

FOREIGN KEY는 자식 테이블의 컬럼이 부모 테이블의 PK 또는 UNIQUE 값을 참조하도록 만드는 제약조건입니다.
존재하지 않는 부모 데이터를 참조하지 못하게 하여 참조 무결성을 유지합니다.

Q5. ON DELETE CASCADE와 ON DELETE SET NULL의 차이는 무엇인가요?

ON DELETE CASCADE는 부모 행이 삭제될 때 자식 행도 함께 삭제합니다. ON DELETE SET NULL은 자식 행은 남기고 FK 컬럼만 NULL로 변경합니다.

Q6. CHECK 제약조건은 언제 사용하나요?

CHECK는 컬럼에 들어갈 수 있는 값을 조건으로 제한할 때 사용합니다. 예를 들어 성별은 M/F, 활성 여부는 Y/N만 허용하도록 만들 수 있습니다.

Q7. 컬럼 레벨 제약조건과 테이블 레벨 제약조건의 차이는 무엇인가요?

컬럼 레벨 제약조건은 컬럼 옆에 바로 작성하고,
테이블 레벨 제약조건은 컬럼 선언 후 마지막에 CONSTRAINT 이름과 함께 작성합니다.
테이블 레벨 방식은 제약조건 이름 관리와 복합키 설정에 유리합니다.

Oracle의 CREATE TABLE과 제약조건은 Spring Boot의 Entity, DTO 검증, Repository, Service 비즈니스 로직과 직접 연결됩니다.

Oracle

  • CREATE TABLE USERS
  • PRIMARY KEY
  • NOT NULL
  • UNIQUE
  • FOREIGN KEY
  • CHECK
  • DEFAULT

Java / Spring

  • @Entity
  • @Table
  • @Id
  • @Column(nullable = false)
  • @UniqueConstraint
  • @ManyToOne
  • @JoinColumn

실무 기능

  • 회원가입 중복 방지
  • 로그인 ID 고유성 보장
  • 게시글 작성자 FK 보장
  • 매물 등록자 FK 보장
  • 조회 로그 데이터 무결성
  • 관리자 통계 신뢰성 확보

8-1. Entity 연결 예시

@Entity
@Table(
    name = "STUDY_USERS",
    uniqueConstraints = {
        @UniqueConstraint(name = "UK_STUDY_USERS_LOGIN", columnNames = "LOGIN_ID"),
        @UniqueConstraint(name = "UK_STUDY_USERS_EMAIL", columnNames = "EMAIL")
    }
)
public class StudyUser {

    @Id
    @Column(name = "USER_ID")
    private Long userId;

    @Column(name = "LOGIN_ID", nullable = false, length = 30)
    private String loginId;

    @Column(name = "PASSWORD", nullable = false, length = 100)
    private String password;

    @Column(name = "EMAIL", nullable = false, length = 100)
    private String email;

    @Column(name = "USER_TYPE", length = 1)
    private String userType;
}

9. 최종 정리

CREATE TABLE
→ 데이터를 저장할 테이블 구조를 만드는 DDL 문법

PRIMARY KEY
→ 행 식별, 중복 불가, NULL 불가

NOT NULL
→ 필수 입력

UNIQUE
→ 중복 방지

DEFAULT
→ 값 생략 시 기본값 자동 입력

CHECK
→ 허용 값 제한

FOREIGN KEY
→ 부모 테이블과 관계 유지

USER_CONSTRAINTS
→ 생성된 제약조건 확인

USER_CONS_COLUMNS
→ 제약조건이 적용된 컬럼 확인

학습 메모

오늘의 핵심

CREATE TABLE은 테이블 구조를 만들고, 제약조건은 잘못된 데이터를 DB 단계에서 막습니다.

제약조건 한눈에 보기

  • PRIMARY KEY : 중복 불가, NULL 불가
  • NOT NULL : 필수 입력
  • UNIQUE : 중복 방지
  • DEFAULT : 기본값 입력
  • CHECK : 조건 검사
  • FOREIGN KEY : 부모 테이블 참조

실행 순서

  1. DROP TABLE로 기존 실습 테이블 정리
  2. 부모 테이블 먼저 생성
  3. 자식 테이블 생성
  4. DESC로 구조 확인
  5. USER_CONSTRAINTS 조회
  6. 정상 INSERT 실행
  7. 오류 INSERT로 제약조건 확인

암기 SQL

SELECT table_name,
       constraint_name,
       constraint_type
FROM user_constraints
WHERE table_name = '테이블명';
