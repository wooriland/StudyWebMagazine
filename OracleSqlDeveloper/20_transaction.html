Oracle SQL Developer - Chapter 20

트랜잭션 Transaction

트랜잭션은 여러 개의 INSERT, UPDATE, DELETE 작업을 하나의 논리적인 작업 단위로 묶는 개념입니다.
모든 작업이 정상적으로 끝나면 COMMIT으로 확정하고, 하나라도 실패하거나 잘못되면 ROLLBACK으로 되돌립니다.
이번 단원에서는 COMMIT, ROLLBACK, SAVEPOINT, 자동 COMMIT, 자동 ROLLBACK, TRUNCATE와 DELETE의 차이,
그리고 Java·Spring의 @Transactional 연결까지 학습합니다.

학습 목표 트랜잭션의 의미와
COMMIT / ROLLBACK / SAVEPOINT를
SQL Developer에서 직접 실행하며 이해한다.
실습 방향 DML 실행 → SELECT 확인 →
COMMIT 또는 ROLLBACK →
SAVEPOINT로 부분 되돌리기 확인
프로젝트 연결 Spring @Transactional, Service 계층,
주문·회원가입·매물 등록의 작업 단위와 연결한다.
Overview

1. 트랜잭션이란?

트랜잭션은 데이터베이스에서 하나의 논리적인 작업 단위를 의미합니다.
하나의 SQL 문장만으로 끝나는 경우도 있지만, 실제 프로젝트에서는 여러 개의 SQL이 모여 하나의 기능을 완성하는 경우가 많습니다.

Transaction 여러 SQL을 하나의 작업 단위로 묶는다.
COMMIT 정상 작업 결과를 영구적으로 반영한다.
ROLLBACK 잘못된 작업을 COMMIT 전 상태로 되돌린다.

한 줄 정리

트랜잭션은 여러 DML 작업을 하나의 작업처럼 묶고, 모두 성공하면 COMMIT, 하나라도 실패하면 ROLLBACK하는 데이터 안정성 장치입니다.

Concept Card

2. 단원 개념 카드

2-1. 왜 트랜잭션을 쓰는가?

실제 기능 하나는 여러 테이블 작업으로 구성되는 경우가 많습니다.
예를 들어 회원가입은 회원 정보만 저장하는 것이 아니라, 프로필, 이메일 인증 정보, 활동 로그까지 함께 저장될 수 있습니다.

회원가입

회원 정보, 기본 프로필, 이메일 인증 정보, 가입 로그가 함께 저장되어야 합니다.

USERS INSERT
USER_PROFILE INSERT
EMAIL_AUTH INSERT
ACTIVITY_LOG INSERT

게시글 작성

게시글, 첨부파일, 태그, 사용자 활동 로그가 함께 저장될 수 있습니다.

BOARD INSERT
BOARD_FILE INSERT
BOARD_TAG INSERT
USER_ACTIVITY_LOG INSERT

주문 처리

주문, 주문 상세, 결제, 재고 차감, 포인트 사용이 하나의 흐름으로 묶입니다.

ORDER INSERT
ORDER_ITEM INSERT
PAYMENT INSERT
STOCK UPDATE
POINT UPDATE

매물 등록

매물 기본 정보, 이미지, 옵션, 위치 정보, 등록 로그가 함께 저장됩니다.

PROPERTY INSERT
PROPERTY_IMAGE INSERT
PROPERTY_OPTION INSERT
PROPERTY_LOCATION INSERT
PUBLISHER_LOG INSERT

2-2. 트랜잭션의 기본 원칙

트랜잭션은 여러 작업이 모두 성공해야 의미가 있는 상황에서 사용합니다. 중간에 하나라도 실패하면 전체를 취소해야 데이터가 어긋나지 않습니다.

모두 성공 COMMIT

모든 변경 내용을 데이터베이스에 확정합니다.

하나라도 실패 ROLLBACK

아직 확정하지 않은 변경 내용을 원래대로 되돌립니다.

중간 지점 필요 SAVEPOINT

트랜잭션 안에서 부분적으로 되돌릴 수 있는 지점을 표시합니다.

2-3. 트랜잭션과 DML

트랜잭션과 가장 직접적으로 연결되는 SQL은 INSERT, UPDATE, DELETE입니다. 이 세 명령은 데이터를 실제로 변경하기 때문에 COMMIT 또는 ROLLBACK으로 제어해야 합니다.

명령어 의미 트랜잭션 관계 예시 기능
INSERT 새 행 입력 COMMIT 전 ROLLBACK 가능 회원가입, 게시글 작성
UPDATE 기존 행 수정 COMMIT 전 ROLLBACK 가능 회원정보 수정, 매물 수정
DELETE 기존 행 삭제 COMMIT 전 ROLLBACK 가능 좋아요 취소, 임시 데이터 삭제

3. 실행 순서 그림 카드

3-1. COMMIT과 ROLLBACK 흐름

DML을 실행한 뒤에는 바로 COMMIT하지 않고 SELECT로 결과를 확인해야 합니다. 결과가 맞으면 COMMIT, 틀리면 ROLLBACK합니다.

1 DML 실행

INSERT / UPDATE / DELETE

→
2 SELECT 확인

변경 결과 검증

→
정상

COMMIT

오류

ROLLBACK

3-2. SAVEPOINT 흐름

SAVEPOINT는 트랜잭션 중간에 표시점을 만드는 기능입니다. 전체를 취소하지 않고 특정 지점 이후의 작업만 되돌릴 수 있습니다.

START 트랜잭션 시작
UPDATE 1 첫 번째 수정
SAVEPOINT point1
UPDATE 2 두 번째 수정
SAVEPOINT point2
UPDATE 3 잘못된 수정
ROLLBACK TO point2

그림으로 정리

ROLLBACK TO point2는 point2 이후의 작업만 취소합니다. ROLLBACK만 실행하면 트랜잭션 시작 이후의 모든 미확정 변경이 취소됩니다.

3-3. 자동 COMMIT 흐름

DML을 실행한 뒤 COMMIT하지 않았더라도, 중간에 DDL을 실행하면 앞의 DML까지 자동 COMMIT될 수 있습니다.

Setup

4. SQL Developer 실습 준비

트랜잭션은 반드시 직접 실행하면서 확인해야 합니다. 이번 실습에서는 회원, 주문, 주문 상세 테이블을 만들어서 여러 DML을 하나의 트랜잭션으로 묶는 연습을 합니다.

4-1. 기존 실습 객체 정리

DROP TABLE tx_order_item PURGE;
DROP TABLE tx_order PURGE;
DROP TABLE tx_member PURGE;
DROP SEQUENCE tx_member_seq;
DROP SEQUENCE tx_order_seq;
DROP SEQUENCE tx_order_item_seq;

처음 실행하면 ORA-00942 또는 ORA-02289 오류가 날 수 있습니다.
삭제할 테이블이나 시퀀스가 없다는 뜻이므로 첫 실습에서는 괜찮습니다.

4-2. 회원 테이블 생성

CREATE TABLE tx_member (
    member_id NUMBER,
    login_id VARCHAR2(30) NOT NULL,
    member_name VARCHAR2(50) NOT NULL,
    active CHAR(1) DEFAULT 'Y',
    created_at DATE DEFAULT SYSDATE,
    CONSTRAINT pk_tx_member PRIMARY KEY (member_id),
    CONSTRAINT uk_tx_member_login UNIQUE (login_id),
    CONSTRAINT chk_tx_member_active CHECK (active IN ('Y', 'N'))
);

4-3. 주문 테이블 생성

CREATE TABLE tx_order (
    order_id NUMBER,
    member_id NUMBER NOT NULL,
    order_status VARCHAR2(20) DEFAULT 'READY',
    order_amount NUMBER DEFAULT 0,
    created_at DATE DEFAULT SYSDATE,
    CONSTRAINT pk_tx_order PRIMARY KEY (order_id),
    CONSTRAINT fk_tx_order_member
        FOREIGN KEY (member_id)
        REFERENCES tx_member(member_id),
    CONSTRAINT chk_tx_order_status
        CHECK (order_status IN ('READY', 'PAID', 'CANCELED'))
);

4-4. 주문 상세 테이블 생성

CREATE TABLE tx_order_item (
    order_item_id NUMBER,
    order_id NUMBER NOT NULL,
    product_name VARCHAR2(100) NOT NULL,
    quantity NUMBER DEFAULT 1,
    price NUMBER NOT NULL,
    CONSTRAINT pk_tx_order_item PRIMARY KEY (order_item_id),
    CONSTRAINT fk_tx_order_item_order
        FOREIGN KEY (order_id)
        REFERENCES tx_order(order_id),
    CONSTRAINT chk_tx_order_item_quantity CHECK (quantity > 0),
    CONSTRAINT chk_tx_order_item_price CHECK (price >= 0)
);

4-5. 시퀀스 생성

CREATE SEQUENCE tx_member_seq
START WITH 1
INCREMENT BY 1
NOCACHE
NOCYCLE;

CREATE SEQUENCE tx_order_seq
START WITH 1
INCREMENT BY 1
NOCACHE
NOCYCLE;

CREATE SEQUENCE tx_order_item_seq
START WITH 1
INCREMENT BY 1
NOCACHE
NOCYCLE;

4-6. 생성 결과 확인

DESC tx_member;
DESC tx_order;
DESC tx_order_item;

SELECT table_name
FROM user_tables
WHERE table_name IN ('TX_MEMBER', 'TX_ORDER', 'TX_ORDER_ITEM')
ORDER BY table_name;

SELECT table_name,
       constraint_name,
       constraint_type
FROM user_constraints
WHERE table_name IN ('TX_MEMBER', 'TX_ORDER', 'TX_ORDER_ITEM')
ORDER BY table_name, constraint_name;

SELECT sequence_name,
       increment_by,
       cycle_flag,
       cache_size,
       last_number
FROM user_sequences
WHERE sequence_name IN ('TX_MEMBER_SEQ', 'TX_ORDER_SEQ', 'TX_ORDER_ITEM_SEQ')
ORDER BY sequence_name;

5. SQL 예제 카드 - COMMIT

COMMIT은 현재 트랜잭션에서 발생한 변경 내용을 데이터베이스에 영구적으로 반영합니다. COMMIT 이후에는 일반적인 ROLLBACK으로 되돌릴 수 없습니다.

5-1. 회원 데이터 입력

INSERT INTO tx_member (
    member_id,
    login_id,
    member_name
) VALUES (
    tx_member_seq.NEXTVAL,
    'hong',
    '홍길동'
);

5-2. 입력 결과 확인

SELECT *
FROM tx_member
ORDER BY member_id;

5-3. COMMIT 실행

COMMIT;

이 시점부터 hong 데이터는 확정됩니다.

5-4. COMMIT 후 ROLLBACK 확인

ROLLBACK;

SELECT *
FROM tx_member
ORDER BY member_id;

hong 데이터가 그대로 남아 있다면
COMMIT 후에는 ROLLBACK되지 않는다는 것을 확인한 것입니다.

ROLLBACK

6. SQL 예제 카드 - ROLLBACK

ROLLBACK은 아직 COMMIT하지 않은 변경 내용을 취소합니다. 실수한 INSERT, UPDATE, DELETE를 되돌릴 때 사용합니다.

6-1. 테스트 데이터 입력

INSERT INTO tx_member (
    member_id,
    login_id,
    member_name
) VALUES (
    tx_member_seq.NEXTVAL,
    'rollback_test',
    '롤백테스트'
);

6-2. 입력 결과 확인

SELECT *
FROM tx_member
WHERE login_id = 'rollback_test';

6-3. ROLLBACK 실행

ROLLBACK;

6-4. ROLLBACK 결과 확인

SELECT *
FROM tx_member
WHERE login_id = 'rollback_test';

결과가 없으면 ROLLBACK 성공입니다.

Multiple DML

7. 여러 DML을 하나의 트랜잭션으로 묶기

주문 생성처럼 여러 SQL이 하나의 기능을 구성할 때는 모든 SQL이 함께 성공해야 합니다. 주문 생성, 주문 상세 생성, 주문 금액 업데이트를 하나의 트랜잭션으로 묶어 실습합니다.

7-1. 주문 생성

INSERT INTO tx_order (
    order_id,
    member_id,
    order_status
) VALUES (
    tx_order_seq.NEXTVAL,
    1,
    'READY'
);

SELECT tx_order_seq.CURRVAL
FROM dual;

7-2. 주문 상세 생성

INSERT INTO tx_order_item (
    order_item_id,
    order_id,
    product_name,
    quantity,
    price
) VALUES (
    tx_order_item_seq.NEXTVAL,
    tx_order_seq.CURRVAL,
    '키보드',
    1,
    30000
);

INSERT INTO tx_order_item (
    order_item_id,
    order_id,
    product_name,
    quantity,
    price
) VALUES (
    tx_order_item_seq.NEXTVAL,
    tx_order_seq.CURRVAL,
    '마우스',
    2,
    15000
);

7-3. 주문 금액 업데이트

UPDATE tx_order
SET order_amount = (
    SELECT SUM(quantity * price)
    FROM tx_order_item
    WHERE order_id = tx_order_seq.CURRVAL
)
WHERE order_id = tx_order_seq.CURRVAL;

7-4. 결과 확인 후 COMMIT

SELECT *
FROM tx_order
ORDER BY order_id;

SELECT *
FROM tx_order_item
ORDER BY order_item_id;

COMMIT;
Failure Case

8. 실패 상황과 ROLLBACK

하나의 트랜잭션 안에서 중간 작업이 실패하면 전체 작업을 ROLLBACK해야 합니다. 이번에는 CHECK 제약조건 오류를 일부러 발생시켜 전체 취소 흐름을 확인합니다.

8-1. 주문 생성

INSERT INTO tx_order (
    order_id,
    member_id,
    order_status
) VALUES (
    tx_order_seq.NEXTVAL,
    1,
    'READY'
);

8-2. 정상 주문 상세 입력

INSERT INTO tx_order_item (
    order_item_id,
    order_id,
    product_name,
    quantity,
    price
) VALUES (
    tx_order_item_seq.NEXTVAL,
    tx_order_seq.CURRVAL,
    '모니터',
    1,
    200000
);

8-3. 오류 주문 상세 입력

INSERT INTO tx_order_item (
    order_item_id,
    order_id,
    product_name,
    quantity,
    price
) VALUES (
    tx_order_item_seq.NEXTVAL,
    tx_order_seq.CURRVAL,
    '오류상품',
    0,
    10000
);

quantity > 0 CHECK 제약조건 때문에 ORA-02290 오류가 발생해야 정상입니다.

8-4. 전체 ROLLBACK

ROLLBACK;

SELECT *
FROM tx_order
ORDER BY order_id;

SELECT *
FROM tx_order_item
ORDER BY order_item_id;

방금 시도한 주문과 주문 상세가 사라졌다면 전체 취소가 정상적으로 된 것입니다.

SAVEPOINT

9. SAVEPOINT SQL 예제

SAVEPOINT는 전체 트랜잭션을 취소하지 않고 특정 지점 이후의 작업만 되돌릴 때 사용합니다.

9-1. 첫 번째 UPDATE 후 point1 생성

UPDATE tx_member
SET member_name = '홍길동_1차수정'
WHERE login_id = 'hong';

SAVEPOINT point1;

SELECT *
FROM tx_member
WHERE login_id = 'hong';

9-2. 두 번째 UPDATE 후 point2 생성

UPDATE tx_member
SET active = 'N'
WHERE login_id = 'hong';

SAVEPOINT point2;

SELECT *
FROM tx_member
WHERE login_id = 'hong';

9-3. 세 번째 잘못된 UPDATE

UPDATE tx_member
SET member_name = '잘못된수정'
WHERE login_id = 'hong';

SELECT *
FROM tx_member
WHERE login_id = 'hong';

9-4. point2로 되돌리기

ROLLBACK TO point2;

SELECT *
FROM tx_member
WHERE login_id = 'hong';

잘못된수정만 취소되고 active = 'N' 상태는 유지됩니다.

9-5. point1로 되돌리기

ROLLBACK TO point1;

SELECT *
FROM tx_member
WHERE login_id = 'hong';

point1 이후의 변경이 취소됩니다.

9-6. 전체 ROLLBACK

ROLLBACK;

SELECT *
FROM tx_member
WHERE login_id = 'hong';

마지막 COMMIT 이후 상태로 되돌아갑니다.

Auto Commit

10. 자동 COMMIT과 자동 ROLLBACK

Oracle에서는 DDL이나 DCL 문장이 완료되면 자동 COMMIT이 발생할 수 있습니다.
또한 SQL*PLUS 정상 종료 시 자동 COMMIT, 비정상 종료 또는 시스템 실패 시 자동 ROLLBACK이 발생할 수 있습니다.

10-1. DML 후 DDL 실행

INSERT INTO tx_member (
    member_id,
    login_id,
    member_name
) VALUES (
    tx_member_seq.NEXTVAL,
    'auto_commit_test',
    '자동커밋테스트'
);

SELECT *
FROM tx_member
WHERE login_id = 'auto_commit_test';

CREATE TABLE tx_auto_commit_test (
    id NUMBER
);

CREATE TABLE은 DDL이므로 앞의 INSERT까지 자동 COMMIT될 수 있습니다.

10-2. ROLLBACK 후 확인

ROLLBACK;

SELECT *
FROM tx_member
WHERE login_id = 'auto_commit_test';

데이터가 남아 있다면 DDL로 인해 자동 COMMIT된 것입니다.

10-3. 실습 테이블 정리

DROP TABLE tx_auto_commit_test PURGE;

자동 COMMIT 주의

DML 실습 중간에 CREATE, ALTER, DROP, TRUNCATE 같은 DDL을 섞으면 이전 DML이 의도치 않게 확정될 수 있습니다.

TRUNCATE

11. TRUNCATE와 트랜잭션

TRUNCATE는 테이블 구조는 유지하고 데이터만 빠르게 전체 삭제하는 명령입니다.
DELETE와 비슷해 보이지만, ROLLBACK을 위한 로깅 정보를 남기지 않기 때문에 일반적인 ROLLBACK을 기대하면 안 됩니다.

구분 DELETE TRUNCATE
명령 성격 DML DDL 성격
WHERE 사용 가능 불가능
일부 삭제 가능 불가능
ROLLBACK COMMIT 전 가능 일반적으로 어려움
속도 상대적으로 느림 빠름
사용 상황 조건 삭제, 복구 가능성 필요 임시 테이블 전체 비우기

조건 삭제가 필요할 때

DELETE FROM tx_order_item
WHERE order_id = 1;

임시 테이블 전체 삭제

TRUNCATE TABLE temp_order_result;
Result Check

12. 결과 화면 확인

트랜잭션 실습은 실행 후 결과 확인이 가장 중요합니다. COMMIT, ROLLBACK, SAVEPOINT마다 SELECT를 실행하여 실제 상태를 확인해야 합니다.

12-1. 현재 데이터 확인

SELECT *
FROM tx_member
ORDER BY member_id;

SELECT *
FROM tx_order
ORDER BY order_id;

SELECT *
FROM tx_order_item
ORDER BY order_item_id;

12-2. COMMIT 확인

COMMIT;

ROLLBACK;

SELECT *
FROM tx_member
ORDER BY member_id;

COMMIT 후 ROLLBACK을 해도 데이터가 남아 있으면 정상입니다.

12-3. ROLLBACK 확인

ROLLBACK;

SELECT *
FROM tx_member
WHERE login_id = 'rollback_test';

ROLLBACK 후 결과가 없으면 정상입니다.

Mistake

14. 자주 하는 실수

  • 실수 1. COMMIT을 너무 빨리 실행함 DML 실행 후 바로 COMMIT하지 말고 SELECT로 결과를 확인한 뒤 확정해야 합니다.
  • 실수 2. DDL이 자동 COMMIT을 일으킨다는 것을 모름 CREATE, ALTER, DROP, TRUNCATE 같은 DDL은 이전 DML까지 확정시킬 수 있습니다.
  • 실수 3. TRUNCATE도 ROLLBACK될 거라고 생각함 TRUNCATE는 ROLLBACK을 위한 로깅 정보를 남기지 않으므로 복구가 어렵습니다.
  • 실수 4. SAVEPOINT 후 COMMIT하면 SAVEPOINT로 돌아갈 수 있다고 생각함 COMMIT하면 트랜잭션이 종료되므로 이전 SAVEPOINT로 돌아갈 수 없습니다.
  • 실수 5. 여러 작업 중 하나가 실패했는데 일부만 남겨둠 하나의 기능으로 묶인 작업 중 일부가 실패하면 전체 ROLLBACK해야 합니다.
  • 실수 6. Spring에서 @Transactional을 Controller에 무조건 붙임 트랜잭션은 보통 여러 Repository 작업을 묶는 Service 계층에 적용합니다.
  • 실수 7. Checked Exception 발생 시 롤백이 안 되는 상황을 모름 Spring 기본 트랜잭션은 주로 RuntimeException에서 롤백되므로 필요한 경우 rollbackFor 설정이 필요합니다.
Interview

15. 면접 질문 카드

Q1. 트랜잭션이란 무엇인가요?

트랜잭션은 데이터베이스에서 하나의 논리적인 작업 단위입니다.
여러 SQL 작업이 하나의 업무 흐름을 구성할 때, 모두 성공하면 COMMIT하고 하나라도 실패하면 ROLLBACK하여 데이터 일관성을 유지합니다.

Q2. COMMIT과 ROLLBACK의 차이는 무엇인가요?

COMMIT은 현재 트랜잭션의 변경 내용을 영구적으로 반영하는 명령입니다. ROLLBACK은 아직 COMMIT하지 않은 변경 내용을 취소하고 원래 상태로 되돌리는 명령입니다.

Q3. SAVEPOINT는 무엇인가요?

SAVEPOINT는 트랜잭션 안에서 중간 저장 지점을 만드는 명령입니다.
전체 트랜잭션을 되돌리지 않고 특정 지점까지만 되돌리고 싶을 때 ROLLBACK TO SAVEPOINT명을 사용합니다.

Q4. 자동 COMMIT이 발생하는 경우는 언제인가요?

Oracle에서는 DDL이나 DCL 문장이 완료될 때 자동 COMMIT이 발생할 수 있습니다. SQL*PLUS 정상 종료 시에도 자동 COMMIT이 발생할 수 있습니다.

Q5. 자동 ROLLBACK이 발생하는 경우는 언제인가요?

SQL*PLUS가 비정상 종료되거나 시스템 실패가 발생하면, 아직 COMMIT되지 않은 작업은 자동 ROLLBACK될 수 있습니다.

Q6. DELETE와 TRUNCATE의 트랜잭션 차이는 무엇인가요?

DELETE는 DML이므로 COMMIT 전에는 ROLLBACK할 수 있습니다.
반면 TRUNCATE는 DDL 성격이 강하고 ROLLBACK을 위한 로깅 정보를 남기지 않으므로 일반적인 ROLLBACK이 어렵습니다.

Q7. Spring에서 트랜잭션은 어디에 적용하는 것이 좋나요?

보통 Service 계층에 적용합니다.
Service는 여러 Repository 호출을 하나의 비즈니스 작업 단위로 묶는 역할을 하기 때문에,
해당 메서드에 @Transactional을 붙여 정상 종료 시 COMMIT, 예외 발생 시 ROLLBACK되도록 관리합니다.

Q8. 트랜잭션이 없으면 어떤 문제가 생길 수 있나요?

여러 테이블을 수정하는 작업 중 일부만 성공하고 일부가 실패하면 데이터가 불완전해질 수 있습니다.
예를 들어 주문은 생성되었지만 주문 상세가 저장되지 않거나, 좋아요 기록은 저장되었는데 좋아요 수가 증가하지 않는 문제가 생길 수 있습니다.

16. Java·Spring·Oracle 프로젝트 연결

Oracle의 COMMIT, ROLLBACK 개념은 Spring의 @Transactional과 직접 연결됩니다. Spring에서는 보통 Service 계층에서 여러 Repository 작업을 하나의 트랜잭션으로 묶습니다.

Spring @Transactional 기본 구조

@Transactional
public void createOrder(OrderRequest request) {
    Order order = orderRepository.save(request.toOrder());

    orderItemRepository.saveAll(request.toOrderItems(order));

    stockRepository.decreaseStock(
        request.getProductId(),
        request.getQuantity()
    );
}

메서드가 정상 종료되면 COMMIT,
RuntimeException이 발생하면 ROLLBACK됩니다.

회원가입 트랜잭션

@Transactional
public void signup(SignupRequest request) {
    User user = userRepository.save(request.toUser());

    userProfileRepository.save(
        UserProfile.createDefault(user)
    );

    emailAuthRepository.save(
        EmailAuth.create(user)
    );

    activityLogRepository.save(
        ActivityLog.signup(user)
    );
}

게시글 작성 트랜잭션

@Transactional
public Long createBoard(BoardCreateRequest request) {
    Board board = boardRepository.save(request.toBoard());

    fileRepository.saveAll(request.toFiles(board));

    activityLogRepository.save(
        ActivityLog.boardCreate(board)
    );

    return board.getBoardId();
}

주문 처리 트랜잭션

@Transactional
public void order(OrderRequest request) {
    Order order = orderRepository.save(request.toOrder());

    orderItemRepository.saveAll(
        request.toOrderItems(order)
    );

    paymentRepository.save(
        request.toPayment(order)
    );

    stockRepository.decrease(
        request.getProductId(),
        request.getQuantity()
    );

    pointRepository.usePoint(
        request.getUserId(),
        request.getPoint()
    );
}

내집마련 좋아요 등록

@Transactional
public void likeProperty(Long userId, Long propertyId) {
    propertyLikeRepository.save(
        PropertyLike.of(userId, propertyId)
    );

    propertyRepository.increaseLikeCount(propertyId);

    activityLogRepository.save(
        ActivityLog.propertyLike(userId, propertyId)
    );
}

좋아요 기록 저장은 성공했지만 좋아요 수 증가가 실패하면 전체 ROLLBACK되어야 합니다.

Checked Exception 롤백 설정

@Transactional(rollbackFor = Exception.class)
public void processWithCheckedException() throws Exception {
    // checked exception 발생 시에도 rollback 필요할 때 사용
}

17. 최종 정리

트랜잭션
→ 하나의 논리적인 작업 단위

COMMIT
→ 변경 내용을 확정한다.

ROLLBACK
→ COMMIT 전 변경 내용을 취소한다.

SAVEPOINT
→ 트랜잭션 중간 지점을 표시한다.

ROLLBACK TO SAVEPOINT
→ 특정 지점 이후의 변경만 취소한다.

자동 COMMIT
→ DDL/DCL 완료 시, SQL*PLUS 정상 종료 시 발생할 수 있다.

자동 ROLLBACK
→ SQL*PLUS 비정상 종료 또는 시스템 실패 시 발생할 수 있다.

DELETE
→ DML, COMMIT 전 ROLLBACK 가능

TRUNCATE
→ DDL 성격, 빠르지만 일반적인 ROLLBACK이 어렵다.

학습 메모

오늘의 핵심

트랜잭션은 여러 DML 작업을 하나의 작업 단위로 묶고, 모두 성공하면 COMMIT, 하나라도 실패하면 ROLLBACK하는 구조입니다.

암기 문법

COMMIT;

ROLLBACK;

SAVEPOINT point1;

ROLLBACK TO point1;

안전 실행 순서

  1. INSERT / UPDATE / DELETE 실행
  2. SELECT로 결과 확인
  3. 맞으면 COMMIT
  4. 틀리면 ROLLBACK
  5. 부분 되돌리기가 필요하면 SAVEPOINT 사용

자동 COMMIT

  • DDL 문장 완료 시
  • DCL 문장 완료 시
  • SQL*PLUS 정상 종료 시

자동 ROLLBACK

  • SQL*PLUS 비정상 종료 시
  • 시스템 실패 시

주의할 점

DML 실습 중간에 CREATE, ALTER, DROP, TRUNCATE 같은 DDL을 실행하면 이전 DML이 자동 COMMIT될 수 있습니다.

프로젝트 연결

Oracle의 COMMIT / ROLLBACK은 Spring의 @Transactional과 연결됩니다. Service 메서드가 정상 종료되면 COMMIT, 예외가 발생하면 ROLLBACK됩니다.
