Oracle SQL Developer - Chapter 21

VIEW

VIEW는 하나 이상의 테이블 또는 SELECT 결과를 바탕으로 만든 가상의 테이블입니다.
실제 데이터를 별도로 저장하는 것이 아니라, SELECT문에 이름을 붙여 테이블처럼 조회할 수 있게 만든 객체입니다.
이번 단원에서는 VIEW의 개념, 보안성, 편의성, 간결성,
CREATE VIEW, CREATE OR REPLACE VIEW, WITH READ ONLY, USER_VIEWS, VIEW DML 제한,
Java·Spring 프로젝트의 조회 API 연결까지 학습합니다.

학습 목표 VIEW가 무엇인지 이해하고
TABLE과의 차이를 설명할 수 있다.
실습 방향 실습 테이블 생성 → VIEW 생성 → 조회 →
READ ONLY 확인 → USER_VIEWS 확인
프로젝트 연결 게시판 목록, 매물 카드, 관리자 대시보드,
조회 전용 DTO / Entity와 연결한다.
Overview

1. VIEW란?

VIEW는 기존 테이블이나 SELECT 결과를 바탕으로 만든 가상의 테이블입니다.
실제 데이터를 새로 저장하는 것이 아니라, SELECT문에 이름을 붙여서 테이블처럼 조회할 수 있도록 만든 데이터베이스 객체입니다.

VIEW SELECT 결과를 테이블처럼 보여주는 가상 객체
원본 테이블 실제 데이터가 저장되어 있는 테이블
USER_VIEWS 생성된 VIEW와 정의 SQL을 확인하는 데이터 딕셔너리

한 줄 정리

VIEW는 원본 테이블을 직접 보여주지 않고, 필요한 컬럼과 행만 선택해서 보여주는 가상의 조회 창구입니다.

Concept Card

2. 단원 개념 카드

2-1. VIEW를 왜 사용하는가?

VIEW는 단순히 SELECT문을 줄이기 위한 도구가 아닙니다. 보안성, 편의성, 간결성 측면에서 실무에서 많이 사용됩니다.

보안성

원본 테이블의 비밀번호, 전화번호, 내부 코드 같은 민감한 컬럼을 숨기고
필요한 컬럼만 공개할 수 있습니다. 

CREATE OR REPLACE VIEW v_member_safe AS
SELECT member_id,
       login_id,
       member_name,
       email
FROM view_member;

편의성

복잡한 JOIN 결과에 이름을 붙여두면 매번 긴 SQL을 작성하지 않고
VIEW를 테이블처럼 조회할 수 있습니다. 

SELECT *
FROM v_board_list
ORDER BY board_id DESC;

간결성

MyBatis Mapper나 Repository SQL에서 복잡한 조인을 줄이고,
화면이나 API에서 필요한 형태로 데이터를 쉽게 조회할 수 있습니다. 

SELECT *
FROM v_property_card
WHERE listing_status = 'ACTIVE';

재사용성

여러 화면에서 같은 조회 결과가 필요할 때 VIEW를 만들어 재사용할 수 있습니다. 

SELECT *
FROM v_member_board_count;

2-2. VIEW와 TABLE의 차이

구분 TABLE VIEW
의미 실제 데이터를 저장하는 객체 SELECT 결과를 테이블처럼 보여주는 가상 객체
데이터 저장 실제 저장함 일반적으로 저장하지 않음
생성 방식 컬럼명과 자료형을 직접 정의 SELECT문으로 정의
주요 목적 업무 데이터 저장 조회 구조 제공, 보안, SQL 간소화
DML INSERT, UPDATE, DELETE 가능 구조에 따라 가능 또는 제한
삭제 명령 DROP TABLE DROP VIEW

2-3. VIEW의 핵심 특징

가상 테이블 Virtual Table

일반 VIEW는 데이터를 직접 저장하지 않고 SELECT 정의를 저장합니다.

원본 참조 Source Table

VIEW 조회 결과는 원본 테이블의 현재 데이터에 따라 달라집니다.

DML 주의 Updatable View

단순 VIEW에 대한 DML은 원본 테이블에 반영될 수 있습니다.

읽기 전용 WITH READ ONLY

조회 전용 VIEW는 WITH READ ONLY로 원본 테이블 변경을 막을 수 있습니다.

3. 실행 순서 그림 카드

3-1. VIEW는 창문이다

VIEW는 원본 테이블 전체를 직접 보여주는 것이 아니라, 필요한 컬럼만 골라 보여주는 창문처럼 이해하면 쉽습니다.

원본 테이블 VIEW_MEMBER

MEMBER_ID LOGIN_ID PASSWORD_HASH MEMBER_NAME EMAIL PHONE ACTIVE CREATED_AT
→

VIEW V_MEMBER_PUBLIC

MEMBER_ID LOGIN_ID MEMBER_NAME EMAIL ACTIVE

3-2. 복잡한 SELECT에 이름 붙이기

VIEW는 복잡한 SELECT문에 이름을 붙여서 테이블처럼 조회할 수 있게 합니다.

1 복잡한 SELECT 작성

JOIN, WHERE, GROUP BY 등을 포함한 조회문

2 CREATE VIEW

SELECT문에 VIEW 이름 부여

3 VIEW 조회

SELECT * FROM view_name

3-3. VIEW DML은 원본에 반영될 수 있다

단순 VIEW에 UPDATE를 실행하면 VIEW만 바뀌는 것이 아니라 원본 테이블이 바뀔 수 있습니다.

UPDATE v_member_simple

VIEW에 UPDATE 실행

→
VIEW_MEMBER 변경

원본 테이블에 반영될 수 있음

중요

조회용 VIEW라면 WITH READ ONLY 옵션을 붙여 VIEW를 통한 INSERT, UPDATE, DELETE를 막는 것이 안전합니다.

4. VIEW 기본 문법

교안의 VIEW 생성 문법은 다음과 같습니다. 기존 VIEW를 수정할 때는 OR REPLACE를 사용하고, 조회 전용 VIEW를 만들 때는 WITH READ ONLY를 붙입니다. 

CREATE [OR REPLACE] VIEW 뷰명 [(컬럼별칭1, 컬럼별칭2, ...)]
AS
SELECT 구문
[WITH READ ONLY];

4-1. 기본 VIEW 생성

CREATE VIEW v_emp_basic AS
SELECT empno,
       ename,
       job,
       sal,
       deptno
FROM emp;

4-2. CREATE OR REPLACE VIEW

CREATE OR REPLACE VIEW v_emp_basic AS
SELECT empno,
       ename,
       job,
       sal,
       deptno,
       hiredate
FROM emp;

4-3. 컬럼 별칭 지정

CREATE OR REPLACE VIEW v_emp_summary (
    emp_no,
    emp_name,
    job_name,
    salary
) AS
SELECT empno,
       ename,
       job,
       sal
FROM emp;

4-4. 읽기 전용 VIEW

CREATE OR REPLACE VIEW v_emp_readonly AS
SELECT empno,
       ename,
       job,
       sal,
       deptno
FROM emp
WITH READ ONLY;

4-5. VIEW 삭제

DROP VIEW v_emp_basic;

VIEW를 삭제해도 원본 테이블과 데이터는 삭제되지 않습니다.

4-6. USER_VIEWS 확인

SELECT view_name,
       text
FROM user_views
WHERE view_name = 'V_EMP_BASIC';

5-1. 기존 실습 객체 정리

처음 실행하면 ORA-00942 또는 ORA-02289 오류가 날 수 있습니다. 삭제할 객체가 없다는 뜻이므로 첫 실습에서는 괜찮습니다.

5-2. 회원 테이블 생성

CREATE TABLE view_member (
    member_id NUMBER,
    login_id VARCHAR2(30) NOT NULL,
    password_hash VARCHAR2(100) NOT NULL,
    member_name VARCHAR2(50) NOT NULL,
    email VARCHAR2(100),
    phone VARCHAR2(20),
    active CHAR(1) DEFAULT 'Y',
    created_at DATE DEFAULT SYSDATE,
    CONSTRAINT pk_view_member PRIMARY KEY (member_id),
    CONSTRAINT uk_view_member_login UNIQUE (login_id),
    CONSTRAINT chk_view_member_active CHECK (active IN ('Y', 'N'))
);

5-3. 게시글 테이블 생성

CREATE TABLE view_board (
    board_id NUMBER,
    title VARCHAR2(200) NOT NULL,
    content VARCHAR2(1000),
    member_id NUMBER NOT NULL,
    deleted_yn CHAR(1) DEFAULT 'N',
    created_at DATE DEFAULT SYSDATE,
    CONSTRAINT pk_view_board PRIMARY KEY (board_id),
    CONSTRAINT fk_view_board_member
        FOREIGN KEY (member_id)
        REFERENCES view_member(member_id),
    CONSTRAINT chk_view_board_deleted CHECK (deleted_yn IN ('Y', 'N'))
);

5-4. 시퀀스 생성

CREATE SEQUENCE view_member_seq
START WITH 1
INCREMENT BY 1
NOCACHE
NOCYCLE;

CREATE SEQUENCE view_board_seq
START WITH 1
INCREMENT BY 1
NOCACHE
NOCYCLE;

VIEW Examples

6. SQL 예제 카드 - VIEW 생성과 조회

단순 VIEW, 조건 VIEW, READ ONLY VIEW, 조인 VIEW, 통계 VIEW를 순서대로 만들어보며 VIEW의 사용 목적을 익힙니다.

6-1. 공개 회원 VIEW

CREATE OR REPLACE VIEW v_view_member_public AS
SELECT member_id,
       login_id,
       member_name,
       email,
       active,
       created_at
FROM view_member;

SELECT *
FROM v_view_member_public
ORDER BY member_id;

PASSWORD_HASH와 PHONE 컬럼이 보이지 않아야 합니다.

6-2. 활성 회원 VIEW

CREATE OR REPLACE VIEW v_view_active_member AS
SELECT member_id,
       login_id,
       member_name,
       email,
       created_at
FROM view_member
WHERE active = 'Y';

SELECT *
FROM v_view_active_member
ORDER BY member_id;

active = 'N'인 회원은 결과에서 제외됩니다.

6-3. 읽기 전용 VIEW

CREATE OR REPLACE VIEW v_view_member_readonly AS
SELECT member_id,
       login_id,
       member_name,
       email,
       active,
       created_at
FROM view_member
WITH READ ONLY;

SELECT *
FROM v_view_member_readonly
ORDER BY member_id;

6-4. READ ONLY VIEW 수정 시도

UPDATE v_view_member_readonly
SET member_name = '수정시도'
WHERE member_id = 1;

읽기 전용 VIEW이므로 오류가 발생해야 정상입니다.

6-5. 게시글 목록 조인 VIEW

CREATE OR REPLACE VIEW v_view_board_list AS
SELECT b.board_id,
       b.title,
       b.content,
       b.deleted_yn,
       b.created_at,
       m.member_id,
       m.login_id,
       m.member_name
FROM view_board b
JOIN view_member m
  ON b.member_id = m.member_id
WHERE b.deleted_yn = 'N';

SELECT *
FROM v_view_board_list
ORDER BY board_id DESC;

6-6. 회원별 게시글 수 VIEW

CREATE OR REPLACE VIEW v_view_member_board_count AS
SELECT m.member_id,
       m.login_id,
       m.member_name,
       COUNT(b.board_id) AS board_count
FROM view_member m
LEFT JOIN view_board b
  ON m.member_id = b.member_id
GROUP BY m.member_id,
         m.login_id,
         m.member_name;

SELECT *
FROM v_view_member_board_count
ORDER BY member_id;

6-7. VIEW 수정

CREATE OR REPLACE VIEW v_view_member_public AS
SELECT member_id,
       login_id,
       member_name,
       email,
       phone,
       active,
       created_at
FROM view_member;

SELECT *
FROM v_view_member_public
ORDER BY member_id;

VIEW 컬럼이 변경되면 DTO, Mapper, 화면에도 영향이 생길 수 있습니다.

6-8. VIEW 삭제

DROP VIEW v_view_member_public;

SELECT view_name
FROM user_views
WHERE view_name = 'V_VIEW_MEMBER_PUBLIC';

SELECT *
FROM view_member;

VIEW를 삭제해도 원본 VIEW_MEMBER 테이블은 그대로 남아 있어야 합니다.

6-9. USER_VIEWS 확인

SELECT view_name
FROM user_views
ORDER BY view_name;

SELECT view_name,
       text
FROM user_views
WHERE view_name = 'V_VIEW_BOARD_LIST';
VIEW DML

7. VIEW에 DML 실습

VIEW는 조회용으로 많이 사용하지만, 단순 VIEW의 경우 INSERT, UPDATE, DELETE가 가능한 경우도 있습니다. 단, VIEW에 대한 DML은 VIEW만 바꾸는 것이 아니라 원본 테이블에 반영될 수 있으므로 주의해야 합니다.

7-1. 단순 VIEW 생성

CREATE OR REPLACE VIEW v_view_member_simple AS
SELECT member_id,
       login_id,
       member_name,
       email,
       active
FROM view_member;

7-2. 단순 VIEW에 UPDATE

UPDATE v_view_member_simple
SET email = 'hong_view_update@test.com'
WHERE member_id = 1;

SELECT member_id,
       login_id,
       email
FROM view_member
WHERE member_id = 1;

ROLLBACK;

원본 VIEW_MEMBER의 EMAIL 값이 바뀌었다면 VIEW DML이 원본에 반영된 것입니다.

7-3. READ ONLY VIEW에 UPDATE

UPDATE v_view_member_readonly
SET email = 'readonly_update@test.com'
WHERE member_id = 1;

읽기 전용 VIEW이므로 오류가 발생해야 정상입니다.

7-4. 조인 VIEW에 INSERT 시도

INSERT INTO v_view_board_list (
    board_id,
    title,
    content,
    deleted_yn,
    created_at,
    member_id,
    login_id,
    member_name
) VALUES (
    999,
    '뷰 INSERT 테스트',
    '조인 뷰에 INSERT 시도',
    'N',
    SYSDATE,
    1,
    'hong',
    '홍길동'
);

조인 VIEW는 대부분 INSERT가 어렵습니다. 오류가 발생해도 정상적인 실습 결과입니다.

VIEW DML 제한 정리

  • 단순 VIEW는 DML이 가능할 수 있습니다.
  • DISTINCT, GROUP BY, 집계 함수, UNION 등이 포함되면 DML이 제한될 수 있습니다.
  • 조인 VIEW는 대부분 INSERT가 어렵습니다.
  • VIEW에 대한 DML은 원본 테이블에 반영될 수 있습니다.
  • 조회 전용 VIEW는 WITH READ ONLY를 사용하는 것이 안전합니다.
ORDER BY

8. VIEW 생성 시 ORDER BY 주의

교안에서는 Oracle 8i 이전 버전까지 VIEW 생성 시 ORDER BY를 포함할 수 없다고 설명합니다. 현대 버전에서는 일부 상황에서 허용될 수 있지만, 실무에서는 VIEW 정의에 정렬을 고정하기보다 VIEW를 조회할 때 ORDER BY를 작성하는 방식이 더 명확합니다.

비권장

CREATE OR REPLACE VIEW v_member_public AS
SELECT member_id,
       login_id,
       member_name
FROM view_member
ORDER BY member_id;

권장

CREATE OR REPLACE VIEW v_member_public AS
SELECT member_id,
       login_id,
       member_name
FROM view_member;

SELECT *
FROM v_member_public
ORDER BY member_id;

10. 실습 문제 카드

첨부된 실습 문제의 JOIN, OUTER JOIN, TOP-N 문제는 VIEW와 잘 연결됩니다. 반복되는 SELECT문을 VIEW로 만들어두면 매번 긴 SQL을 작성하지 않아도 됩니다.

문제 1. 공개 회원 VIEW 만들기

CREATE OR REPLACE VIEW v_member_public AS
SELECT member_id,
       login_id,
       member_name,
       email,
       active,
       created_at
FROM view_member;

SELECT *
FROM v_member_public;

문제 2. 활성 회원 VIEW 만들기

CREATE OR REPLACE VIEW v_active_member AS
SELECT member_id,
       login_id,
       member_name,
       email
FROM view_member
WHERE active = 'Y';

SELECT *
FROM v_active_member;

문제 3. 게시글 목록 VIEW 만들기

CREATE OR REPLACE VIEW v_board_list AS
SELECT b.board_id,
       b.title,
       b.content,
       b.created_at,
       m.login_id,
       m.member_name
FROM view_board b
JOIN view_member m

Mistake

11. 자주 하는 실수

  • 실수 1. VIEW를 실제 테이블로 착각함 VIEW는 일반적으로 데이터를 저장하지 않고 SELECT문을 저장한 가상 객체입니다.
  • 실수 2. VIEW에 DML하면 VIEW만 바뀐다고 생각함 단순 VIEW에 대한 UPDATE는 원본 테이블에 반영될 수 있습니다.
  • 실수 3. 조인 VIEW에 INSERT하려고 함 두 개 이상 테이블로 만든 VIEW는 대부분 INSERT가 어렵습니다.
  • 실수 4. VIEW를 만들면 성능이 무조건 빨라진다고 생각함 일반 VIEW는 조회 시 내부 SELECT가 실행되므로 성능은 인덱스, 조인 구조, 실행 계획과 함께 봐야 합니다.
  • 실수 5. USER_VIEWS로 정의를 확인하지 않음 VIEW 내부 SELECT를 모르면 유지보수가 어려워집니다.
  • 실수 6. VIEW 수정 후 DTO나 Mapper를 수정하지 않음 VIEW 컬럼이 바뀌면 MyBatis resultMap, DTO, JPA Entity, React 화면에도 영향이 생길 수 있습니다.
  • 실수 7. 조회 전용 VIEW에 WITH READ ONLY를 붙이지 않음 조회용 VIEW라면 원본 테이블 보호를 위해 WITH READ ONLY를 고려해야 합니다.
Interview

12. 면접 질문 카드

Q1. VIEW란 무엇인가요?

VIEW는 하나 이상의 테이블 또는 SELECT 결과를 바탕으로 만든 가상의 테이블입니다.
실제 데이터를 저장하는 것이 아니라 SELECT문을 저장해두고, 조회할 때 원본 테이블에서 데이터를 가져옵니다.

Q2. VIEW와 TABLE의 차이는 무엇인가요?

TABLE은 실제 데이터를 저장하는 객체이고, VIEW는 일반적으로 데이터를 저장하지 않는 가상의 테이블입니다.
VIEW는 SELECT문으로 정의되며, 원본 테이블의 데이터를 필요한 형태로 보여줍니다.

Q3. VIEW를 사용하는 이유는 무엇인가요?

VIEW는 민감한 컬럼을 숨겨 보안을 유지하고, 복잡한 JOIN이나 집계 쿼리를 재사용하며,
애플리케이션에서 긴 SQL을 반복하지 않아도 되도록 소스를 간결하게 만들기 위해 사용합니다.

Q4. CREATE VIEW와 CREATE OR REPLACE VIEW의 차이는 무엇인가요?

CREATE VIEW는 새로운 VIEW를 생성합니다. 같은 이름의 VIEW가 이미 있으면 오류가 발생할 수 있습니다.
CREATE OR REPLACE VIEW는 같은 이름의 VIEW가 없으면 새로 만들고, 이미 있으면 기존 VIEW 정의를 교체합니다.

Q5. WITH READ ONLY는 무엇인가요?

WITH READ ONLY는 VIEW를 조회 전용으로 만드는 옵션입니다. 이 옵션이 붙은 VIEW는 SELECT는 가능하지만 VIEW를 통한 INSERT, UPDATE, DELETE는 제한됩니다.

Q6. VIEW에 INSERT, UPDATE, DELETE가 가능한가요?

가능한 경우도 있고 제한되는 경우도 있습니다.
하나의 테이블로 만든 단순 VIEW는 DML이 가능한 경우가 있지만, DISTINCT, GROUP BY, 집계 함수, UNION, 조인 등이 포함된 VIEW는 제한될 수 있습니다.

Q7. VIEW에 UPDATE하면 데이터가 어디에 반영되나요?

단순 VIEW에 UPDATE가 가능하다면, 그 변경은 VIEW 자체가 아니라 원본 테이블에 반영됩니다. VIEW는 일반적으로 데이터를 직접 저장하지 않기 때문입니다.

Q8. USER_VIEWS는 무엇인가요?

USER_VIEWS는 현재 사용자가 생성한 VIEW 정보를 확인할 수 있는 데이터 딕셔너리 뷰입니다.
VIEW_NAME으로 VIEW 이름을 확인하고, TEXT 컬럼으로 VIEW를 만들 때 사용한 SELECT 정의를 확인할 수 있습니다.

Q9. VIEW를 만들면 성능이 좋아지나요?

일반 VIEW는 성능 향상 도구라기보다 SQL 재사용과 간결화를 위한 도구입니다.
VIEW는 조회 시 내부 SELECT를 실행하므로 성능은 원본 테이블의 인덱스, 조건절, 조인 방식, 실행 계획에 영향을 받습니다.

학습 메모

오늘의 핵심

VIEW는 SELECT문에 이름을 붙여 만든 가상의 테이블입니다. 원본 테이블의 필요한 컬럼과 행만 보여줄 수 있어 보안성과 편의성이 좋습니다.

암기 문법

CREATE OR REPLACE VIEW 뷰명 AS
SELECT 컬럼명
FROM 테이블명
WHERE 조건;

CREATE OR REPLACE VIEW 뷰명 AS
SELECT 컬럼명
FROM 테이블명
WITH READ ONLY;

DROP VIEW 뷰명;

VIEW 정보 확인

SELECT view_name,
       text
FROM user_views;

VIEW의 장점

  • 보안성 : 민감한 컬럼 숨김
  • 편의성 : 복잡한 JOIN 재사용
  • 간결성 : 애플리케이션 SQL 단순화
  • 재사용성 : 여러 화면에서 같은 조회 구조 사용

주의할 점

단순 VIEW에 UPDATE하면 원본 테이블이 바뀔 수 있습니다. 조회 전용 VIEW라면 WITH READ ONLY를 고려해야 합니다.
