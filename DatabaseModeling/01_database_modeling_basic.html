01

데이터베이스 모델링을 공부한다는 것

데이터베이스 모델링은 현실 세계의 업무를 데이터베이스로 옮기기 위한 설계도 작성 과정입니다.
현실 업무에는 사람, 사물, 사건, 장소, 개념이 섞여 있습니다. 개발자는 이 복잡한 업무 문장 속에서 관리해야 할 데이터를 찾아야 합니다.
예를 들어 “고객이 자동차를 구매한다”라는 문장을 바로 테이블로 만들면 안 됩니다.
먼저 고객, 자동차, 판매라는 대상을 찾고, 각각이 어떤 정보를 가지며, 서로 어떤 관계인지 분석해야 합니다.

Modeling

현실 업무를 데이터 구조로 번역

사람이 처리하는 업무를
DB가 저장하고 조회할 수 있는
테이블, 컬럼, 키, 관계 구조로
바꾸는 과정입니다.

Data First

데이터 중심 분석

화면이나 버튼보다 먼저
“이 시스템이 어떤 데이터를
관리해야 하는가?”를 기준으로 분석합니다.

Design

테이블 이전의 설계도

바로 CREATE TABLE을 작성하지 않고,
ERD와 관계 구조를
먼저 정리해 설계 오류를 줄입니다.

Connection

전체 개발 구조의 시작점

DB 모델링 결과는 Java 객체,
Spring Entity, DTO, API 요청 구조,
React Form까지 영향을 줍니다.

핵심 문장

데이터베이스 모델링은 현실 업무 → 실체/속성/관계 → ERD → Table → Entity → API → React 화면 으로 이어지는 전체 개발 흐름의 출발점입니다.

02

현실 업무에서 React 화면까지의 흐름도

데이터베이스 모델링은 DB 안에서만 끝나는 작업이 아닙니다. 처음에는 현실 업무를 분석하지만, 최종적으로는 Spring Boot API와 React 화면 구조까지 연결됩니다.

1 현실 업무

사람이 실제로 처리하는 업무입니다.
예: 고객 등록, 자동차 판매, 예배 기록 작성.

→
2 데이터 중심 분석

화면이 아니라 저장해야 할 데이터가 무엇인지 찾습니다.
명사 중심으로 관리 대상을 뽑습니다.

→
3 ERD

엔터티, 속성, 식별자, 관계를 그림으로 정리합니다.
이 단계가 설계의 중심입니다.

4 Table

ERD를 테이블, 컬럼, PK, FK 구조로 변환합니다.
Oracle SQL로 구현할 준비를 합니다.

→
5 Spring Entity / DTO

테이블은 Entity로 연결되고,
화면 요청 데이터는 DTO로 주고받습니다.

→
6 React Form / Axios

React 입력값은 DTO 형태의 JSON으로 변환되어
Spring Boot API로 전송됩니다.

개념적 모델링

무엇을 관리할 것인가?

업무 관점에서 관리해야 할 대상을 찾습니다. 아직 데이터 타입이나 인덱스를 깊게 정하지 않습니다.

논리적 모델링

어떤 구조로 나눌 것인가?

엔터티, 속성, 관계를 테이블 구조에 가깝게 정리합니다. Mapping Rule과 정규화가 연결됩니다.

물리적 모델링

실제 DBMS에서 어떻게 만들 것인가?

Oracle 기준으로 NUMBER, VARCHAR2, DATE, PK, FK, INDEX, SEQUENCE 등을 구체화합니다.

화면 하나가 테이블 하나라는 뜻은 아닙니다

예배 기록 작성 화면
  ├─ WORSHIP_SERVICE
  │   ├─ service_id
  │   ├─ service_title
  │   ├─ service_date
  │   └─ writer_name
  │
  └─ WORSHIP_TIMETABLE_ITEM
      ├─ item_id
      ├─ service_id
      ├─ sort_order
      ├─ item_title
      └─ manager_name
03

그림처럼 이해하기

데이터베이스 모델링은 건축 설계도와 비슷합니다. 집을 지을 때 바로 벽돌부터 쌓지 않고, 먼저 방의 위치, 배관, 전기선, 문 위치를 설계합니다.
DB도 마찬가지로 바로 테이블부터 만들면 나중에 구조를 크게 고쳐야 할 수 있습니다.

현실 업무

건축 의뢰 내용

“예배 기록을 저장하고 싶다”,
“고객 판매 내역을 관리하고 싶다”처럼
사람이 원하는 업무 요구사항입니다.

ERD

건축 설계도

어떤 공간이 필요한지,
공간끼리 어떻게 연결되는지를
그린 설계도처럼 엔터티와 관계를 표현합니다.

Table

실제 시공 구조

설계도를 바탕으로
실제 DBMS에 테이블과 컬럼,
PK, FK, 제약 조건을 생성합니다.

API / React

사람이 사용하는 입구

사용자는 React 화면에서 데이터를 입력하고,
Spring Boot API가 그 데이터를
DB 구조에 맞게 저장합니다.

초보자가 가장 많이 하는 착각

화면이 하나라고 테이블도 하나라고 생각하는 것입니다. 화면은 사용자 경험 기준이고, 테이블은 데이터 구조 기준입니다.
하나의 React 화면에서 여러 테이블의 데이터를 동시에 입력할 수 있습니다.

public class Customer {

    private String address;

        this.customerId = customerId;

4) Spring Boot Entity

import jakarta.persistence.*;
import java.time.LocalDateTime;

@Entity
@Table(name = "CUSTOMER")
public class CustomerEntity {

    @Id
    @GeneratedValue(strategy = GenerationType.SEQUENCE, generator = "seq_customer")
    @SequenceGenerator(
            name = "seq_customer",
            sequenceName = "SEQ_CUSTOMER",
            allocationSize = 1
    )
    @Column(name = "CUSTOMER_ID")
    private Long customerId;

    @Column(name = "NAME", nullable = false, length = 50)
    private String name;

    @Column(name = "PHONE", length = 30)
    private String phone;

    @Column(name = "ADDRESS", length = 200)
    private String address;

    @Column(name = "CREATED_AT")
    private LocalDateTime createdAt;

    protected CustomerEntity() {
    }

    public CustomerEntity(String name, String phone, String address) {
        this.name = name;
        this.phone = phone;
        this.address = address;
        this.createdAt = LocalDateTime.now();
    }
}

7) Controller

import org.springframework.web.bind.annotation.*;

@RestController
@RequestMapping("/api/customers")
public class CustomerController {

    private final CustomerRepository customerRepository;

    public CustomerController(CustomerRepository customerRepository) {
        this.customerRepository = customerRepository;
    }

    @PostMapping
    public CustomerEntity createCustomer(@RequestBody CustomerCreateRequest request) {
        CustomerEntity customer = new CustomerEntity(
                request.getName(),
                request.getPhone(),
                request.getAddress()
        );

        return customerRepository.save(customer);
    }
}

9) React Form 입력값을 DTO로 보내기

import { useState } from "react";
import { createCustomer } from "../api/customerApi";

function CustomerCreatePage() {
  const [form, setForm] = useState({
    name: "",
    phone: "",
    address: "",
  });

  const handleChange = (event) => {
    const { name, value } = event.target;

    setForm((prev) => ({
      ...prev,
      [name]: value,
    }));
  };

  const handleSubmit = async (event) => {
    event.preventDefault();

    try {
      const savedCustomer = await createCustomer(form);
      console.log("저장된 고객:", savedCustomer);
      alert("고객 등록 성공");
    } catch (error) {
      console.error(error);
      alert("고객 등록 실패");
    }
  };

  return (
    <form onSubmit={handleSubmit}>
      <input
        name="name"
        value={form.name}
        onChange={handleChange}
        placeholder="고객 이름"
      />

      <input
        name="phone"
        value={form.phone}
        onChange={handleChange}
        placeholder="연락처"
      />

      <input
        name="address"
        value={form.address}
        onChange={handleChange}
        placeholder="주소"
      />

      <button type="submit">등록</button>
    </form>
  );
}

export default CustomerCreatePage;

인증이 있는 프로젝트에서는 로그인 후 받은 JWT를 저장하고, Axios 요청마다 Authorization 헤더에 붙입니다.

const ACCESS_TOKEN_KEY = "accessToken";

export function saveToken(token) {
  localStorage.setItem(ACCESS_TOKEN_KEY, token);
}

export function getToken() {
  return localStorage.getItem(ACCESS_TOKEN_KEY);
}

apiClient.interceptors.request.use((config) => {
  const token = getToken();

  if (token) {
    config.headers.Authorization = `Bearer ${token}`;
  }

  return config;
});
05

자주 하는 실수 카드

1. 바로 CREATE TABLE부터 작성

요구사항 분석 없이 테이블을 만들면 반복 컬럼,
중복 데이터, FK 혼란이 생기기 쉽습니다.

2. 화면 하나를 테이블 하나로 생각

화면은 사용자 경험 기준이고, 테이블은 데이터 구조 기준입니다.

3. 반복 데이터를 컬럼으로 계속 추가

item1_title, item2_title처럼 반복 컬럼을 만들면
데이터가 늘어날수록 구조가 깨집니다.

4. PK 없이 테이블 만들기

각 행을 구분할 식별자가 없으면
수정, 삭제, 참조 관계 설정이 어려워집니다.

5. FK 관계를 늦게 생각하기

부모와 자식 관계를 뒤늦게 정하면
저장 순서와 삭제 정책이 복잡해집니다.

6. DTO 필드명과 React 필드명 불일치

React가 customerName으로 보내는데
DTO가 name을 기대하면 값이 제대로 매핑되지 않을 수 있습니다.

06

오류 메시지 카드

DB 모델링이 부족하면 오류는 Oracle에서만 끝나지 않습니다. Spring Boot Entity, DTO, API 요청, React 화면까지 연쇄적으로 문제가 발생할 수 있습니다.

오류 메시지 / 현상
의미
확인할 것
ORA-01400
NOT NULL 컬럼에 NULL을 넣으려고 했습니다.
필수 입력값인지, React Form과 DTO 필드명이 일치하는지 확인합니다.
ORA-00001
PK 또는 UNIQUE 값이 중복되었습니다.
무엇이 유일해야 하는지, 시퀀스 설정이 올바른지 확인합니다.
ORA-02291
자식 데이터가 참조하는 부모 데이터가 없습니다.
부모 데이터를 먼저 저장했는지, FK 값이 실제로 존재하는지 확인합니다.
ORA-02292
부모 데이터를 삭제하려는데 자식 데이터가 남아 있습니다.
삭제 정책을 정합니다. 자식 먼저 삭제, cascade, 상태값 처리 중 선택합니다.
ids must be manually assigned
JPA Entity의 ID 자동 생성 설정이 없거나 잘못되었습니다.
@Id, @GeneratedValue, @SequenceGenerator,
DB Sequence 존재 여부를 확인합니다.
JSON parse error
React JSON을 Spring DTO로 변환하지 못했습니다.
DTO 구조와 요청 body 구조, 날짜 형식, 배열 구조가 맞는지 확인합니다.
Cannot read properties of undefined
React에서 아직 없는 값을 읽으려고 했습니다.
API 응답 DTO 구조와 React 화면에서 접근하는 구조가 같은지 확인합니다.

오류를 읽는 기본 순서

  1. DB 컬럼 제약 조건을 확인합니다. 예: NOT NULL, UNIQUE, FK.
  2. Spring Entity의 컬럼 매핑과 ID 생성 전략을 확인합니다.
  3. DTO 필드명과 React 요청 body 필드명이 일치하는지 확인합니다.
  4. 부모 데이터와 자식 데이터의 저장 순서가 맞는지 확인합니다.
  5. API 응답 구조와 React 화면 출력 구조가 일치하는지 확인합니다.

실무 연결 카드

면접 포인트 카드

Q1. 데이터베이스 모델링이란 무엇인가요?

현실 세계의 업무 요구사항을 분석해서 관리해야 할 데이터 대상, 속성, 관계를 찾고 이를 데이터베이스 테이블 구조로 설계하는 과정입니다.

Q2. 데이터베이스 모델링을 왜 해야 하나요?

모델링 없이 바로 테이블을 만들면 데이터 중복, NULL 문제, FK 관계 혼란, API 구조 변경 같은 문제가 생길 수 있기 때문입니다.

Q3. 개념적, 논리적, 물리적 모델링의 차이는 무엇인가요?

개념적 모델링은 업무 관점에서 관리 대상을 찾는 단계, 논리적 모델링은 테이블 관계에 가깝게 정리하는 단계, 물리적 모델링은 실제 DBMS에 맞게 타입과 제약 조건을 정하는 단계입니다.

Q4. 왜 바로 CREATE TABLE을 작성하면 안 되나요?

요구사항 분석 없이 테이블을 만들면 반복 컬럼과 중복 데이터가 생기고, 나중에 Entity, DTO, React Form까지 연쇄 수정해야 할 수 있습니다.

Q5. 데이터 중심 분석이란 무엇인가요?

화면이나 기능보다 먼저 시스템이 관리해야 할 데이터를 중심으로 분석하는 방법입니다. 화면 하나가 곧 테이블 하나라는 뜻은 아닙니다.

Q6. DB 모델링이 Spring Boot 개발과 어떻게 연결되나요?

테이블은 Entity로 매핑되고, 컬럼은 필드가 되며, PK와 FK는 식별자와 관계 구조에 반영됩니다. DTO와 API 구조도 DB 설계의 영향을 받습니다.

Q7. DB 모델링이 React 개발과도 관련이 있나요?

관련이 있습니다. React Form의 상태 구조와 Axios 요청 body는 Spring DTO와 연결되고, DTO는 Entity와 DB 테이블 구조에 연결됩니다.

학습 메모

오늘의 암기 포인트

  • 데이터베이스 모델링은 현실 업무를 데이터 구조로 번역하는 설계 과정입니다.
  • 바로 CREATE TABLE부터 작성하지 않고, 먼저 관리 대상과 관계를 찾습니다.
  • 개념적 모델링은 업무 언어, 논리적 모델링은 데이터 구조 언어, 물리적 모델링은 DBMS 구현 언어에 가깝습니다.
  • 좋은 DB 모델링은 Java Entity, DTO, API, React Form 구조를 안정적으로 만듭니다.

헷갈리기 쉬운 개념

화면 vs 테이블

화면 하나가 테이블 하나라는 뜻은 아닙니다.
하나의 화면에서 여러 테이블 데이터를 입력할 수 있습니다.

Java Class vs DB Table

서로 연결되지만 완전히 같은 개념은 아닙니다.
DB는 저장 구조, Java는 객체 구조에 더 가깝습니다.

개념적 vs 논리적 모델링

개념적 모델링은 업무 대상을 찾는 단계이고,
논리적 모델링은 테이블 관계에 가깝게 정리하는 단계입니다.

논리적 vs 물리적 모델링

논리 모델은 구조 중심이고,
물리 모델은 Oracle 같은 실제 DBMS 구현 중심입니다.

DTO 필드명

React 요청 body와 Spring DTO 필드명이 다르면
값이 제대로 들어오지 않을 수 있습니다.

PK와 FK

PK는 자기 테이블의 행을 구분하고,
FK는 다른 테이블과의 관계를 연결합니다.
