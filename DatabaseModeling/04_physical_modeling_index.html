Database Modeling - Chapter 04

물리 모델링과 인덱스

물리 모델링은 논리적으로 설계한 테이블 구조를 실제 Oracle DBMS에서 동작 가능한 형태로 구현하는 단계입니다.
이 단계에서는 VARCHAR2, NUMBER, DATE, TIMESTAMP 같은 데이터 타입을 정하고, PK, FK, NOT NULL, UNIQUE, CHECK, INDEX를 설계합니다.
좋은 물리 설계는 단순히 테이블을 만드는 것을 넘어 Spring Boot Entity, DTO Validation, Repository Query, React Form 입력 제한까지 안정적으로 연결합니다.

Physical Modeling DBMS Oracle VARCHAR2 NUMBER DATE TIMESTAMP Column Size Primary Key Foreign Key NOT NULL UNIQUE CHECK DEFAULT INDEX SEQUENCE View Stored Procedure Trigger Denormalization
학습 목표 1

논리 모델을 실제 Oracle 테이블로 구현할 때
데이터 타입, 컬럼 크기, 제약 조건을
어떻게 정하는지 설명할 수 있습니다.

학습 목표 2

인덱스가 왜 필요한지,
언제 사용해야 하는지,
그리고 남용하면 왜 위험한지 이해할 수 있습니다.

학습 목표 3

물리 모델링 결과가 Spring Boot Entity,
DTO Validation, Repository Query,
React Form 제한과 연결되는 흐름을 이해할 수 있습니다.

01

물리 모델링을 공부한다는 것

앞 단원에서는 ERD를 테이블 구조로 바꾸고, 정규화를 통해 중복과 이상 현상을 줄이는 방법을 배웠습니다.
하지만 논리적으로 테이블을 나누는 것만으로는 실제 데이터베이스가 완성되지 않습니다. Oracle에 테이블을 만들려면 컬럼마다 데이터 타입, 크기, 제약 조건, 인덱스, 시퀀스를 정해야 합니다.
예를 들어 논리 모델에서 고객명이라는 속성이 있다고 해도, 실제 DB에서는 이 값을 VARCHAR2(50) NOT NULL로 만들지, VARCHAR2(100)으로 만들지 결정해야 합니다.
이 결정은 Spring Boot의 @Column(length = 50), DTO의 @Size(max = 50), React의 maxLength={50}까지 영향을 줍니다.

DBMS 선정

어떤 DBMS에 맞출 것인가?

Oracle, MySQL, PostgreSQL은
타입과 자동 증가 방식이 다릅니다.
현재 학습에서는 Oracle 기준으로
물리 설계를 이해합니다.

Data Type

어떤 값이 들어오는가?

문자열은 VARCHAR2,
숫자는 NUMBER, 업무 날짜는 DATE,
생성/수정 시각은 TIMESTAMP로
설계할 수 있습니다.

Constraint

잘못된 데이터 막기

PK, FK, NOT NULL, UNIQUE,
CHECK, DEFAULT는 DB 차원에서
데이터 무결성을 지키는 규칙입니다.

Index

조회 성능 확보

WHERE, JOIN, ORDER BY에
자주 쓰이는 컬럼은 인덱스 후보입니다.
단, 모든 컬럼에 만들면
쓰기 성능이 떨어질 수 있습니다.

핵심 문장

물리 모델링은 논리 모델 → Oracle CREATE TABLE → 제약 조건 → 인덱스 → Entity → DTO → React Form 으로 이어지는 실제 구현 설계 단계입니다.

업무 날짜는 DATE,

03

그림처럼 이해하기

물리 모델링은 건축 도면을 실제 자재와 시공 방식으로 바꾸는 것과 같습니다.
논리 모델이 “고객 방, 자동차 방, 판매 기록 방이 필요하다”는 도면이라면, 물리 모델은 “이 방의 벽 재료는 무엇이고, 문 크기는 얼마이며, 자주 오가는 통로는 넓혀야 하는가?”를 정하는 단계입니다.

논리 모델

건축 도면

CUSTOMER, CAR, SALE이 필요하고,
SALE은 CUSTOMER와 CAR를
참조한다는 구조를 정합니다.

데이터 타입

건축 자재 선택

문자열은 VARCHAR2, 숫자는 NUMBER,
날짜는 DATE, 생성 시각은 TIMESTAMP로
정합니다.

제약 조건

안전 규칙

PK, FK, NOT NULL, UNIQUE,
CHECK로 잘못된 데이터가
들어오지 못하게 막습니다.

인덱스

빠른 이동 통로

자주 찾는 데이터로 빠르게
이동할 수 있게 돕지만,
너무 많이 만들면 유지 비용이 커집니다.

Oracle → Java 
VARCHAR2  → String
NUMBER    → Long / Integer / BigDecimal
DATE      → LocalDate
TIMESTAMP → LocalDateTime
Oracle → Spring Validation 
NOT NULL        → @NotNull / @NotBlank
VARCHAR2(50)    → @Size(max = 50)
CHECK(price>=0) → @PositiveOrZero
UNIQUE          → 중복 검사 로직 + DB 제약
Oracle → React Form 
VARCHAR2(50) → maxLength={50}
NOT NULL     → required
DATE         → type="date"
NUMBER       → type="number"

인덱스는 빠른 검색표이지만 공짜가 아닙니다

장점

WHERE 조건, JOIN 조건, 날짜 범위 검색, 정렬에서 원하는 데이터를 더 빠르게 찾을 수 있습니다.

단점

저장 공간을 더 사용하고, INSERT, UPDATE, DELETE 때
인덱스도 함께 갱신해야 하므로 쓰기 비용이 증가합니다.

정규화와 역정규화의 균형

정규화는 중복을 줄여 데이터 무결성을 높이고, 역정규화는 조회 성능이나 업무 편의성을 위해 일부 중복을 허용합니다.
처음부터 역정규화를 남발하기보다, 먼저 정규화된 구조를 만들고 실제 조회 성능 문제가 명확할 때 신중하게 적용하는 것이 좋습니다.

1) Oracle CREATE TABLE과 제약 조건

CREATE TABLE customer (
    customer_id NUMBER PRIMARY KEY,
    name VARCHAR2(50) NOT NULL,
    phone VARCHAR2(30),
    address VARCHAR2(200),
    status VARCHAR2(20) DEFAULT 'ACTIVE',
    created_at TIMESTAMP DEFAULT SYSTIMESTAMP,

    CONSTRAINT chk_customer_status
        CHECK (status IN ('ACTIVE', 'INACTIVE'))
);

CREATE TABLE car (
    car_id NUMBER PRIMARY KEY,
    car_name VARCHAR2(100) NOT NULL,
    model_year NUMBER(4),
    created_at TIMESTAMP DEFAULT SYSTIMESTAMP
);

CREATE TABLE sale (
    sale_id NUMBER PRIMARY KEY,
    customer_id NUMBER NOT NULL,
    car_id NUMBER NOT NULL,
    sale_date DATE NOT NULL,
    sale_price NUMBER,

    CONSTRAINT fk_sale_customer
        FOREIGN KEY (customer_id)
        REFERENCES customer(customer_id),

    CONSTRAINT fk_sale_car
        FOREIGN KEY (car_id)
        REFERENCES car(car_id),

    CONSTRAINT chk_sale_price
        CHECK (sale_price >= 0)
);

4) View 예제

CREATE VIEW v_sale_list AS
SELECT
    s.sale_id,
    c.name AS customer_name,
    car.car_name,
    s.sale_date,
    s.sale_price
FROM sale s
JOIN customer c
    ON s.customer_id = c.customer_id
JOIN car car
    ON s.car_id = car.car_id;

5) Trigger 예제

CREATE TABLE sale_audit_log (
    audit_id NUMBER PRIMARY KEY,
    sale_id NUMBER NOT NULL,
    action_type VARCHAR2(20) NOT NULL,
    created_at TIMESTAMP DEFAULT SYSTIMESTAMP
);

CREATE SEQUENCE seq_sale_audit_log
    START WITH 1
    INCREMENT BY 1;

CREATE OR REPLACE TRIGGER trg_sale_audit
AFTER INSERT ON sale
FOR EACH ROW
BEGIN
    INSERT INTO sale_audit_log (
        audit_id,
        sale_id,
        action_type,
        created_at
    )
    VALUES (
        seq_sale_audit_log.NEXTVAL,
        :NEW.sale_id,
        'INSERT',
        SYSTIMESTAMP
    );
END;

import jakarta.persistence.*;
import java.time.LocalDateTime;

@Entity
@Table(
        name = "CUSTOMER",
        indexes = {
                @Index(name = "idx_customer_name", columnList = "NAME")
        }
)
public class CustomerEntity {

    @Id
    @GeneratedValue(strategy = GenerationType.SEQUENCE, generator = "seq_customer")
    @SequenceGenerator(
            name = "seq_customer",
            sequenceName = "SEQ_CUSTOMER",
            allocationSize = 1
    )
    @Column(name = "CUSTOMER_ID")
    private Long customerId;

    @Column(name = "NAME", nullable = false, length = 50)
    private String name;

    @Column(name = "PHONE", length = 30)
    private String phone;

    @Column(name = "ADDRESS", length = 200)
    private String address;

    @Column(name = "STATUS", length = 20)
    private String status;

    @Column(name = "CREATED_AT")
    private LocalDateTime createdAt;

    protected CustomerEntity() {
    }

    public CustomerEntity(String name, String phone, String address) {
        this.name = name;
        this.phone = phone;
        this.address = address;
        this.status = "ACTIVE";
        this.createdAt = LocalDateTime.now();
    }
}
import jakarta.persistence.*;

8) Repository / Service / Controller

import org.springframework.data.jpa.repository.JpaRepository;
import java.time.LocalDate;
import java.util.List;

public interface SaleRepository extends JpaRepository<SaleEntity, Long> {
    List<SaleEntity> findByCustomerId(Long customerId);
    List<SaleEntity> findByCarId(Long carId);
    List<SaleEntity> findBySaleDateBetween(LocalDate startDate, LocalDate endDate);
}
import org.springframework.stereotype.Service;

@Service
public class SaleService {

    private final SaleRepository saleRepository;
    private final CustomerRepository customerRepository;
    private final CarRepository carRepository;

    public SaleService(
            SaleRepository saleRepository,
            CustomerRepository customerRepository,
            CarRepository carRepository
    ) {
        this.saleRepository = saleRepository;
        this.customerRepository = customerRepository;
        this.carRepository = carRepository;
    }

    public SaleEntity createSale(SaleCreateRequest request) {
        if (!customerRepository.existsById(request.getCustomerId())) {
            throw new IllegalArgumentException("존재하지 않는 고객입니다.");
        }

        if (!carRepository.existsById(request.getCarId())) {
            throw new IllegalArgumentException("존재하지 않는 자동차입니다.");
        }

        SaleEntity sale = new SaleEntity(
                request.getCustomerId(),
                request.getCarId(),
                request.getSaleDate(),
                request.getSalePrice()
        );

        return saleRepository.save(sale);
    }
}
import jakarta.validation.Valid;
import org.springframework.web.bind.annotation.*;

@RestController
@RequestMapping("/api/sales")
public class SaleController {

    private final SaleService saleService;

    public SaleController(SaleService saleService) {
        this.saleService = saleService;
    }

    @PostMapping
    public SaleEntity createSale(@Valid @RequestBody SaleCreateRequest request) {
        return saleService.createSale(request);
    }
}

9) React Axios API 호출

import axios from "axios";

const apiClient = axios.create({
  baseURL: "http://localhost:8080/api",
  headers: {
    "Content-Type": "application/json",
  },
});

export async function createSale(form) {
  const requestBody = {
    customerId: Number(form.customerId),
    carId: Number(form.carId),
    saleDate: form.saleDate,
    salePrice: Number(form.salePrice),
  };

  const response = await apiClient.post("/sales", requestBody);
  return response.data;
}

export async function getSalesByCustomer(customerId) {
  const response = await apiClient.get(`/sales?customerId=${customerId}`);
  return response.data;
}

10) React Form 입력값을 DTO로 보내기

import { useState } from "react";
import { createSale } from "../api/saleApi";

function SaleCreatePage() {
  const [form, setForm] = useState({
    customerId: "",
    carId: "",
    saleDate: "",
    salePrice: "",
  });

  function handleChange(event) {
    const { name, value } = event.target;

    setForm((prev) => ({
      ...prev,
      [name]: value,
    }));
  }

  async function handleSubmit(event) {
    event.preventDefault();

    try {
      const savedSale = await createSale(form);
      console.log("저장된 판매:", savedSale);
      alert("판매 등록 성공");
    } catch (error) {
      console.error(error);
      alert("판매 등록 실패");
    }
  }

  return (
    <form onSubmit={handleSubmit}>
      <select
        name="customerId"
        value={form.customerId}
        onChange={handleChange}
        required
      >
        <option value="">고객 선택</option>
        <option value="1">홍길동</option>
        <option value="2">김민수</option>
      </select>

      <select
        name="carId"
        value={form.carId}
        onChange={handleChange}
        required
      >
        <option value="">자동차 선택</option>
        <option value="10">아반떼</option>
        <option value="11">쏘나타</option>
      </select>

      <input
        type="date"
        name="saleDate"
        value={form.saleDate}
        onChange={handleChange}
        required
      />

      <input
        type="number"
        name="salePrice"
        value={form.salePrice}
        onChange={handleChange}
        min="0"
        placeholder="판매금액"
      />

      <button type="submit">판매 등록</button>
    </form>
  );
}

export default SaleCreatePage;

const ACCESS_TOKEN_KEY = "accessToken";

export function saveToken(token) {
  localStorage.setItem(ACCESS_TOKEN_KEY, token);
}

export function getToken() {
  return localStorage.getItem(ACCESS_TOKEN_KEY);
}

export function removeToken() {
  localStorage.removeItem(ACCESS_TOKEN_KEY);
}

apiClient.interceptors.request.use((config) => {

자주 하는 실수 카드

1. 모든 컬럼에 인덱스 만들기

인덱스는 조회를 빠르게 할 수 있지만,
INSERT, UPDATE, DELETE 때 갱신 비용이 생깁니다.
조회 패턴을 보고 필요한 곳에만 만들어야 합니다.

2. VARCHAR2 크기 과도 설정

모든 문자열을 VARCHAR2(4000)으로 잡으면
입력 기준이 흐려지고,
DTO와 React 검증 기준도 잡기 어려워집니다.

3. 날짜를 문자열로 저장

sale_date를 VARCHAR2로 저장하면
날짜 비교, 기간 검색, 정렬이 어려워집니다.
날짜는 DATE 또는 TIMESTAMP로 저장합니다.

4. DB 제약 조건을 생략

React와 Spring에서 검증하더라도
DB 제약 조건은 마지막 방어선입니다.
NOT NULL, FK, CHECK를 적절히 사용해야 합니다.

5. FK 컬럼 인덱스 미고려

FK 컬럼은 JOIN과 조건 검색에 자주 사용됩니다.
데이터가 많아질 테이블이라면 인덱스 후보로 검토해야 합니다.

6. 역정규화 성급 적용

조회가 편하다는 이유로
처음부터 customer_name을 SALE에 중복 저장하면
데이터 불일치 위험이 커집니다.

7. Sequence와 Entity 설정 불일치

DB 시퀀스 이름과 @SequenceGenerator의
sequenceName이 다르면
JPA 저장 시 오류가 발생할 수 있습니다.

8. Trigger 로직 남용

Trigger는 자동 실행되어 편리하지만
로직이 DB 안에 숨어 디버깅과 유지보수가 어려울 수 있습니다.

08

면접 포인트 카드

Q1. 물리 모델링이란 무엇인가요?

물리 모델링은 논리적으로 설계한 테이블 구조를 실제 DBMS에서 구현 가능한 형태로 구체화하는 과정입니다. 데이터 타입, 컬럼 크기, 제약 조건, 인덱스, 시퀀스, View, Procedure, Trigger 등을 정합니다.

Q2. 논리 모델링과 물리 모델링의 차이는 무엇인가요?

논리 모델링은 엔터티, 속성, 관계 같은 데이터 구조의 의미를 정리하는 단계이고, 물리 모델링은 그 구조를 Oracle 같은 실제 DBMS에서 어떤 타입, 크기, 제약 조건, 인덱스로 만들지 결정하는 단계입니다.

Q3. VARCHAR2 크기는 어떻게 정하나요?

실제 업무에서 입력될 최대 길이, 화면 입력 제한, DTO 검증 기준, 검색과 인덱스 여부, 향후 확장 가능성을 고려해 정합니다. 무조건 크게 잡는 것이 아니라 의미에 맞는 크기를 정해야 합니다.

Q4. NUMBER, DATE, TIMESTAMP는 언제 사용하나요?

NUMBER는 ID, 금액, 수량, 정렬 순서 같은 숫자 데이터에 사용합니다. DATE는 업무 날짜에 사용하고, TIMESTAMP는 생성일시와 수정일시처럼 시간까지 명확히 저장해야 하는 값에 사용합니다.

Q5. 인덱스란 무엇인가요?

인덱스는 데이터를 빠르게 찾기 위한 DB의 찾아보기 구조입니다. WHERE, JOIN, ORDER BY, 날짜 범위 검색에 자주 사용되는 컬럼에 만들면 조회 성능을 높일 수 있습니다.

Q6. 인덱스의 장단점은 무엇인가요?

장점은 조회 성능을 높일 수 있다는 점입니다. 단점은 저장 공간을 더 사용하고 INSERT, UPDATE, DELETE 시 인덱스도 함께 갱신되어 쓰기 성능이 느려질 수 있다는 점입니다.

Q7. 제약 조건은 왜 필요한가요?

제약 조건은 잘못된 데이터가 DB에 저장되는 것을 막기 위한 규칙입니다.
PK는 행을 구분하고, FK는 참조 무결성을 지키며, NOT NULL은 필수값을 보장하고, UNIQUE는 중복을 막고, CHECK는 값의 범위를 제한합니다.

Q8. 정규화와 역정규화의 차이는 무엇인가요?

정규화는 중복을 줄이고 데이터 무결성을 높이기 위해 테이블을 의미별로 분리하는 과정입니다. 역정규화는 조회 성능이나 업무 편의성을 위해 일부 중복을 허용하는 설계입니다.

Q9. View는 언제 사용하나요?

View는 복잡한 SELECT나 JOIN 결과를 가상의 테이블처럼 재사용하고 싶을 때 사용합니다. 판매 목록처럼 여러 테이블을 JOIN해서 화면에 보여줄 데이터 구조를 만들 때 사용할 수 있습니다.

Q10. Stored Procedure와 Trigger는 언제 사용하나요?

Stored Procedure는 월마감, 대량 배치, 통계 생성처럼 DB 내부 처리에 사용할 수 있습니다.
Trigger는 INSERT, UPDATE, DELETE 이벤트가 발생했을 때 자동으로 감사 로그나 이력을 기록할 때 사용할 수 있습니다.

Q11. Spring Boot와 물리 모델링은 어떻게 연결되나요?

DB 컬럼 타입과 제약 조건은 Entity 타입과 @Column 설정, DTO Validation에 연결됩니다.
예를 들어 NAME VARCHAR2(50) NOT NULL은 String name, @Column(length = 50, nullable = false), @NotBlank, @Size(max = 50)로 연결됩니다.

Q12. React와 물리 모델링은 어떻게 연결되나요?

React는 DB를 직접 다루지 않지만 DB 컬럼 크기와 필수값 여부가 Form 입력 제한에 반영됩니다. VARCHAR2(50) NOT NULL 컬럼은 maxLength와 required로 연결할 수 있습니다.

학습 메모

오늘의 암기 포인트

  • 물리 모델링은 논리 모델을 실제 DBMS에서 구현 가능한 구조로 구체화하는 단계입니다.
  • Oracle 기준 문자열은 VARCHAR2, 숫자는 NUMBER, 업무 날짜는 DATE, 생성/수정 시각은 TIMESTAMP를 주로 사용합니다.
  • 컬럼 크기는 DB, DTO Validation, React Form 입력 제한을 함께 고려해서 정합니다.
  • PK, FK, NOT NULL, UNIQUE, CHECK, DEFAULT는 데이터 무결성을 지키는 제약 조건입니다.
  • 인덱스는 조회 성능을 높일 수 있지만 쓰기 성능과 저장 공간 비용이 생깁니다.
  • 역정규화는 조회 성능을 위해 일부 중복을 허용하는 설계이지만 데이터 불일치 위험을 관리해야 합니다.
  • View, Procedure, Trigger는 필요한 경우에 사용하되 남용하면 유지보수가 어려워질 수 있습니다.

헷갈리기 쉬운 개념

논리 모델 vs 물리 모델

논리 모델은 데이터 구조의 의미를 정하고,
물리 모델은 실제 DBMS 구현 방식을 정합니다.

DATE vs TIMESTAMP

DATE는 업무 날짜에 주로 사용하고,
TIMESTAMP는 생성일시와 수정일시처럼
시각이 중요한 값에 사용합니다.

인덱스 vs 제약 조건

인덱스는 조회 성능을 위한 구조이고,
제약 조건은 데이터 무결성을 지키는 규칙입니다.

정규화 vs 역정규화

정규화는 중복을 줄이고,
역정규화는 성능을 위해 일부 중복을 허용합니다.

View vs Table

View는 SELECT 결과를 가상의 테이블처럼 보여주는 객체이며,
일반 View는 데이터를 별도로 저장하지 않습니다.

Trigger vs Service Logic

Trigger는 DB 이벤트에 자동 반응하고,
Service는 애플리케이션 계층에서
명시적으로 실행되는 로직입니다.
