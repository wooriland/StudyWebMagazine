01

프로젝트 DB 설계 실습을 공부한다는 것

지금까지는 데이터베이스 모델링의 개별 개념을 배웠습니다.
먼저 모델링이 무엇인지 배웠고, 그다음 엔터티, 속성, 식별자, 관계를 정리했습니다.
이후 ERD를 테이블 구조로 바꾸는 Mapping Rule과 정규화, 실제 Oracle DBMS에 맞춰 구현하는 물리 모델링과 인덱스까지 공부했습니다.

03

그림처럼 이해하기

이번 실습은 고객과 자동차를 연결하는 업무 흐름을 데이터 구조로 바꾸는 과정입니다.
처음에는 “고객이 자동차를 구매한다”라는 단순 문장처럼 보이지만, 실제로는 고객 정보, 자동차 정보, 판매 사건, 폐차 정보가 서로 연결된 구조입니다.

요구사항

고객이 자동차를 산다

현실 업무 문장입니다.
아직 테이블도, 컬럼도, API도 없습니다.

엔터티

고객 / 자동차 / 판매 / 폐차

관리해야 할 대상과 사건을 나누어
독립된 데이터 상자로 만듭니다.

관계

ID로 연결하기

SALE은 customer_id와 car_id로

화면

선택하고 등록하기

React 판매 등록 화면에서는
고객과 자동차를 선택하고
판매일과 금액을 입력합니다.

판매는 관계인가, 엔터티인가?

처음에는 고객과 자동차 사이의 관계처럼 보입니다.
하지만 판매일, 판매금액, 계약번호, 판매상태 같은 속성을 가질 수 있으므로
별도 엔터티로 분리하는 것이 좋습니다. 

CUSTOMER 1 : N SALE
CAR      1 : N SALE

폐차 정보는 자동차 테이블에 넣을까?

모든 자동차가 폐차되는 것은 아니므로
CAR 테이블에 disposal_date를 넣으면
NULL이 많아질 수 있습니다. 폐차 정보는 선택 관계로 보고 DISPOSAL 테이블로 분리합니다. 

CAR 1 : 0..1 DISPOSAL

프로젝트 설계 판단 기준

어떤 정보가 독립적으로 여러 건 생기고, 자기 속성을 가지며, 다른 데이터와 관계를 맺는다면 엔터티 후보입니다.
반대로 어떤 대상을 설명하는 단순 값이라면 속성 후보입니다. 이 기준으로 고객, 자동차, 판매, 폐차를 나누면 테이블 책임이 명확해집니다.

1) Oracle Table 설계

CREATE TABLE customer (
    customer_id NUMBER PRIMARY KEY,
    name VARCHAR2(50) NOT NULL,
    resident_number VARCHAR2(20),
    phone VARCHAR2(30),
    address VARCHAR2(200),
    company_name VARCHAR2(100),
    job_title VARCHAR2(50),
    created_at TIMESTAMP DEFAULT SYSTIMESTAMP,
    updated_at TIMESTAMP
);

CREATE TABLE car (
    car_id NUMBER PRIMARY KEY,
    car_type VARCHAR2(50),
    car_name VARCHAR2(100) NOT NULL,
    model_year NUMBER(4),
    created_at TIMESTAMP DEFAULT SYSTIMESTAMP,
    updated_at TIMESTAMP
);

CREATE TABLE sale (
    sale_id NUMBER PRIMARY KEY,
    customer_id NUMBER NOT NULL,
    car_id NUMBER NOT NULL,
    sale_date DATE NOT NULL,
    sale_price NUMBER,
    created_at TIMESTAMP DEFAULT SYSTIMESTAMP,

    CONSTRAINT fk_sale_customer
        FOREIGN KEY (customer_id)
        REFERENCES customer(customer_id),

    CONSTRAINT fk_sale_car
        FOREIGN KEY (car_id)
        REFERENCES car(car_id),

    CONSTRAINT chk_sale_price
        CHECK (sale_price >= 0)
);

CREATE TABLE disposal (
    disposal_id NUMBER PRIMARY KEY,
    car_id NUMBER NOT NULL,
    disposal_date DATE NOT NULL,
    disposal_reason VARCHAR2(300),
    created_at TIMESTAMP DEFAULT SYSTIMESTAMP,

    CONSTRAINT fk_disposal_car
        FOREIGN KEY (car_id)
        REFERENCES car(car_id),

    CONSTRAINT uk_disposal_car
        UNIQUE (car_id)
);

import jakarta.persistence.*;
import java.time.LocalDateTime;

@Entity
@Table(name = "CUSTOMER")
public class CustomerEntity {

    @Id
    @GeneratedValue(strategy = GenerationType.SEQUENCE, generator = "seq_customer")
    @SequenceGenerator(
            name = "seq_customer",
            sequenceName = "SEQ_CUSTOMER",
            allocationSize = 1
    )
    @Column(name = "CUSTOMER_ID")
    private Long customerId;

    @Column(name = "NAME", nullable = false, length = 50)
    private String name;

    @Column(name = "RESIDENT_NUMBER", length = 20)
    private String residentNumber;

    @Column(name = "PHONE", length = 30)
    private String phone;

    @Column(name = "ADDRESS", length = 200)
    private String address;

    @Column(name = "COMPANY_NAME", length = 100)
    private String companyName;

    @Column(name = "JOB_TITLE", length = 50)
    private String jobTitle;

    @Column(name = "CREATED_AT")
    private LocalDateTime createdAt;

    protected CustomerEntity() {
    }

    public CustomerEntity(String name, String phone, String address) {
        this.name = name;
        this.phone = phone;
        this.address = address;
        this.createdAt = LocalDateTime.now();
    }
}

6) Service / Controller

import org.springframework.stereotype.Service;

@Service
public class SaleService {

    private final SaleRepository saleRepository;
    private final CustomerRepository customerRepository;
    private final CarRepository carRepository;

    public SaleService(
            SaleRepository saleRepository,
            CustomerRepository customerRepository,
            CarRepository carRepository
    ) {
        this.saleRepository = saleRepository;
        this.customerRepository = customerRepository;
        this.carRepository = carRepository;
    }

    public SaleEntity createSale(SaleCreateRequest request) {
        if (!customerRepository.existsById(request.getCustomerId())) {
            throw new IllegalArgumentException("존재하지 않는 고객입니다.");
        }

        if (!carRepository.existsById(request.getCarId())) {
            throw new IllegalArgumentException("존재하지 않는 자동차입니다.");
        }

        SaleEntity sale = new SaleEntity(
                request.getCustomerId(),
                request.getCarId(),
                request.getSaleDate(),
                request.getSalePrice()
        );

        return saleRepository.save(sale);
    }
}
import org.springframework.stereotype.Service;

@Service
public class DisposalService {

    private final DisposalRepository disposalRepository;
    private final CarRepository carRepository;

    public DisposalService(
            DisposalRepository disposalRepository,
            CarRepository carRepository
    ) {
        this.disposalRepository = disposalRepository;
        this.carRepository = carRepository;
    }

    public DisposalEntity createDisposal(DisposalCreateRequest request) {
        if (!carRepository.existsById(request.getCarId())) {
            throw new IllegalArgumentException("존재하지 않는 자동차입니다.");
        }

        if (disposalRepository.existsByCarId(request.getCarId())) {
            throw new IllegalArgumentException("이미 폐차 정보가 등록된 자동차입니다.");
        }

        DisposalEntity disposal = new DisposalEntity(
                request.getCarId(),
                request.getDisposalDate(),
                request.getDisposalReason()
        );

        return disposalRepository.save(disposal);
    }
}

import axios from "axios";

const apiClient = axios.create({
  baseURL: "http://localhost:8080/api",
  headers: {
    "Content-Type": "application/json",
  },
});

export async function createSale(form) {
  const response = await apiClient.post("/sales", {
    customerId: Number(form.customerId),
    carId: Number(form.carId),
    saleDate: form.saleDate,
    salePrice: Number(form.salePrice),
  });

  return response.data;
}

export async function createDisposal(form) {
  const response = await apiClient.post("/disposals", {
    carId: Number(form.carId),
    disposalDate: form.disposalDate,
    disposalReason: form.disposalReason,
  });

  return response.data;

자주 하는 실수 카드

1. 판매 정보를 고객 테이블에 넣기

고객이 자동차를 여러 대 구매하면 고객 정보가 반복됩니다.
판매는 고객의 속성이 아니라 별도 사건으로 봐야 합니다.

2. 판매 정보를 자동차 테이블에 넣기

자동차 기본 정보와 판매 사건 정보가 섞이면
판매 이력, 판매 통계, 계약 상태 관리가 어려워집니다.

3. 폐차 정보를 CAR에 무조건 넣기

모든 자동차가 폐차되는 것은 아니므로
폐차 관련 컬럼이 대부분 NULL이 될 수 있습니다.
DISPOSAL 테이블로 분리하는 것이 명확합니다.

4. 주민번호를 PK로 사용하기

주민번호는 민감한 개인정보입니다.
FK로 여러 테이블에 전파되면 보안 위험이 커지므로
customer_id 같은 대리 식별자를 사용합니다.

5. FK 저장 순서 무시하기

SALE은 CUSTOMER와 CAR를 참조합니다.
따라서 고객과 자동차를 먼저 저장한 뒤 판매를 저장해야 합니다.

6. DTO 필드명 불일치

React는 customerId로 보내는데
Spring DTO가 customerNo로 받으면
값이 들어오지 않을 수 있습니다.

7. JSON 구조를 DTO와 다르게 보내기

DTO가 ID 중심이면 React도 customerId, carId처럼
ID 중심으로 보내야 합니다.
객체 중첩 구조를 보내면 매핑 오류가 날 수 있습니다.

8. 화면 구조만 보고 테이블 만들기

화면 하나가 테이블 하나는 아닙니다.
데이터의 의미와 관계를 기준으로 테이블을 나누어야 합니다.

면접 포인트 카드

Q1. 요구사항에서 테이블을 도출하는 방법을 설명해 주세요.

먼저 요구사항 문장에서 명사를 추출합니다.
그중 독립적으로 관리해야 하고 여러 건의 데이터가 생기며 자기 속성과 관계를 가지는 대상을 엔터티 후보로 봅니다.
이후 속성을 정리하고 식별자를 선택한 뒤, 엔터티 간 관계를 설계합니다.
마지막으로 엔터티는 테이블로, 속성은 컬럼으로, 관계는 외래키로 변환합니다.

Q2. 자동차 판매 관리 요구사항의 핵심 엔터티는 무엇인가요?

핵심 엔터티는 CUSTOMER, CAR, SALE, DISPOSAL입니다. 고객은 구매자 정보, 자동차는 판매 대상 정보, 판매는 고객과 자동차 사이의 구매 사건, 폐차는 자동차가 폐차되었을 때의 정보를 관리합니다.

Q3. 판매는 관계인가요, 엔터티인가요?

판매는 처음에는 고객과 자동차 사이의 관계처럼 보입니다. 하지만 판매일, 판매금액, 계약번호, 판매상태 같은 자기 속성을 가질 수 있으므로 별도 엔터티로 분리하는 것이 좋습니다.

Q4. 폐차 정보는 자동차 테이블에 넣어야 하나요?

모든 자동차가 폐차되는 것은 아니므로 자동차 테이블에 넣으면 NULL이 많아질 수 있습니다. 폐차 정보는 자동차와 선택적 1:1 관계로 보고 DISPOSAL 테이블로 분리하는 것이 좋습니다.

Q5. CUSTOMER와 SALE의 관계는 어떻게 설계하나요?

고객 1명은 여러 판매 기록을 가질 수 있고, 판매 기록 1건은 반드시 고객 1명과 연결됩니다. 따라서 CUSTOMER 1:N SALE 관계이며 FK는 SALE.customer_id에 둡니다.

Q6. CAR와 DISPOSAL의 관계는 어떻게 설계하나요?

자동차 1대는 폐차 정보가 없을 수도 있고 1건 있을 수도 있습니다. 폐차 정보는 반드시 자동차 1대와 연결되어야 하므로 CAR 1:0..1 DISPOSAL 관계로 설계합니다.

Q7. 주민번호를 PK로 사용하면 안 되는 이유는 무엇인가요?

주민번호는 민감한 개인정보입니다. PK로 사용하면 여러 테이블에 FK로 전파되어 노출 위험이 커집니다. 따라서 customer_id 같은 대리 식별자를 PK로 사용하는 것이 좋습니다.

Q8. FK 순서 오류는 왜 발생하나요?

자식 테이블이 부모 테이블을 참조하는데 부모 데이터가 아직 없을 때 발생합니다. SALE은 CUSTOMER와 CAR를 참조하므로 판매를 등록하기 전에 고객과 자동차가 먼저 저장되어야 합니다.

Q9. DTO 필드명 불일치는 왜 문제가 되나요?

React가 customerId로 보내는데 Spring DTO가 customerNo로 받으면 값이 매핑되지 않을 수 있습니다. 그러면 customerId가 NULL이 되어 NOT NULL 오류나 FK 오류로 이어질 수 있습니다.

Q10. 좋은 프로젝트 DB 설계의 기준은 무엇인가요?

요구사항을 반영하면서 각 테이블의 책임이 명확해야 합니다.
중복 데이터와 NULL 남발을 줄이고, PK/FK 관계가 분명해야 하며, Spring Boot Entity, DTO, API, React Form으로 자연스럽게 이어져야 합니다.

헷갈리기 쉬운 개념

명사 vs 엔터티

모든 명사가 엔터티는 아닙니다.
독립적으로 관리해야 하고 여러 건이 생기는지를 확인해야 합니다.

판매 관계 vs 판매 엔터티

판매가 판매일, 판매금액 같은 속성을 가지면
별도 엔터티로 분리하는 것이 좋습니다.

폐차 속성 vs 폐차 테이블

폐차 정보가 항상 존재하지 않으면
CAR에 넣기보다 DISPOSAL로 분리하는 것이 명확합니다.

PK vs 개인정보

주민번호처럼 민감한 값은 PK로 쓰기보다
대리 식별자를 사용하는 것이 안전합니다.

DB 구조 vs JSON 구조

DB가 FK를 가진다고 해서
React JSON을 항상 객체 중첩으로 보내는 것은 아닙니다.
DTO 구조에 맞춰 보내야 합니다.

화면 vs 테이블

화면 하나가 테이블 하나는 아닙니다.
테이블은 화면이 아니라
데이터의 의미와 관계를 기준으로 설계합니다.

데이터베이스 모델링 전체 마무리

이번 5개 단원은 모델링 기초, ERD와 관계, 정규화와 Mapping Rule, 물리 모델링과 인덱스, 프로젝트 DB 설계 실습 순서로 이어졌습니다.

요구사항 분석 → 엔터티 도출 → 관계 설계 → ERD → 정규화 → 물리 설계 →
Table → Entity → DTO → API → React Form
