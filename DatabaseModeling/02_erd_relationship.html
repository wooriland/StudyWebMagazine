Database Modeling - Chapter 02

ERD와 엔터티 관계

ERD와 엔터티 관계는 데이터베이스 모델링의 핵심 재료를 배우는 단원입니다.
현실 업무 문장에서 엔터티(Entity)를 찾고, 엔터티가 가지는 속성(Attribute)을 정리하며,
각 데이터를 구분할 식별자(Identifier)를 선택하고, 엔터티 사이의 관계(Relationship)를 연결합니다.
이 과정을 통해 나중에 테이블, 컬럼, PK, FK, DTO, API, React Form 구조가 자연스럽게 결정됩니다.

ERD Entity Attribute Identifier Candidate Identifier Primary Identifier Alternate Identifier Foreign Identifier Surrogate Identifier Relationship Cardinality Optionality 1:1 1:N N:N PK FK JPA Entity

현실 업무 문장에서

후보 식별자, 주 식별자,

학습 목표 3

1:1, 1:N, N:N 관계와 선택성을 이해하고,
ERD를 Table, Entity, DTO,
React Form으로 연결할 수 있습니다.

01

ERD와 엔터티 관계를 공부한다는 것

이전 단원에서는 데이터베이스 모델링이 현실 업무를 데이터 구조로 번역하는 과정이라고 배웠습니다.
이번 단원에서는 그 번역에 필요한 핵심 재료를 배웁니다. 바로 엔터티, 속성, 식별자, 관계입니다.
예를 들어 “고객이 자동차를 구매한다”라는 문장이 있을 때, 개발자는 이 문장을 그대로 테이블로 만들지 않습니다.
먼저 고객, 자동차, 판매라는 대상을 찾고, 고객과 판매, 자동차와 판매 사이의 관계를 정리합니다.

Entity

업무에서 관리할 대상

고객, 자동차, 판매, 폐차처럼
시스템이 데이터로 관리해야 하는
독립적인 대상입니다.
나중에 테이블 후보가 됩니다.

Attribute

대상의 세부 정보

고객명, 연락처, 주소, 차명, 연식, 판매일처럼
엔터티를 설명하는 값입니다.
나중에 컬럼 후보가 됩니다.

Identifier

인스턴스를 구분하는 기준

customer_id, car_id, sale_id처럼
각 데이터 한 건을 유일하게 구분하는 값입니다.
나중에 PK 후보가 됩니다.

Relationship

대상 사이의 업무 연결

고객은 판매 기록을 가지고,
판매는 자동차와 연결됩니다.
관계는 나중에 FK 구조로 구현되는 경우가
많습니다.

핵심 문장

ERD는 엔터티 → 속성 → 식별자 → 관계 → 차수성 → 선택성 을 정리해서 현실 업무를 데이터베이스 설계도로 표현한 것입니다.

02

명사 찾기에서 ERD까지의 흐름도

ERD는 한 번에 완성되는 그림이 아닙니다. 먼저 요구사항 문장에서 명사를 찾고, 그중에서 엔터티 후보를 골라낸 뒤, 속성, 식별자, 관계를 순서대로 정리해야 합니다.

1 현실 업무 문장

자동차 판매 사원은 고객 정보를 관리하고,
고객이 구매한 자동차와 폐차 정보를 관리합니다.

→
2 명사 찾기

고객, 자동차, 판매, 폐차 정보, 연락처,
주소, 차명, 연식 같은 명사를 찾습니다.

→
3 엔터티 후보 정리

독립적으로 관리해야 하는 고객, 자동차,
판매, 폐차를 엔터티 후보로 정리합니다.

4 속성 정리

고객명, 연락처, 주소, 차종, 차명,
연식, 판매일, 판매금액을 정리합니다.

→
5 식별자 선택

customer_id, car_id, sale_id,
disposal_id처럼 각 행을 구분할 기준을 정합니다.

→
6 관계 연결

CUSTOMER 1:N SALE,
CAR 1:N SALE,
CAR 1:0..1 DISPOSAL 관계를 연결합니다.

엔터티

테이블 후보

업무에서 독립적으로
관리해야 하는 대상입니다.
여러 인스턴스가 생기고
자기만의 속성을 가집니다.

속성

컬럼 후보

엔터티를 설명하는 세부 정보입니다.
데이터 타입, 크기, NULL 허용 여부 같은
도메인과 연결됩니다.

식별자

PK 후보

엔터티의 인스턴스를
유일하게 구분하는 기준입니다.
주 식별자는 나중에 기본키가 됩니다.

관계

FK 후보

두 엔터티 사이의 업무적 연결입니다.
부모의 식별자가 자식 쪽으로 전달되면
외래키가 됩니다.

자동차 판매 관리 텍스트 ERD

CUSTOMER
  PK customer_id
     name
     phone
     address
     company_name
     job_title

CAR
  PK car_id
     car_type
     car_name
     model_year

SALE
  PK sale_id
  FK customer_id
  FK car_id
     sale_date
     sale_price

DISPOSAL
  PK disposal_id
  FK car_id
     disposal_date
     disposal_reason

관계:
  CUSTOMER 1 ───< SALE >─── 1 CAR
  CAR      1 ─── 0..1 DISPOSAL
03

그림처럼 이해하기

ERD는 업무 데이터를 보관할 서류함 구조를 설계하는 것처럼 이해하면 쉽습니다. 고객 서류함, 자동차 서류함, 판매 서류함을 따로 만들고, 어떤 서류가 어떤 서류와 연결되는지 줄을 긋는 과정입니다.

명사 찾기

후보 명단 만들기

요구사항 문장에서
고객, 자동차, 판매, 폐차처럼
중요한 명사를 먼저 추려냅니다.

엔터티

독립 서류함

여러 건을 따로 관리해야 하는 대상은
별도 서류함, 즉 별도 테이블 후보로 봅니다.

속성

서류 안의 항목

고객 서류 안의 이름, 연락처, 주소처럼
엔터티를 설명하는 세부 항목입니다.

관계

서류함 사이의 연결선

판매 서류는
고객 서류와 자동차 서류를 참조합니다.
이 연결선이 나중에 FK로 구현됩니다.

N:N 관계는 중간 엔터티로 풀어야 합니다

처음 보이는 관계 
CUSTOMER N : N CAR

고객은 여러 자동차를 구매할 수 있고,
자동차도 판매 이력 관점에서
여러 거래와 연결될 수 있습니다.
좋은 ERD 구조 
CUSTOMER 1 : N SALE
CAR      1 : N SALE

SALE은 고객과 자동차 사이의
중간 엔터티입니다.

속성과 엔터티를 구분하는 기준

고객명은 고객을 설명하는 값이므로 속성입니다. 하지만 판매는 고객과 자동차 사이에서 발생하는 사건이고, 판매일, 판매금액 같은 자기 속성을 가지므로 엔터티로 보는 것이 자연스럽습니다.

2) Oracle Table 설계

CREATE TABLE customer (
    customer_id NUMBER PRIMARY KEY,
    name VARCHAR2(50) NOT NULL,
    resident_number VARCHAR2(20),
    phone VARCHAR2(30),
    address VARCHAR2(200),
    company_name VARCHAR2(100),
    job_title VARCHAR2(50)
);

CREATE TABLE car (
    car_id NUMBER PRIMARY KEY,
    car_type VARCHAR2(50),
    car_name VARCHAR2(100) NOT NULL,
    model_year NUMBER(4)
);

CREATE TABLE sale (
    sale_id NUMBER PRIMARY KEY,
    customer_id NUMBER NOT NULL,
    car_id NUMBER NOT NULL,
    sale_date DATE NOT NULL,
    sale_price NUMBER,

    CONSTRAINT fk_sale_customer
        FOREIGN KEY (customer_id)
        REFERENCES customer(customer_id),

    CONSTRAINT fk_sale_car
        FOREIGN KEY (car_id)
        REFERENCES car(car_id)
);

CREATE TABLE disposal (
    disposal_id NUMBER PRIMARY KEY,
    car_id NUMBER NOT NULL,
    disposal_date DATE NOT NULL,
    disposal_reason VARCHAR2(300),

    CONSTRAINT fk_disposal_car
        FOREIGN KEY (car_id)
        REFERENCES car(car_id),

    CONSTRAINT uk_disposal_car
        UNIQUE (car_id)
);

        return saleDate;

import org.springframework.stereotype.Service;
import java.util.List;

@Service
public class SaleService {

    private final SaleRepository saleRepository;
    private final CustomerRepository customerRepository;
    private final CarRepository carRepository;

    public SaleService(
            SaleRepository saleRepository,
            CustomerRepository customerRepository,
            CarRepository carRepository
    ) {
        this.saleRepository = saleRepository;
        this.customerRepository = customerRepository;
        this.carRepository = carRepository;
    }

    public SaleResponse createSale(SaleCreateRequest request) {
        if (!customerRepository.existsById(request.getCustomerId())) {
            throw new IllegalArgumentException("존재하지 않는 고객입니다.");
        }

        if (!carRepository.existsById(request.getCarId())) {
            throw new IllegalArgumentException("존재하지 않는 자동차입니다.");
        }

        SaleEntity sale = new SaleEntity(
                request.getCustomerId(),
                request.getCarId(),
                request.getSaleDate(),
                request.getSalePrice()
        );

        SaleEntity savedSale = saleRepository.save(sale);

        return new SaleResponse(savedSale);
    }

    public List<SaleResponse> getSalesByCustomer(Long customerId) {
        return saleRepository.findByCustomerId(customerId)
                .stream()
                .map(SaleResponse::new)
                .toList();
    }
}
import org.springframework.web.bind.annotation.*;
import java.util.List;

@RestController
@RequestMapping("/api/sales")
public class SaleController {

    private final SaleService saleService;

    public SaleController(SaleService saleService) {
        this.saleService = saleService;
    }

    @PostMapping
    public SaleResponse createSale(@RequestBody SaleCreateRequest request) {
        return saleService.createSale(request);
    }

    @GetMapping("/customers/{customerId}")
    public List<SaleResponse> getSalesByCustomer(@PathVariable Long customerId) {
        return saleService.getSalesByCustomer(customerId);
    }
}

6) React Axios API 호출

import axios from "axios";

const apiClient = axios.create({
  baseURL: "http://localhost:8080/api",
  headers: {
    "Content-Type": "application/json",
  },
});

export async function createSale(form) {
  const requestBody = {
    customerId: Number(form.customerId),
    carId: Number(form.carId),
    saleDate: form.saleDate,
    salePrice: Number(form.salePrice),
  };

  const response = await apiClient.post("/sales", requestBody);
  return response.data;
}

export async function getSalesByCustomer(customerId) {
  const response = await apiClient.get(`/sales/customers/${customerId}`);
  return response.data;
}

7) React Form 입력값을 DTO로 보내기

import { useState } from "react";
import { createSale } from "../api/saleApi";

function SaleCreatePage() {
  const [form, setForm] = useState({
    customerId: "",
    carId: "",
    saleDate: "",
    salePrice: "",
  });

  function handleChange(event) {
    const { name, value } = event.target;

    setForm((prev) => ({
      ...prev,
      [name]: value,
    }));
  }

  async function handleSubmit(event) {
    event.preventDefault();

    try {
      const savedSale = await createSale(form);
      console.log("저장된 판매 기록:", savedSale);
      alert("판매 등록 성공");
    } catch (error) {
      console.error(error);
      alert("판매 등록 실패");
    }
  }

  return (
    <form onSubmit={handleSubmit}>
      <label>
        고객 ID
        <input
          name="customerId"
          value={form.customerId}
          onChange={handleChange}
          placeholder="고객 ID"
        />
      </label>

      <label>
        자동차 ID
        <input
          name="carId"
          value={form.carId}
          onChange={handleChange}
          placeholder="자동차 ID"
        />
      </label>

      <label>
        판매일
        <input
          name="saleDate"
          type="date"
          value={form.saleDate}
          onChange={handleChange}
        />
      </label>

      <label>
        판매금액
        <input
          name="salePrice"
          value={form.salePrice}
          onChange={handleChange}
          placeholder="판매금액"
        />
      </label>

      <button type="submit">판매 등록</button>
    </form>
  );
}

export default SaleCreatePage;

8) 토큰 저장과 인증 API 요청 연결

판매 등록 API를 로그인 사용자만 사용할 수 있게 만들면, React는 로그인 후 받은 토큰을 저장하고 요청마다 Authorization 헤더에 붙여야 합니다. 

const ACCESS_TOKEN_KEY = "accessToken";

export function saveToken(token) {
  localStorage.setItem(ACCESS_TOKEN_KEY, token);
}

export function getToken() {
  return localStorage.getItem(ACCESS_TOKEN_KEY);
}

export function removeToken() {
  localStorage.removeItem(ACCESS_TOKEN_KEY);
}

apiClient.interceptors.request.use((config) => {
  const token = getToken();

  if (token) {
    config.headers.Authorization = `Bearer ${token}`;
  }

  return config;
});
05

자주 하는 실수 카드

1. 속성을 엔터티로 만들기

고객명, 연락처, 주소는 보통 고객 엔터티의 속성입니다.
독립적으로 관리하지 않는 값을
무조건 테이블로 분리하면 구조가 복잡해집니다.

2. 엔터티를 속성으로 넣기

판매는 고객의 단순 속성이 아니라
고객과 자동차 사이에서 발생하는 사건입니다.
판매일, 판매금액을 가지므로
엔터티로 분리하는 것이 자연스럽습니다.

3. N:N 관계를 그대로 구현하기

CUSTOMER에 car1_id, car2_id처럼
반복 컬럼을 만들면 안 됩니다.
SALE 같은 중간 엔터티로 풀어야 합니다.

4. PK 없이 테이블 만들기

각 행을 구분할 식별자가 없으면
수정, 삭제, 조회, FK 연결이 어려워집니다.
대부분의 엔터티에는 PK가 필요합니다.

5. FK 위치를 반대로 잡기

1:N 관계에서는 보통 N쪽에 FK를 둡니다.
CUSTOMER 1:N SALE이면
FK는 SALE.customer_id에 들어갑니다.

6. 자연 식별자를 무조건 PK로 쓰기

주민번호, 이메일, 전화번호는
변경 가능성과 보안 문제가 있습니다.
실무에서는 customer_id 같은 대리 식별자를 자주 사용합니다.

7. 선택성을 고려하지 않기

모든 자동차가 폐차된 것은 아닙니다.
폐차 정보는 선택 관계이므로
별도 DISPOSAL 테이블로 관리하는 것이 자연스러울 수 있습니다.

8. DTO와 React 필드명 불일치

React는 customerId로 보내는데
DTO는 customerNo로 받으면 값이 매핑되지 않을 수 있습니다.
요청 body와 DTO 필드명을 맞춰야 합니다.

  5. JPA Entity의 ID 생성 전략과 DB Sequence가 일치하는지 확인합니다.

08

면접 포인트 카드

Q1. 엔터티란 무엇인가요?

엔터티는 업무에서 관리해야 하는 독립적인 대상입니다. 사람, 장소, 사물, 사건, 개념이 될 수 있으며 데이터베이스에서는 보통 테이블 후보가 됩니다.

Q2. 속성이란 무엇인가요?

속성은 엔터티가 가지는 세부 정보입니다. 예를 들어 고객 엔터티의 속성은 고객 ID, 이름, 연락처, 주소 등이 될 수 있습니다.

Q3. 엔터티와 속성은 어떻게 구분하나요?

독립적으로 관리되고 여러 건의 데이터가 생기며 자기 속성과 관계를 가지면 엔터티입니다. 반면 어떤 엔터티를 설명하는 단일 값이면 속성입니다.

Q4. PK와 FK의 차이는 무엇인가요?

PK는 자기 테이블의 각 행을 유일하게 구분하는 기본키입니다. FK는 다른 테이블의 PK를 참조해서 테이블 사이의 관계를 연결하는 외래키입니다.

Q5. 후보 식별자와 주 식별자의 차이는 무엇인가요?

후보 식별자는 인스턴스를 유일하게 구분할 수 있는 모든 후보입니다. 주 식별자는 그중 대표로 선택한 식별자이며 물리 테이블에서는 보통 기본키가 됩니다.

Q6. 외부 식별자란 무엇인가요?

외부 식별자는 부모 엔터티의 주 식별자가 관계를 통해 자식 엔터티로 전달된 식별자입니다. 물리 테이블에서는 Foreign Key로 구현됩니다.

Q7. N:N 관계는 어떻게 해결하나요?

N:N 관계는 중간 엔터티를 만들어 두 개의 1:N 관계로 해결합니다. 예를 들어 고객과 자동차 사이에는 SALE이라는 중간 엔터티를 둘 수 있습니다.

Q8. 선택성이란 무엇인가요?

선택성은 관계가 필수인지 선택인지를 나타냅니다. 예를 들어 판매 기록은 반드시 고객과 연결되어야 하지만, 고객은 아직 판매 기록이 없을 수도 있습니다.

Q9. ERD가 Spring Boot 개발에 어떻게 연결되나요?

ERD의 엔터티는 Entity가 되고, 속성은 필드가 되며, 식별자는 @Id로 매핑되고, 관계는 FK 또는 JPA 관계 매핑으로 연결됩니다.

Q10. ERD가 React 개발과도 관련 있나요?

관련 있습니다. ERD에서 정한 관계는 DTO와 API 요청 body에 반영됩니다. 판매가 고객과 자동차를 참조한다면 React Form에서도 customerId와 carId 값을 보내야 합니다.

학습 메모

오늘의 암기 포인트

  • 엔터티는 업무에서 관리해야 하는 독립적인 대상이며 테이블 후보입니다.
  • 속성은 엔터티를 설명하는 세부 정보이며 컬럼 후보입니다.
  • 식별자는 인스턴스를 유일하게 구분하는 기준이며 PK 후보입니다.
  • 외부 식별자는 관계를 통해 자식 엔터티로 전달된 부모 식별자이며 FK 후보입니다.
  • 1:N 관계에서는 보통 N쪽 테이블에 FK가 들어갑니다.
  • N:N 관계는 중간 엔터티를 만들어 두 개의 1:N 관계로 풀어야 합니다.
  • 선택성은 관계가 반드시 필요한지, 없어도 되는지를 나타냅니다.
  • ERD는 Entity, Attribute, Identifier, Relationship을 표현한 DB 설계도입니다.

헷갈리기 쉬운 개념

엔터티 vs 속성

고객은 엔터티이고 고객명은 속성입니다.
독립적으로 관리해야 하는지가 핵심 기준입니다.

PK vs FK

PK는 자기 테이블의 행을 구분하고,
FK는 다른 테이블과의 관계를 연결합니다.

후보 식별자 vs 주 식별자

후보 식별자는 가능한 식별자 후보이고,
주 식별자는 그중 대표로 선택한 식별자입니다.

부 식별자 vs 대리 식별자

부 식별자는 선택되지 않은 후보 식별자이고,
대리 식별자는 시스템이 인위적으로 만든 식별자입니다.

1:N vs N:N

1:N은 한쪽이 여러 건을 가지는 관계이고,
N:N은 중간 엔터티로 풀어야 하는 관계입니다.

필수 관계 vs 선택 관계

판매는 반드시 고객이 필요하지만,
고객은 아직 판매 기록이 없을 수도 있습니다.
