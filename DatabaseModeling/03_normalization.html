Database Modeling - Chapter 03

정규화와 Mapping Rule

정규화와 Mapping Rule은 ERD를 실제 테이블 구조로 바꾸는 단원입니다.
ERD에서 찾은 엔터티는 테이블로, 속성은 컬럼으로, 주식별자는 기본키로, 관계는 외래키로 변환합니다.
그리고 정규화를 통해 반복 컬럼, 중복 데이터, 수정 이상, 삭제 이상, 삽입 이상을 줄이고
Spring Boot Entity, Repository, DTO, API, React Form이 안정적으로 이어질 수 있는 테이블 구조를 만듭니다.

Mapping Rule Normalization 1NF 2NF 3NF Entity → Table Attribute → Column Identifier → Primary Key Relationship → Foreign Key Data Duplication Insert Anomaly Update Anomaly Delete Anomaly Functional Dependency Composite Key DTO Repository React Form
학습 목표 1

ERD의 엔터티, 속성, 식별자, 관계가
실제 테이블, 컬럼, PK, FK로
어떻게 변환되는지 설명할 수 있습니다.

학습 목표 2

제1정규화, 제2정규화, 제3정규화가
각각 어떤 문제를 해결하는지
구분할 수 있습니다.

학습 목표 3

정규화된 테이블 구조가
Spring Boot Entity, Repository, DTO,
React Form 구조에 어떻게 연결되는지 이해할 수 있습니다.

01

정규화와 Mapping Rule을 공부한다는 것

앞 단원에서 우리는 현실 업무 문장에서 엔터티, 속성, 식별자, 관계를 찾는 법을 배웠습니다. 하지만 ERD는 아직 그림에 가깝습니다.
실제 데이터베이스에 저장하려면 ERD를 테이블, 컬럼, 기본키, 외래키 구조로 바꾸어야 합니다. 이 변환 규칙이 바로 Mapping Rule입니다.
그리고 단순히 테이블로 바꾸는 것만으로 끝나지 않습니다. 같은 고객 정보가 여러 행에 반복되거나, 판매 기록을 삭제했더니 고객 정보까지 사라지는 문제가 생기면 좋은 설계라고 할 수 없습니다.
이런 문제를 줄이기 위해 테이블을 의미별로 정리하는 과정이 정규화입니다.

Mapping Rule

ERD를 테이블로 번역

엔터티는 테이블로, 속성은 컬럼으로,
주식별자는 기본키로,
관계는 외래키로 변환합니다.

Normalization

데이터를 제자리로 배치

고객 정보는 CUSTOMER에,
자동차 정보는 CAR에,
판매 정보는 SALE에 위치하도록 정리합니다.

Anomaly

이상 현상 방지

삽입 이상, 수정 이상, 삭제 이상처럼
잘못된 테이블 구조에서 생기는
문제를 줄입니다.

Development

개발 구조 안정화

테이블 구조가 안정되어야
Entity, Repository, DTO,
API, React Form도 흔들리지 않습니다.

핵심 문장

Mapping Rule은 ERD → Table 변환 규칙이고, 정규화는 중복 제거 → 이상 현상 방지 → 올바른 테이블 배치 를 위한 구조 정리 과정입니다.

ERD를 테이블로 바꿀 때는 먼저 Mapping Rule을 적용하고, 그다음 정규화를 통해 반복 컬럼과 잘못된 의존 관계를 제거합니다.

03

그림처럼 이해하기

정규화는 한 장짜리 복잡한 엑셀 파일을 의미별 폴더로 나누는 것과 비슷합니다. 처음에는 모든 정보가 한 표에 있으면 편해 보이지만, 데이터가 많아질수록 중복, 불일치, 삭제 문제, 삽입 문제가 생깁니다.

복잡한 표

모든 정보를 한 장에 몰아넣기

고객명, 연락처, 자동차명, 판매일,
판매금액을 한 테이블에 모두 넣으면
중복이 빠르게 늘어납니다.

분리

의미별 서랍으로 나누기

고객 정보는 고객 서랍에,
자동차 정보는 자동차 서랍에,
판매 정보는 판매 서랍에 넣습니다.

연결

ID로 서로 연결하기

판매 테이블은
customer_id와 car_id를 사용해
고객과 자동차를 참조합니다.

조회

필요할 때 조합하기

화면에 고객명과 자동차명을
함께 보여줘야 할 때는
JOIN이나 응답 DTO로 조합합니다.

정규화는 무조건 테이블을 많이 나누는 작업이 아닙니다

    car_id NUMBER NOT NULL,

import java.time.LocalDate;

public class SaleCreateRequest {
    private Long customerId;
    private Long carId;
    private LocalDate saleDate;
    private Integer salePrice;

    public Long getCustomerId() {
        return customerId;
    }

    public Long getCarId() {
        return carId;

    public LocalDate getSaleDate() {
        return saleDate;
    }

    public Integer getSalePrice() {
        return salePrice;

import org.springframework.stereotype.Service;

@Service
public class SaleService {

    private final SaleRepository saleRepository;
    private final CustomerRepository customerRepository;
    private final CarRepository carRepository;

    public SaleService(
            SaleRepository saleRepository,
            CustomerRepository customerRepository,
            CarRepository carRepository
    ) {
        this.saleRepository = saleRepository;
        this.customerRepository = customerRepository;
        this.carRepository = carRepository;
    }

    public SaleEntity createSale(SaleCreateRequest request) {
        if (!customerRepository.existsById(request.getCustomerId())) {
            throw new IllegalArgumentException("존재하지 않는 고객입니다.");
        }

        if (!carRepository.existsById(request.getCarId())) {
            throw new IllegalArgumentException("존재하지 않는 자동차입니다.");
        }

        SaleEntity sale = new SaleEntity(
                request.getCustomerId(),
                request.getCarId(),
                request.getSaleDate(),
                request.getSalePrice()
        );

        return saleRepository.save(sale);
    }
}
import org.springframework.web.bind.annotation.*;

@RestController
@RequestMapping("/api/sales")
public class SaleController {

    private final SaleService saleService;

    public SaleController(SaleService saleService) {
        this.saleService = saleService;
    }

    @PostMapping
    public SaleEntity createSale(@RequestBody SaleCreateRequest request) {
        return saleService.createSale(request);
    }
}

8) React Form 입력값을 DTO로 보내기

import { useState } from "react";
import { createSale } from "../api/saleApi";

function SaleCreatePage() {
  const [form, setForm] = useState({
    customerId: "",
    carId: "",
    saleDate: "",
    salePrice: "",
  });

  function handleChange(event) {
    const { name, value } = event.target;

    setForm((prev) => ({
      ...prev,
      [name]: value,
    }));
  }

  async function handleSubmit(event) {
    event.preventDefault();

    try {
      const savedSale = await createSale(form);
      console.log("저장된 판매 기록:", savedSale);
      alert("판매 등록 성공");
    } catch (error) {
      console.error(error);
      alert("판매 등록 실패");
    }
  }

  return (
    <form onSubmit={handleSubmit}>
      <select name="customerId" value={form.customerId} onChange={handleChange}>
        <option value="">고객 선택</option>
        <option value="1">홍길동</option>
        <option value="2">김민수</option>
      </select>

      <select name="carId" value={form.carId} onChange={handleChange}>
        <option value="">자동차 선택</option>
        <option value="10">아반떼</option>
        <option value="11">쏘나타</option>
      </select>

      <input
        name="saleDate"
        type="date"
        value={form.saleDate}
        onChange={handleChange}
      />

      <input
        name="salePrice"
        value={form.salePrice}
        onChange={handleChange}
        placeholder="판매금액"
      />

      <button type="submit">판매 등록</button>
    </form>
  );
}

export default SaleCreatePage;

export function removeToken() {

car1_name, car2_name, car3_name처럼
반복 컬럼을 만들면 데이터가 늘어날 때마다
테이블 구조를 바꿔야 합니다.

3. 제2정규화 대상 오해

제2정규화는 복합키가 있는 테이블에서 특히 중요합니다.
복합키 일부에만 의존하는 컬럼을 찾아야 합니다.

4. 제3정규화 무시

zip_code가 city와 district를 결정하는데도
같은 테이블에 계속 두면
일반 컬럼 간 종속 문제가 생길 수 있습니다.

5. 정규화를 무조건 분리로 이해

정규화는 테이블을 많이 만드는 것이 아니라
데이터를 있어야 할 위치에 두는 과정입니다.

6. 저장 DTO와 조회 DTO를 같게 만들기

저장할 때는 customerId, carId가 자연스럽고,
조회할 때는 customerName, carName이 필요할 수 있습니다.

7. JOIN이 싫어서 중복 저장하기

조회가 편하다는 이유로
customerName을 SALE에 계속 저장하면
수정 이상과 데이터 불일치가 생깁니다.

8. FK 필수값 누락

정규화 후 SALE은 CUSTOMER와 CAR를 참조해야 하므로
customerId와 carId를 반드시 받아야 합니다.

06

정규화가 잘못되면 SQL 문법 오류보다 더 위험한 데이터 품질 문제가 생깁니다. 또한 정규화된 테이블의 FK, NOT NULL, PK 전략을 제대로 맞추지 않으면 Oracle, JPA, React에서 오류가 발생합니다.

08

면접 포인트 카드

Q1. Mapping Rule이란 무엇인가요?

Mapping Rule은 ERD를 관계형 데이터베이스 스키마로 변환하는 규칙입니다. 엔터티는 테이블로, 속성은 컬럼으로, 주식별자는 기본키로, 관계는 외래키로 변환합니다.

Q2. 정규화란 무엇인가요?

정규화는 데이터 중복을 줄이고 이상 현상을 방지하기 위해 속성을 올바른 테이블에 배치하는 과정입니다.

Q3. 정규화를 왜 하나요?

정규화를 하지 않으면 같은 데이터가 여러 곳에 반복되어 삽입 이상, 수정 이상, 삭제 이상이 발생할 수 있기 때문입니다.

Q4. 제1정규화를 설명해 주세요.

제1정규화는 반복되는 속성이나 그룹 속성을 제거하는 과정입니다. car1_name, car2_name 같은 반복 컬럼 대신 별도 SALE 테이블의 여러 행으로 관리합니다.

Q5. 제2정규화를 설명해 주세요.

제2정규화는 복합키를 가진 테이블에서 일반 컬럼이 복합키 전체에 의존하도록 만드는 과정입니다. 복합키 일부에만 의존하는 컬럼은 별도 테이블로 분리합니다.

Q6. 제3정규화를 설명해 주세요.

제3정규화는 기본키가 아닌 컬럼에 종속되는 컬럼을 분리하는 과정입니다. 예를 들어 zip_code가 city와 district를 결정한다면 별도 ZIP_CODE 테이블로 분리할 수 있습니다.

Q7. 수정 이상이란 무엇인가요?

수정 이상은 같은 데이터가 여러 곳에 중복 저장되어 일부만 수정되는 문제입니다. 고객 연락처가 판매 기록마다 반복 저장되면 이런 문제가 생길 수 있습니다.

Q8. 삭제 이상이란 무엇인가요?

삭제 이상은 특정 데이터를 삭제했을 때 보존해야 할 데이터까지 함께 사라지는 문제입니다. 마지막 판매 기록을 삭제했더니 고객 정보까지 사라지는 경우가 예입니다.

Q9. 삽입 이상이란 무엇인가요?

삽입 이상은 일부 정보가 없어서 저장해야 할 데이터를 저장하지 못하는 문제입니다. 구매 기록이 없는 고객을 등록하지 못하는 구조가 대표적인 예입니다.

Q10. 정규화된 구조에서는 조회가 복잡해지지 않나요?

맞습니다. 저장 구조는 안정적이지만 조회 시 JOIN이 필요할 수 있습니다. 실무에서는 저장 구조는 정규화하고, 화면 응답은 DTO나 Query로 조합합니다.

Q11. Spring Boot Entity를 만들기 전에 왜 테이블 구조가 안정되어야 하나요?

테이블 구조가 바뀌면 Entity, Repository, DTO, Service, Controller, React 요청 구조까지 함께 수정해야 하기 때문입니다.

Q12. React 개발과 정규화는 어떤 관련이 있나요?

React는 DB를 직접 다루지 않지만 DTO 구조를 통해 DB 설계의 영향을 받습니다. 정규화된 구조에서는 customerId, carId 같은 FK 값을 Form에서 선택해 API로 전송합니다.

  • Mapping Rule은 ERD를 테이블 구조로 번역하는 규칙입니다.
  • 엔터티는 테이블로, 속성은 컬럼으로, 주식별자는 기본키로, 관계는 외래키로 변환됩니다.
  • 정규화는 데이터를 있어야 할 테이블에 배치해 중복과 이상 현상을 줄이는 과정입니다.
  • 제1정규화는 반복 속성이나 그룹 속성을 제거합니다.
  • 제2정규화는 복합키 일부에만 의존하는 컬럼을 제거합니다.
  • 제3정규화는 기본키가 아닌 컬럼에 종속되는 컬럼을 제거합니다.
  • 저장 구조는 정규화하고, 조회 응답은 JOIN과 DTO로 화면에 맞게 조합합니다.

헷갈리기 쉬운 개념

Mapping Rule vs 정규화

Mapping Rule은 ERD를 테이블로 바꾸는 규칙이고,
정규화는 테이블 구조를 안정적으로 다듬는 과정입니다.

제1정규화

반복 컬럼과 그룹 속성을 제거합니다.
car1_name, car2_name 같은 구조를 피합니다.

제2정규화

복합키 일부에만 의존하는 컬럼을 분리합니다.
복합키가 없으면 보통 주요 대상이 아닙니다.

제3정규화

기본키가 아닌 컬럼에 의존하는 컬럼을 분리합니다.
zip_code와 city 관계를 떠올리면 쉽습니다.

정규화 vs JOIN

정규화하면 저장 구조는 안정적이지만,
화면 조회에는 JOIN이나 응답 DTO 조합이 필요할 수 있습니다.

Request DTO vs Response DTO

저장 요청은 ID 중심,
조회 응답은 화면 표시용 이름 중심으로 다르게 만들 수 있습니다.
